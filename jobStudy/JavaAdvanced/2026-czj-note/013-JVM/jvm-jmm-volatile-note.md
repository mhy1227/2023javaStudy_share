# JVM JMM 与 volatile 笔记

## 1. 这份文档怎么用

这一篇专门整理 JVM / 并发交界处最常见的一组高频题：

- 什么是 JMM
- `volatile` 有什么作用
- `volatile` 能不能保证线程安全
- 什么是 happens-before
- `volatile` 和 `synchronized` 怎么区分

它通常是 JVM 主线讲到后半段时，自然会接上的部分。

## 2. 先记住最重要的 6 句

先把下面这 6 句记住：

1. JMM 是 Java 内存模型，用来约束多线程下变量可见性、有序性和原子性相关规则。
2. `volatile` 主要保证可见性和有序性。
3. `volatile` 通常不保证复合操作的原子性。
4. `volatile` 很适合做状态标记和开关变量。
5. happens-before 可以理解成“前一个操作的结果对后一个操作可见”的规则。
6. `synchronized` 比 `volatile` 更重，但还能保证互斥和原子性。

## 3. 什么是 JMM

标准回答：

> JMM 就是 Java 内存模型。它不是具体某一块真实内存，而是一套规范，用来定义多线程场景下变量是怎么在主内存和工作内存之间交互的，以及线程之间怎么保证可见性、有序性和原子性。

更口语一点：

> JMM 本质上是在回答一个问题，多线程同时操作共享变量时，Java 到底怎么尽量保证结果正确。

## 4. 为什么要有 JMM

可以这样答：

> 因为多线程环境下，如果每个线程都只操作自己本地缓存的变量副本，又没有统一规则，就很容易出现一个线程改了值，另一个线程看不到，或者执行顺序被重排导致结果异常。JMM 的存在就是为了给这些行为定规则。

## 5. `volatile` 的作用是什么

更稳的回答：

> `volatile` 的核心作用我一般总结成两点。第一是保证可见性，也就是一个线程修改了变量之后，其他线程能尽快看到最新值；第二是限制指令重排，也就是保证一定的有序性。

一句话记忆：

> `volatile` 主要解决可见性和有序性问题。

## 6. `volatile` 能不能保证线程安全

这是非常高频的一题。

更稳的回答：

> 不能简单说它就能保证线程安全。它能保证单次读写的可见性，但像 `i++` 这种复合操作，本质上还是读、改、写三步，`volatile` 通常保证不了整个过程的原子性。

## 7. 为什么 `volatile` 不能保证 `i++` 安全

可以这样答：

> 因为 `i++` 不是一个原子操作，它本质上包含读当前值、加一、再写回三个步骤。多个线程同时做这件事时，中间步骤可能互相覆盖，所以结果仍然可能不对。`volatile` 只能保证线程看到的是最新值，但不能把这三个步骤打包成一个整体。

## 8. 什么场景适合用 `volatile`

更稳的回答：

> `volatile` 比较适合做状态标记、开关变量、停止信号这类场景，也就是“一个线程改状态，其他线程需要尽快看到”的情况。

这类题的重点不是讲得特别底层，而是知道它适合“轻量可见性通知”，不适合复杂共享数据修改。

## 9. 什么是 happens-before

标准回答：

> happens-before 可以理解成 Java 内存模型里的一套可见性规则。如果一个操作 happens-before 另一个操作，就可以认为前一个操作的结果对后一个操作是可见的，而且前一个操作的执行顺序也排在前面。

## 10. 为什么要有 happens-before

可以这样答：

> 因为多线程下如果只谈代码书写顺序，实际执行未必是那个顺序。happens-before 的意义就是告诉你，在哪些规则下，一个线程的写操作一定能被另一个线程看到，这样程序员才有办法推断并发程序的行为。

## 11. 你至少要知道的几条 happens-before 规则

面试里不用全背，但至少会这几条：

1. 程序次序规则：同一个线程里，前面的操作先于后面的操作。
2. `volatile` 变量规则：对一个 `volatile` 变量的写，先于后续对这个变量的读。
3. 锁规则：解锁先于后续的加锁。
4. 线程启动规则：`start()` 先于线程中的动作。
5. 线程终止规则：线程中的所有操作先于其他线程检测到它结束。

## 12. `volatile` 和 `synchronized` 怎么区分

更稳的回答：

> 我一般这么区分，`volatile` 更轻量，主要解决可见性和一定程度上的有序性问题；`synchronized` 更重，但它除了可见性，还能保证互斥，也就是同一时刻只有一个线程进入临界区，所以它还能保证原子性。

一句话记忆：

> 只是做状态标记，`volatile` 够用；涉及共享资源修改，锁通常更稳。

## 13. 这题最容易和什么继续连起来

最常继续追：

- `synchronized`
- CAS
- 原子类
- happens-before 具体规则
- 双重检查锁为什么要配合 `volatile`

也就是说，这一篇是后面并发主线的入口之一。

## 14. 项目里可以怎么举例

如果面试官问项目里有没有类似场景，可以这样稳一点回答：

> 在业务项目里我不一定会直接手写很多底层并发控制，但我知道像开关变量、任务停止标记这类场景，`volatile` 会比较合适；如果涉及共享数据一致性，通常还是要上锁或者用并发工具类。

## 15. 一版适合面试的回答

> JMM 就是 Java 内存模型，我理解它是一套多线程下的内存访问规范，主要解决可见性、有序性和原子性相关问题。`volatile` 在这里最核心的作用是保证可见性和限制指令重排，也就是一个线程修改后的值，其他线程能尽快看到，同时避免一些危险的重排序。但它通常不保证复合操作的原子性，比如 `i++` 这种场景就不安全。至于 happens-before，我会把它理解成一套可见性规则，用来说明前一个操作的结果什么时候对后一个操作一定可见。

## 16. 最容易答错的点

- 把 JMM 说成某块真实内存区域
- 把 `volatile` 直接等同于线程安全
- 说不清为什么 `i++` 不安全
- 只会背 happens-before 名字，不会说它解决什么问题

## 17. 一句话收尾

> JMM 这条线最该先讲顺的，是“为什么多线程会看不到彼此的修改、`volatile` 能解决什么、又解决不了什么”。
