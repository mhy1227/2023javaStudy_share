# JVM OOM 与线上排查笔记

## 1. 这份文档怎么用

这一篇专门整理 JVM 面试里最像真实工作场景的一组问题：

- 什么是 OOM
- 什么是内存泄漏
- OOM 和内存泄漏有什么区别
- Full GC 频繁怎么排查
- CPU 飙高怎么查
- JVM 调优一般怎么答

这部分通常会在 JVM 基础主线之后被继续追问，尤其在社招和有项目经验的面试里很常见。

## 2. 先记住最重要的 7 句

先把下面这 7 句记住：

1. OOM 是 JVM 在申请内存时发现没有足够可用空间，导致分配失败。
2. 内存泄漏是本来没用了的对象还被引用着，导致回收不掉。
3. 内存泄漏可能长期积累，最后拖成 OOM。
4. Full GC 频繁不一定先改参数，要先看对象为什么回收不掉。
5. CPU 飙高、OOM、GC 频繁，排查思路不一样，不能混着讲。
6. JVM 调优不是先背参数，而是先定位问题、再判断原因、最后调整。
7. 很多 JVM 问题根因其实是代码或对象生命周期设计问题，不只是参数问题。

## 3. 什么是 OOM

最稳的回答：

> OOM 就是 OutOfMemoryError，也就是 JVM 在申请内存时，发现已经没有足够可用内存来完成分配了。它不是简单的“程序报错”，而是 JVM 级别的内存分配失败。

更口语一点：

> OOM 可以理解成 JVM 想再要一块内存，但已经拿不到了。

## 4. 什么是内存泄漏

标准回答：

> 内存泄漏是本来业务上已经不再需要的对象，却因为还被某些引用链持有，导致垃圾回收器无法回收。也就是说，内存不是不够，而是“该回收的没回收掉”。

## 5. 内存泄漏和 OOM 有什么区别

更稳的回答：

> OOM 是结果，内存泄漏常常是原因之一。内存泄漏强调的是对象本来没用了，但还活着；OOM 强调的是 JVM 最终没有足够内存可分配。所以可以说，内存泄漏如果长期积累，很容易把系统拖到 OOM。

一句话记忆：

> 内存泄漏更像病因，OOM 更像病发结果。

## 6. 常见 OOM 类型有哪些

面试里至少会这几种：

1. Java 堆内存溢出
2. 栈相关错误，比如栈溢出
3. 元空间溢出
4. 直接内存相关问题

重点不是一次背全，而是知道：

> OOM 不只有堆内存这一种。

## 7. 堆溢出怎么讲

更稳的回答：

> 堆溢出一般意味着对象在堆里分配不下了。常见原因有对象创建太多、缓存堆积、集合无限增长，或者内存泄漏导致对象一直回收不掉。

## 8. 栈溢出怎么讲

可以这样答：

> 栈溢出更偏方法调用层面，常见场景是递归太深或者调用层次过深，导致线程栈空间耗尽。它和堆溢出不一样，重点不在对象太多，而在方法栈帧压得太深。

## 9. 元空间溢出怎么讲

更稳的回答：

> 元空间主要存类的元数据。如果系统不断生成大量类，或者类加载器相关资源一直不释放，就可能导致元空间溢出。所以它和普通对象堆内存问题也不是一个维度。

## 10. 内存泄漏常见场景有哪些

面试里可以说这些高频例子：

1. 长生命周期集合不断持有对象
2. 缓存只放不清
3. 静态集合或静态引用持有大量对象
4. 连接、流、资源没及时释放
5. ThreadLocal 使用不当

这些例子不只是为了背，而是为了让面试官感觉你知道“对象为什么会回收不掉”。

## 11. Full GC 频繁怎么排查

更稳的回答：

> 如果 Full GC 很频繁，我第一反应不是直接调大堆，而是先看为什么对象一直回收不掉。常见会先看 GC 日志、看老年代占用情况、看是不是有大对象或者长生命周期对象堆积。如果有必要，再结合堆快照看哪些对象占内存最多，最后再判断是不是代码问题、缓存问题或者参数问题。

一句话记忆：

> Full GC 频繁，先查对象为什么活着，不是先拍脑袋调参数。

## 12. OOM 怎么排查

更稳的回答：

> 如果是 OOM，我会先区分是哪一类 OOM，比如堆内存溢出、元空间溢出还是栈相关问题。然后结合错误日志和监控看内存走势，再通过堆转储文件或者分析工具去看哪些对象占用最多、引用链为什么断不掉。重点不是只说“内存不够”，而是找出到底是谁把内存吃掉了。

## 13. CPU 飙高怎么查

可以这样答：

> CPU 飙高和 OOM 不一样，排查时我会先看是不是某个进程或者某个线程占用异常，再去看线程栈，判断是在死循环、频繁竞争锁，还是在做大量计算。也就是说，CPU 问题更偏线程执行层面，不是只看堆内存。

## 14. 常见 JVM 排查工具知道哪些

面试里至少能说出这些：

1. `jps`：看 Java 进程
2. `jstat`：看 GC 情况
3. `jmap`：看内存、导出堆快照
4. `jstack`：看线程栈
5. `jconsole`、`jvisualvm`：可视化看 JVM 状态

如果你不想讲太碎，这一层就够用了。

## 15. 为什么说调优不是只改参数

更稳的回答：

> 因为很多 JVM 问题，根因其实是代码或业务对象生命周期设计不合理。比如缓存堆积、集合无限增长、对象引用没释放，如果只靠调大内存，可能只是把问题往后拖。所以调优一定要区分，哪些是参数问题，哪些是代码问题。

## 16. 面试里如果没真实调优经验怎么说

这是很实用的一题。

可以稳一点这样答：

> 我实际线上没有做过特别复杂的大规模 JVM 调优，但我知道基本排查思路。我的理解是，不是上来就调参数，而是先区分问题类型，再结合 GC、内存、线程这几个维度去定位。

这比硬吹自己调过大型系统安全得多。

## 17. 一版适合面试的完整回答

> 我理解 JVM 调优和线上排查的前提是先明确问题，而不是先改参数。比如如果是 Full GC 频繁，我会先看 GC 日志和老年代占用，判断是不是对象存活太多或者回收不掉；如果是 OOM，我会先区分是哪类 OOM，再看堆快照找出是谁占了大量内存；如果是 CPU 飙高，我会更关注线程栈和热点线程。常见工具我知道 `jps`、`jstat`、`jmap`、`jstack` 这些。我的理解是，调优本质上是定位问题和验证原因，不是背几个参数模板。

## 18. 最容易答错的点

- 把 OOM 和内存泄漏混成一回事
- 一讲 Full GC 频繁就只会说“加内存”
- 把 CPU 飙高也讲成“看堆”
- 只会背工具名，说不清大概怎么用
- 一上来就硬吹自己做过复杂调优

## 19. 一句话收尾

> JVM 排障这条线最重要的不是参数名，而是你能不能先把问题分对类，再按 GC、内存、线程这几个维度去定位。
