# JVM 考前复盘 Checklist

## 1. 这份文档怎么用

这一份不是详细笔记，而是考前快速扫一遍的版本。

建议用法：

- 面试前 30 分钟：按模块快速过一遍。
- 面试前 10 分钟：只看“最值得先背”的部分。
- 如果某一块卡住，就回跳到对应主线文档补一遍。

## 2. 基础认知 Checklist

- 我能说清 JVM 是什么。
- 我能说清 JVM、JDK、JRE 的区别。
- 我知道为什么 Java 能跨平台和 JVM 有关系。
- 我知道 JVM 不只是“解释执行”，还负责内存管理、垃圾回收、类加载等。
- 我知道 JVM 这一章的核心主线不是背名词，而是把运行、内存、回收、加载、并发、排障串起来。

最值得先背：

> JVM 就是 Java 虚拟机，它负责把 Java 程序加载进来并运行，同时提供字节码执行、内存管理、垃圾回收、类加载这些能力。Java 之所以强调跨平台，本质上是因为不同平台各自实现了 JVM，Java 代码编译成字节码后可以交给对应平台上的 JVM 去运行。

## 3. 内存结构 Checklist

- 我能说清 JVM 运行时内存结构有哪些。
- 我能区分程序计数器、虚拟机栈、本地方法栈、堆、方法区。
- 我知道哪些区域是线程私有，哪些区域是线程共享。
- 我能说清堆和栈的区别。
- 我知道为什么面试里最爱问堆、栈、方法区。
- 我知道对象通常主要分配在堆里。
- 我知道栈更关注方法调用、局部变量、栈帧这些概念。

最值得先背：

> JVM 运行时内存结构常见会讲程序计数器、虚拟机栈、本地方法栈、堆和方法区。面试里最核心的是堆、栈、方法区。堆主要存对象，线程共享，重点和对象生命周期、GC 有关；栈主要存方法执行过程中的局部变量和调用信息，线程私有，重点和方法调用、递归、栈溢出有关。

## 4. GC Checklist

- 我能说清什么是 GC。
- 我知道 JVM 为什么需要 GC。
- 我能解释 JVM 怎么判断对象是不是垃圾。
- 我知道为什么引用计数不是主流答案。
- 我能解释可达性分析和 GC Roots。
- 我能区分标记清除、复制、标记整理。
- 我知道为什么 JVM 会引入分代思想。
- 我能区分新生代、老年代。
- 我知道 Minor GC、Major GC、Full GC 这几个词大概在说什么。
- 我知道 Stop-The-World 是什么。

最值得先背：

> GC 就是垃圾回收，本质上是 JVM 自动回收不再使用对象内存的机制。JVM 常用可达性分析来判断对象能不能回收，而不是单纯依赖引用计数。常见回收算法有标记清除、复制和标记整理。因为很多对象朝生夕死，少数对象会长期存活，所以 JVM 引入了分代思想，把对象放到不同代里做更合适的回收。

## 5. 垃圾回收器 Checklist

- 我知道常见垃圾回收器有哪些。
- 我能简单解释 Serial、ParNew、Parallel Scavenge、CMS、G1 这些名字至少各自是什么定位。
- 我能说清 CMS 和 G1 的基本差异。
- 我知道为什么现在面试更常听到 G1。
- 我知道 Full GC 频繁为什么危险。

最值得先背：

> 垃圾回收器是 GC 算法的具体实现。面试里常见会问 CMS 和 G1。CMS 更偏低停顿，G1 更强调对停顿时间的可控性，也更符合现代 JVM 场景里的主流回答。如果 Full GC 很频繁，通常意味着老年代压力大，或者对象长期回收不掉，这时候不能只会说“加内存”。

## 6. 类加载 Checklist

- 我能说清什么是类加载。
- 我知道类从 `.class` 变成 JVM 可用数据的大致过程。
- 我能说清加载、验证、准备、解析、初始化。
- 我知道什么是类加载器。
- 我知道常见类加载器有哪些。
- 我能解释双亲委派。
- 我知道为什么要有双亲委派。
- 我知道双亲委派主要解决什么问题。

最值得先背：

> 类加载本质上是 JVM 把 `.class` 文件变成可运行类数据的过程。常见步骤包括加载、验证、准备、解析和初始化。双亲委派的核心思想是类加载请求先交给父加载器尝试处理，只有父加载器处理不了，子加载器才自己处理，这样可以避免类重复加载，也能保护 Java 核心类不被随意替换。

## 7. JMM 与 volatile Checklist

- 我能说清什么是 JMM。
- 我知道 JMM 不是某一块真实内存，而是一套并发访问规范。
- 我能说清 JMM 想解决什么问题。
- 我能解释 `volatile` 的两个核心作用。
- 我知道 `volatile` 不能保证完整线程安全。
- 我能解释为什么 `volatile` 不能保证 `i++`。
- 我知道 happens-before 是什么。
- 我至少知道几条常见 happens-before 规则。
- 我能区分 `volatile` 和 `synchronized`。

最值得先背：

> JMM 就是 Java 内存模型，它主要解决多线程下的可见性、有序性和原子性相关问题。`volatile` 的核心作用一般总结成两点，一个是保证可见性，一个是限制指令重排。但它不能保证复合操作的原子性，所以像 `i++` 这种场景不能只靠 `volatile`。如果既要可见性又要互斥，通常还是要讲 `synchronized` 或锁。

## 8. 对象与引用 Checklist

- 我能说清 `new` 一个对象大概发生了什么。
- 我知道对象创建会涉及类加载检查、内存分配、零值初始化、对象头设置等概念。
- 我知道对象通常分配在堆里。
- 我能区分强引用、软引用、弱引用、虚引用。
- 我知道软引用和弱引用至少各自一个典型使用场景。

最值得先背：

> `new` 一个对象时，JVM 通常会先检查类是否已经加载，然后在堆上分配内存，做默认初始化，再设置对象头，最后执行构造方法。引用类型里最常问的是强软弱虚。强引用最普通，软引用一般和内存敏感缓存联系，弱引用常和更积极的回收策略联系，虚引用更多是配合引用队列做更底层的资源跟踪。

## 9. OOM 与排查 Checklist

- 我能说清什么是 OOM。
- 我能说清什么是内存泄漏。
- 我能区分 OOM 和内存泄漏。
- 我知道常见 OOM 类型不只堆内存这一种。
- 我能区分堆溢出、栈溢出、元空间溢出的大概方向。
- 我知道内存泄漏常见场景有哪些。
- 我能说清 OOM 大概怎么排查。
- 我知道 CPU 飙高和 OOM 的排查思路不是一套。

最值得先背：

> OOM 是 JVM 在申请内存时发现没有足够可用空间，最终分配失败；内存泄漏强调的是本来没用的对象还被引用着，导致回收不掉，所以内存泄漏经常会长期积累并最终拖成 OOM。排查 OOM 不能只说“内存不够”，而要先分清是哪一类 OOM，再结合日志、监控、堆快照去找是谁把内存吃掉了。

## 10. JVM 工具与调优 Checklist

- 我知道 `jps`、`jstat`、`jmap`、`jstack` 大概是干什么的。
- 我知道可视化工具比如 `jconsole`、`jvisualvm` 的用途。
- 我知道 JVM 调优不应该上来就调参数。
- 我能说清 Full GC 频繁怎么排查。
- 我能说清 CPU 飙高怎么查。
- 我知道很多 JVM 问题根因可能在代码和对象生命周期，而不是 JVM 参数本身。
- 如果我没有复杂线上调优经验，我知道怎么稳一点回答。

最值得先背：

> JVM 调优不是先背参数，而是先定位问题。比如 Full GC 频繁时，先看 GC 日志和老年代占用；如果是 OOM，就结合错误日志和堆快照去看大对象和引用链；如果是 CPU 飙高，就更关注线程和线程栈。很多 JVM 问题根因其实是对象生命周期设计不合理、缓存堆积、集合无限增长，而不是参数配得不对。

## 13. 如果时间只剩 2 分钟

只背这段：

> JVM 面试最核心的主线就是内存结构、GC、类加载、JMM 和线上排查。内存结构里高频是堆、栈、方法区；GC 常用可达性分析判断对象是否可回收，并通过分代思想提升回收效率；类加载常问加载过程和双亲委派；JMM 主要解决多线程下的可见性、有序性和原子性问题，`volatile` 能保证可见性和一定有序性，但不能保证像 `i++` 这样的复合操作原子性；如果往真实工作场景讲，还要补 OOM、Full GC 频繁、CPU 飙高和 JVM 排查工具这些内容。

## 14. 最值得先背的一版

> 我一般把 JVM 分成几条主线来理解。第一条是运行时内存结构，也就是堆、栈、方法区这些区域分别干什么；第二条是垃圾回收，也就是对象什么时候回收、为什么要分代、常见回收器怎么理解；第三条是类加载和双亲委派，也就是类怎么从 `.class` 变成 JVM 里的可用数据；第四条是 JMM 和 `volatile`，更偏多线程下的可见性和有序性；最后如果面试往真实工作场景追问，我会再补 OOM、Full GC 和线上排查思路。这样一条线讲下来，JVM 这一章会比较完整。
