# JVM 高频面试题回答版

## 1. 这份文档怎么用

这一篇不是标准教材答案，而是更适合一面、二面口语化复述的版本。

建议：

- 先自己答一遍
- 再对照这里修正表达
- 不要求逐字背，但要把主线说顺

## 2. 什么是 JVM

可以这样答：

> JVM 就是 Java 虚拟机，它负责把 Java 程序加载进来并运行，提供字节码执行、内存管理、垃圾回收等能力。你也可以把它理解成 Java 程序真正跑起来的环境。

## 3. JVM 内存结构有哪些

可以这样答：

> JVM 运行时内存结构常见会讲程序计数器、虚拟机栈、本地方法栈、堆和方法区。不过面试里真正高频深入问的，主要还是堆、栈和方法区。

## 4. 堆和栈有什么区别

可以这样答：

> 堆主要存对象，线程共享；栈主要存方法执行过程中的局部变量和调用信息，线程私有。一般来说，堆更关注对象生命周期和垃圾回收，栈更关注方法调用和线程执行过程。

## 5. 什么是 GC

可以这样答：

> GC 就是垃圾回收，本质上是 JVM 自动回收不再使用对象内存的机制。它的核心作用是避免堆内存一直增长，让程序能持续稳定运行。

## 6. JVM 怎么判断对象能不能回收

可以这样答：

> JVM 面试里更常讲可达性分析。也就是从 GC Roots 出发，如果一个对象已经无法从这些根继续访问到，就可以认为它是可回收的。

## 7. 为什么不主讲引用计数

可以这样答：

> 因为引用计数虽然直观，但它解决不了循环引用问题。所以 JVM 面试里更常讲可达性分析，而不是把引用计数当成主流方案。

## 8. 为什么要有分代思想

可以这样答：

> 因为不同对象的生命周期差异很大，很多对象创建后很快就死掉了，少数对象会长期存活。既然对象特征不同，回收策略也没必要完全一样，所以 JVM 引入了分代思想。

## 9. CMS 和 G1 怎么讲

可以这样答：

> CMS 更强调低停顿，适合对响应时间比较敏感的场景；G1 是后面更常见的一种回收器，它的目标是尽量兼顾吞吐和停顿时间控制，在现代 JVM 场景里更常见一些。

## 10. 什么是类加载

可以这样答：

> 类加载本质上是 JVM 把 `.class` 文件变成可运行类数据的过程。常见步骤包括加载、验证、准备、解析和初始化。

## 11. 什么是双亲委派

可以这样答：

> 双亲委派机制的核心思想是类加载请求先交给父加载器尝试处理，只有父加载器处理不了，子加载器才自己处理。这样做的主要目的是避免类重复加载，同时保证 Java 核心类由更高层加载器统一加载。

## 12. 什么是 JMM

可以这样答：

> JMM 就是 Java 内存模型，它不是某一块真实内存，而是一套多线程下的内存访问规范，主要解决可见性、有序性和原子性相关问题。

## 13. `volatile` 有什么作用

可以这样答：

> `volatile` 最核心的作用我一般总结成两点。第一是保证可见性，也就是一个线程修改了变量之后，其他线程能尽快看到最新值；第二是限制指令重排，也就是保证一定的有序性。

## 14. `volatile` 为什么不能保证 `i++`

可以这样答：

> 因为 `i++` 不是一个原子操作，它本质上包含读当前值、加一、再写回三个步骤。`volatile` 只能保证看到最新值，但不能把这三个步骤打包成一个整体。

## 15. happens-before 是什么

可以这样答：

> happens-before 可以理解成 Java 内存模型里的一套可见性规则。如果一个操作 happens-before 另一个操作，就可以认为前一个操作的结果对后一个操作是可见的，而且前一个操作的执行顺序也排在前面。

## 16. `volatile` 和 `synchronized` 怎么区分

可以这样答：

> `volatile` 更轻量，主要解决可见性和一定程度上的有序性问题；`synchronized` 更重，但它除了可见性，还能保证互斥，也就是同一时刻只有一个线程进入临界区，所以它还能保证原子性。

## 17. 什么是 OOM

可以这样答：

> OOM 就是 OutOfMemoryError，也就是 JVM 在申请内存时发现没有足够可用空间，导致分配失败。它不是简单的“程序报错”，而是 JVM 级别的内存问题。

## 18. 什么是内存泄漏

可以这样答：

> 内存泄漏是本来业务上已经不再需要的对象，却因为还被某些引用链持有，导致垃圾回收器无法回收。也就是说，内存不是单纯不够，而是“该回收的没回收掉”。

## 19. 内存泄漏和 OOM 有什么区别

可以这样答：

> OOM 是结果，内存泄漏常常是原因之一。内存泄漏强调的是对象本来没用了，但还活着；OOM 强调的是 JVM 最终没有足够内存可分配。所以可以说，内存泄漏如果长期积累，很可能最终演变成 OOM。

## 20. Full GC 频繁怎么排查

可以这样答：

> 如果 Full GC 很频繁，我第一反应不是直接调大堆，而是先看为什么对象一直回收不掉。常见会先看 GC 日志、看老年代占用情况、看是不是有大对象或者长生命周期对象堆积。如果有必要，再结合堆快照看哪些对象占内存最多，最后再判断是不是代码问题、缓存问题或者参数问题。

## 21. OOM 怎么排查

可以这样答：

> 如果是 OOM，我会先区分是哪一类 OOM，比如堆内存溢出、元空间溢出还是栈相关问题。然后结合错误日志和监控看内存走势，再通过堆转储文件或者分析工具去看哪些对象占用最多、引用链为什么断不掉。重点不是只说“内存不够”，而是找出到底是谁把内存吃掉了。

## 22. CPU 飙高怎么查

可以这样答：

> CPU 飙高和 OOM 不一样，排查时我会先看是不是某个进程或者某个线程占用异常，再去看线程栈，判断是在死循环、频繁竞争锁，还是在做大量计算。也就是说，CPU 问题更偏线程执行层面，不是只看堆内存。

## 23. JVM 调优一般怎么回答

可以这样答：

> 我理解 JVM 调优不是一上来就去改参数，而是先看线上到底出现了什么问题，比如 Full GC 频繁、响应慢、内存占用过高或者 OOM。先通过监控和工具定位问题，再去判断是对象存活太多、内存分配不合理，还是参数设置不合适，最后再做针对性的调整。

## 24. 如果面试官让你整体讲一遍 JVM

可以这样答：

> JVM 我一般从几条主线来理解。第一条是运行时内存结构，比如堆、栈、方法区这些区域分别干什么；第二条是垃圾回收，也就是对象什么时候回收、常见回收器怎么理解；第三条是类加载和双亲委派，也就是类怎么从 `.class` 变成 JVM 可用的数据；第四条是 JMM 和 `volatile`，更偏并发下的可见性和有序性；最后如果往真实工作场景讲，我会再补 OOM、Full GC 和 JVM 排查思路。
