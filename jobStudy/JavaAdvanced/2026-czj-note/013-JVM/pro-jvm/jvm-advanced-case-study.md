# JVM 进阶案例整理

## 1. 这份文档怎么用

这一篇不是理论定义，而是把 JVM 进阶知识放进几个更像线上排查的案例里。

目标是让你从：

- 我知道术语

走到：

- 我知道这种现象大概该往哪条线判断

## 2. 案例一：堆没满，但服务还是越来越吃内存

现象：

- 堆监控看起来没有特别夸张
- 但进程整体内存占用持续上涨
- 系统越来越不稳

先想到什么：

- 不只盯堆
- 想到堆外内存 / Direct Memory
- 想到线程栈、元空间这类非堆方向

更稳的表达：

> 如果堆看起来还好，但进程内存持续上涨，我不会只盯着堆。我会怀疑是不是堆外内存、Direct Memory、线程栈或者元空间方向出了问题，也就是说，这时候要把 JVM 内存视角从“只看堆”扩展开。

## 3. 案例二：Full GC 很频繁，但回收效果一般

现象：

- Full GC 频繁发生
- 每次回收后内存下降不明显
- 老年代长期压力高

先想到什么：

- 长生命周期对象太多
- 缓存堆积
- 大对象
- 对象其实回收不掉

更稳的表达：

> 这种情况我不会先拍脑袋调参数，而会先看为什么对象一直活着。比如是不是缓存、集合无限增长，或者某些大对象和长生命周期对象长期压在老年代。GC 日志这时候很重要，因为可以先看回收前后内存变化和老年代压力。

## 4. 案例三：CPU 飙高，但堆占用并不高

现象：

- CPU 打满
- 堆没明显爆
- 服务响应变慢

先想到什么：

- 热点线程
- 死循环
- 锁竞争
- 高计算量代码

更稳的表达：

> 这种情况更偏线程执行问题，而不是单纯堆内存问题。我会先找热点线程，再结合线程栈去看是在死循环、锁竞争，还是在做高计算量工作。也就是说，CPU 问题通常优先往线程维度走。

## 5. 案例四：服务运行一段时间后变快

现象：

- 刚启动时性能一般
- 跑一段时间后吞吐更稳定

先想到什么：

- 热点探测
- JIT
- 方法内联

更稳的表达：

> 这种现象常常和 JIT 有关。因为 JVM 在运行过程中会识别热点代码，再把它们编译成本地机器码，后续执行效率会更高。所以程序跑一会儿之后变快，并不奇怪。

## 6. 案例五：热部署几次后系统越来越不稳定

现象：

- 热部署后旧版本下线了
- 但内存压力越来越明显
- 元空间方向有异常迹象

先想到什么：

- 类加载器释放不掉
- 类卸载不理想
- Metaspace 压力

更稳的表达：

> 这种情况我会先怀疑类加载器和类卸载这条线。因为热部署、容器、插件化这些场景下，类加载器隔离更复杂，如果旧类加载器释放不掉，就更容易让类元数据长期占着元空间。

## 7. 案例六：接口抖动明显，但不是 SQL 慢

现象：

- RT 抖动
- SQL 排查没明显问题
- 应用层偶发性卡顿

先想到什么：

- STW
- GC 停顿
- Safepoint 相关影响

更稳的表达：

> 如果 SQL、网络都不像主因，但服务会偶发性卡顿，我会往 GC 停顿、STW 这条线想。因为业务线程被暂停时，外部看到的就是接口突然抖一下，这时候就要结合 GC 日志和停顿时间去看。

## 8. 这些案例在面试里怎么答更像真人

一个很稳的统一模板是：

1. 先说现象
2. 再说你第一反应判断方向
3. 最后说你会怎么进一步验证

也就是：

> 现象 -> 判断方向 -> 验证思路

这样答，比一上来直接背术语自然得多。

## 9. 面试里最喜欢借案例考什么

通过这些案例，面试官最想看的是：

- 你是不是只会背堆和 GC
- 你知不知道还有堆外内存、JIT、元空间、线程这几条线
- 你会不会把不同现象归到不同方向

## 10. 一版适合面试的总结回答

> 我理解 JVM 进阶题真正想考的，不是我会不会背更多名词，而是我能不能把不同现象分到不同方向。比如堆没满但内存涨，要想到堆外内存；CPU 高要先往线程走；服务跑久了更快要想到 JIT；热部署后元空间压力高要想到类加载器和类卸载。这种把现象和 JVM 机制连起来的能力，比单独背术语更重要。

## 11. 一句话收尾

> JVM 的 pro 版不是把知识点学得更碎，而是把不同现象更快地映射到正确的判断方向上。
