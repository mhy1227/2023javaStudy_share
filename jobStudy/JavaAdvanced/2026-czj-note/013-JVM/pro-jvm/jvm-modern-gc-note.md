# JVM 现代 GC 视角笔记

## 1. 这份文档怎么用

这一篇不是让你把所有 GC 名字都背一遍，而是让你建立“现代 GC 视角”。

它主要解决：

- 你已经会讲 CMS / G1 了，下一步还能怎么提升
- 什么是低停顿 GC 思路
- ZGC、Shenandoah 大概在解决什么问题
- 为什么有些老八股今天听起来不完全够用了

## 2. 先记住最核心的 5 句

1. CMS / G1 已经足够覆盖大多数基础面试。
2. 现代 GC 这条线更强调低停顿，而不只是“能不能回收”。
3. ZGC、Shenandoah 常被当成低停顿 GC 的代表。
4. 现代 GC 的核心卖点，往往不是吞吐极致，而是停顿控制更友好。
5. 面试里这条线通常是加分项，不是基础必答项。

## 3. 为什么还要讲“现代 GC”

因为很多 JVM 八股会停在：

- Serial
- Parallel
- CMS
- G1

这当然没错，但更进一层时，面试官可能会想看：

> 你知不知道 JVM 近些年的 GC 设计方向，已经越来越强调低停顿体验了。

## 4. 什么叫“低停顿 GC”

最稳的回答：

> 低停顿 GC 可以理解成，一类更强调减少业务线程长时间停顿感受的回收思路。它的重点不只是“把垃圾清掉”，还在于尽量控制对应用响应的影响。

一句话记忆：

> 传统视角更偏“回收效率”，现代视角更强调“停顿体验”。

## 5. ZGC 怎么理解

更稳的回答：

> ZGC 面试里不一定要求你讲特别深，但至少可以理解成一种更强调低停顿目标的现代回收器。它的讨论重点通常不在“会不会回收”，而在“如何把停顿影响做得更小”。

## 6. Shenandoah 怎么理解

可以这样答：

> Shenandoah 和 ZGC 一样，经常被放在低停顿 GC 这个语境下讨论。面试里如果提到它，通常不是要求你背实现细节，而是看你知不知道现代 GC 越来越重视停顿时间控制。

## 7. 现代 GC 和传统 GC 的思路差异

更稳的回答：

> 如果粗略理解，传统 GC 更容易让人先关注吞吐和回收行为本身，而现代 GC 更强调低停顿体验，也就是尽量减轻 STW 对业务线程的影响。面试里不一定要把技术实现讲细，但至少要知道这个设计方向上的区别。

一句话记忆：

> 现代 GC 更像是在追求“用户感觉别卡”。

## 8. 为什么这条线越来越像加分项

因为 Java 版本不断演进，很多系统不再只满足于：

> 能跑就行。

而会更关心：

- 延迟
- 响应时间
- 长时间停顿

所以现代 GC 视角能体现你不是只停留在老版本八股。

## 9. 面试里一般怎么问

常见问法：

- 你除了 CMS、G1 还知道什么 GC
- ZGC / Shenandoah 了解吗
- 现代 GC 更关注什么
- 为什么现在越来越强调低停顿

这类题的稳法：

> 不要硬装懂实现细节，重点讲“低停顿 GC 的目标和现代方向”。

## 10. 一版适合面试的完整回答

> 如果基础层面讲 GC，我通常会先讲 CMS 和 G1，因为这已经能覆盖大多数常规面试。如果再往深一点，我会补现代 GC 这条线，比如 ZGC、Shenandoah。我的理解是，这些回收器更像是在强调低停顿目标，也就是尽量减少业务线程长时间卡顿的感受。面试里如果问到这块，我会把重点放在设计目标和方向上，而不是硬背特别细的底层实现。

## 11. 最容易答错的点

- 以为不知道 ZGC / Shenandoah 就等于 JVM 不会
- 一上来硬背细节，结果越讲越乱
- 说不清现代 GC 和传统 GC 的方向差异

## 12. 一句话收尾

> 现代 GC 这条线最大的价值，不是让你记更多名字，而是让你知道 JVM 近些年的一个重要优化方向就是“更低停顿”。
