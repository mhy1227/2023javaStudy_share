# JVM 现场分析题笔记

## 1. 这份文档怎么用

这一篇不是新知识点，而是把前面 JVM 的多条线压成“现场分析题”。

它主要训练你：

- 给你一个现象，你先怀疑哪条 JVM 线
- 你会先看什么
- 你怎么避免一上来就答错方向

这类题在真实面试里很像“你到底会不会判断问题”。

## 2. 先记住最核心的思路

面对 JVM 现场题，不要一上来背术语。

更稳的顺序是：

1. 先描述现象
2. 再判断它更像哪条线
3. 最后说你会怎么验证

也就是：

> 现象 -> 判断方向 -> 验证思路

## 3. 题一：堆没满，但内存一直涨

现象：

- 堆占用不算夸张
- 但进程整体内存持续上涨

优先怀疑哪条线：

- 堆外内存
- Direct Memory
- 线程栈
- 元空间

更稳的答法：

> 如果堆没明显打满，但进程内存一直涨，我不会只盯堆。我会先怀疑是不是堆外内存、Direct Memory、线程栈或者元空间方向的问题，因为 JVM 的内存压力不只来自堆。

## 4. 题二：GC 不频繁，但 RT 抖动

现象：

- GC 次数不算多
- 但接口响应时间有明显抖动

优先怀疑哪条线：

- STW
- 某些停顿阶段
- 热点线程或锁竞争

更稳的答法：

> 如果 GC 次数不高，但 RT 明显抖动，我不会立刻排除 JVM，因为还可能有 STW、线程竞争或者其他运行时停顿问题。这时候我会结合 GC 日志、线程状态和停顿时间一起看。

## 5. 题三：CPU 高，但线程数不多

现象：

- CPU 飙高
- 线程数量不算特别夸张

优先怀疑哪条线：

- 热点线程
- 死循环
- 高计算量代码
- JIT 热点区域

更稳的答法：

> CPU 高不代表线程一定很多，我会先找热点线程和线程栈，看是不是少数线程在死循环、锁竞争或者执行高计算量代码。必要时再结合热点方法和 JIT 方向去理解为什么某段代码这么热。

## 6. 题四：服务跑久了更快

现象：

- 刚启动一般
- 跑一段时间后更顺

优先怀疑哪条线：

- JIT
- 热点代码
- 方法内联

更稳的答法：

> 这种现象常和 JIT 有关。因为 JVM 会在运行过程中识别热点代码，再做编译和优化，所以服务跑一段时间后性能更稳定，是比较合理的现象。

## 7. 题五：热部署几次后越来越不稳

现象：

- 热部署后服务还能跑
- 但内存和稳定性越来越差

优先怀疑哪条线：

- 类加载器
- 类卸载
- Metaspace

更稳的答法：

> 这种情况我会先怀疑类加载器和元空间方向，尤其是在热部署、容器、插件化场景里，如果旧类加载器释放不掉，就可能让类元数据长期占用空间。

## 8. 题六：Full GC 不断发生，但回收效果一般

现象：

- Full GC 频繁
- 每次回收后内存下降不明显

优先怀疑哪条线：

- 长生命周期对象
- 缓存堆积
- 大对象
- 晋升压力

更稳的答法：

> 这种情况我会先看对象为什么一直活着，而不是先改参数。比如是不是缓存、集合长期持有对象，或者大对象、晋升压力导致老年代持续紧张。GC 日志在这里非常关键。

## 9. 题七：对象创建很多，吞吐掉得快

现象：

- 系统对象分配非常频繁
- 吞吐下降明显

优先怀疑哪条线：

- Eden / Minor GC 压力
- TLAB 分配效率
- 小对象风暴

更稳的答法：

> 如果对象创建非常频繁，我会优先往对象分配这条线想，比如 Eden 压力、Minor GC 频率，以及高频对象分配本身带来的成本。更深一点还可以联系 TLAB 这类线程本地分配优化。

## 10. 这类题最容易翻车的地方

- 一上来就说“肯定是堆不够”
- 只会背一个方向，不会区分不同现象
- 不知道该先看日志、线程还是内存
- 把 JVM 所有问题都答成调参数

## 11. 一版通用答题模板

如果你临场很慌，可以套这个模板：

> 这个现象我不会先下结论，我会先判断它更偏内存、GC、线程还是类加载这条线。然后再结合对应的日志、监控或者工具去验证。比如如果更偏内存，我会看堆和非堆；如果更偏 GC，我会看 GC 日志和老年代压力；如果更偏线程，我会看热点线程和线程栈。

## 12. 一句话收尾

> JVM 现场分析题真正考的不是你会不会背术语，而是你能不能先把方向判断对。
