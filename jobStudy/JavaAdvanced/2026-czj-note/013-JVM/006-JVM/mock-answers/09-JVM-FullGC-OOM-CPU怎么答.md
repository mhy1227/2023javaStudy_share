# JVM：Full GC、OOM、CPU 怎么答

## 1. 这份文档怎么用
这份文档只解决一个问题：

> 面试官问你怎么看 Full GC、OOM、CPU 飙高时，怎么答得简短、自然、像真人。

这份不要背太长，抓住主线就行。

---

## 2. 一段总回答
你可以直接这样说：

> 我会先分问题类型，因为 Full GC、OOM、CPU 飙高，这三类问题排查方向不一样。如果是 Full GC 频繁，我会先看 GC 日志和老年代占用，判断对象是不是一直回收不掉；如果是 OOM，我会先分清是哪一类 OOM，再结合日志和堆快照看是什么对象占了大量内存；如果是 CPU 飙高，我会更偏向线程维度去看，先找热点线程，再结合线程栈判断是死循环、锁竞争还是某段代码执行特别重。也就是说，我理解这类问题先是排查题，不是一上来就调 JVM 参数。

---

## 3. Full GC 怎么答
简短版：

> 如果是 Full GC 频繁，我会先看 GC 日志和监控，重点看老年代占用是不是一直很高，以及 GC 后内存有没有明显回落。如果 GC 后占用还是很高，我会怀疑是不是有长生命周期对象、缓存堆积或者对象一直回收不掉。

再短一点：

> Full GC 我重点看两个点，一个是老年代是不是一直高，另一个是 GC 后内存能不能回落。

---

## 4. OOM 怎么答
简短版：

> 如果是 OOM，我会先看报错信息，先确认是哪一类 OOM，比如堆内存问题还是元空间问题。然后再结合日志、监控和堆快照去看，到底是什么对象占用了大量内存，为什么这些对象没有被释放。

再短一点：

> OOM 我先分类型，再看堆快照，重点找谁占内存最多、为什么一直不释放。

---

## 5. CPU 飙高怎么答
简短版：

> 如果是 CPU 飙高，我会更偏向线程维度排查。先看是不是 Java 进程占用异常，再定位热点线程，最后结合线程栈判断是死循环、锁竞争，还是某段代码执行特别重。

再短一点：

> CPU 问题我主要看热点线程和线程栈，因为它更偏执行层面，不是单纯看内存。

---

## 6. 最关键的一句区分
你一定要记住这句：

> Full GC 更偏内存回收压力问题，OOM 更偏内存分配失败问题，CPU 飙高更偏线程执行问题，所以这三类不能用一套思路去查。

这句很好用。

---

## 7. 如果面试官继续追
你可以这样接：

1. Full GC  
   我先看 GC 日志、老年代占用、GC 后是否回落。

2. OOM  
   我先看报错类型，再看内存走势和堆快照。

3. CPU 飙高  
   我先找热点线程，再看线程栈。

---

## 8. 最后速记版
先背这 5 句：

1. Full GC、OOM、CPU 飙高不是一套排查思路。
2. Full GC 先看老年代和 GC 后内存是否回落。
3. OOM 先分类型，再看堆快照。
4. CPU 飙高先找热点线程，再看线程栈。
5. 这类题先是排查题，不是一上来就调 JVM 参数。
