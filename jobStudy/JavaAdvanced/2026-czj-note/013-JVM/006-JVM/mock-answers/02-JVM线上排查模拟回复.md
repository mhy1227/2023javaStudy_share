# JVM 线上排查模拟回复

## 1. 这份文档怎么用
这份文档专门用来应对 JVM 里更像“真实工作场景”的问题。

核心是三件事：

1. 不要假装自己做过很大规模调优
2. 但要把排查思路说顺
3. 要区分不同问题的排查方向

---

## 2. JVM 调优一般怎么讲
模拟回复：

> 我理解 JVM 调优不是先背参数，而是先明确问题是什么。比如是 Full GC 频繁、响应时间变长、内存占用过高，还是直接 OOM。先通过监控和日志把问题定位到大方向，再判断是对象生命周期问题、代码问题，还是 JVM 参数设置问题，最后再做针对性调整。

---

## 3. Full GC 频繁怎么排查
模拟回复：

> 如果 Full GC 很频繁，我第一反应不会是直接调大堆，而是先看为什么对象一直回收不掉。通常会先看 GC 日志和监控，确认老年代占用是不是持续很高，再判断是不是有长生命周期对象、缓存堆积或者大对象问题。如果有必要，再结合堆快照看具体是哪些对象占内存最多。

---

## 4. OOM 怎么排查
模拟回复：

> OOM 的排查我一般会先分类型，比如是堆内存溢出、元空间问题，还是别的内存区域异常。然后结合日志和监控看内存走势，再用堆快照分析工具去看哪些对象占用最多，为什么它们一直不能被回收。我的理解是，OOM 的重点不是只说“内存不够”，而是找出到底是谁把内存吃掉了。

---

## 5. CPU 飙高怎么查
模拟回复：

> CPU 飙高和 OOM 的思路不一样。我会先看是不是某个 Java 进程占用异常，再去看热点线程，最后结合线程栈判断是死循环、锁竞争，还是某段代码在做大量计算。也就是说，CPU 问题更偏线程执行层面，不只是看堆内存。

---

## 6. 常见 JVM 工具知道哪些
模拟回复：

> 常见工具我知道 `jps`、`jstat`、`jmap`、`jstack`。如果只是快速定位 Java 进程会先看 `jps`；看 GC 情况会想到 `jstat`；看内存和堆快照会用 `jmap`；看线程状态和线程栈会用 `jstack`。如果需要可视化观察，也可以用 `jconsole` 或 `jvisualvm`。

---

## 7. 如果没有真实线上调优经验怎么说
模拟回复：

> 我实际没有做过特别重的大规模 JVM 调优，但我知道基本排查思路。我的理解是，不要一上来就改参数，而是先区分问题类型，再结合 GC、内存和线程这几个维度去定位。这样说会更真实，也更符合实际工作里的排查方式。

---

## 8. 内存泄漏怎么讲
模拟回复：

> 如果怀疑内存泄漏，我会先看内存是不是持续上涨，再看 GC 后内存能不能明显回落。如果 GC 后占用仍然很高，就要怀疑是不是有对象本来没用了，但还被引用链持有着。接下来会通过堆快照去看哪些对象占用最多，以及它们为什么还活着。

---

## 9. 一段总回答模板
如果面试官问：

> 你讲讲 JVM 线上问题怎么排查。

你可以这样答：

> 我理解 JVM 线上排查首先要先分清问题类型，不同问题看重点不一样。如果是 Full GC 频繁，我会先看对象为什么回收不掉；如果是 OOM，我会先区分是哪类 OOM，再看堆快照找大对象和引用链；如果是 CPU 飙高，我会更关注热点线程和线程栈。常见工具我知道 `jps`、`jstat`、`jmap`、`jstack`。我认为 JVM 排查的核心不是背参数，而是先定位问题，再验证原因。

---

## 10. 你最该稳住的 5 句
先背这几句：

1. JVM 调优先定位问题，再调参数。
2. Full GC 频繁先看对象为什么回收不掉。
3. OOM 先分类型，再看堆快照。
4. CPU 飙高更偏线程栈排查。
5. 很多 JVM 问题根因其实是代码问题，不只是参数问题。
