# Collection 场景强化题

## 1. 这份题的目标

前面的练习题偏基础，这一份更偏：

- 真实业务场景
- 集合选型
- 为什么选这个，不选那个

## 2. 场景题

### 场景 1

你要维护一个“评论列表”，要求：

- 保留插入顺序
- 允许重复评论内容

问题：

1. 你会选什么集合？
2. 为什么？
3. 为什么不选 `Set`？

### 场景 2

你要保存一批“用户标签”，要求：

- 标签不能重复
- 不关心输出顺序

问题：

1. 你会选什么集合？
2. 为什么不选 `List`？

### 场景 3

你要保存一批“去重后的标签”，并且希望遍历时尽量保持加入顺序。

问题：

1. 你会选什么集合？
2. 为什么不选普通 `HashSet`？

### 场景 4

你要统计一篇文章里每个单词出现的次数。

问题：

1. 你会选什么集合？
2. key 和 value 分别是什么？
3. 为什么不选 `List`？

### 场景 5

你要保存学生成绩，要求：

- 学号 -> 成绩
- 后续经常通过学号查成绩

问题：

1. 你会选什么集合？
2. 为什么？

### 场景 6

你要保存“按 id 查询用户对象”的关系，并且希望遍历时 key 的顺序更稳定。

问题：

1. 你会选什么集合？
2. 为什么不只用 `HashMap`？

### 场景 7

你要维护一组数字，要求：

- 不重复
- 自动按从小到大排列

问题：

1. 你会选什么集合？
2. 为什么不选 `HashSet`？

### 场景 8

你要维护一个任务处理结构，要求：

- 谁先入队谁先处理

问题：

1. 你会选什么集合？
2. 为什么这里不优先想到 `List`？

### 场景 9

你要维护一个结构，要求：

- 两端都可能插入和删除

问题：

1. 你会选什么接口或实现类？
2. 为什么？

### 场景 10

多线程环境下多个线程共享同一个 `Map`。

问题：

1. 你会选什么集合？
2. 为什么不直接选 `HashMap`？
3. 为什么面试里更常提 `ConcurrentHashMap` 而不是 `Hashtable`？

## 3. 参考作答方向

### 场景 1

- `ArrayList`
- 因为需要有序、可重复

### 场景 2

- `HashSet`
- 因为需求核心是去重

### 场景 3

- `LinkedHashSet`
- 因为去重 + 顺序表现更稳定

### 场景 4

- `HashMap<String, Integer>`
- key 是单词，value 是次数

### 场景 5

- `HashMap<String, Integer>` 或类似映射结构
- 因为重点是“通过 key 找 value”

### 场景 6

- `LinkedHashMap`
- 因为既要 key-value，又想顺序更稳定

### 场景 7

- `TreeSet`
- 因为去重 + 排序

### 场景 8

- `Queue`
- 或具体实现如 `LinkedList`

### 场景 9

- `Deque`
- 或具体实现如 `ArrayDeque` / `LinkedList`

### 场景 10

- `ConcurrentHashMap`
- 因为并发场景下更常见

## 4. 一句话总结

场景题的关键不是死背类名，而是先抓需求关键词：

- 有序
- 可重复
- 去重
- 排序
- key-value
- 队列
- 并发
