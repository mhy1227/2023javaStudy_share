# MyBatis 与数据库访问链路

## 1. 这份文档是干什么的
这份文档主要解决 Spring 主线里非常常见的一组问题：

1. 什么是 MyBatis
2. Mapper 是干什么的
3. 一次请求从 Controller 到数据库大概怎么走
4. MyBatis、JDBC、连接池之间是什么关系
5. 为什么项目里通常不会直接手写 JDBC

这组题在 Java 后端面试里非常常见。

因为面试官在问完：

- Spring MVC
- Service 层
- 事务

之后，通常就会顺着追：

> 你这条请求到底怎么走到数据库的？

所以这份很值得单独整理。

---

## 2. 先说最重要的结论
你先记住这四句：

1. **MyBatis 是持久层框架，主要负责 SQL 映射和数据库访问。**
2. **Mapper 可以理解成数据库访问接口。**
3. **一次典型链路通常是 `Controller -> Service -> Mapper -> MyBatis -> JDBC -> MySQL`。**
4. **连接池负责管理数据库连接，MyBatis 负责更方便地使用这些连接执行 SQL。**

这四句基本就是整条数据库访问链路的骨架。

---

## 3. 什么是 MyBatis
最稳的回答：

> MyBatis 是 Java 里常见的持久层框架，主要作用是帮我们做 SQL 执行、参数绑定和结果映射，让数据库访问这件事更方便、更清晰。

更口语一点：

> 你可以把 MyBatis 理解成一个“帮你更优雅访问数据库”的框架。

---

## 4. 为什么项目里常用 MyBatis，而不是直接手写 JDBC
这题很常见。

你可以这样答：

> 因为直接手写 JDBC 会有很多样板代码，比如获取连接、创建语句、设置参数、处理结果集、释放资源，代码会比较繁琐。MyBatis 把这些通用工作做了封装，同时还保留了 SQL 可控性，所以在很多项目里用得很多。

一句话版：

> MyBatis 不是不用 SQL，而是让你写 SQL 这件事更可维护。

---

## 5. Mapper 是什么
这是高频基础题。

你可以这样答：

> Mapper 可以理解成数据库访问接口，业务代码通过调用 Mapper 方法，把数据库查询和更新操作交给 MyBatis 去执行。

更口语一点：

> Mapper 就像数据库操作的门面接口。

---

## 6. Mapper 和 SQL 的关系怎么讲
你可以这样说：

> Mapper 接口方法本身是一个抽象入口，真正的 SQL 通常由 MyBatis 通过 XML、注解或者其他配置方式去绑定和执行。

这句话够面试用了。

---

## 7. MyBatis 主要帮我们做了什么
这个问题答好了会很像你真的做过。

你可以这样答：

> MyBatis 主要帮我们做三件事。第一，执行 SQL；第二，做参数绑定；第三，把数据库查询结果映射成 Java 对象。

这三件事非常关键。

---

## 8. 参数绑定是什么意思
你可以这样理解：

> 业务代码把参数传给 Mapper 方法后，MyBatis 会把这些参数绑定到对应 SQL 的占位符上，再去执行真正的数据库操作。

一句话版：

> 方法参数怎么进 SQL，这件事就是参数绑定在处理。

---

## 9. 结果映射是什么意思
你可以这样答：

> SQL 查询出来的是结果集，而业务代码更希望拿到 Java 对象。MyBatis 会把查询结果映射成对应的实体对象、DTO 或集合。

这句话也很值钱。

---

## 10. 一次请求从 Controller 到数据库怎么走
这是最实用的一题。

你可以先记最简单版本：

1. 请求进入 Controller
2. Controller 调 Service
3. Service 调 Mapper
4. Mapper 由 MyBatis 执行对应 SQL
5. MyBatis 通过 JDBC 和数据库交互
6. 数据返回后逐层往上返回给前端

这已经够你先回答主干了。

---

## 11. 数据库连接池在这条链路里是什么位置
这个点容易被忽略，但其实很重要。

你可以这样说：

> MyBatis 自己不是数据库连接池，它在执行 SQL 时通常会从数据源或连接池里拿连接。连接池负责连接的创建、复用和回收，MyBatis 负责在这些连接之上执行 SQL 和做映射。

一句话版：

> 连接池管连接，MyBatis 管访问。

---

## 12. JDBC 在这条链路里是什么位置
这题很适合把“框架”和“底层”关系讲清楚。

你可以这样答：

> JDBC 是 Java 访问数据库的底层标准接口，MyBatis 本质上还是建立在 JDBC 之上的，只不过它把很多数据库访问细节做了封装。

更口语一点：

> JDBC 是底层能力，MyBatis 是更上层、更好用的封装。

---

## 13. 为什么不建议面试时把 MyBatis 和 JDBC 说成对立关系
因为这样很容易显得理解不清。

更稳的说法是：

> MyBatis 不是替代 JDBC 的另一套底层，它本质上还是对 JDBC 的封装和增强。

这句话很关键。

---

## 14. Spring 和 MyBatis 是怎么配合的
你可以这样说：

> 在项目里，Spring 负责对象管理、依赖注入和事务整合，MyBatis 负责持久层数据库访问。也就是说，Spring 负责把 Mapper、Service 这些对象组织起来，MyBatis 负责具体执行 SQL。

这段很像真实项目表达。

---

## 15. 一段标准面试回答
如果面试官问你“从 Controller 到数据库这条链路怎么走”，你可以直接这样答：

> 一次典型请求一般是先到 Controller，由 Controller 接收参数后调用 Service 层处理业务，Service 再调用 Mapper 访问数据库。Mapper 本身可以理解成数据库访问接口，真正的 SQL 由 MyBatis 去执行。MyBatis 在执行过程中会做参数绑定和结果映射，底层本质上还是通过 JDBC 和数据库交互，只是实际拿连接时通常会从数据源或连接池里获取。简单说，这条链路一般就是 `Controller -> Service -> Mapper -> MyBatis -> JDBC -> MySQL`。 

---

## 16. 更口语一点的版本
你也可以这样讲：

> 请求进来以后，Controller 先接住，然后调 Service 处理业务，Service 再去调 Mapper。Mapper 这层真正执行 SQL 的是 MyBatis，MyBatis 底层还是靠 JDBC 去跟 MySQL 交互，只不过连接一般不是临时现建，而是从连接池里拿。所以你可以把它理解成，Spring 负责把对象和流程串起来，MyBatis 负责把数据库访问这件事做得更顺。 

---

## 17. 最值得先背的 10 句
1. **MyBatis 是持久层框架，主要负责 SQL 映射和数据库访问。**
2. **Mapper 可以理解成数据库访问接口。**
3. **MyBatis 主要做 SQL 执行、参数绑定和结果映射。**
4. **一次典型链路通常是 `Controller -> Service -> Mapper -> MyBatis -> JDBC -> MySQL`。**
5. **JDBC 是底层标准接口，MyBatis 是在它之上的封装。**
6. **连接池负责连接的创建、复用和回收。**
7. **MyBatis 不是连接池。**
8. **Spring 负责对象管理和事务整合，MyBatis 负责持久层访问。**
9. **项目里不用手写大量 JDBC，是因为 MyBatis 把样板代码封装了。**
10. **MyBatis 和 JDBC 不是对立关系，而是封装关系。**

---
