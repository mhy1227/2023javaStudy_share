# Spring 常规高频模拟回复

## 1. 这份文档怎么用
这份不是标准答案，而是“可以直接开口说”的模拟回复模板。

你用的时候这样处理：

1. 保留主干
2. 删掉不符合你项目的内容
3. 换成你自己的模块名
4. 每段控制在 20 到 40 秒

---

## 2. 什么是 IOC
模拟回复：

> 我理解 IOC 的核心就是对象不要自己在代码里手动 new 和管理依赖，而是交给 Spring 容器统一去创建、注入和管理。这样做的好处是对象之间耦合度更低，也更方便扩展和测试。简单说，就是对象控制权从业务代码转交给了容器。

---

## 3. 什么是 AOP
模拟回复：

> AOP 我一般理解成把事务、日志、权限这种横切逻辑从业务代码里抽出来统一处理。这样核心业务方法就只关心主流程，不用每个地方都重复写这些公共逻辑。Spring 里事务本身其实就是一个很典型的 AOP 应用。

---

## 4. `@Autowired` 和 `@Resource` 有什么区别
模拟回复：

> 我一般这样理解，`@Autowired` 是 Spring 原生注解，默认按类型注入；`@Resource` 是标准注解，通常按名称注入。项目里这两个都能完成依赖注入，但如果在 Spring 体系里，我更常见的是 `@Autowired`。另外实际开发里我会更倾向构造器注入，因为依赖关系更明确，也更利于测试。

---

## 5. `@Bean` 和 `@Component` 有什么区别
模拟回复：

> `@Component` 更像是把自己写的类直接交给 Spring 扫描注册，通常标在类上；`@Bean` 则一般写在配置类的方法上，通过方法返回值把对象注册进容器。我的理解是，`@Component` 更适合业务类，`@Bean` 更适合第三方组件或者需要手动定制初始化过程的对象。

---

## 6. `@Controller`、`@ResponseBody`、`@RestController` 的关系
模拟回复：

> `@Controller` 更偏传统 MVC，通常返回页面；`@ResponseBody` 表示方法返回值直接写到响应体；`@RestController` 可以理解成 `@Controller + @ResponseBody`。如果是前后端分离项目，一般直接用 `@RestController` 更自然，因为接口基本都是返回 JSON。

---

## 7. `@RequestBody` 和 `@RequestParam` 有什么区别
模拟回复：

> `@RequestBody` 主要是接请求体里的 JSON 数据，通常会转成一个 Java 对象；`@RequestParam` 更常见于接查询参数或者表单参数。前后端分离项目里，复杂对象提交通常是 `@RequestBody`，简单查询条件更常见的是 `@RequestParam`。

---

## 8. `@Transactional` 是怎么理解的
模拟回复：

> `@Transactional` 的核心作用就是声明事务，让多个数据库操作在一个统一的事务边界里执行，要么一起成功，要么一起回滚。实际项目里我一般会把事务放在 Service 层，因为事务本质上是业务层的边界控制，而不是 Controller 层或者 Mapper 层的职责。

---

## 9. `@Transactional` 为什么会失效
模拟回复：

> 最常见的失效场景有几类，比如同一个类内部自调用、方法不是 `public`、异常被自己 catch 住但没有继续抛出、或者抛出的异常不在默认回滚范围里。它本质上通常是因为事务是基于 Spring AOP 代理实现的，没有真正经过代理链，事务就不生效。

---

## 10. Spring MVC 执行流程怎么讲
模拟回复：

> 我一般会把 Spring MVC 执行流程讲成一条链路。请求先到 `DispatcherServlet`，它相当于前端控制器；然后根据请求找到要执行的 Controller 方法；再做参数绑定；方法执行完之后，如果是前后端分离场景，通常会通过消息转换器把返回对象转成 JSON 响应给前端。

---

## 11. Spring Boot 自动装配怎么讲
模拟回复：

> 我理解 Spring Boot 自动装配的核心不是“全自动”，而是“按条件自动装配”。也就是 Spring Boot 会根据当前项目的依赖、配置和环境，自动决定要加载哪些配置类和 Bean。这样我们不需要每次都手动写大量样板配置，开发效率会高很多。

---

## 12. Bean 生命周期怎么讲
模拟回复：

> Bean 生命周期我一般概括成：先实例化对象，再做依赖注入，然后执行初始化逻辑，之后 Bean 才真正可用；容器关闭时再执行销毁相关逻辑。面试里如果继续往下问，我会再补充初始化前后有扩展点，比如 BeanPostProcessor。

---

## 13. 拦截器和过滤器有什么区别
模拟回复：

> 过滤器是 Servlet 规范里的，更底层；拦截器是 Spring MVC 里的，更贴近 Controller 和 Handler。我的理解是，像字符编码、跨域这种更底层的通用处理可以放过滤器；像登录校验、权限判断、接口日志这种更贴近业务接口的处理，更适合放拦截器。

---

## 14. 一段总回答模板
如果面试官说：

> 你讲讲 Spring 这块你掌握的内容。

你可以这样答：

> Spring 这块我平时接触比较多的主要有几条线。第一条是 IOC 和依赖注入，也就是对象交给容器管理；第二条是 AOP，像事务这类能力本质上就是通过 AOP 落地的；第三条是 Web 层，比如 `@RestController`、参数绑定、消息转换器这些；第四条是 Spring Boot 自动装配，解决的是项目整合和快速启动问题。如果结合我自己的项目，最常用的还是事务、接口分层、统一异常处理、配置管理这些能力。
