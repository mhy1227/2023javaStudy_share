# Spring 高频对比详细回答

## 1. 这份文档怎么用

这一篇是 `spring-compare-note.md` 的展开版。

你可以这样理解两篇文档的分工：

- `spring-compare-note.md`：速记版，适合最后几分钟扫一遍
- `spring-compare-long-answer-note.md`：详细版，适合面试里需要展开说 30 秒到 1 分钟

如果你觉得原来的对比回复太短，这篇就是专门补这个问题的。

## 2. Spring vs Spring MVC vs Spring Boot

### 详细版回答

> 我一般会把这三个理解成三层。Spring 是基础框架底座，主要解决的是对象管理、依赖注入、AOP、事务这些通用能力；Spring MVC 是 Spring 在 Web 层的解决方案，负责把请求组织起来，比如通过 `DispatcherServlet` 接住请求，再路由到对应 Controller；Spring Boot 则不是替代 Spring，而是站在 Spring 生态上的快速开发和工程化整合方案，它把项目启动、配置管理和常见中间件整合做得更顺。所以这三者不是并列替代关系，而是底座、Web 层和工程化整合层的关系。

### 面试里再补一句

> 如果结合项目讲，我会把 Spring 理解成运行时底座，把 Spring MVC 理解成接口请求处理层，把 Spring Boot 理解成让这套东西更快搭起来的工程化方案。

## 3. IOC vs DI

### 详细版回答

> 这两个概念经常一起出现，但严格来说不是一回事。IOC 是一种思想，核心是对象的创建和依赖管理不再由业务代码自己控制，而是交给容器统一处理；DI 则是 IOC 最常见的实现方式，也就是容器在创建对象时，把它依赖的其他对象一并注入进去。所以如果用面试里更稳的说法，就是 IOC 讲的是控制权转移，DI 讲的是容器具体怎么把依赖塞进来。

### 面试里再补一句

> 所以我一般会说，IOC 是理念，DI 是它在 Spring 里的典型落地方式。

## 4. AOP vs OOP

### 详细版回答

> OOP 主要解决的是业务建模和职责封装问题，强调把状态和行为组织到类和对象里；AOP 则更适合处理事务、日志、权限、监控这种横切逻辑。也就是说，OOP 更像把业务本身组织好，AOP 更像在不改业务代码的前提下，把很多通用能力统一织进去。所以这两者不是对立关系，而是处理问题的维度不同，一个偏业务结构，一个偏横切增强。

### 面试里再补一句

> 我通常会把 AOP 理解成对 OOP 的补充，而不是替代。

## 5. BeanFactory vs ApplicationContext

### 详细版回答

> `BeanFactory` 可以理解成更基础的容器接口，核心能力是 Bean 的获取和管理；`ApplicationContext` 则是在这个基础上进一步扩展出来的完整容器，除了 Bean 管理之外，还整合了国际化、资源访问、事件发布等更多能力。项目里真正高频接触到的通常是 `ApplicationContext`，因为它代表的是更完整的 Spring 容器能力。

### 面试里再补一句

> 如果问到区别，我一般会强调“一个更底层基础，一个更完整常用”。

## 6. `@Autowired` vs `@Resource`

### 详细版回答

> 这两个注解都能完成依赖注入，但语义来源和默认匹配思路不完全一样。`@Autowired` 更偏 Spring 原生体系，默认更常被理解为按类型注入；`@Resource` 是标准注解，常被理解为按名称注入。所以面试里我一般不会把它答成绝对规则，而是会说：在 Spring 项目里更常见的是 `@Autowired`，如果要强调标准注解或者名称匹配，也可能用 `@Resource`。

### 面试里再补一句

> 如果再往实践上说，我会更推荐构造器注入，而不是纠结字段上到底写哪个注解。

## 7. `@Component` vs `@Bean`

### 详细版回答

> `@Component` 更适合标在类上，让 Spring 通过组件扫描自动把这个类注册进容器；`@Bean` 一般写在配置类的方法上，表示把方法返回对象显式注册成 Bean。前者更像“框架扫描发现”，后者更像“开发者主动声明注册”。如果是自己写的业务类，通常更常用 `@Component`、`@Service` 这种方式；如果是第三方组件、需要自定义初始化逻辑，或者这个类本身不能直接加注解，那更适合用 `@Bean`。

### 面试里再补一句

> 所以这题的关键不是谁更高级，而是谁更适合当前对象的注册方式。

## 8. `@Controller` vs `@RestController`

### 详细版回答

> `@Controller` 更常用于传统 MVC 页面场景，返回值更容易和视图解析关联起来；`@RestController` 更适合前后端分离项目，它本质上可以理解成 `@Controller + @ResponseBody`，也就是方法返回值通常直接写到响应体里，常见形式就是 JSON。所以如果是现代后端接口项目，我通常会更常用 `@RestController`。

### 面试里再补一句

> 一句话记忆就是，一个偏页面，一个偏接口数据返回。

## 9. `@RequestBody` vs `@RequestParam`

### 详细版回答

> `@RequestBody` 主要是接收请求体里的 JSON 数据，再把它转换成 Java 对象；`@RequestParam` 更常用于接收 URL 查询参数或者表单参数。面试里更稳的说法不是只讲注解名，而是顺手补场景：复杂对象提交、前后端分离接口更常见用 `@RequestBody`，而简单筛选条件、分页参数、搜索条件更常见用 `@RequestParam`。

### 面试里再补一句

> 所以一个更偏请求体反序列化，一个更偏简单参数提取。

## 10. `@ResponseBody` vs `@RestController`

### 详细版回答

> `@ResponseBody` 一般标在方法上，表示当前方法返回值直接写到 HTTP 响应体里，而不是走页面视图解析；`@RestController` 一般标在类上，相当于统一给这个类下的方法都加上了 `@ResponseBody`。所以如果是前后端分离项目，通常直接用 `@RestController` 会更自然，不需要在每个方法上重复写 `@ResponseBody`。

### 面试里再补一句

> 一个是方法级别，一个是类级别。

## 11. `ApplicationRunner` vs `CommandLineRunner`

### 详细版回答

> 这两个接口都属于 Spring Boot 启动后执行逻辑的钩子。它们的时机非常接近，核心区别主要在参数封装方式上：`CommandLineRunner` 接收原始的字符串数组参数，`ApplicationRunner` 接收封装后的 `ApplicationArguments`。所以如果只是从面试角度看，这题最稳的回答不是展开太多，而是说明“作用很像，区别主要在参数形态”。

### 面试里再补一句

> 如果我要处理启动参数，会更偏向 `ApplicationRunner`，因为参数封装更友好。

## 12. Listener vs Runner

### 详细版回答

> 这两个都和 Spring Boot 启动有关，但职责不一样。Listener 更偏事件监听，也就是在某个启动阶段收到事件通知后做处理；Runner 更偏应用启动末尾执行逻辑的入口。也就是说，Listener 关注“什么时候发生了什么事”，Runner 关注“启动到这个阶段后我要执行什么逻辑”。所以如果面试官问区别，最稳的切入点就是“一个偏事件，一个偏执行入口”。

### 面试里再补一句

> 这题本质上考的是你有没有把启动流程、事件和钩子区分开。

## 13. `ApplicationStartedEvent` vs `ApplicationReadyEvent`

### 详细版回答

> 这两个事件经常容易被混。`ApplicationStartedEvent` 更早，表示容器 refresh 已经完成，但 `ApplicationRunner` / `CommandLineRunner` 还没有全部执行完；`ApplicationReadyEvent` 更晚，表示这些 Runner 都跑完了，应用真正进入 ready 状态，更接近可以正式接流量。所以这题最关键的一句其实是：Started 不等于 Ready。

### 面试里再补一句

> 如果让我选哪个更接近“应用真的起来了”，我会选 `ApplicationReadyEvent`。

## 14. `starter` vs 自动装配

### 详细版回答

> 这两个概念也很容易被说混。`starter` 更偏依赖整合，它把某类场景常用的依赖先带进来；自动装配更偏配置生效，它会根据当前依赖、配置和条件，决定哪些配置类和 Bean 要真正生效。所以如果一句话区分，就是 `starter` 负责“把材料带进来”，自动装配负责“按条件把材料拼起来”。这题如果答成一个东西，面试官一般会继续追问。

### 面试里再补一句

> 一个偏依赖入口，一个偏配置生效。

## 15. Spring 事务 vs 数据库事务

### 详细版回答

> Spring 事务和数据库事务不是两个互斥概念，而是应用层和数据库层的配合。Spring 事务更多是在应用层统一管理事务边界，核心是通过 AOP 和代理把事务控制织入到业务方法调用链里；数据库事务则是数据库自身提供的提交、回滚、一致性和隔离性能力。也就是说，Spring 负责“什么时候开启和结束这段事务边界”，数据库负责“底层怎么真正完成事务能力”。

### 面试里再补一句

> 所以我一般会说，Spring 管边界，数据库管底层实现。

## 16. 同步事务 vs `@Async` 场景下事务

### 详细版回答

> 同步事务很多时候和当前线程上下文绑定，而 `@Async` 的核心特点是切线程。所以当事务和异步放在一起时，问题就不再只是“两个注解能不能共存”，而是“线程切走以后，原来的事务上下文还在不在”。因此这题最稳的答法是：同步事务和异步线程场景下事务的关键差别在于线程边界，一旦切线程，事务上下文通常不会自然传过去。

### 面试里再补一句

> 这题本质上考的是线程切换、代理和事务边界的关系。

## 17. Profile vs 配置覆盖

### 详细版回答

> Profile 更偏“按环境切换配置”，比如 dev、test、prod 属于不同环境分组；配置覆盖则更偏“同一个配置项来自多个来源时，最终谁生效”。一个是环境划分维度，一个是优先级覆盖维度。很多人会把这两个混成一回事，但面试里其实问的是两个不同问题。

### 面试里再补一句

> Profile 管环境，覆盖规则管优先级。

## 18. Spring Boot 2.x vs 3.x

### 详细版回答

> 面试里如果问版本差异，最值得讲的是 2.x 到 3.x 这条主线，而不是去背很多零散小改动。更稳的回答是：Boot 3 抬高了 Java 基线到 17，底层切到了 Spring Framework 6，Jakarta 命名空间从 `javax.*` 到 `jakarta.*`，再往后还有自动配置口径和部分依赖生态的变化。所以这题的核心不是记很多版本号，而是知道升级到 Boot 3 时，技术基线和依赖生态都一起变了。

### 面试里再补一句

> 所以这题最值钱的关键词通常是 Java 17、Framework 6、Jakarta。

## 19. 这些对比题怎么练最有效

建议练法：

1. 先看 `spring-compare-note.md` 的速记版
2. 再回到这篇练详细版
3. 每次只抽一个对比项，用 30 秒到 1 分钟讲清

这样比一口气背很多更有效。
