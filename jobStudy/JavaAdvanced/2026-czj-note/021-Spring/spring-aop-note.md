# Spring AOP 笔记

## 1. 这份文档怎么用

这一篇专门补 Spring 很核心的一条线：

- 什么是 `AOP`
- 为什么 Spring 事务和 `AOP` 强相关
- 为什么日志、权限、事务这类能力能统一增强
- 代理为什么是这条线的关键

如果 `IOC` 解决的是“对象为什么能被管理”，那 `AOP` 解决的就是“通用逻辑为什么能被统一织进去”。

## 2. 先记住最核心的 5 句

1. `AOP` 就是面向切面编程。
2. 它的核心是在不改业务代码的前提下，把通用逻辑织入进去。
3. Spring 的事务、本质上就是 `AOP` 的典型应用。
4. Spring 做 `AOP`，代理是关键。
5. 业务逻辑负责业务，横切逻辑由代理统一增强。

## 3. 什么是 AOP

最稳的回答：

> `AOP` 就是面向切面编程，核心思想是在不改业务代码的前提下，把日志、事务、权限、监控这类通用横切逻辑统一织入到目标方法前后。

更口语一点：

> 业务代码只管业务，通用能力由框架帮你统一加上去。

## 4. 为什么 Spring 需要 AOP

可以这样答：

> 因为很多项目里都会有一些反复出现的通用逻辑，比如事务控制、日志记录、权限校验、性能统计。如果这些逻辑都手写在每个业务方法里，代码会非常重复，也很难维护，所以 Spring 通过 `AOP` 来统一处理。

一句话记忆：

> `AOP` 是为了把重复的横切逻辑从业务代码里抽出来。

## 5. 什么叫“横切逻辑”

更稳的回答：

> 横切逻辑指的就是那种会横跨很多业务方法、很多模块反复出现的通用逻辑，比如日志、事务、权限、异常处理这类能力。

## 6. Spring AOP 为什么离不开代理

这是高频追问。

最稳的回答：

> 因为 Spring 要在方法执行前后织入增强逻辑，前提是调用不能直接打到原始对象本身，而是先经过代理对象。只有这样，Spring 才有机会在方法前后做统一增强。

一句话版：

> 不是业务对象自己有魔法，而是代理对象在起作用。

## 7. AOP 和事务的关系怎么讲

这题非常关键。

更稳的回答：

> Spring 事务本质上就是 `AOP` 在事务场景下的一种典型应用。业务方法负责处理业务，事务的开启、提交、回滚这些横切逻辑由 Spring 通过代理统一织入。

## 8. AOP 在项目里常见用法

你可以举最常见的几个方向：

- 事务
- 日志
- 权限
- 监控
- 统一埋点

重点不是举得越多越好，而是知道：

> `AOP` 更适合处理“很多地方都要做，但又不想侵入业务代码”的逻辑。

## 9. 面试里一般怎么问

常见问法：

- 什么是 `AOP`
- Spring 为什么需要 `AOP`
- `AOP` 能解决什么问题
- Spring 事务和 `AOP` 有什么关系
- Spring AOP 为什么离不开代理

面试里的稳法是：

> 先讲概念，再讲为什么需要，再讲事务是最典型应用。

## 10. 一版适合面试的完整回答

> `AOP` 就是面向切面编程，核心思想是在不改业务代码的前提下，把日志、事务、权限这些通用横切逻辑统一织入到目标方法前后。Spring 之所以需要 `AOP`，是因为项目里很多通用逻辑会横跨很多模块反复出现，如果全部手写在业务代码里，会非常重复。Spring 做 `AOP` 时，代理对象是关键，因为调用先经过代理，Spring 才有机会做增强。像 `@Transactional` 这种事务能力，本质上就是 `AOP` 在事务场景下的一种典型应用。

## 11. 最容易答错的点

- 把 `AOP` 讲成“就是代理”，但不知道它解决什么问题
- 只会说 `AOP` 很强，却举不出事务这种典型应用
- 不知道为什么代理是这条线的关键

## 12. 一句话收尾

> AOP 这条线真正要讲清的，不是名词，而是 Spring 为什么能在不改业务代码的前提下，把通用能力统一增强进去。
