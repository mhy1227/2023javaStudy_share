# Spring 事务失效场景笔记

## 1. 这份文档怎么用

这一篇单独只讲一个高频问题：

`@Transactional` 为什么会失效？

这个问题几乎是 Spring 面试里的固定追问，所以单独拆出来更方便复习。

## 2. 先记住最核心的 5 句

1. `@Transactional` 失效，很多时候不是注解写错了，而是代理没有真正生效。
2. 同类内部自调用，是最经典的事务失效场景。
3. 异常被吃掉了，Spring 感知不到需要回滚，也会看起来像“事务没生效”。
4. 不是所有异常都会默认触发回滚。
5. 多线程、异步切线程后，事务上下文也可能传不过去。

## 3. 先给一句总定义

面试里你可以先这样开口：

> `@Transactional` 失效，本质上通常和三件事有关：调用有没有经过代理、异常有没有按预期传播、执行上下文有没有发生变化。

这句很适合先稳住全局。

## 4. 最经典的一题：同类内部自调用

这是最常见的问法。

标准回答：

> Spring 事务依赖代理对象生效，而同类内部方法互相调用时，调用通常直接落在当前对象本身，不会先经过 Spring 代理，所以事务增强就没有机会织入。

一句话版：

> 核心不是注解坏了，而是调用没走代理。

## 5. 为什么“自己调自己”不行

更口语一点可以这样讲：

> 外部调这个 Bean 时，拿到的是 Spring 包装过的代理对象；但类里面一个方法直接调另一个方法，相当于对象内部直接调用，绕开了代理层，所以事务逻辑没拦到。

## 6. 方法可见性为什么也常被问

可以这样答：

> Spring 事务增强通常依赖代理机制，所以事务方法最好放在符合代理增强预期的位置上。实际开发里一般建议放在 `public` 方法上，这样更稳，也符合大多数项目习惯。

这里不需要讲得过深，但要知道：

- 事务方法不要写得太随意
- 常规项目里把事务边界放在 `public` 的 `Service` 方法上最稳

## 7. 异常被吃掉为什么会像事务失效

这题特别像真实项目排查。

标准回答：

> 如果业务代码里自己把异常 `catch` 住了，而且没有继续往外抛，Spring 可能就感知不到当前事务应该回滚，最后表现出来就像事务没有生效。

一句话版：

> Spring 不是不会回滚，而是它根本不知道你这里应该回滚。

## 8. 为什么“抛异常了”还可能不回滚

可以这样答：

> 因为 Spring 默认更常见的是对运行时异常进行回滚，如果异常类型不在默认回滚范围内，或者你没有按预期配置回滚规则，也会出现抛了异常但事务没有回滚的情况。

这题的关键点是：

- 不是所有异常都会自动回滚
- 不要把“抛异常”和“一定回滚”画等号

## 9. Bean 没交给 Spring 管理也会有问题

这题有时会被顺带问。

可以这样答：

> 如果这个对象根本不是 Spring 容器里的 Bean，那它就不会被 Spring 代理，自然也谈不上事务增强。

一句话记忆：

> 没进容器，就没代理；没代理，就没事务增强。

## 10. 多线程 / 异步场景为什么也危险

可以这样答：

> Spring 事务很多时候和当前线程上下文绑定，如果方法执行过程中切到了别的线程，比如线程池、异步任务，那原来的事务上下文就可能传不过去，这时也容易出现事务不符合预期的问题。

这句很像做过项目的人会说的话。

## 11. 为什么很多人觉得“加了注解就一定安全”

这是一个常见误区。

更好的理解是：

> `@Transactional` 只是告诉 Spring 这里需要事务能力，但真正能不能按预期生效，还要看调用路径、代理机制、异常传播和线程上下文是不是都成立。

## 12. 一版适合面试的标准回答

> `@Transactional` 失效，最常见的原因不是注解写错了，而是事务增强没有真正生效。比如最经典的同类内部自调用，因为这种调用不会经过 Spring 代理；再比如异常被业务代码自己捕获了，Spring 感知不到需要回滚；或者抛出的异常类型不在默认回滚范围内；另外如果对象没交给 Spring 管理，或者执行过程中切到了异步线程，事务上下文也可能丢失。所以这类问题本质上通常都和代理、异常传播以及执行上下文有关。 

## 13. 更口语一点的版本

> 我一般会先看这个调用有没有经过 Spring 代理，因为同类内部自调用是最经典的坑。然后再看异常是不是被吃掉了，或者异常类型是不是没有触发默认回滚。再往后就是看对象是不是 Spring Bean，以及有没有切线程导致事务上下文丢失。 

## 14. 最容易答错的点

- 只会背“同类内部调用失效”，但说不清为什么
- 认为事务失效就一定是 Spring bug
- 不知道异常被吃掉也会导致回滚失败
- 不知道异步线程会让事务上下文断掉
- 把“没回滚”和“没加事务”混成一回事

## 15. 一句话收尾

> 事务失效问题表面看是注解问题，实质上大多还是代理有没有接住调用、异常有没有正确传出去、执行有没有一直留在原来的事务上下文里。
