# Spring 项目场景题

## 1. 这份文档怎么用

这一篇专门补一种特别像真实一面的题：

不是问你“什么是 IOC”，而是问你：

- 你项目里 Spring 怎么用
- 事务在项目里怎么落
- Spring Boot 为什么适合你们项目
- 你们项目里有什么 Spring 相关坑点

这种题的核心不是背定义，而是把知识点说得像“真的在项目里落过”。

## 2. 场景一：你项目里 Spring 主要承担什么作用

### 问题

你项目里 Spring 主要承担什么作用？

### 回答方向

> 可以从对象管理、依赖注入、事务控制、Web 请求处理、统一增强这几层展开。

### 口语版

> 如果结合项目看，Spring 对我来说更像是运行时基础设施。Controller、Service、配置类这些对象都交给容器管理；事务、参数校验、统一异常处理这类通用能力也都建立在 Spring 这一层上；再往外 Spring MVC 负责把请求路由到对应接口，所以很多业务模块其实都是搭在 Spring 这套底座上的。

## 3. 场景二：你们项目里为什么会用 Spring Boot

### 问题

你们项目里为什么会用 Spring Boot？

### 回答方向

> 重点讲工程化整合和启动效率。

### 口语版

> 因为常规后端项目里，Spring Boot 基本已经是默认底座了。它的价值不只是让我少配几个注解，而是把项目启动、配置管理和常见中间件整合都做得更顺。如果不用 Spring Boot，很多 Web、配置、数据库访问这类能力要自己拼装，成本会更高。

## 4. 场景三：你们项目里事务一般怎么划边界

### 问题

你们项目里的事务边界一般怎么划？

### 回答方向

> 重点讲 Service 层、多表写入、完整业务流程。

### 口语版

> 我一般会把事务边界放在 Service 层，因为这层更接近完整业务流程，能统一包住多个 DAO 或 Mapper 的调用。像多表写入、状态流转、结果汇总这类场景，就更适合在这一层通过 `@Transactional` 去控制一致性。

## 5. 场景四：你们项目里统一异常处理怎么做

### 问题

你们项目里的统一异常处理怎么做？

### 回答方向

> 重点讲 `@ControllerAdvice` 和 `@ExceptionHandler`。

### 口语版

> 一般会通过 `@ControllerAdvice` 和 `@ExceptionHandler` 做统一异常处理，把接口里分散的 try-catch 尽量收掉。这样返回体格式能统一，错误码和错误信息也更容易维护，接口层也会更干净。

## 6. 场景五：你们项目里最常用哪些 Spring 注解

### 问题

你们项目里最常用哪些 Spring 注解？

### 回答方向

> 按层分类讲，不要只罗列。

### 口语版

> 如果结合项目分层看，接口层最常用的是 `@RestController`、`@RequestMapping`、`@RequestBody`；业务层常见的是 `@Service` 和 `@Transactional`；配置层会有 `@Configuration`、`@Bean`；统一异常和参数校验这边则会涉及 `@ControllerAdvice`、`@ExceptionHandler`、`@Validated`。

## 7. 场景六：项目里有没有遇到 Spring 的坑

### 问题

你项目里有没有遇到过 Spring 相关问题？

### 回答方向

> 优先说真实又高频的问题，比如事务失效、扫描范围、参数绑定、统一异常处理。

### 口语版

> 这种我一般优先说事务失效，因为它既高频又像真实排查场景。比如同类内部自调用导致没有走 Spring 代理，或者异常被业务代码吃掉，最后事务没有按预期回滚。除此之外，像配置扫描范围、参数绑定、统一异常处理这些工程问题也比较常见。

## 8. 场景七：如果让你评价 Spring Boot 在项目里的价值

### 问题

如果结合项目，你怎么评价 Spring Boot 的价值？

### 回答方向

> 不要只说“少写配置”，要说工程效率。

### 口语版

> 我觉得 Spring Boot 的价值主要体现在工程效率。它把项目启动、默认配置、中间件整合这些事情工程化了，所以团队不需要每次都从零去拼搭项目底座。对开发来说，最大的感受其实不是少写了几个配置项，而是整个项目搭建和扩展都更顺。

## 9. 场景八：如果让你把 Spring 和 Spring Boot 一起讲

### 问题

如果结合项目，把 Spring 和 Spring Boot 一起讲，你会怎么说？

### 口语版

> 如果结合项目整体看，Spring 更像运行时基础设施，负责对象管理、依赖注入、事务控制和 Web 请求处理这些底座能力；Spring Boot 则是在这套底座上把项目启动、配置和常见中间件整合工程化了。所以我一般会把 Spring 理解成“运行时底座”，把 Spring Boot 理解成“工程化整合方案”。

## 10. 这类场景题怎么练最有效

建议练法：

1. 先选一个你熟悉的项目
2. 把每道题都强行往这个项目上落
3. 不要空讲概念
4. 每题控制在 30 秒到 1 分钟

如果一题说到 2 分钟还没说清，多半就是没抓住重点。

## 11. 一句话收尾

> 项目场景题的关键，不是把 Spring 八股再背一遍，而是让面试官感觉到你知道这些能力在项目里到底是怎么落下去的。
