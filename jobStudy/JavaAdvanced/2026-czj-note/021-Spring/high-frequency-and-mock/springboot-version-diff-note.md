# Spring Boot 版本差异笔记

## 1. 这份文档怎么用

这一篇不是为了把所有版本差异都背下来，而是专门整理面试里最可能问到、也最容易答错的几类版本问题：

- Spring Boot 2.x 和 3.x 的差异
- 迁移时最容易踩的坑
- 现在再讲老口径时，哪些地方已经需要更新

## 2. 先说最重要的结论

如果是面试视角，最值得记住的是：

1. 真实项目和面试里，最常见的还是 `2.x -> 3.x` 差异。
2. 这条线最值钱的不是背版本号，而是知道 Java 版本、Jakarta 命名空间、Spring Framework 基线和自动配置相关变化。
3. 旧口径里只说 `spring.factories`，现在最好补一句较新版本已经转向 `AutoConfiguration.imports` 这类机制。
4. 如果你能把“运行时基础能力没变，但工程层和依赖生态变了”讲清，已经很像做过升级的人。

## 3. 当前官方文档的稳定版本线

根据 Spring Boot 官方文档总览页，截至 2026-04-22，页面展示的稳定文档线包括：

- `4.0.5`
- `3.5.13`
- `3.4.13`
- `3.3.13`

这意味着：

> 面试里如果只是讲 Boot 2 的老口径而完全不提 3 甚至 4 的变化，容易显得有点旧。

但对大多数 Java 面试来说，最核心的版本差异主线依然是：

> `2.x -> 3.x`

## 4. 2.x -> 3.x 最值得讲的差异

### 4.1 Java 基线提高

官方迁移资料里明确提到，Spring Boot 3.0 要求 Java 17 或更高版本。

面试里你可以这样答：

> Boot 3 相比 2.x，一个很直接的变化就是最低 Java 版本抬到了 17，所以升级前首先要看 JDK 基线是不是满足。

### 4.2 Spring Framework 基线变化

Boot 3 建立在 Spring Framework 6 上。

一句话版：

> 升级到 Boot 3，不只是 Boot 自己变了，底层 Spring Framework 基线也一起抬升了。

### 4.3 Jakarta 命名空间变化

这是最常见的高频点。

官方迁移资料里明确提到，Boot 3 使用 Jakarta EE 10，对应包名从 `javax.*` 迁移到 `jakarta.*`。

面试里稳一点的表达是：

> 升级到 Boot 3 时，一个最典型的变化就是 Jakarta 命名空间迁移，很多原来 `javax.*` 的依赖和 import 都要切到 `jakarta.*`。

### 4.4 Spring Security 基线变化

官方迁移资料也提到，Boot 3 使用 Spring Security 6。

一句话记忆：

> 安全相关项目升级到 Boot 3 时，往往不只是 Boot 自己的改动，Security 这条线也得一起看。

### 4.5 配置属性迁移

官方升级文档提到，一些配置项会被重命名或移除，并推荐使用 `spring-boot-properties-migrator` 做过渡检查。

面试里可直接说：

> 升级时不能只改依赖版本，还要检查配置属性变化，Spring Boot 官方也提供了 `spring-boot-properties-migrator` 这类迁移辅助模块。

## 5. 自动配置发现机制这条线怎么讲

这是近几年很值得补的一点。

如果你只讲老版本口径，可能会说：

> 自动配置类通过 `spring.factories` 被发现。

这在旧版本语境下没问题，但更稳的说法应该是：

> 如果从老版本视角讲，很多人会先想到 `spring.factories`；但在较新的 Spring Boot 版本语境里，自动配置发现机制已经转向 `META-INF/spring/org.springframework.boot.autoconfigure.AutoConfiguration.imports` 这类方式，所以面试里最好把这点顺手更新一下。

你不一定要展开到源码，但至少别把旧说法讲成唯一正确答案。

## 6. 版本差异这题最稳的答法

如果面试官问：

> 你了解 Spring Boot 版本差异吗？

你可以这样答：

> 如果按面试里最常见的主线，我会重点看 2.x 到 3.x 的变化。第一是 Java 基线提高到 17，第二是底层切到 Spring Framework 6，第三是 Jakarta 命名空间从 `javax.*` 迁移到 `jakarta.*`，这在依赖和 import 上都很明显。再往后还有 Spring Security 6、部分配置属性迁移，以及自动配置发现机制的更新。如果是更新的版本线，我知道现在官方稳定文档已经到了 4.x 和 3.5.x 这些线，但大多数项目和面试里，2 到 3 的迁移差异还是最值得先讲清的。 

## 7. 升级场景里最容易被追问的坑

- Java 版本不够
- 旧的 `javax.*` 还没清理干净
- 第三方依赖生态没跟上 Boot 3 / Framework 6
- 配置属性变更没检查
- 还在只用旧版本自动配置口径回答问题

## 8. 这篇怎么和面试结合

如果你项目没做过真实升级，也可以这样说：

> 我没有完整做过大规模 Boot 升级项目，但我知道版本差异最关键的点主要集中在 Java 基线、Jakarta 命名空间、Spring Framework 基线、安全依赖和自动配置机制这些地方。

这样会比硬说“我做过很多升级”更稳。

## 9. 一句话收尾

> Spring Boot 版本差异题最重要的，不是背很多小改动，而是把 2.x 到 3.x 的主线变化和新版本口径更新说清。
