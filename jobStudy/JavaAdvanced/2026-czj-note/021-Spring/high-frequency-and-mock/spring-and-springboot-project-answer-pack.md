# Spring / Spring Boot 项目回答包

## 1. 这份文档怎么用

这一篇不是纯八股，而是专门处理一个更贴近真实一面的场景：

面试官不一定会直接问你“什么是 IOC”，他更可能这样问：

- 你项目里 Spring 用得多吗
- Spring 在你项目里主要承担什么作用
- 你们项目为什么用 Spring Boot
- 你们项目里的事务、注解、配置是怎么落的

所以这份文档的重点是：

把 Spring 和 Spring Boot 的知识点，变成更像真实项目表达的话术。

## 2. 如果被问“Spring 在你项目里主要起什么作用”

### 口语版

> 如果结合项目看，我会把 Spring 的作用分成几层。第一层是对象管理和依赖注入，也就是 Controller、Service、配置类这些对象都交给 Spring 容器统一管理；第二层是通用能力增强，比如事务、参数校验、统一异常处理；第三层是 Web 请求处理，也就是通过 Spring MVC 把请求路由到对应接口。整体上它更像项目运行时的一套基础设施。

## 3. 如果被问“你项目里最常用哪些 Spring 能力”

### 口语版

> 如果结合常规后端项目，我最常接触的还是几块：一是 `@Service`、`@RestController`、`@Autowired` 这些基础组件和依赖注入；二是 `@Transactional` 做事务边界控制；三是 `@RequestBody`、`@RequestParam` 这些 Web 层参数处理；四是 `@ControllerAdvice`、`@ExceptionHandler` 做统一异常处理；再往外一点就是配置类和 Spring Boot 启动整合。

## 4. 如果被问“你项目里为什么用 Spring Boot”

### 口语版

> 这个我一般会从工程效率去讲。因为传统 Spring 项目本身能力很强，但配置和整合成本偏高。Spring Boot 的好处是把很多样板配置和中间件整合工作收掉了，让项目启动、配置管理和依赖整合都更高效，所以现在大多数项目基本都会直接基于 Spring Boot 来搭。

## 5. 如果被问“Spring Boot 在你项目里带来了什么实际收益”

### 口语版

> 最直接的收益是项目搭建效率和整合效率更高。比如 Web、数据库访问、配置绑定这些能力，很多默认方案已经准备好了，我们不用再从零拼装。第二是项目结构更统一，启动入口、组件扫描、配置加载这些都比较自然。第三是后面接 Redis、MyBatis、消息队列这类能力时，整体整合成本会更低。

## 6. 如果被问“你项目里的事务一般怎么做”

### 口语版

> 事务这块我一般会放在 Service 层来做边界控制，因为 Service 层更接近完整业务流程，能统一包住多个 DAO 或 Mapper 的调用。像多表写入、状态流转、汇总更新这类场景，就比较适合在这一层用 `@Transactional` 控住。

## 7. 如果被问“你们项目里统一异常处理怎么做”

### 口语版

> 一般会通过 `@ControllerAdvice` 和 `@ExceptionHandler` 做统一异常处理，把接口里分散的 try-catch 尽量收掉。这样一方面返回体格式能统一，另一方面错误码和提示信息也更容易维护。

## 8. 如果被问“你们项目里注解主要集中在哪些地方”

### 口语版

> 我一般会分层看。接口层主要是 `@RestController`、`@RequestMapping`、`@RequestBody`；业务层主要是 `@Service`、`@Transactional`；配置层会有 `@Configuration`、`@Bean`；再往外是统一异常处理和参数校验这类增强注解。

## 9. 如果被问“你怎么理解 Spring 在项目里的地位”

### 口语版

> 我会把 Spring 理解成项目运行时的基础设施，不是单独一个功能模块。因为对象管理、依赖注入、事务、Web 请求处理、统一增强这些能力，基本都建立在 Spring 这一层上。很多业务模块能顺利拼起来，背后其实都在吃 Spring 的这套底座能力。

## 10. 如果被问“你怎么理解 Spring Boot 在项目里的地位”

### 口语版

> Spring Boot 更像是把 Spring 这套基础设施工程化了。它不是替代 Spring，而是让 Spring 项目的启动、整合、配置和扩展变得更顺手。所以如果 Spring 更偏运行时底座，那 Spring Boot 就更偏工程化启动和整合底座。

## 11. 如果被问“结合你项目说一下 Spring MVC”

### 口语版

> 如果结合项目讲，Spring MVC 对我来说就是接口请求处理这一层。前端请求先通过 Spring MVC 路由到对应 Controller，再由 Controller 去调 Service，最后把结果通过 JSON 返回出去。像参数绑定、路径映射、统一返回处理，其实都在这条链上。

## 12. 如果被问“结合项目说一下自动装配”

### 口语版

> 如果从项目视角讲，我不会先说源码，我会先说结果。也就是很多常见能力像 Web、配置绑定、数据库访问，它们之所以能比较自然接进项目，背后就是 Spring Boot 自动装配在按条件把对应配置接起来。也就是说，它先帮你把常规路径打通了，你再在这个基础上做定制。

## 13. 如果被问“你项目里有没有碰到 Spring 的坑”

### 口语版

> 这类问题我一般会优先说真实又高频的坑，比如事务失效，尤其是同类内部自调用导致没有走代理；或者参数绑定、统一异常处理、配置扫描范围这种比较常见的工程问题。这样既能体现你做过项目，也不会显得是硬编特别离谱的故障。

## 14. 一个适合一面使用的项目总答法

### 口语版

> 如果结合项目整体看，Spring 主要帮我们解决了对象管理、依赖注入、事务控制和 Web 请求处理这些基础能力，很多 Controller、Service、统一异常处理、参数校验其实都建立在这套底座上。Spring Boot 则进一步把项目启动、配置管理和常见中间件整合这件事做得更高效，所以我一般会把 Spring 理解成运行时基础设施，把 Spring Boot 理解成工程化整合方案。这样项目开发时，不管是接口层、业务层还是配置层，整体协作都会顺很多。 

## 15. 一句话收尾

> 项目表达题的关键，不是把 Spring 和 Spring Boot 拆成纯八股，而是让面试官感觉到你真的知道它们在项目里分别承担什么角色。
