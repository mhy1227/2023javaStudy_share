# JDK 版本题陷阱问法

## 1. 这份文档怎么用

这一篇专门整理 JDK 版本题里最容易踩坑的地方。

它不是让你学更多新知识，而是帮你避免这种情况：

- 明明知道一点，但一答就散
- 会背特性，但不会结合项目
- 说得很热闹，结果一听就像没真正理解

## 2. 最常见的坑

### 坑 1：一提 JDK 8 只会说 Lambda

这是最常见的问题。

如果你回答：

> `JDK 8` 主要就是 `Lambda`

那基本等于把后面的分全丢了。

更稳的讲法应该是：

> `JDK 8` 的主线是函数式和声明式写法明显进入 Java，代表点包括 `Lambda`、函数式接口、`Stream`、`Optional`、接口默认方法和新的时间 API。

### 坑 2：一提 JDK 17 只会说 LTS

`LTS` 当然重要，但如果你只会说：

> 因为它是长期支持版本

那回答太薄了。

更完整一点应该是：

> `JDK 17` 既是 `LTS`，也是很多现代框架和新项目的基线。它的价值不只是长期支持，还有工程升级上的现实意义。

### 坑 3：JDK 9 说不清主线

很多人知道 `List.of`，但忘了 `JDK 9` 真正最值钱的是模块化。

所以如果被问：

- `JDK 9` 你知道什么
- 你觉得 `JDK 9` 最值得记什么

最稳的答法仍然是先保住：

> 模块化系统

然后再顺手带一句集合工厂方法、接口私有方法、`JShell`。

### 坑 4：把语言特性和项目升级混成一团

很多人会这样答：

> `JDK 17` 有 `record`、密封类、文本块，所以项目就应该升级

这就容易显得太学生气。

真实项目升级不会因为“语法更炫”就做。

应该改成：

> 语言新特性是加分项，但项目升级更看重长期支持、框架基线、依赖兼容、测试成本和迁移收益。

### 坑 5：把“为什么不升级”答成“因为我们菜”

最差的一类答法：

- 我们项目老，所以没升
- 公司不让升
- 我也不知道，反正一直就是 8

这种说法最大的问题是没有工程判断。

更好的说法应该是：

> 老项目不升级通常是综合考虑稳定性、历史依赖、升级收益和迁移成本，不是简单的“不会升级”。

## 3. 面试官可能怎么追问你

### 3.1 追问：你们为什么一直用 JDK 8

如果你前面只说：

> 因为项目老

面试官就会继续问：

- 老在哪里
- 升级风险是什么
- 是框架问题还是依赖问题

所以你最好一次就带上：

- 稳定性
- 依赖链
- 兼容性
- 回归测试成本

### 3.2 追问：那为什么别人又升级到 JDK 17

如果你前面只说：

> 因为它新

也不行。

更像面试回答的是：

> 因为它是 `LTS`，而且和现在不少新框架、新项目的技术基线更匹配，升级收益在某些团队里已经比较明确了。

### 3.3 追问：你到底用过没有

这是版本题里非常常见的真伪判断。

如果你没真用过，不要硬装。

更稳的回答：

> 线上主项目还是 `JDK 8`，但我在学习或 demo 里关注过 `JDK 17`，我更看重它在工程升级和框架基线里的意义。

## 4. 几组常见错误答法和修正版

### 错误答法 1

> `JDK 8` 新特性很多，最主要是 `Lambda`。

### 修正版

> `JDK 8` 不是单点更新，它带来的变化更像编码风格升级。常见的包括 `Lambda`、`Stream`、`Optional`、接口默认方法和新的时间 API。

### 错误答法 2

> `JDK 17` 就是更高级、更快。

### 修正版

> `JDK 17` 最值钱的不是“更高级”这种泛化说法，而是它作为 `LTS` 和现代技术栈基线的现实意义。

### 错误答法 3

> `JDK 9` 我记不太清了，好像有些 API。

### 修正版

> `JDK 9` 我会先记住模块化系统，这是它最核心的主线，其他再补集合工厂方法、接口私有方法和 `JShell`。

## 5. 最后记一条总原则

JDK 版本题最容易翻车的原因，不是知识点不够，而是主线不清。

你只要把这三句抓住，基本就稳很多：

- `JDK 8`：函数式和声明式写法进入 Java
- `JDK 9`：模块化
- `JDK 17`：现代工程基线 + `LTS`

## 6. 一句话收尾

> 版本题最怕的不是不会，而是答得像背零散更新日志，听不出你有主线、有项目判断。
