# JDK 8 中 Stream、Optional、Lambda 的常见误用

## 1. 这份文档怎么用

这一篇不是重复讲 `JDK 8` 新特性定义。

它专门补一个面试里很容易被问深的点：

- 你会不会用 `Stream`
- 你有没有滥用 `Optional`
- `Lambda` 是不是真的理解了

所以这篇更偏“易错点”和“选型意识”。

## 2. 先记住一个总原则

`JDK 8` 的这些能力很有价值，但它们不是“只要能用就全用”。

面试里更稳的表达是：

> 我会根据可读性、场景复杂度和团队习惯来选，不会机械地把所有代码都改成 `Stream`、`Optional` 或 `Lambda` 风格。

## 3. Stream 常见误用

### 3.1 误用一：把简单循环硬改成很长的 Stream 链

比如原本一个简单的遍历就能讲清楚，你非要写成特别长的链式调用。

问题在于：

- 可读性不一定更好
- 调试不一定更方便
- 新人不一定更容易接

### 面试里怎么说

> `Stream` 更适合过滤、映射、分组、聚合这类声明式处理，如果逻辑本来就很简单，或者中间步骤很多、调试成本高，我不一定会强行用 `Stream`。

### 3.2 误用二：把有副作用的逻辑塞进 Stream

比如在 `map`、`forEach` 里做复杂状态修改、外部变量写入、甚至数据库操作。

问题在于：

- 语义会变乱
- 容易让链式处理不纯
- 后面别人接手很难看

### 面试里怎么说

> 我会尽量让 `Stream` 保持在数据转换和聚合层面，而不是在里面塞太多带副作用的业务逻辑。

### 3.3 误用三：以为 Stream 一定更高性能

这是一个常见误区。

更稳的表达：

> `Stream` 的价值更多在于表达力和可读性，不是天然就比普通循环更快。性能还是得结合具体场景判断。

## 4. Optional 常见误用

### 4.1 误用一：到处都包 Optional

比如：

- 实体类字段用 `Optional`
- DTO 字段用 `Optional`
- 很多内部变量也都包一层

这样往往会让代码变得很别扭。

### 面试里怎么说

> `Optional` 更适合作为一种显式表达“可能为空”的边界工具，而不是把所有地方都包起来。滥用反而会让代码更重。

### 4.2 误用二：把 Optional 当成防空指针万能药

很多人觉得只要用了 `Optional` 就不会空指针了。

这显然不对。

更合理的理解是：

- 它改善的是表达方式
- 不会自动消灭所有空值问题
- 关键还是你怎么设计边界和数据流

### 4.3 误用三：写成更绕的判空

有些代码用了 `Optional` 之后，比 `if (x == null)` 还难读。

这时候就本末倒置了。

### 面试里怎么说

> 如果用了 `Optional` 之后代码更绕，那我宁愿回到直接、清晰的判空写法。工具是为可读性服务的，不是为“显得高级”服务的。

## 5. Lambda 常见误用

### 5.1 误用一：以为 Lambda 只是“匿名内部类简写”

这句话不算错，但太浅。

更好的理解是：

- 它代表行为可以作为参数传递
- 更适合函数式接口场景
- 和 `Stream` 等 API 结合才真正体现价值

### 面试里怎么说

> `Lambda` 不只是匿名内部类写短了，而是让 Java 在行为表达上更自然，尤其适合和函数式接口、`Stream` 这种 API 一起用。

### 5.2 误用二：复杂业务逻辑全塞进 Lambda

如果 Lambda 里已经塞了很多判断、循环、异常处理，那通常就不优雅了。

更好的做法往往是：

- 提取方法
- 保持 Lambda 简短
- 让表达重点清楚

### 5.3 误用三：为了用而用

这是最常见的问题。

比如本来普通方法调用就很好理解，非要改成一堆函数式风格。

### 面试里怎么说

> 我不会为了用 `Lambda` 而用 `Lambda`。如果它能明显减少样板代码、让行为表达更清晰，我会用；如果只是形式更花，我不会硬套。

## 6. 一组面试里很实用的总结句

### 6.1 讲 Stream

> `Stream` 更适合做声明式的数据处理，但不是所有循环都值得改成 `Stream`。

### 6.2 讲 Optional

> `Optional` 是表达“可能为空”的工具，不是到处都包一层的统一规范。

### 6.3 讲 Lambda

> `Lambda` 的价值不只是缩短代码，而是让行为表达和函数式接口配合得更自然。

## 7. 如果面试官问“你在项目里怎么用这些特性”

比较稳的回答可以是：

> 我在项目里会比较克制地用这些能力。像集合过滤、映射、聚合，我会考虑 `Stream`；像需要显式表达返回值可能为空的边界场景，我会考虑 `Optional`；像一些回调、排序、函数式接口实现，我会考虑 `Lambda`。但我不会为了“语法更现代”就到处套这些写法，核心还是可读性和场景适配。

## 8. 一句话收尾

> 面试里真正加分的，不是你知道 `Stream`、`Optional`、`Lambda`，而是你知道什么时候该用、什么时候别滥用。
