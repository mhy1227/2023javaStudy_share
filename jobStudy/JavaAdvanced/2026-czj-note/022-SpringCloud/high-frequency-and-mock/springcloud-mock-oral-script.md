# Spring Cloud 模拟追问脚本

## 1. 使用方式

这份文档用于模拟一整条 Spring Cloud 连续追问链。

建议回答结构：

1. 先说微服务带来的问题
2. 再说对应组件
3. 再说项目场景
4. 最后补边界和取舍

## 2. 第一轮：整体认知

### 面试官：你了解 Spring Cloud 吗？

候选人回答重点：

- 微服务治理方案
- 不是单个组件
- 解决拆分后的基础设施问题

可答：

> Spring Cloud 是一套微服务治理组件组合，用来解决微服务拆分后的注册发现、服务调用、负载均衡、网关、熔断限流降级、配置中心和分布式事务等问题。它的核心不是某一个组件，而是把微服务拆分后产生的问题系统化解决。

### 面试官追问：为什么不继续用单体？

候选人回答重点：

- 单体简单但后期耦合重
- 部署扩容不灵活
- 微服务提升拆分、协作和扩展能力
- 同时带来复杂度

可答：

> 单体项目早期开发和部署简单，但业务复杂后会出现模块耦合、整体发布、整体扩容和团队协作困难。微服务按业务边界拆分后，可以独立开发、部署和扩容。但它也带来服务注册、远程调用、网关、限流熔断、分布式事务等新问题，所以需要 Spring Cloud 这类治理体系。

## 3. 第二轮：注册中心和配置中心

### 面试官：为什么需要 Nacos？

候选人回答重点：

- 服务地址动态变化
- 注册发现
- 配置集中管理

可答：

> 微服务实例可能会扩容、缩容、重启，如果调用方写死 IP 和端口，维护成本很高。Nacos 可以作为注册中心，让服务启动后自动注册，调用方根据服务名获取实例列表。同时它也可以作为配置中心，集中管理配置并支持动态刷新。

### 面试官追问：注册中心和负载均衡有什么关系？

候选人回答重点：

- 注册中心提供实例列表
- LoadBalancer 选择实例

可答：

> 注册中心负责告诉调用方目标服务有哪些可用实例，LoadBalancer 负责从这些实例里选一个。简单说，Nacos 提供候选列表，LoadBalancer 决定这次请求打哪台机器。

## 4. 第三轮：服务调用

### 面试官：OpenFeign 是什么？

候选人回答重点：

- 声明式 HTTP 调用
- 接口方式调用远程服务
- 本质仍是远程调用

可答：

> OpenFeign 是声明式 HTTP 调用组件，它通过接口和注解描述远程调用，底层生成代理对象，把接口方法调用转换成 HTTP 请求。它让服务间调用写起来像本地接口，但本质仍然是远程网络调用，所以要考虑超时、失败和降级。

### 面试官追问：Feign 调用失败怎么办？

候选人回答重点：

- 超时
- 谨慎重试
- 熔断降级
- Sentinel

可答：

> Feign 调用本质是远程调用，所以首先要配置合理超时，避免线程一直阻塞。重试要谨慎，尤其是写操作不能随便重试。对于下游异常，可以结合 Sentinel 做熔断降级，返回业务可接受的兜底结果，避免故障扩散。

## 5. 第四轮：网关

### 面试官：为什么需要 Gateway？

候选人回答重点：

- 统一入口
- 路由转发
- 鉴权限流日志

可答：

> 微服务拆分后，如果前端直接访问各个服务，会暴露服务地址，也会让前端调用变复杂。Gateway 提供统一入口，根据路由规则转发到具体服务，同时可以统一做鉴权、限流、跨域、日志和灰度等入口治理能力。

### 面试官追问：Gateway 和 Nginx 有什么区别？

候选人回答重点：

- Nginx 更偏基础入口
- Gateway 更偏微服务治理
- 可以分层配合

可答：

> Nginx 更偏基础设施层，适合做反向代理、静态资源、HTTPS 和基础负载均衡；Gateway 更偏微服务网关，适合和注册中心、鉴权、限流、灰度、链路追踪等能力结合。两者不是简单替代关系，实际项目里可以请求先到 Nginx，再进入 Gateway。

## 6. 第五轮：稳定性保护

### 面试官：什么是服务雪崩？

候选人回答重点：

- 下游变慢
- 上游阻塞
- 故障扩散

可答：

> 服务雪崩是指某个下游服务变慢或故障后，上游线程一直等待，导致请求堆积和线程资源耗尽，最终故障沿调用链扩散，影响更多服务。

### 面试官：Sentinel 解决什么问题？

候选人回答重点：

- 限流
- 熔断
- 降级
- 系统保护

可答：

> Sentinel 是流量治理和服务保护组件，主要用于限流、熔断、降级和系统保护。限流挡住过高流量，熔断在下游不稳定时暂时停止调用，降级在失败后返回兜底结果，从而避免故障继续扩散。

## 7. 第六轮：分布式事务

### 面试官：为什么微服务里事务更难？

候选人回答重点：

- 数据分散
- 本地事务不够
- 网络调用不可靠

可答：

> 单体应用里很多操作可以放在一个本地事务里，但微服务拆分后，订单、库存、账户可能属于不同服务和数据库，本地事务管不到所有参与者。再加上网络调用不可靠，所以跨服务写数据的一致性会变复杂。

### 面试官：Seata 和 MQ 最终一致怎么选？

候选人回答重点：

- Seata 强协调
- MQ 最终一致轻量
- 看业务一致性要求

可答：

> Seata 更偏分布式事务协调，适合一致性要求较高的同步链路。MQ 最终一致更适合允许短时间不一致的异步场景，比如日志、通知、统计。选哪个要看业务能不能接受最终一致，而不是所有场景都上 Seata。

## 8. 最后一段总结

如果最后让你总结 Spring Cloud，可以这样说：

> Spring Cloud 的核心不是背组件名，而是解决微服务拆分后的治理问题。Nacos 解决服务发现和配置管理，Feign 和 LoadBalancer 解决服务调用和实例选择，Gateway 解决统一入口，Sentinel 解决限流熔断降级，Seata 或 MQ 解决跨服务一致性。面试回答时要从问题出发，再讲组件和取舍。
