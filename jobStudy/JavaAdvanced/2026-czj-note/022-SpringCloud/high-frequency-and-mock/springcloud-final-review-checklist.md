# Spring Cloud 最终复盘清单

## 1. 使用方式

这份文档适合面试前最后复盘。

不用逐字背，重点检查：

- 能不能讲清楚为什么要微服务
- 能不能说清楚每个组件解决什么问题
- 能不能把组件串成完整链路
- 能不能讲出边界和取舍

## 2. 总主线

必须记住：

> Spring Cloud 不是背组件名，而是回答“微服务拆分后新增了哪些问题，以及用哪些组件解决”。

最核心问题链：

1. 为什么从单体演进到微服务
2. 服务地址怎么动态发现
3. 服务之间怎么调用
4. 多个实例怎么负载均衡
5. 外部请求怎么统一入口
6. 下游异常怎么防止雪崩
7. 跨服务写数据怎么保持一致

## 3. 组件职责速记

`Nacos`：

- 服务注册与发现
- 配置中心
- 提供服务实例列表

`OpenFeign`：

- 声明式 HTTP 调用
- 让远程调用写起来像接口调用
- 本质还是远程网络调用

`LoadBalancer`：

- 客户端负载均衡
- 从多个实例中选择一个

`Gateway`：

- 统一入口
- 路由转发
- 鉴权、限流、跨域、日志、灰度

`Sentinel`：

- 流量治理
- 限流、熔断、降级
- 防止服务雪崩

`Seata`：

- 分布式事务
- 协调多个本地事务
- 解决跨服务写数据一致性

## 4. 高频一句话

微服务：

> 用架构复杂度换取服务独立开发、独立部署、独立扩容能力。

注册中心：

> 服务实例动态变化，不能写死地址，所以需要注册发现。

Feign：

> 写法像本地接口，本质还是远程 HTTP 调用。

LoadBalancer：

> Nacos 给实例列表，LoadBalancer 选本次请求打哪台。

Gateway：

> Gateway 做统一入口，不只是转发，还承接入口治理。

Sentinel：

> 限流挡流量，熔断断依赖，降级给兜底。

Seata：

> Seata 不是替代本地事务，而是协调多个本地事务。

## 5. 必会对比题

`Nacos` 和 `Eureka`：

- 都能做注册中心
- Nacos 还常做配置中心
- 面试中可强调 Nacos 一体化能力更强

`OpenFeign` 和 `Dubbo`：

- Feign 常走 HTTP REST
- Dubbo 更偏 RPC 框架
- 两者都是服务调用方案，但体系和通信模型不同

`Gateway` 和 `Nginx`：

- Nginx 更偏基础入口和反向代理
- Gateway 更偏微服务治理
- 两者可以分层配合

`Sentinel` 和 Hystrix：

- 都解决熔断降级
- Sentinel 更强调流量治理和规则能力
- 国内 Spring Cloud Alibaba 体系更常见 Sentinel

`Seata` 和 MQ 最终一致：

- Seata 更偏同步事务协调
- MQ 更偏异步最终一致
- 选型看业务能否接受短时间不一致

## 6. 项目回答检查

回答项目时至少带上：

1. 为什么拆微服务
2. Nacos 怎么发现服务
3. Feign 怎么做服务调用
4. Gateway 怎么做入口鉴权和路由
5. Sentinel 怎么防止下游异常扩散
6. 跨服务写数据时为什么不一定都用 Seata

不要只说：

> 我用了 Nacos、Feign、Gateway。

要说：

> 我用这些组件分别解决服务发现、服务调用、入口治理、稳定性保护和一致性问题。

## 7. 容易翻车点

不要说：

- Feign 是本地调用
- Gateway 可以放所有业务逻辑
- Sentinel 只是限流
- 分布式事务所有场景都要 Seata
- 重试越多越可靠
- 降级可以随便返回成功

应该说：

- Feign 只是写法像本地调用
- Gateway 做入口通用治理，业务逻辑放业务服务
- Sentinel 是流量治理和服务保护
- Seata 要看一致性要求和成本
- 重试要限制次数，并且只适合幂等场景
- 降级结果必须业务可接受

## 8. 最后 10 分钟背这段

可以这样背：

`Spring Cloud` 本质是一套微服务治理组件。微服务拆分后，服务地址动态变化，所以需要 `Nacos` 做注册发现；服务之间需要远程调用，所以用 `OpenFeign`；一个服务有多个实例，所以用 `LoadBalancer` 选择实例；外部请求需要统一入口，所以用 `Gateway` 做路由、鉴权、限流、跨域；下游异常可能引发雪崩，所以用 `Sentinel` 做限流、熔断、降级；跨服务写数据会有一致性问题，所以根据业务要求选择 `Seata` 或 MQ 最终一致。回答 Spring Cloud 时，关键不是背组件，而是讲清楚问题、方案和取舍。

## 9. 一句话总结

Spring Cloud 最终复盘只抓一条线：微服务拆分带来问题，组件负责治理问题，项目里要讲清楚取舍。
