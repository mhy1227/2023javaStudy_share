# Spring Cloud 微服务主线介绍

## 1. 这一份怎么用

这一份不是题库问答，而是 `Spring Cloud` 这一章的主线开场。

它要解决的不是“某个组件怎么记”，而是先把下面这条线讲清楚：

1. 为什么单体系统会逐步走向微服务
2. 微服务拆分后会新增哪些基础设施问题
3. `Spring Cloud` 这套体系分别在补什么
4. 面试里怎么先把主线讲顺，再去接高频追问

如果这条主线没立住，后面背 `Nacos`、`Gateway`、`OpenFeign`、`Sentinel`、`Seata` 都会显得很散。

## 2. 为什么会引入微服务

最早很多 Java 后端项目都是单体架构。

单体架构的好处很直接：

- 开发简单
- 部署简单
- 调试简单
- 本地跑起来也快

但它的问题也很快会暴露出来。

### 2.1 单体架构常见问题

1. 代码越堆越大，模块之间耦合越来越重
2. 一个小改动可能牵动整个项目重新构建和部署
3. 不同模块的访问压力不一样，但扩容只能整体扩
4. 团队协作边界不清晰，大家容易在同一个大工程里互相影响
5. 一旦某个模块出问题，可能拖累整个应用

所以当业务规模、团队规模、调用复杂度都上来以后，单体架构的维护成本就会持续升高。

### 2.2 微服务想解决什么

微服务不是为了“更高级”，而是为了把大系统拆成多个职责更单一的服务。

它想达到的目标一般是：

- 服务可以独立开发
- 服务可以独立部署
- 服务可以独立扩容
- 服务可以按照业务边界拆分
- 团队可以按服务分工

也就是说，微服务的核心不是拆，而是让系统在更大规模下还能继续维护和演进。

## 3. 微服务拆开后会多出什么问题

很多人只看到微服务“拆开更清晰”，但拆完以后新的问题也马上来了。

### 3.1 服务地址怎么找

单体系统里，方法调用就是方法调用。

但微服务拆开后，一个服务要调用另一个服务，就先得知道对方在哪。

如果把地址写死：

- 服务扩容后不好维护
- 实例上下线时要频繁改配置
- 环境切换也很麻烦

所以就需要注册中心来做服务发现。

### 3.2 服务之间怎么调用

拆成多个服务后，原来一个进程内的方法调用，变成了网络调用。

这会带来：

- 序列化和反序列化
- 网络延迟
- 调用失败重试
- 超时控制
- 调用链排查

所以需要统一的服务调用方式，而不是每个服务自己随便拼 HTTP 请求。

### 3.3 配置怎么统一管理

单体里配置写在一个地方还算好管理。

微服务一多，每个服务都可能有自己的配置：

- 数据库地址
- Redis 地址
- 第三方接口地址
- 开关配置
- 环境差异配置

如果全散在各个服务里，后期维护会很痛苦。

### 3.4 入口流量怎么治理

服务拆分后，外部请求不应该直接散到每个服务上。

因为你通常还需要：

- 统一鉴权
- 统一限流
- 统一路由
- 统一日志
- 统一灰度

这就需要网关。

### 3.5 某个服务挂了怎么办

微服务调用链一长，就很容易出现一个下游慢，带着上游也变慢，最后整个链路都被拖垮的情况。

这时候就需要：

- 熔断
- 限流
- 降级
- 超时控制

### 3.6 跨服务改数据怎么办

单体里一个事务就能兜住的事情，拆成多个服务后就不那么简单了。

比如订单服务、库存服务、支付服务可能都要一起参与。

这时就会碰到分布式事务问题。

## 4. Spring Cloud 在补什么

`Spring Cloud` 可以理解成一套围绕微服务常见问题的基础设施组合。

它不是单个组件，而是一组“把微服务跑起来、管起来、稳住”的能力集合。

### 4.1 注册中心和服务发现

解决的问题是：

- 服务地址不能写死
- 服务实例要动态上下线
- 调用方要能找到可用实例

常见组件：

- `Nacos`
- `Eureka`
- `Zookeeper`

### 4.2 服务调用

解决的问题是：

- 服务之间怎么发请求
- 怎么做负载均衡
- 怎么减少手写模板代码

常见组件：

- `OpenFeign`
- `LoadBalancer`

### 4.3 配置中心

解决的问题是：

- 配置分散
- 环境切换麻烦
- 配置刷新不方便

常见组件：

- `Nacos Config`

### 4.4 网关

解决的问题是：

- 统一入口
- 统一路由
- 统一鉴权
- 统一限流
- 统一灰度

常见组件：

- `Spring Cloud Gateway`

### 4.5 熔断、限流、降级

解决的问题是：

- 防止某个服务拖垮整个链路
- 控制系统在高压下的行为
- 保留核心可用能力

常见组件：

- `Sentinel`

### 4.6 分布式事务

解决的问题是：

- 多服务协作改数据时的一致性问题
- 让跨服务事务有一种可实现的协调方式

常见组件：

- `Seata`

## 5. 这些组件之间是什么关系

可以先按“问题 -> 组件”记。

1. 服务地址怎么找 -> 注册中心
2. 服务之间怎么调用 -> `OpenFeign`
3. 多实例怎么分流 -> 负载均衡
4. 配置怎么集中管理 -> 配置中心
5. 外部请求怎么统一接入 -> 网关
6. 某个下游慢了怎么办 -> 熔断、限流、降级
7. 多服务改数据怎么办 -> 分布式事务

这样去理解，比单纯背组件名更稳。

## 6. 面试里怎么先介绍 Spring Cloud

可以先这样说：

`Spring Cloud` 不是单个框架，而是一套围绕微服务常见问题的解决方案集合。微服务拆分后，会带来服务发现、服务调用、配置管理、统一网关、熔断限流和分布式事务这些问题，`Spring Cloud` 就是把这些基础能力系统化地补齐，让微服务能够更稳定地落地。

如果想再往下补一句，可以接：

它本质上不是在替代业务代码，而是在给分布式系统提供一层标准化的基础设施。

## 7. 当前这一章先记住什么

先记住下面这几个结论就够开始往下扩了：

1. 微服务不是为了复杂化，而是为了让大系统继续可维护
2. 拆分后最先出现的是服务发现、调用、配置、网关、熔断和事务问题
3. `Spring Cloud` 的价值是补齐这些基础设施能力
4. 后面所有高频题，基本都可以从“微服务带来了什么问题”这个角度往下展开

## 8. 一句话总结

`Spring Cloud` 的主线不是组件列表，而是：单体为什么不够用了，拆成微服务以后多出哪些问题，以及这套体系分别怎么把这些问题补上。
