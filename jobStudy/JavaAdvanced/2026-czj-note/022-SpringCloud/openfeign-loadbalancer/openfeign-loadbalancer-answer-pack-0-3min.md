# OpenFeign 与 LoadBalancer 0-3 分钟回答包

## 1. 请你介绍一下 OpenFeign

`OpenFeign` 是 Spring Cloud 里常用的声明式 HTTP 调用组件，主要解决微服务拆分之后服务之间调用代码重复、可读性差的问题。

以前如果用比较底层的方式调用远程服务，可能要自己拼 URL、组装参数、处理请求头、解析返回值，代码会比较分散。OpenFeign 的做法是定义一个接口，在接口上通过 `@FeignClient` 指定目标服务名，再用 `@GetMapping`、`@PostMapping` 这类注解描述请求路径和参数。业务代码调用这个接口时，底层其实会通过动态代理把接口调用转换成 HTTP 请求。

但是要注意，Feign 只是写法上像本地调用，本质上还是远程网络调用。所以项目里不能把它当成本地方法一样使用，还要考虑超时、异常处理、重试、降级、请求头透传和日志排查等问题。

## 2. OpenFeign 的完整调用流程是什么？

可以从服务注册和服务调用两部分讲。

首先，服务启动后会把自己的服务名、IP、端口注册到注册中心，比如 Nacos。调用方定义 Feign 接口，通过 `@FeignClient(name = "user-service")` 指定要调用哪个服务。

当业务代码调用 Feign 接口方法时，Feign 会通过代理对象解析方法上的请求路径、请求方式和参数。然后它会根据服务名去获取目标服务的实例列表。这个实例列表通常来自注册中心。拿到多个实例之后，`LoadBalancer` 会按照轮询等策略选择一个具体实例。最后 Feign 通过底层 HTTP Client 发起请求，并把返回结果反序列化成 Java 对象返回给业务代码。

所以完整链路可以概括为：Feign 负责声明式调用，注册中心负责提供实例列表，LoadBalancer 负责选择实例，HTTP Client 负责真正发请求。

## 3. LoadBalancer 是什么？为什么需要它？

`LoadBalancer` 是 Spring Cloud 里的客户端负载均衡组件。它解决的是多实例部署时，请求该打到哪个实例的问题。

微服务一般不会只部署一个实例，因为单实例有单点风险，也不方便承载高并发流量。比如 `user-service` 部署了三台机器，调用方不能固定访问其中一台，否则这台机器压力会很大，其它实例却没有被利用起来。

客户端负载均衡的做法是：调用方先从注册中心获取目标服务的可用实例列表，然后在本地通过负载均衡策略选择一个实例，再直接请求这个实例。Spring Cloud LoadBalancer 就是做这个选择动作的。

它和 Nacos 的关系也比较清楚：Nacos 提供实例列表，LoadBalancer 负责从列表里选一个。两者是配合关系，不是替代关系。

## 4. 客户端负载均衡和服务端负载均衡有什么区别？

客户端负载均衡是调用方自己选择实例。比如服务 A 调服务 B 时，服务 A 从注册中心拿到服务 B 的实例列表，然后在本地通过 LoadBalancer 选择一台机器请求。

服务端负载均衡是请求先到一个统一入口，比如 Nginx、SLB 或网关，再由这个中间层把请求转发到后端实例。也就是说，客户端并不知道最终是哪台机器处理请求。

两者的区别主要是实例选择发生的位置不同。客户端负载均衡发生在调用方本地，适合微服务内部调用；服务端负载均衡发生在代理层，适合统一入口流量，比如外部请求进入系统时的流量分发。

## 5. Feign 调用为什么要配置超时？

Feign 本质是远程 HTTP 调用，只要涉及网络和下游服务，就一定可能出现响应慢、网络抖动、服务不可用等问题。

如果没有超时控制，调用方线程可能一直等待下游返回。请求量一大，线程会逐渐被占满，导致上游接口也变慢，甚至引发级联故障。

所以超时的作用不是保证请求成功，而是控制失败等待时间，防止资源被无限占用。一般会关注连接超时和读取超时：连接超时表示建立连接的等待时间，读取超时表示连接建立后等待响应数据的时间。

## 6. Feign 重试能不能随便开？

不能随便开。

重试可以处理短暂网络抖动，但如果下游本来已经很慢或者压力很大，上游再不断重试，会把请求量进一步放大，导致下游更难恢复。

另外，写操作和非幂等接口尤其不能随便重试，比如下单、扣库存、支付。如果第一次请求其实已经成功了，只是响应超时，上游再次重试就可能造成重复下单、重复扣减或者重复支付。

所以重试要谨慎，一般更适合幂等读请求或偶发网络失败，而且要限制次数和时间，并结合超时、熔断、限流一起使用。

## 7. Feign 调用失败后怎么做降级？

降级的思路是：当下游服务不可用或响应太慢时，不再强依赖下游结果，而是返回一个业务上可以接受的兜底结果。

比如商品详情页的评论服务失败，可以先不展示评论；推荐服务失败，可以返回默认推荐列表；用户积分服务失败，可以显示默认值或提示稍后刷新。

降级不是让所有功能都正常，而是保证核心链路不要被非核心依赖拖垮。实际项目里通常会结合 Sentinel 这类组件，根据异常比例、慢调用比例或并发数做熔断和降级。当下游明显异常时，暂时不继续打请求，而是直接走兜底逻辑。

## 8. 项目里怎么介绍 OpenFeign 和 LoadBalancer？

可以这样说：

在项目里，服务之间的同步调用主要通过 OpenFeign 实现。我们会通过 Feign 接口声明要调用的下游服务和接口路径，业务代码直接注入接口调用。服务实例由 Nacos 统一注册和发现，Feign 发起调用时会根据服务名获取实例列表，再由 Spring Cloud LoadBalancer 在多个实例中选择一个目标实例。

同时，因为 Feign 本质是远程调用，我们不会只关注调用是否写得方便，还会配置合理的连接超时和读取超时，对非幂等写接口谨慎开启重试，对核心链路结合 Sentinel 做熔断降级，避免下游服务异常时拖垮上游服务。

## 9. 一段完整口述版

`OpenFeign` 和 `LoadBalancer` 一般可以放在一起讲，因为它们共同解决微服务之间怎么调用、以及多个实例时调用哪个实例的问题。OpenFeign 是声明式 HTTP 调用组件，通过接口和注解描述远程调用，底层生成代理对象，把接口调用转换成 HTTP 请求。LoadBalancer 是客户端负载均衡组件，负责从注册中心提供的多个服务实例中选择一个目标实例。

完整调用流程是：目标服务先注册到 Nacos，调用方通过 Feign 接口发起调用，Feign 根据服务名获取实例列表，再由 LoadBalancer 选择一个实例，最后发起 HTTP 请求并把响应转换成 Java 对象。虽然 Feign 写起来像本地接口调用，但它本质是远程调用，所以项目里必须考虑超时、重试、熔断和降级。超时是为了防止线程长时间阻塞，重试只能谨慎用于幂等场景，降级则用于下游异常时保护核心链路。
