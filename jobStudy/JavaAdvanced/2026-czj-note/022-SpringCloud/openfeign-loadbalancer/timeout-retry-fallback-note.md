# 超时、重试与降级专题

## 1. 为什么这块很重要

`OpenFeign` 面试里经常会追问稳定性。

因为服务调用不是本地方法调用，而是远程调用。

远程调用一定会遇到：

- 网络延迟
- 下游服务变慢
- 下游服务宕机
- 部分实例异常
- 请求堆积
- 线程资源被占满

所以面试里不能只讲“Feign 怎么调用”，还要讲“调用失败怎么办”。

## 2. 超时是什么

超时就是给远程调用设置一个最大等待时间。

如果超过这个时间还没有返回，就不继续等待，而是认为本次调用失败。

常见超时包括：

- 连接超时
- 读取超时

连接超时关注的是：

> 能不能建立连接。

读取超时关注的是：

> 连接建立后，对方迟迟不返回数据怎么办。

## 3. 为什么必须设置超时

如果没有合理超时，调用方线程可能长时间阻塞。

后果包括：

- 请求越积越多
- Tomcat 线程被占满
- 上游接口也被拖慢
- 故障沿调用链扩散

所以超时不是为了让请求更快成功，而是为了控制失败等待时间，避免资源被无限占用。

## 4. 重试是什么

重试指一次调用失败后，再尝试调用一次或多次。

重试听起来很有用，但不能滥用。

适合重试的场景通常是：

- 短暂网络抖动
- 偶发连接失败
- 读请求
- 幂等接口

不适合随便重试的场景包括：

- 下单
- 扣库存
- 支付
- 写入类非幂等接口

因为重试可能导致重复写入、重复扣减、重复请求下游。

## 5. 为什么重试可能放大故障

假设下游服务已经很慢。

如果上游大量请求失败后继续重试，会导致：

1. 原请求没处理完
2. 新请求又进来
3. 重试请求继续打过来
4. 下游压力更大

这就是故障放大。

所以重试要满足几个条件：

- 次数有限
- 时间有限
- 只对幂等接口开启
- 和超时、熔断、限流配合

## 6. 降级是什么

降级指下游服务不可用或响应太慢时，不再强依赖下游结果，而是返回一个兜底结果。

比如：

- 商品详情页评论服务失败，先不展示评论
- 用户积分服务失败，先显示默认积分
- 推荐服务失败，返回默认推荐列表

降级的目标不是保证所有功能都正常，而是保证核心链路还能继续。

## 7. 熔断和降级的关系

熔断更偏保护机制。

当失败率或慢调用比例过高时，系统暂时不再继续请求下游，避免把下游打得更崩。

降级更偏兜底策略。

当调用失败或被熔断后，返回一个可接受的备用结果。

可以这样理解：

- 熔断：先别再打了
- 降级：失败后给个兜底

## 8. Feign 调用失败一般怎么处理

实际项目里可以从这些方面处理：

1. 配置合理超时时间
2. 对读请求或幂等接口谨慎开启重试
3. 对核心接口设置降级策略
4. 对异常进行统一包装
5. 打印必要的调用日志
6. 结合 Sentinel 等组件做限流、熔断、降级

## 9. 面试怎么说

可以这样说：

Feign 本质是远程 HTTP 调用，所以必须考虑超时、重试和降级。超时是为了避免调用方线程长时间阻塞，控制失败等待时间；重试可以处理短暂网络抖动，但不能滥用，尤其是写操作和非幂等接口，否则可能造成重复写入或故障放大；降级是在下游不可用时返回兜底结果，保证核心链路可用。实际项目里一般会结合 Sentinel 这类组件做限流、熔断和降级，避免单个下游故障拖垮整个调用链。

## 10. 一句话总结

服务调用稳定性的核心是：超时控制等待时间，重试处理短暂失败，降级保护核心链路。
