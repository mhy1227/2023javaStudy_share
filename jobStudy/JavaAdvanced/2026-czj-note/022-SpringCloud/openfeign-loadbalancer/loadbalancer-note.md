# LoadBalancer 专题

## 1. LoadBalancer 是什么

`LoadBalancer` 是 Spring Cloud 里负责客户端负载均衡的组件。

它解决的问题是：

> 一个服务有多个实例时，调用方这次请求到底应该打到哪个实例。

比如 `user-service` 有三台实例：

- `192.168.1.10:8080`
- `192.168.1.11:8080`
- `192.168.1.12:8080`

调用方不应该固定访问其中一台，而是应该按一定策略在多个实例中选择。

## 2. 为什么需要负载均衡

微服务通常会多实例部署，原因包括：

1. 提高并发处理能力
2. 避免单点故障
3. 支持横向扩容
4. 方便滚动发布

如果没有负载均衡，所有请求都打到一个实例上，就会出现：

- 某个实例压力过高
- 其它实例空闲
- 单点故障风险
- 扩容后流量没有真正分散

负载均衡的目标不是“绝对平均”，而是让请求尽量合理地分布到可用实例上。

## 3. 客户端负载均衡是什么意思

客户端负载均衡指的是：

> 调用方在发请求之前，自己根据服务实例列表选择一个目标实例。

流程大致是：

1. 调用方要访问 `user-service`
2. 从注册中心拿到 `user-service` 的实例列表
3. `LoadBalancer` 根据策略选择一个实例
4. 调用方直接请求这个实例

所以选择动作发生在调用方本地。

## 4. 服务端负载均衡是什么意思

服务端负载均衡指的是：

> 请求先到一个统一入口，再由这个入口转发到后端实例。

常见例子：

- Nginx
- SLB
- 网关层

流程大致是：

1. 客户端请求统一入口
2. 统一入口维护后端服务实例
3. 统一入口选择一个后端实例
4. 请求被转发过去

所以选择动作发生在中间代理层。

## 5. 客户端和服务端负载均衡对比

| 对比点 | 客户端负载均衡 | 服务端负载均衡 |
|---|---|---|
| 选择实例的位置 | 调用方本地 | 中间代理层 |
| 是否依赖注册中心 | 通常依赖 | 不一定 |
| 调用链路 | 调用方直接访问实例 | 先访问代理，再转发 |
| 常见组件 | Spring Cloud LoadBalancer | Nginx、SLB、网关 |
| 优点 | 少一层转发，和注册中心结合紧密 | 对客户端透明，集中治理 |
| 缺点 | 客户端需要集成负载均衡逻辑 | 多一层转发，代理层也要高可用 |

面试里不用把它们说成谁一定更好。

更准确的说法是：

> 客户端负载均衡更适合微服务内部调用，服务端负载均衡更适合统一入口或外部流量接入。

## 6. 常见负载均衡策略

常见策略包括：

- 轮询
- 随机
- 权重
- 最少连接
- 根据实例健康状态选择
- 根据区域或标签选择

Spring Cloud LoadBalancer 默认常见的是轮询思路。

轮询不是最复杂的策略，但简单、稳定、可预测，适合作为基础默认策略。

## 7. LoadBalancer 和 Nacos 的关系

这两个组件经常一起被问。

`Nacos` 负责提供服务实例列表。

`LoadBalancer` 负责从实例列表中选择一个。

可以用一句话记：

> Nacos 告诉你有哪些机器，LoadBalancer 决定这次请求打哪台机器。

两者不是替代关系，而是配合关系。

## 8. LoadBalancer 和 OpenFeign 的关系

`OpenFeign` 负责把接口调用转换成远程 HTTP 请求。

`LoadBalancer` 负责选择具体服务实例。

一次常见调用链是：

1. 业务代码调用 Feign 接口
2. Feign 根据服务名发起请求
3. LoadBalancer 从服务实例列表里选择一个实例
4. Feign 发送 HTTP 请求

所以这两个经常放在一起讲。

## 9. 面试怎么说

可以这样说：

`LoadBalancer` 主要解决微服务多实例部署下，请求应该打到哪个实例的问题。它属于客户端负载均衡，也就是调用方在发请求前，先从注册中心获取服务实例列表，再按照轮询等策略选择一个实例进行调用。它通常和 `OpenFeign` 配合使用，Feign 负责声明式远程调用，LoadBalancer 负责实例选择。和 Nginx 这类服务端负载均衡相比，它的实例选择发生在客户端本地，更适合微服务内部调用。

## 10. 一句话总结

`LoadBalancer` 的核心作用，是在多个可用服务实例之间为本次调用选择一个目标实例。
