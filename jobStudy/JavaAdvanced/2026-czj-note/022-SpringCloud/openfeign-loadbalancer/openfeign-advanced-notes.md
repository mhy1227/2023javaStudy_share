# OpenFeign 进阶与常见坑

## 1. 这份文档补什么

## 2. OpenFeign 能不能透传请求头

可以。

常见场景包括：

- 登录态
- 用户身份标识
- 链路追踪 ID
- 灰度标识

一般会通过拦截器统一把上游请求头带到下游请求里。

面试里可以这样说：

> Feign 本质是远程调用，所以如果链路里需要保留身份信息或追踪信息，就可以通过请求拦截器统一透传请求头，避免每个接口手动处理。

注意点：

- 不要把不必要的头全量透传
- 不要透传敏感信息
- 不要把和业务无关的脏数据带到下游

## 3. OpenFeign 能不能做文件上传

可以，但要注意写法和限制。

文件上传本质上还是走 HTTP multipart/form-data。

项目里经常会遇到：

- 头像上传
- 附件上传
- 资料文件上传

这时 Feign 也可以封装成接口调用，但要保证：

- 请求类型正确
- 编码器配置正确
- 下游接口支持 multipart

面试里如果被问到，可以说：

> Feign 可以支持文件上传，但它本质还是 HTTP 请求封装，不是专门的文件传输协议。实际项目里要配合 multipart 配置，并注意文件大小、超时和网关限制。

## 4. fallback 和 fallbackFactory 有什么区别

这也是常见追问点。

`fallback` 更像是直接指定一个兜底实现。

`fallbackFactory` 更适合拿到失败原因后再做处理。

你可以简单记成：

- `fallback`：失败后直接走默认实现
- `fallbackFactory`：失败后可以拿到异常，再根据异常决定怎么兜底

面试里不用讲得太细，但要知道它们都是为了在调用失败时返回替代逻辑。

## 5. OpenFeign 的日志有什么用

Feign 日志主要用于排查：

- 调了哪个接口
- 传了什么参数
- 返回了什么结果
- 出了什么异常

常见使用场景：

- 联调阶段定位请求问题
- 排查参数不一致
- 排查下游返回异常

注意点：

- 生产环境不要无限制打详细日志
- 注意日志量和性能
- 注意敏感信息脱敏

## 6. 全局配置一般配什么

OpenFeign 的全局配置通常会关注这些：

- 连接超时
- 读取超时
- 日志级别
- 编码器和解码器
- 拦截器
- 重试策略

面试里可以这样说：

> OpenFeign 在项目里通常不会只用默认配置，而是会根据业务场景统一配置超时时间、日志级别和拦截器，保证调用行为可控。

## 7. OpenFeign 和自定义编码解码是什么关系

当接口参数或返回值比较复杂时，底层需要把 Java 对象转换成 HTTP 请求体，再把响应结果转换回来。

这个过程通常会涉及：

- 编码器
- 解码器
- 序列化工具

一般项目里不需要把实现细节讲得太深，但要知道：

> Feign 不只是“帮你发请求”，它还负责参数编码和结果解码。

## 8. OpenFeign 常见坑有哪些

常见坑主要有这些：

1. 把 Feign 当成本地方法调用，忽略超时和异常
2. 对非幂等接口随便开重试
3. 下游慢时没有降级兜底
4. 请求头没有统一透传
5. 文件上传没有配 multipart
6. 日志开太大影响性能
7. 只关注调用成功，不关注调用链稳定性

## 9. 面试怎么补充这块

如果面试官继续追问，你可以这样补：

`OpenFeign` 不只是一个“把 HTTP 调用写成接口”的工具，它在项目里通常还会配合请求头透传、统一超时配置、日志、编码解码和降级策略一起使用。尤其是远程调用场景里，不能只关注调用方便，还要关注失败处理和稳定性。比如文件上传时要注意 multipart 配置，写接口时要避免对非幂等请求随便重试，调用链上还要根据情况做兜底或者熔断，避免下游故障扩散。

## 10. 一句话总结

OpenFeign 的进阶重点不是“怎么写接口”，而是“怎么让远程调用在项目里稳定可控地跑起来”。
