# OpenFeign 与 LoadBalancer 基础

## 1. 这一份是干什么的

这一份是 `OpenFeign` 和 `LoadBalancer` 的基础入门。

它要先讲清楚一条线：

1. 微服务拆开以后，服务之间怎么调用
2. 调用时为什么还要考虑多个实例怎么选
3. `OpenFeign` 和 `LoadBalancer` 各自负责什么
4. 为什么这两个经常放在一起讲

## 2. 为什么服务调用和负载均衡要放在一起

微服务拆分后，服务之间不再是单体里的方法调用，而是远程调用。

远程调用会马上带来两个问题：

1. 调谁
2. 多个实例时选谁

所以这条线天然就是一组：

- 服务调用
- 负载均衡

`OpenFeign` 更偏“怎么发起调用”，`LoadBalancer` 更偏“如果有多个实例，这次请求选哪个”。

## 3. 什么是服务间调用

服务间调用可以先理解成：

> 一个微服务通过网络去请求另一个微服务提供的能力。

比如：

- 订单服务调用库存服务
- 用户服务调用积分服务
- 支付服务调用订单服务

这时已经不是单体里的本地方法调用了，而是要经过网络请求、序列化、超时控制等过程。

## 4. OpenFeign 是什么

`OpenFeign` 可以先理解成一个更方便的服务调用方式。

它的核心价值是：

- 让远程调用写起来更像本地接口调用
- 减少手写 HTTP 请求模板代码
- 让服务之间的调用更统一

### 4.1 它解决什么问题

如果没有这类工具，服务调用会比较麻烦：

1. 要手写请求地址
2. 要自己拼参数
3. 要自己处理返回值
4. 每个调用点代码都比较重复

`OpenFeign` 的作用，就是把这些调用细节封装起来。

## 5. LoadBalancer 是什么

`LoadBalancer` 可以先理解成客户端侧的负载均衡组件。

当一个服务有多个实例时，请求不应该永远打到同一个实例上。

`LoadBalancer` 要做的事情就是：

- 在多个可用实例之间做选择
- 让请求尽量分散
- 避免压力全堆到某一个实例

## 6. 注册中心和负载均衡是什么关系

这两个也很容易被一起问。

注册中心负责：

- 告诉你有哪些服务实例
- 哪些实例当前可用

负载均衡负责：

- 在这些可用实例里选一个

所以可以先记成：

1. 注册中心提供候选列表
2. 负载均衡从候选列表里做选择
3. 调用方真正发请求

## 7. 客户端负载均衡和服务端负载均衡有什么区别

先不用记太复杂，先抓住方向。

### 7.1 客户端负载均衡

由调用方自己在发请求前选择实例。

也就是说，请求还没发出去，客户端就先决定打到哪台机器。

### 7.2 服务端负载均衡

由中间代理或网关层去选择实例。

请求先到中间层，再由中间层分发到后面的服务实例。

### 7.3 先怎么理解

先记住一句：

- 客户端负载均衡是在调用方这边选
- 服务端负载均衡是在中间层选

## 8. 为什么调用链里还要考虑超时和失败

服务调用不是本地方法调用，网络上会有很多不确定性。

常见问题包括：

- 网络延迟
- 下游服务变慢
- 实例临时不可用
- 响应超时
- 调用失败重试

所以服务调用这一层不只是“能发请求”，还要考虑稳定性。

## 9. 为什么 `OpenFeign` 和 `LoadBalancer` 常常一起出现

因为 `OpenFeign` 负责发起调用，而 `LoadBalancer` 负责在多个实例里选一个。

这两个连起来，才能完成一条完整的服务调用链。

可以这样理解：

1. 调用方先通过 `OpenFeign` 发起远程调用
2. 调用时如果有多个实例，就由 `LoadBalancer` 选择实例
3. 最终真正完成请求发送

## 10. 面试里怎么介绍这一组组件

可以这样说：

`OpenFeign` 负责把微服务之间的远程调用封装得更像接口调用，减少手写 HTTP 请求的样板代码；`LoadBalancer` 负责在多个服务实例之间做请求分配。它们放在一起，就是在解决微服务里“怎么调用”和“调用哪个实例”的问题。

## 11. 当前这一份先记住什么

先记住下面几句：

1. 微服务拆开后，服务之间要通过网络调用
2. `OpenFeign` 负责把调用封装得更方便
3. `LoadBalancer` 负责多个实例的选择
4. 注册中心提供候选实例，负载均衡决定本次调用谁
5. 客户端负载均衡和服务端负载均衡要区分

## 12. 一句话总结

`OpenFeign` 和 `LoadBalancer` 这一条线，核心就是把“服务之间怎么调用”以及“多个服务实例怎么选”讲清楚。
