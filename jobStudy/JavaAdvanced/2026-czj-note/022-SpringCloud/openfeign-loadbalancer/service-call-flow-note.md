# 服务调用流程专题

## 1. 为什么要单独理解调用流程

面试官问 `OpenFeign` 和 `LoadBalancer` 时，经常不是单独问概念，而是问：

> 一个服务调用另一个服务时，完整流程是什么？

如果只回答“Feign 调接口，LoadBalancer 做负载均衡”，会显得比较浅。

更好的回答方式是把调用链路讲完整。

## 2. 一个典型场景

假设有两个服务：

- `order-service`
- `user-service`

订单服务创建订单时，需要查询用户信息。

那么调用关系是：

> `order-service` 通过 OpenFeign 调用 `user-service`。

## 3. 完整调用流程

可以按下面步骤讲：

1. `user-service` 启动后，把自己的服务名、IP、端口注册到注册中心
2. `order-service` 启动后，也注册到注册中心，同时会注入 Feign 客户端代理对象
3. 业务代码调用 `UserClient.getById(userId)`
4. Feign 代理对象解析接口上的请求路径、请求方式和参数
5. 根据 `@FeignClient(name = "user-service")` 找到目标服务名
6. 从注册中心获取 `user-service` 的可用实例列表
7. `LoadBalancer` 从实例列表中选择一个实例
8. Feign 组装并发送 HTTP 请求
9. `user-service` 处理请求并返回数据
10. Feign 将响应反序列化成 Java 对象，返回给业务代码

## 4. 这条链路里各组件负责什么

| 组件 | 作用 |
|---|---|
| 注册中心 | 维护服务实例列表 |
| OpenFeign | 把接口调用转换成 HTTP 请求 |
| LoadBalancer | 从多个实例中选择一个目标实例 |
| HTTP Client | 真正发起网络请求 |
| Jackson 等序列化组件 | 处理请求响应对象转换 |

## 5. 为什么不能把 Feign 当成本地方法

因为 Feign 只是写法像本地方法。

它本质上仍然经过：

- 网络传输
- 序列化和反序列化
- 超时控制
- 连接管理
- 下游服务处理
- 异常返回

所以它会有远程调用的一切问题：

- 下游不可用
- 网络抖动
- 响应慢
- 超时
- 重试
- 数据不一致

面试里可以强调：

> Feign 的便利性容易让人误以为它是普通方法调用，但项目里必须按远程调用来设计超时、异常和降级。

## 6. 如果注册中心里没有实例怎么办

如果目标服务没有可用实例，调用一般会失败。

可能原因包括：

- 目标服务没有启动
- 注册失败
- 实例被健康检查剔除
- 网络隔离
- 服务名写错

这种情况不能靠 Feign 自己“变出实例”，只能从注册中心、服务名、网络和实例健康状态排查。

## 7. 如果某个实例挂了怎么办

正常情况下，注册中心会通过心跳或健康检查感知实例状态。

实例不可用后，会从可用实例列表中剔除。

调用方下一次获取或刷新实例列表后，负载均衡就不会再选择它。

但这里有一个时间差：

> 服务实例刚挂掉到注册中心发现并剔除之间，仍然可能有请求打到这个异常实例。

所以服务调用层还需要超时、重试、熔断或降级兜底。

## 8. 面试怎么说完整流程

可以这样说：

在 Spring Cloud 里，一个服务调用另一个服务时，目标服务会先把自己的实例信息注册到注册中心。调用方通过 `OpenFeign` 定义远程调用接口，业务代码调用这个接口时，Feign 会生成代理并解析接口上的请求信息。然后它会根据服务名从注册中心获取目标服务的实例列表，再交给 `LoadBalancer` 选择一个具体实例，最终发起 HTTP 请求。响应返回后，Feign 会把结果反序列化成 Java 对象。虽然写法像本地调用，但本质还是远程网络调用，所以必须考虑超时、失败、重试和降级。

## 9. 一句话总结

服务调用流程可以概括为：服务注册，Feign 代理发起调用，注册中心提供实例列表，LoadBalancer 选择实例，HTTP 请求真正执行。
