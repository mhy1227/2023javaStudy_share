# Nacos 基础介绍

## 1. 这一份是干什么的

这一份是 `Nacos` 的基础入门。

先不讲太深的源码和一致性协议，只先讲清楚：

1. `Nacos` 是什么
2. 为什么微服务里需要它
3. 它作为注册中心在解决什么问题
4. 它作为配置中心在解决什么问题
5. 面试里怎么先说清它的定位

## 2. Nacos 是什么

`Nacos` 在 Spring Cloud 体系里最常见的定位有两个：

- 注册中心
- 配置中心

也就是说，它既可以帮服务之间做“地址发现”，也可以帮多个服务做“配置集中管理”。

如果先用一句话理解：

> Nacos 是一个同时支持服务注册发现和配置管理的基础设施组件。

## 3. 为什么需要注册中心

在单体项目里，模块之间通常是进程内调用。

但微服务拆开后，一个服务要调用另一个服务，就会变成远程调用。

这时候就会遇到一个很现实的问题：

调用方怎么知道被调用方在哪？

如果把服务地址写死，会有很多问题：

1. 服务扩容后地址会变
2. 服务下线后地址可能失效
3. 环境切换要改很多配置
4. 多实例情况下不知道该调哪个实例

所以需要注册中心。

## 4. Nacos 作为注册中心做什么

作为注册中心时，`Nacos` 主要做三件事：

1. 服务注册
2. 服务发现
3. 服务健康感知

### 4.1 服务注册

服务启动后，会把自己的信息注册到 `Nacos`。

这些信息通常包括：

- 服务名
- IP
- 端口
- 实例状态
- 分组或命名空间信息

这样注册中心就知道当前有哪些服务实例可用。

### 4.2 服务发现

调用方不再写死具体 IP 和端口，而是根据服务名去找可用实例。

比如订单服务要调用库存服务，不需要写死库存服务某台机器的地址，而是通过注册中心拿到可用实例列表。

### 4.3 服务健康感知

如果某个服务实例挂了，注册中心需要感知到，并让调用方尽量不要再调用这个坏实例。

这就是服务健康检查和实例上下线相关的能力。

## 5. 为什么需要配置中心

微服务一多，配置也会变多。

常见配置包括：

- 数据库连接
- Redis 连接
- 线程池参数
- 开关配置
- 第三方接口地址
- 不同环境的配置

如果这些配置都散在各个服务自己的配置文件里，后期会很难维护。

比如：

1. 改一个配置要改多个服务
2. 不同环境配置容易弄混
3. 配置变更后需要重新发版
4. 配置缺少集中管理和审计

所以需要配置中心。

## 6. Nacos 作为配置中心做什么

作为配置中心时，`Nacos` 主要解决的是配置集中管理。

它可以把配置从服务本地抽出来，统一放到配置中心。

这样做的好处是：

1. 配置集中管理
2. 不同环境更好隔离
3. 配置变更更容易同步
4. 一些开关类配置可以更灵活调整

## 7. 注册中心和配置中心有什么区别

这两个概念要分清。

注册中心关注的是：

- 服务在哪里
- 哪些实例可用
- 调用方怎么找到被调用方

配置中心关注的是：

- 配置放在哪里
- 配置怎么统一管理
- 配置变更怎么让服务感知

所以一句话区分：

- 注册中心管服务地址
- 配置中心管服务配置

## 8. 面试里怎么介绍 Nacos

可以这样说：

`Nacos` 在 Spring Cloud 里通常同时承担注册中心和配置中心的角色。作为注册中心，它负责服务注册与发现，让服务之间不用写死地址；作为配置中心，它负责集中管理配置，减少多服务配置分散带来的维护成本。

如果想再补一句：

所以 `Nacos` 解决的是微服务拆分后“服务怎么找”和“配置怎么管”这两个基础问题。

## 9. 当前这一份先记住什么

先记住下面几句：

1. `Nacos` 常见定位是注册中心和配置中心
2. 注册中心解决服务发现问题
3. 配置中心解决配置集中管理问题
4. 注册中心管服务地址，配置中心管服务配置
5. 面试里先讲清定位，再讲细节

## 10. 一句话总结

`Nacos` 的核心不是背配置项，而是理解它在微服务里分别解决“服务怎么动态发现”和“配置怎么集中管理”这两个问题。
