# Spring Cloud 基础入门

## 1. 这一份是干什么的

这一份是 `Spring Cloud` 的基础入门笔记。

它和主线介绍的区别是：

- 主线介绍更偏“为什么要引入微服务，以及 Spring Cloud 在补什么”
- 这一份更偏“先把 Spring Cloud 的基本概念认清楚”

如果你对下面这些还不够熟，这一份可以先读：

- 什么是微服务
- `Spring Boot` 和 `Spring Cloud` 的关系
- `Spring Cloud` 常见组件大概干什么
- 为什么这些组件会一起被放在微服务体系里讲

## 2. 什么是 Spring Cloud

`Spring Cloud` 可以先理解成一套围绕微服务的基础设施工具集合。

它不是单个框架，而是一组用来解决分布式系统常见问题的组件和规范。

这些问题包括：

- 服务地址怎么找
- 服务之间怎么调用
- 配置怎么统一管理
- 外部流量怎么进入系统
- 某个服务慢了或挂了怎么办
- 多服务一起改数据怎么办

所以 `Spring Cloud` 的定位，不是写业务逻辑，而是给微服务提供一层通用能力。

## 3. Spring Boot 和 Spring Cloud 的关系

这两个经常一起出现，但关注点不同。

### 3.1 Spring Boot

`Spring Boot` 更像是在解决“怎么把 Spring 项目更快跑起来”。

它关注的是：

- 自动配置
- 简化依赖管理
- 简化启动方式
- 减少样板代码

### 3.2 Spring Cloud

`Spring Cloud` 更像是在解决“微服务系统怎么更好地组织起来”。

它关注的是：

- 服务发现
- 服务调用
- 配置管理
- 网关
- 熔断限流
- 分布式事务

### 3.3 两者关系

简单说：

- `Spring Boot` 负责把单个服务跑起来
- `Spring Cloud` 负责把多个服务组织起来

所以它们通常不是替代关系，而是组合关系。

## 4. 为什么需要 Spring Cloud

当系统还是单体时，很多问题不明显。

但一旦拆成多个服务，下面这些问题就会出现：

1. 服务地址不能写死
2. 服务之间要统一调用
3. 配置不能散在各处
4. 外部请求需要统一入口
5. 下游慢了要防止拖垮上游
6. 跨服务改数据会变复杂

`Spring Cloud` 就是把这些基础问题补起来。

## 5. 常见组件大概是干什么的

下面先只记“作用”，不用一上来记太多细节。

### 5.1 `Nacos`

常见用法：

- 注册中心
- 配置中心

它负责解决：

- 服务实例怎么被发现
- 配置怎么集中管理

### 5.2 `OpenFeign`

它负责解决：

- 服务和服务之间怎么发请求

可以把它理解成一个更方便的服务调用方式。

### 5.3 `LoadBalancer`

它负责解决：

- 多个服务实例之间怎么分配请求

简单说就是负载均衡。

### 5.4 `Spring Cloud Gateway`

它负责解决：

- 外部请求怎么统一进入系统
- 统一做路由、鉴权、限流等

### 5.5 `Sentinel`

它负责解决：

- 熔断
- 限流
- 降级

简单说就是让系统在高压下别一下子崩掉。

### 5.6 `Seata`

它负责解决：

- 分布式事务

也就是多个服务一起改数据时的一致性问题。

## 6. 这些组件为什么经常一起出现

因为它们解决的不是一个孤立问题，而是微服务拆分后连带出现的一组问题。

你可以把它们理解成：

1. 先有微服务拆分
2. 再出现服务发现、调用、配置、入口、稳定性、事务这些问题
3. `Spring Cloud` 相关组件就是一套补基础设施的工具

所以在面试里，不要把它们讲成零散工具，而要讲成一套系统问题的配套方案。

## 7. 当前这一份先记住什么

先记住下面几句就够了：

1. `Spring Cloud` 不是单个框架，而是一组微服务基础设施能力
2. `Spring Boot` 更偏“启动单个服务”，`Spring Cloud` 更偏“组织多个服务”
3. `Nacos` 管注册和配置
4. `OpenFeign` 管服务调用
5. `Gateway` 管统一入口
6. `Sentinel` 管稳定性
7. `Seata` 管分布式事务

## 8. 一句话总结

`Spring Cloud` 可以先理解成“微服务常见问题的工具箱”，它负责把服务发现、服务调用、配置管理、统一入口、熔断限流和分布式事务这些能力补齐。
