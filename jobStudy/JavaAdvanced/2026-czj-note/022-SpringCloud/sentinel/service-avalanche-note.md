# 服务雪崩专题

## 1. 什么是服务雪崩

服务雪崩指的是：

> 一个服务出现故障或变慢后，影响依赖它的上游服务，最终导致故障沿调用链不断扩散。

它不是单个服务挂掉这么简单，而是一个点的问题被不断放大，最后影响整个系统。

## 2. 一个典型例子

假设调用链是：

```text
订单服务 -> 库存服务 -> 商品服务
```

如果商品服务突然变慢，库存服务调用商品服务时会一直等待。

库存服务线程被大量阻塞后，订单服务调用库存服务也会变慢。

最后可能出现：

- 商品服务慢
- 库存服务线程被占满
- 订单服务请求堆积
- 用户下单接口整体不可用

这就是服务雪崩。

## 3. 服务雪崩常见原因

常见原因包括：

- 下游服务变慢
- 下游服务宕机
- 网络抖动
- 数据库慢查询
- 线程池资源耗尽
- 重试放大流量
- 上游没有超时控制
- 没有限流、熔断、降级

其中比较典型的是：

> 下游慢 + 上游无限等待 + 请求不断进入 = 线程资源被拖垮。

## 4. 为什么下游慢比下游直接失败更危险

下游直接失败时，上游通常能很快得到异常。

下游变慢时，上游线程会长时间等待。

等待时间越长，占用资源越久。

如果请求量继续进来，就会导致：

- 线程被占满
- 队列堆积
- 新请求无法处理
- 上游服务也开始变慢

所以慢调用很容易引发雪崩。

## 5. 怎么防止服务雪崩

常见手段包括：

1. 超时控制
2. 限流
3. 熔断
4. 降级
5. 线程池隔离
6. 舱壁隔离
7. 避免无脑重试
8. 监控告警

其中 Sentinel 重点解决的是：

- 限流
- 熔断
- 降级
- 系统保护

## 6. 超时和 Sentinel 是什么关系

超时负责控制等待时间。

Sentinel 负责在异常趋势出现时进行保护。

比如：

- 单次调用不能无限等，所以要配置超时
- 下游持续慢调用，就需要熔断或降级
- 请求量太高，就需要限流

它们是配合关系。

## 7. 面试怎么说

可以这样说：

服务雪崩是指微服务调用链里某个下游服务变慢或故障后，导致上游请求阻塞、线程被占满，最终故障沿调用链扩散。比如订单服务调用库存服务，库存服务又依赖商品服务，如果商品服务响应很慢，库存服务线程会被大量阻塞，订单服务也会被拖慢。防止雪崩通常需要超时、限流、熔断、降级和隔离机制。Sentinel 的作用就是在流量异常、失败率升高或慢调用增多时及时保护服务，避免故障继续扩大。

## 8. 一句话总结

服务雪崩的本质，是局部故障没有被及时隔离和止损，最终拖垮整条调用链。
