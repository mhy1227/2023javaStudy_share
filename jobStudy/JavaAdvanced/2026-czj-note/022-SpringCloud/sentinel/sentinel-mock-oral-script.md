# Sentinel 模拟追问脚本

## 1. 使用方式

这份文档按模拟面试节奏组织。

建议回答顺序：

1. 先说结论
2. 再说原因
3. 补项目场景
4. 最后讲边界

## 2. 第一轮：基础定位

### 面试官：Sentinel 是什么？

候选人回答重点：

- 流量治理
- 服务保护
- 限流、熔断、降级
- 防雪崩

可答：

> Sentinel 是一个流量治理和服务保护组件，常用于微服务里的限流、熔断和降级。它的核心作用是在流量过高、下游异常或系统资源紧张时及时止损，避免服务被拖垮。

### 面试官追问：为什么微服务里要有 Sentinel？

候选人回答重点：

- 调用链变长
- 局部故障扩散
- 下游慢会拖垮上游

可答：

> 因为微服务拆分后，一个请求可能经过多个服务，只要某个下游变慢或异常，上游线程就可能一直等待，导致请求堆积和故障扩散。Sentinel 的作用就是在这种情况下及时限流、熔断或降级，防止服务雪崩。

## 3. 第二轮：核心机制

### 面试官：限流、熔断、降级有什么区别？

候选人回答重点：

- 限流挡流量
- 熔断断依赖
- 降级给兜底

可答：

> 限流是控制进入系统的请求量，防止流量太大压垮当前服务；熔断是当下游异常率或慢调用比例过高时，暂时停止继续调用下游，避免故障扩散；降级是调用失败后返回兜底结果，保证核心链路可用。简单说就是限流挡流量，熔断断依赖，降级给兜底。

### 面试官追问：什么是服务雪崩？

候选人回答重点：

- 下游故障
- 上游阻塞
- 故障扩散

可答：

> 服务雪崩就是某个下游服务变慢或故障后，上游请求因为一直等待而被阻塞，线程资源被占满，最后故障沿调用链扩散，导致更多服务不可用。

## 4. 第三轮：规则与边界

### 面试官：你们怎么做限流？

候选人回答重点：

- QPS
- 并发
- 热点参数
- 入口层 + 服务层分工

可答：

> 我们会优先对高流量、容易被刷或者不允许瞬时冲太高的接口做限流，比如查询接口、开放接口、文件下载接口等。入口层可以做统一限流，服务内部再针对具体业务接口做更细粒度保护。

### 面试官：熔断后怎么恢复？

候选人回答重点：

- 不是永久关闭
- 统计窗口
- 探测请求
- 逐步恢复

可答：

> 熔断不是永久关闭调用，它的目标是在下游异常期间先止损，等下游恢复后再逐步放开流量。通常会结合统计窗口和探测策略，先进入半开状态，再根据恢复情况重新恢复正常调用。

### 面试官：降级是不是随便返回默认值？

候选人回答重点：

- 不能随便
- 业务可接受
- 不影响决策

可答：

> 降级不能随便返回默认值，必须保证业务可接受，而且不会引入错误决策。比如推荐、评论这类非核心能力可以降级，但支付、扣库存这种核心写操作不能随便降级成成功。

## 5. 第四轮：和 Gateway / Feign 的关系

### 面试官：Sentinel 和 Gateway 什么关系？

候选人回答重点：

- Gateway 入口
- Sentinel 保护
- 分层治理

可答：

> Gateway 负责统一入口和基础鉴权，Sentinel 负责流量治理和服务保护。一般会让 Gateway 先在入口层做基础限流，服务内部再用 Sentinel 做更细粒度的限流、熔断和降级，这样职责更清晰。

### 面试官：Sentinel 和 Feign 什么关系？

候选人回答重点：

- Feign 负责调用
- Sentinel 负责保护调用

可答：

> Feign 负责服务之间的远程调用，Sentinel 负责在调用不稳定时保护系统。比如下游慢调用比例过高或者异常比例过高时，就可以触发熔断降级，避免调用方线程一直等待，也避免继续给异常下游施压。

## 6. 第五轮：项目结合

### 面试官：你们项目里 Sentinel 做了什么？

候选人回答重点：

- 限流接口
- 熔断降级
- 入口层和服务层分工

可答：

> 项目里 Sentinel 主要用于接口流量控制和调用链保护。对于高流量接口会做限流，防止瞬时流量打爆服务；对于依赖下游的非核心能力，比如推荐、评论、积分等，会配置熔断降级，确保下游异常时主流程还能继续。支付、下单这类核心写操作不会随便降级成成功，只会做更谨慎的超时和错误处理。

## 7. 最后一段总结

如果最后让你总结 Sentinel，可以这样说：

> Sentinel 的核心作用是保护系统稳定性。它通过限流、熔断和降级，在流量过高或下游不稳定时及时止损，避免故障沿调用链扩散。项目里通常会把入口层限流、服务层熔断降级结合起来使用，并且要注意规则配置不能拍脑袋，核心写操作也不能随便做错误降级。
