# Sentinel 项目场景表达

## 1. 这份文档解决什么问题

面试里如果项目用了微服务，Sentinel 通常会被问：

- 你们为什么用 Sentinel？
- 限流怎么做的？
- 熔断降级怎么做的？
- 哪些接口做了保护？
- 项目里怎么避免雪崩？

这份文档专门整理项目口述素材。

## 2. 项目里为什么引入 Sentinel

可以这样说：

> 项目拆成多个微服务之后，一个请求可能依赖多个下游服务。如果某个下游服务变慢或异常，上游请求就可能堆积，最终把整条调用链拖垮。我们引入 Sentinel，就是为了在入口流量过高、下游异常或慢调用增多时，及时做限流、熔断和降级，保护核心链路。

重点是讲问题，不只是讲工具。

## 3. 哪些接口适合做限流

适合做限流的通常是：

- 查询热点接口
- 秒杀接口
- 开放接口
- 文件下载接口
- 容易被刷的接口

可以这样说：

> 我们会优先对流量大、容易被刷、或者业务上不允许瞬时冲太高的接口做限流，避免大量请求直接打到后端服务。

## 4. 哪些接口适合做熔断降级

适合做熔断降级的通常是：

- 调下游服务的接口
- 调第三方接口的接口
- 非核心展示类能力

比如：

- 推荐
- 评论
- 积分
- 统计

可以这样说：

> 对于依赖下游、但失败后不会影响核心流程的接口，我们会配置熔断降级。比如推荐服务或评论服务失败时，页面可以返回默认数据或暂时不展示，不影响主流程继续运行。

## 5. 核心写操作怎么说

像支付、扣库存、创建订单这种核心写操作，不能随便降级成“成功”。

可以这样说：

> 核心写操作不适合做随意降级，因为这类接口涉及真实业务状态变更，不能因为下游异常就返回一个假成功。对于这类接口，更多是做超时控制、重试控制和错误兜底，而不是简单返回默认值。

## 6. 和 Feign 怎么结合

可以这样说：

> 我们在 Feign 调用下游服务时会结合 Sentinel 做保护。如果下游慢调用比例或异常比例过高，就触发熔断，后续请求快速失败或走降级逻辑，避免调用方线程一直等待，也避免继续给异常下游施压。

## 7. 和 Gateway 怎么分工

可以这样说：

> Gateway 主要负责统一入口的基础限流和鉴权，Sentinel 负责服务内部的细粒度流量保护。这样入口层先挡住大流量，服务内部再针对具体接口和远程调用做保护，职责更清晰。

## 8. 项目里怎么体现效果

如果面试官问效果，可以保守表达：

> 接入 Sentinel 后，入口层可以提前拦截明显异常的流量，核心接口不会被瞬时流量冲垮；下游服务异常时也能通过熔断和降级快速止损，避免故障继续扩散。实际效果主要体现在系统稳定性更高，故障恢复速度更快。

不要乱报夸张数字，除非你真有。

## 9. 一段项目口述模板

可以这样说：

在项目里，Sentinel 主要用于接口流量控制和调用链保护。对于流量比较高的查询接口或容易被刷的接口，我们会按 QPS 配置限流，防止请求瞬时打爆服务。对于依赖下游服务的接口，比如推荐、评论、积分这类非核心能力，我们会配置熔断降级，当下游慢调用比例或异常比例过高时，返回兜底结果，保证主流程继续运行。

同时网关层会做入口限流，服务内部再用 Sentinel 做更细粒度保护。对于支付、下单、扣库存这类核心写操作，不会随便降级成成功，只会做超时控制和错误处理。整体上，Sentinel 的作用就是在流量过高或下游不稳定时及时止损，避免故障沿调用链扩散。

## 10. 一句话总结

项目里介绍 Sentinel，不要只说“做了限流”，而要讲清楚它保护了哪些接口、为什么这么分层、以及怎么防止雪崩。
