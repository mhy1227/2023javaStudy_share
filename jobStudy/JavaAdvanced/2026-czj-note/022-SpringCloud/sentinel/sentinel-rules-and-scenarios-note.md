# Sentinel 规则与场景

## 1. 为什么要理解规则

Sentinel 不是只会说“限流、熔断、降级”就够了。

面试里常会继续追问：

- 按什么维度限流？
- 什么情况下触发熔断？
- 热点参数怎么保护？
- 系统保护是什么？

这些都和规则有关。

## 2. 流控规则

流控规则主要用于限制请求流量。

常见维度包括：

- QPS
- 并发线程数

比如：

- 某个接口每秒最多 100 个请求
- 某个资源最多允许 50 个并发线程

适用场景：

- 秒杀接口
- 查询热点接口
- 外部开放接口
- 容易被刷的接口

一句话：

> 流控规则解决的是“请求太多了，当前服务扛不住”的问题。

## 3. 熔断规则

熔断规则主要用于保护调用链。

常见触发依据包括：

- 慢调用比例
- 异常比例
- 异常数

比如：

- 下游接口响应时间持续超过阈值
- 一段时间内异常比例过高
- 一段时间内异常数量过多

适用场景：

- 下游服务不稳定
- 第三方接口超时
- 数据库查询异常
- 远程调用失败率升高

一句话：

> 熔断规则解决的是“依赖已经不稳定了，不要继续拖垮调用方”的问题。

## 4. 降级场景

降级通常和熔断、异常处理配合。

比如：

- 推荐服务失败，返回默认推荐
- 评论服务失败，页面不展示评论
- 文件预览服务失败，提示稍后重试
- 积分服务失败，先返回默认值

降级结果要满足两个条件：

1. 用户可以接受
2. 不影响核心链路

如果是支付、扣库存这类强一致核心操作，不能随便降级成“默认成功”。

## 5. 热点参数限流

热点参数限流是针对某些参数值访问特别集中的情况。

比如：

- 某个商品 ID 被大量访问
- 某个用户频繁请求
- 某个文件被大量下载
- 某个活动 ID 流量突然暴增

普通接口限流只能限制整个接口。

热点参数限流可以限制某个具体参数值。

比如：

> 商品详情接口整体流量正常，但某个爆款商品 ID 的访问量异常高，可以只限制这个商品 ID 的请求。

## 6. 系统保护规则

系统保护规则关注整个系统状态。

比如：

- CPU 使用率
- 系统负载
- 平均响应时间
- 入口 QPS
- 并发线程数

当系统整体压力过高时，Sentinel 可以触发保护，防止系统被彻底打垮。

它和普通接口限流不同：

> 普通限流关注某个资源，系统保护关注整机或整个应用的负载状态。

## 7. Sentinel 规则应该怎么配置

配置规则时不能拍脑袋。

一般要结合：

- 压测结果
- 历史流量
- 服务容量
- 下游依赖能力
- 核心链路优先级

比如：

- 核心接口阈值要更谨慎
- 非核心接口可以更早降级
- 读接口和写接口策略不同
- 第三方接口要设置更严格超时和熔断

## 8. 规则配置错误有什么风险

常见风险包括：

- 阈值太低，误伤正常请求
- 阈值太高，保护不及时
- 降级结果不合理，影响业务正确性
- 熔断恢复太快，反复打垮下游
- 没有监控，规则效果不可见

所以 Sentinel 不是“接上就完事”，还要持续观察和调优。

## 9. 面试怎么说

可以这样说：

Sentinel 的规则主要围绕流控、熔断降级、热点参数和系统保护。流控规则一般按 QPS 或并发线程数限制请求量，适合秒杀、热点查询等高流量接口；熔断规则会根据慢调用比例、异常比例或异常数判断下游是否不稳定，触发后暂时不再继续调用；热点参数限流可以针对某个商品 ID、用户 ID 这类高频参数单独保护；系统保护则从 CPU、负载、响应时间等整体状态保护应用。规则配置要结合压测和历史流量，不能随便拍阈值。

## 10. 一句话总结

Sentinel 规则的核心，是根据流量、异常、慢调用、热点参数和系统负载，在合适的位置及时保护系统。
