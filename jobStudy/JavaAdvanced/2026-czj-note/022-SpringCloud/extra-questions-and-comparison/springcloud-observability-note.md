# Spring Cloud 链路追踪与监控补充

## 1. 为什么要补这块

微服务拆分后，一个请求可能经过多个服务。

如果接口慢了，不能只看某一个服务日志。

需要知道：

- 请求从哪里进来
- 经过了哪些服务
- 哪个服务慢
- 哪个调用失败
- 失败是否触发熔断或降级

这就是链路追踪和监控的价值。

## 2. 什么是链路追踪

链路追踪可以理解为：

> 给一次请求打上统一标识，然后记录它经过的每一个服务和调用耗时。

常见概念：

- traceId：一次请求链路的唯一标识
- span：链路中的一次调用片段
- parent span：上游调用片段
- child span：下游调用片段

不用死背概念，先理解：

> traceId 串起整条链路，span 表示链路里的一个调用环节。

## 3. 常见工具

常见工具包括：

- SkyWalking
- Zipkin
- Sleuth
- Micrometer Tracing
- Prometheus
- Grafana

不同工具侧重点不同。

面试里一般不要求你精通所有工具，但要知道微服务需要可观测性。

## 4. 微服务慢调用怎么排查

可以按这个顺序：

1. 先看 Gateway 入口耗时
2. 再看业务服务处理耗时
3. 再看 Feign 下游调用耗时
4. 再看数据库、Redis、MQ 等依赖
5. 最后看 Sentinel 是否触发限流、熔断或降级

不要只说“看日志”。

更好的说法是：

> 通过 traceId 串起整条调用链，定位慢在哪一段。

## 5. 日志里 traceId 有什么用

如果每个服务日志都带同一个 traceId，就可以通过这个 traceId 搜到一次请求在所有服务中的日志。

这对排查很关键。

比如：

- Gateway 收到请求
- 订单服务处理请求
- 订单服务调用库存服务
- 库存服务调用商品服务

这些日志如果都有同一个 traceId，就能还原整条链路。

## 6. 监控和链路追踪有什么区别

监控更偏整体指标。

比如：

- QPS
- 响应时间
- 错误率
- CPU
- 内存
- GC
- 线程数

链路追踪更偏单次请求路径。

比如：

- 这次请求经过哪些服务
- 哪个调用耗时最长
- 哪个服务返回异常

一句话：

> 监控看整体趋势，链路追踪看单次请求路径。

## 7. Sentinel 和监控是什么关系

Sentinel 本身也会提供一些实时流量和规则触发信息。

比如：

- 当前 QPS
- 被限流数量
- 熔断情况
- 慢调用情况

但完整线上观测通常还要结合日志、链路追踪和指标监控。

## 8. 面试怎么说

可以这样说：

微服务拆分后，一个请求会经过多个服务，如果接口变慢或失败，不能只看单个服务日志。一般会通过 traceId 串起整条调用链，结合链路追踪工具查看请求经过哪些服务、每段调用耗时多少，再结合 Prometheus、Grafana 这类监控看整体 QPS、响应时间和错误率。这样可以判断问题是出在网关、业务服务、Feign 下游调用，还是数据库、Redis、MQ 等依赖上。

## 9. 一句话总结

微服务线上排查的关键，是用 traceId 串链路，用监控看趋势，用日志定位细节。
