# Seata 0-3 分钟回答包

## 1. 请你介绍一下 Seata

`Seata` 是一个分布式事务解决方案，主要用于解决微服务拆分后跨服务数据一致性的问题。

在单体应用里，很多操作都可以放在一个本地事务里完成，比如同时写订单表和库存表。但在微服务架构下，订单、库存、账户可能分别属于不同服务，数据也分散在不同数据库里，这时本地事务就管不到所有参与方了。

Seata 的作用就是在分布式场景下协调多个服务的提交和回滚，让一次业务操作在多个参与者之间尽量保持一致。

## 2. Seata 的核心角色是什么？

Seata 里常见的三个角色是 TC、TM 和 RM。

TM 是事务发起者，负责开启全局事务。比如用户点击下单之后，由订单服务发起全局事务。

RM 是资源管理器，负责管理具体资源，通常就是数据库资源。它要配合事务协调执行本地操作。

TC 是事务协调者，负责协调整个全局事务的提交和回滚。可以把它理解成整个分布式事务的总调度中心。

这三者配合起来，才能让多个服务在一次业务操作里形成统一协调。

## 3. AT、TCC、SAGA 有什么区别？

这几个是 Seata 里常见的事务模式，核心区别在于侵入程度、适用场景和一致性要求。

AT 模式可以先理解成自动协调模式，对业务侵入相对较小，比较适合常规 CRUD 场景。它更像是帮你自动处理大部分提交和回滚协调。

TCC 模式需要业务自己实现 Try、Confirm、Cancel 三个阶段，业务侵入较大，但控制力更强，适合核心强一致场景。比如要先预留库存，再确认扣减，失败就取消。

SAGA 模式更偏长事务和补偿思路，它不强调强回滚，而是如果后续步骤失败，就通过补偿动作把前面的步骤修正回来。它更适合流程型业务和最终一致场景。

简单说，AT 偏自动协调，TCC 偏显式三段式控制，SAGA 偏补偿和最终一致。

## 4. Seata 适合什么场景，不适合什么场景？

Seata 适合的是跨服务写数据且一致性要求较高的场景，比如订单创建、库存扣减、账户扣费这类联动操作。

它不太适合纯查询场景，也不适合那些可以用最终一致方案轻松解决、或者对性能特别敏感的高并发场景。因为分布式事务本身会带来协调成本、锁定成本和额外的系统复杂度。

所以项目里不能只看“能不能用 Seata”，还要看“值不值得用 Seata”。如果业务允许最终一致，很多时候消息补偿会更轻量；如果业务要求更高的一致性，再考虑 Seata 这类方案。

## 5. 项目里怎么介绍 Seata？

可以这样说：

在项目里，如果存在订单、库存、账户这类跨服务的数据写操作，而且一致性要求比较高，我们会考虑使用 Seata 做分布式事务协调。它可以在多服务之间统一协调提交和回滚，避免某个环节失败后出现数据不一致。

但我们也不会对所有场景都强上分布式事务。对于纯查询、统计或者可以接受最终一致的场景，通常会优先考虑更轻量的方案，比如消息最终一致或补偿机制。Seata 更适合高一致性要求的核心链路，而不是所有业务都套一层。

## 6. 一段完整口述版

`Seata` 是阿里开源的分布式事务方案，主要解决微服务拆分后跨服务写数据的一致性问题。单体项目里很多操作可以靠一个本地事务解决，但微服务里订单、库存、账户可能分散在不同服务和数据库中，本地事务已经不够用了。

Seata 通过 TC、TM、RM 这几个角色协同工作，协调多个参与者在一次业务操作中完成提交和回滚。常见模式有 AT、TCC 和 SAGA。AT 模式侵入较小，适合常规 CRUD；TCC 需要自己实现 Try、Confirm、Cancel，适合核心强一致场景；SAGA 偏补偿和最终一致，适合长流程业务。

项目选型时不能只看技术名词，而要看业务的一致性要求、性能开销和开发成本。如果业务允许最终一致，消息补偿通常更轻量；如果一致性要求很高，再考虑 Seata 这类方案。
