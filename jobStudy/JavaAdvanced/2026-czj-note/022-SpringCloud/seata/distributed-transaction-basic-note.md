# 分布式事务基础

## 1. 什么是分布式事务

分布式事务指的是：

> 一次业务操作要跨多个服务或多个数据源一起修改数据，并且希望这些修改最终保持一致。

比如：

- 创建订单
- 扣减库存
- 记录账户流水

这些操作可能分散在不同服务中。

## 2. 为什么本地事务不够用

本地事务只适用于同一个数据库、同一个服务内的操作。

比如一个服务里：

- 写订单表
- 写订单日志表
- 更新同库库存表

这些可以由一个本地事务控制。

但微服务拆分后，订单、库存、账户可能分别属于不同服务，甚至不同数据库。

这时单个本地事务已经管不到所有操作了。

## 3. 分布式事务难在哪里

主要难点有几个：

1. 多服务网络调用不可靠
2. 某个分支服务可能超时或失败
3. 不同数据库之间无法直接用一个本地事务包住
4. 回滚逻辑比本地事务复杂
5. 并发和锁的处理更复杂

所以分布式事务的本质不是“多写几张表”，而是：

> 多个独立系统之间如何协调提交和回滚。

## 4. 分布式事务想解决什么

它要解决的是跨服务数据一致性问题。

比如订单创建成功了，但库存扣减失败了：

- 要么订单也撤销
- 要么库存补回
- 要么通过补偿流程修正状态

不同方案对“一致性”的要求不同。

## 5. 常见一致性思路

常见思路包括：

- 强一致
- 最终一致
- 通过消息补偿实现一致
- 通过人工补偿兜底

面试里要强调：

> 并不是所有场景都适合强一致，很多业务更适合接受最终一致。

## 6. 为什么微服务里更容易遇到这个问题

因为单体项目里，很多操作都在一个 JVM、一个数据库、一个本地事务里。

微服务拆分后：

- 服务之间要通过网络调用
- 数据分散到多个库
- 业务步骤拆散到多个服务

这会让一致性问题明显变复杂。

## 7. 面试怎么说

可以这样说：

分布式事务是指一次业务操作跨多个服务或多个数据库一起修改数据，并希望这些修改保持一致。单体项目里很多操作可以用本地事务解决，但微服务拆分后，订单、库存、账户等可能分散在不同服务中，本地事务就不够用了。分布式事务难点在于网络不可靠、回滚复杂、并发更难控制，所以很多场景不会追求强一致，而是根据业务要求选择最终一致、补偿机制或者专门的分布式事务方案。

## 8. 一句话总结

分布式事务的核心，就是跨服务修改数据时，如何协调一致性和失败回滚。
