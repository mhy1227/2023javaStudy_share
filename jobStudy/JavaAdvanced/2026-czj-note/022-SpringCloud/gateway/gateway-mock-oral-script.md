# Gateway 模拟追问脚本

## 1. 使用方式

这份文档按模拟面试节奏组织。

建议练习时不要逐字背，而是按下面结构回答：

1. 先说结论
2. 再说原因
3. 补项目场景
4. 最后补边界和风险

## 2. 第一轮：基础定位

### 面试官：Gateway 是什么？

候选人回答重点：

- 微服务 API 网关
- 统一入口
- 路由转发
- 入口治理

可答：

> Gateway 是 Spring Cloud 体系里的 API 网关组件，通常作为微服务系统的统一入口。客户端请求先进入 Gateway，再由 Gateway 根据路由规则转发到对应服务。同时它还能统一做鉴权、限流、跨域、日志、灰度等入口治理能力。

### 面试官追问：为什么不让前端直接调各个微服务？

候选人回答重点：

- 服务地址暴露
- 前端复杂
- 通用逻辑重复
- 服务调整影响前端

可答：

> 如果前端直接调用各个微服务，前端就要维护多个服务地址，服务拆分和迁移也会影响前端。同时鉴权、跨域、限流这些通用逻辑会分散到各个服务里。Gateway 提供统一入口，可以屏蔽后端服务细节，也能集中处理入口层通用能力。

## 3. 第二轮：核心机制

### 面试官：Gateway 的路由、断言、过滤器分别是什么？

候选人回答重点：

- 路由：去哪
- 断言：是否匹配
- 过滤器：前后处理

可答：

> 路由决定请求最终转发到哪个服务，断言判断请求是否满足这条路由规则，比如路径、方法、请求头是否匹配，过滤器负责在请求转发前后做统一处理，比如鉴权、限流、日志、请求头改写等。简单说，路由决定去哪里，断言决定是否匹配，过滤器决定前后做什么。

### 面试官追问：一次请求进入 Gateway 后流程是什么？

候选人回答重点：

- 接收请求
- 匹配路由
- 执行过滤器
- 转发服务
- 响应返回

可答：

> 请求进入 Gateway 后，先根据断言匹配对应路由，然后执行过滤器逻辑，比如鉴权和限流。通过后 Gateway 会根据路由目标转发到后端服务，后端响应返回后，还可以继续执行响应阶段过滤器，最后再返回给客户端。

## 4. 第三轮：鉴权与安全

### 面试官：你会把鉴权放在 Gateway 吗？

候选人回答重点：

- 基础鉴权可以放
- 复杂业务权限不全放
- 网关和业务服务分工

可答：

> 基础鉴权可以放在 Gateway，比如 token 是否有效、是否登录、黑白名单等。这样可以在入口层直接拦截非法请求。但复杂业务权限不适合全部放网关，比如订单是否属于当前用户、文件目录是否有权限，这类需要业务数据判断的逻辑应该放在业务服务里。

### 面试官追问：用户信息怎么传给下游？

候选人回答重点：

- token 校验
- 解析用户信息
- 请求头透传
- 防止伪造

可答：

> 网关校验 token 后，可以解析出用户 ID、租户 ID 等信息，通过内部请求头传给下游服务。为了防止外部请求伪造身份头，网关要先清理外部传入的内部用户头，再写入自己解析出来的可信信息。

## 5. 第四轮：限流与高可用

### 面试官：Gateway 怎么做限流？

候选人回答重点：

- IP、用户、接口、服务维度
- 令牌桶、漏桶等算法
- 保护后端

可答：

> Gateway 可以在入口层按 IP、用户、接口或服务维度限流，常见算法有令牌桶、漏桶、滑动窗口等。限流的目的不是简单拒绝用户，而是在高流量或异常流量下保护后端服务不被打垮。

### 面试官追问：Gateway 会不会成为单点？

候选人回答重点：

- 单实例会
- 多实例部署
- 前面挂 Nginx 或 SLB
- 健康检查

可答：

> 如果 Gateway 只部署一个实例，就会成为单点。实际项目中应该多实例部署 Gateway，前面通过 Nginx 或 SLB 做负载均衡，并配合健康检查和监控，把异常实例及时摘除。

## 6. 第五轮：和 Nginx 对比

### 面试官：有了 Nginx，为什么还要 Gateway？

候选人回答重点：

- Nginx 基础入口
- Gateway 微服务治理
- 不是替代关系

可答：

> Nginx 和 Gateway 不是简单替代关系。Nginx 更偏基础设施层，适合做反向代理、静态资源、HTTPS 和基础负载均衡。Gateway 更偏微服务网关，适合和注册中心、鉴权、限流、灰度、链路追踪等能力结合。实际项目里可以是请求先到 Nginx，再进入 Gateway。

## 7. 第六轮：项目结合

### 面试官：你的项目里 Gateway 做了什么？

候选人回答重点：

- 统一入口
- 路由转发
- 跨域
- token 校验
- 用户信息透传
- 限流扩展

可答：

> 项目里 Gateway 主要作为统一入口，前端请求先进入网关，再根据路径转发到对应服务。网关层可以统一处理跨域、token 校验、请求日志和基础限流。token 校验通过后，网关会把用户 ID 等信息通过内部请求头传给下游服务。复杂业务权限仍然由业务服务判断。

### 面试官追问：文件上传接口也走 Gateway 吗？

候选人回答重点：

- 可以走
- 网关做入口校验和流量保护
- 上传业务逻辑在文件服务

可答：

> 可以走 Gateway。网关负责登录校验、基础限流、请求大小限制和路由转发，但分块上传、断点续传、秒传、文件合并这些业务逻辑不会放在网关，而是放在文件服务里处理。

## 8. 最后一段总结

如果最后让你总结 Gateway，可以这样说：

> Gateway 的核心价值是给微服务系统提供统一入口，并在入口层承接路由、鉴权、限流、日志、跨域、灰度等通用治理能力。它和 Nginx 不是替代关系，而是职责分层。项目里要注意网关边界，不要把复杂业务逻辑都塞进去，同时 Gateway 本身也要多实例部署，避免成为单点。
