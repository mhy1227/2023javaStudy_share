# Gateway 鉴权、限流与灰度

## 1. 为什么这些能力适合放在网关

鉴权、限流、灰度这些能力有一个共同特点：

> 它们不是某一个业务服务独有，而是入口层经常需要统一处理的通用能力。

如果每个微服务都重复实现这些逻辑，会带来：

- 代码重复
- 规则不统一
- 修改成本高
- 容易遗漏接口

所以很多项目会把一部分通用能力放到 Gateway。

## 2. 网关怎么做鉴权

常见做法是：

1. 请求进入 Gateway
2. Gateway 从请求头或 Cookie 中获取 token
3. 校验 token 是否存在、是否有效
4. 解析用户身份和权限信息
5. 不合法直接拦截
6. 合法则转发到后端服务

网关鉴权适合处理：

- 是否登录
- token 是否有效
- 黑白名单
- 基础权限判断

复杂业务权限不一定都放在网关。

比如订单是否属于当前用户、文件是否有目录权限，这类强业务权限通常还要在业务服务里校验。

## 3. 鉴权信息怎么传给下游

网关校验通过后，可以把用户信息放到请求头里传给下游。

比如：

- 用户 ID
- 租户 ID
- 角色标识
- 链路追踪 ID

下游服务拿到这些信息后，可以构建用户上下文。

注意：

- 不要让外部客户端直接伪造内部用户头
- 网关需要清理外部传入的敏感内部头
- 内部服务也不要完全盲目信任未校验来源的请求头

## 4. 网关怎么做限流

限流的目标是保护系统。

常见限流维度包括：

- 按 IP 限流
- 按用户限流
- 按接口限流
- 按服务限流
- 按租户限流

常见算法包括：

- 固定窗口
- 滑动窗口
- 漏桶
- 令牌桶

面试里可以不用展开算法细节，但要讲清楚限流目的：

> 限流不是为了拒绝用户，而是为了在流量过高时保护后端服务不被打垮。

## 5. 为什么限流适合放在网关

因为 Gateway 是统一入口。

在入口处限流有几个好处：

- 请求还没打到后端就能被拦截
- 可以保护所有下游服务
- 规则集中管理
- 更容易做全局流量控制

但是网关限流不能替代所有服务内部限流。

核心服务内部仍然可能需要更细粒度限流。

## 6. 网关怎么做灰度发布

灰度发布指的是：

> 不把所有流量一次性切到新版本，而是让一部分用户或一部分请求先访问新版本。

网关可以根据条件选择转发到不同版本服务。

常见条件包括：

- 用户 ID
- 请求头
- IP
- 租户
- 灰度标识

比如：

- 请求头 `gray=true` 转发到新版本
- 某些用户 ID 转发到新版本
- 某个租户先试用新版本

## 7. 网关是不是所有逻辑都应该放

不是。

网关适合放通用入口逻辑，不适合塞太多业务逻辑。

适合放网关的：

- 登录校验
- 基础鉴权
- 限流
- 路由
- 跨域
- 灰度
- 日志

不适合过度放网关的：

- 复杂业务规则
- 订单状态判断
- 库存扣减逻辑
- 文件目录权限深度校验
- 和数据库强绑定的业务校验

一句话：

> 网关做通用入口治理，业务服务做业务规则判断。

## 8. 面试怎么说

可以这样说：

Gateway 适合承接一些入口层通用能力，比如统一鉴权、限流、日志、灰度和跨域。鉴权时可以先在网关校验 token，解析用户身份后把必要信息透传给下游；限流可以在请求进入后端服务前按 IP、用户或接口维度拦截过高流量，保护后端；灰度可以根据请求头、用户 ID 或租户标识把一部分流量转发到新版本服务。不过网关不应该承载过多业务逻辑，复杂权限和业务校验仍然应该放在具体业务服务里。

## 9. 一句话总结

网关适合做入口层通用治理，但不应该变成承载复杂业务逻辑的大杂烩。
