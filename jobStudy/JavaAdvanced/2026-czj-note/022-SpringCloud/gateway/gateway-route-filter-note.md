# Gateway 路由、断言与过滤器

## 1. 为什么要理解这三个概念

`Gateway` 面试里最基础、最容易被问的三个概念是：

- 路由
- 断言
- 过滤器

如果这三个说不清楚，后面的鉴权、限流、灰度也很难展开。

## 2. 路由是什么

路由决定一个请求最终转发到哪个服务。

比如：

- `/api/user/**` 转发到 `user-service`
- `/api/order/**` 转发到 `order-service`
- `/api/file/**` 转发到 `file-service`

路由可以理解为：

> 请求入口路径和后端服务之间的映射关系。

## 3. 断言是什么

断言用于判断一个请求是否满足某条路由规则。

比如：

- 路径是否匹配 `/api/user/**`
- 请求方法是否是 `GET`
- 请求头是否包含某个字段
- 参数是否满足条件
- Host 是否匹配

只有断言匹配成功，请求才会进入这条路由。

可以这样说：

> 断言负责判断“这个请求是不是属于这条路由”。

## 4. 过滤器是什么

过滤器用于在请求转发前后做处理。

常见处理包括：

- 鉴权
- 限流
- 打日志
- 添加请求头
- 删除请求头
- 修改请求路径
- 统计耗时
- 统一异常处理

过滤器可以理解为：

> 在请求进入后端服务之前，或者后端响应返回客户端之前，插入一段通用处理逻辑。

## 5. 路由、断言、过滤器的关系

一次请求进入 Gateway 后，可以按这条链路理解：

1. Gateway 接收到请求
2. 根据断言判断请求匹配哪条路由
3. 找到目标路由
4. 执行路由上的过滤器
5. 转发到后端服务
6. 后端返回响应
7. 执行响应阶段过滤逻辑
8. 返回给客户端

一句话：

> 断言决定能不能匹配路由，路由决定转发到哪里，过滤器决定转发前后做什么。

## 6. 全局过滤器和局部过滤器

过滤器通常可以分成两类：

- 全局过滤器
- 局部过滤器

### 6.1 全局过滤器

全局过滤器会对所有请求生效。

适合做：

- 统一日志
- 链路追踪
- 全局鉴权
- 请求耗时统计

### 6.2 局部过滤器

局部过滤器只对某些路由生效。

适合做：

- 某类接口特殊鉴权
- 某个服务路径改写
- 某条路由单独限流

## 7. 过滤器执行顺序重要吗

重要。

比如：

- 先鉴权再转发
- 先限流再进入业务服务
- 先记录请求开始时间，再在响应阶段统计耗时

如果顺序不对，可能会导致：

- 未鉴权请求进入后端
- 限流逻辑失效
- 日志耗时统计不准确

所以过滤器顺序也是实际项目里要注意的点。

## 8. 面试怎么说

可以这样说：

Gateway 的核心概念主要是路由、断言和过滤器。路由决定请求最终转发到哪个服务，断言负责判断请求是否匹配某条路由，过滤器负责在请求转发前后做统一处理，比如鉴权、限流、日志、请求头改写等。一次请求进入 Gateway 后，会先根据断言匹配路由，然后执行对应过滤器，再转发到后端服务，响应回来后也可以继续执行过滤逻辑。

## 9. 一句话总结

路由决定去哪里，断言决定是否匹配，过滤器决定前后做什么。
