# Gateway 高频问题

## 1. Gateway 是什么？

回答方向：

- Spring Cloud 体系里的 API 网关
- 微服务系统统一入口
- 负责路由转发
- 承接鉴权、限流、日志、灰度等入口治理能力

## 2. 为什么需要 Gateway？

回答方向：

- 微服务拆分后服务数量变多
- 客户端不应该直接访问多个微服务
- 避免服务地址暴露
- 统一处理鉴权、限流、跨域、日志等能力
- 降低前端和后端服务拆分的耦合

## 3. Gateway 的核心概念有哪些？

回答方向：

- 路由
- 断言
- 过滤器
- 路由决定转发到哪里
- 断言决定是否匹配路由
- 过滤器决定请求前后做什么

## 4. Gateway 的请求流程是什么？

回答方向：

- 请求进入 Gateway
- 根据断言匹配路由
- 执行过滤器
- 转发到后端服务
- 后端返回响应
- 响应阶段过滤器继续处理
- 返回客户端

## 5. Gateway 可以做哪些事情？

回答方向：

- 路由转发
- 统一鉴权
- 限流
- 日志
- 跨域
- 灰度发布
- 请求头改写
- 链路追踪

## 6. Gateway 和 Nginx 有什么区别？

回答方向：

- Nginx 更偏基础设施层反向代理和负载均衡
- Gateway 更偏微服务网关治理
- Nginx 更靠近外部入口
- Gateway 更靠近微服务内部
- 两者不是简单替代关系，可以分层配合

## 7. Gateway 怎么做鉴权？

回答方向：

- 请求进入网关后先解析 token
- 校验 token 是否有效
- 无效则直接拦截
- 有效则解析用户身份并转发到下游
- 复杂业务权限仍然可以放在业务服务里校验

## 8. Gateway 鉴权后用户信息怎么传给下游？

回答方向：

- 通过请求头透传用户 ID、租户 ID、角色等信息
- 网关需要清理外部伪造的内部请求头
- 下游服务基于可信来源构建用户上下文

## 9. Gateway 怎么做限流？

回答方向：

- 可以按 IP、用户、接口、服务、租户限流
- 常见算法有令牌桶、漏桶、滑动窗口
- 限流目标是保护后端服务
- 核心服务内部也可以继续做更细粒度限流

## 10. Gateway 怎么做灰度发布？

回答方向：

- 根据请求头、用户 ID、IP、租户等条件路由到不同版本
- 小部分流量先访问新版本
- 验证稳定后再逐步放量
- 灰度规则适合放在入口层统一处理

## 11. Gateway 会不会成为瓶颈？

回答方向：

- 会有这个风险
- Gateway 是统一入口，请求都会经过它
- 需要多实例部署
- 前面可以挂 Nginx 或 SLB
- 配合监控、限流和健康检查

## 12. Gateway 是不是单点？

回答方向：

- 单实例部署就是单点
- 实际项目要多实例部署
- 前面挂负载均衡
- 做健康检查和故障摘除

## 13. Gateway 适合放业务逻辑吗？

回答方向：

- 不适合放复杂业务逻辑
- 适合放入口层通用逻辑
- 比如鉴权、限流、路由、跨域、日志
- 复杂权限和业务校验仍然放业务服务

## 14. Gateway 和注册中心怎么配合？

回答方向：

- 后端服务注册到 Nacos
- Gateway 根据服务名进行路由
- 不用写死具体 IP 和端口
- 服务扩缩容后路由可以动态感知

## 15. Gateway 和 OpenFeign 有什么区别？

回答方向：

- Gateway 面向外部请求入口和路由转发
- OpenFeign 面向服务之间的内部调用
- Gateway 是请求进入系统的入口
- Feign 是服务调用另一个服务的客户端

## 16. 项目里怎么介绍 Gateway？

回答方向：

- 作为统一入口接收前端请求
- 根据路径路由到不同微服务
- 在网关层做 token 校验、跨域、日志和限流
- 用户信息校验后通过请求头传给下游
- Gateway 多实例部署避免单点
