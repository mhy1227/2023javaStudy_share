# Gateway 项目场景表达

## 1. 这份文档解决什么问题

面试里如果项目使用了微服务，Gateway 很容易被问：

- 你们项目为什么用网关？
- 网关里做了哪些事情？
- 鉴权怎么做？
- 用户信息怎么传给下游？
- 有没有限流？
- 网关有没有单点问题？

这份文档专门整理项目口述素材。

## 2. 项目里为什么引入 Gateway

可以这样说：

> 项目拆成多个微服务后，前端不适合直接访问每个服务，所以我们引入 Gateway 作为统一入口。它负责把外部请求按照路径路由到对应微服务，同时统一处理跨域、鉴权、日志和基础限流，避免这些通用逻辑在每个服务里重复实现。

重点不是“用了 Gateway”，而是要讲出它解决的问题：

- 统一入口
- 屏蔽后端服务地址
- 降低前端调用复杂度
- 统一入口治理

## 3. 路由怎么设计

可以按业务模块划分路径。

比如：

- `/api/user/**` 转发到用户服务
- `/api/order/**` 转发到订单服务
- `/api/file/**` 转发到文件服务
- `/api/auth/**` 转发到认证服务

面试可以说：

> 我们主要按业务模块设计路由前缀，网关根据请求路径匹配路由，再转发到对应服务。后端服务注册到 Nacos，网关通过服务名转发，避免写死具体 IP 和端口。

## 4. 鉴权怎么讲

可以这样说：

> 请求进入网关后，会先经过全局鉴权过滤器。过滤器从请求头里获取 token，校验 token 是否有效。如果无效，网关直接返回未登录或无权限；如果有效，就解析出用户 ID、租户 ID 等基础信息，并写入内部请求头，转发给下游服务。

再补一句边界：

> 网关只做通用登录态校验和基础权限判断，具体业务权限仍然在业务服务里做，比如文件是否属于当前用户、订单是否允许操作。

## 5. 请求头透传怎么讲

可以这样说：

> 网关校验 token 后，会把可信用户信息通过请求头传给下游服务。为了防止请求头伪造，网关会先清理外部传入的内部身份头，再重新写入自己解析出来的用户信息。下游服务只信任经过网关写入的身份信息。

这句话比较关键，能体现安全意识。

## 6. 跨域怎么讲

可以这样说：

> 因为前端统一访问网关，所以跨域配置也放在网关层集中处理。这样不用每个微服务都单独配置跨域规则，避免规则不一致。生产环境不会简单放开所有来源，而是按前端域名、请求方法和请求头做限制。

## 7. 限流怎么讲

如果项目没有真的做复杂限流，也可以保守说：

> 网关层适合做基础限流，比如按 IP、用户或接口维度控制访问频率，避免异常流量直接打到后端服务。核心接口还可以在服务内部继续做更细粒度的限流。

如果项目没有落地，不要硬说“已经做了令牌桶”。

可以说：

> 项目里预留了网关限流的扩展点，后续可以接 Redis 或 Sentinel 做分布式限流。

## 8. 网关高可用怎么讲

可以这样说：

> Gateway 作为统一入口不能单实例部署，否则会成为单点。生产环境一般会部署多个 Gateway 实例，前面通过 Nginx 或 SLB 做负载均衡，并配合健康检查，把异常实例摘除。

如果面试追问瓶颈，可以补：

> 同时网关层不适合写太重的业务逻辑，否则会影响入口吞吐。我们会把复杂业务判断放在具体服务里，网关只保留通用入口逻辑。

## 9. 和文件上传项目怎么结合

如果你的项目有文件上传，可以这样衔接：

> 文件上传这类接口一般也会先经过 Gateway。网关可以做登录校验、基础限流和请求大小限制，再把请求转发到文件服务。但真正的分块上传、断点续传、秒传、文件合并这些逻辑不会放在网关，而是放在文件服务里处理。

注意这个边界：

- 网关做入口校验和流量保护
- 文件服务做上传业务逻辑

## 10. 一段项目口述模板

可以这样说：

在项目里，Gateway 主要作为后端微服务的统一入口。前端请求统一访问网关，网关根据路径前缀把请求转发到不同服务，比如用户服务、文件服务、权限服务等。这样前端不需要感知每个服务的真实地址，后端服务扩缩容也可以通过注册中心动态发现。

网关层主要做一些通用入口能力，比如跨域处理、token 校验、请求日志和基础限流。请求进入网关后，会先经过全局过滤器校验 token，校验通过后解析用户 ID 等信息，并通过内部请求头传给下游服务。为了避免请求头伪造，网关会先清理外部传入的身份头，再写入可信身份信息。

同时我们不会把复杂业务逻辑都放在网关。比如文件是否有访问权限、分块上传合并、秒传判断这些逻辑仍然放在文件服务里。Gateway 主要负责入口治理，业务服务负责业务规则。为了避免网关成为单点，实际部署时也应该多实例部署，前面通过 Nginx 或 SLB 分发流量。

## 11. 一句话总结

项目里介绍 Gateway，不要只说“负责转发”，要讲清楚统一入口、鉴权透传、限流保护、职责边界和高可用。
