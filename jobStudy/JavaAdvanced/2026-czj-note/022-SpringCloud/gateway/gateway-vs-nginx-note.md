# Gateway 和 Nginx 对比

## 1. 为什么会问这个

面试官问 Gateway 时，经常会顺手问：

> 有了 Nginx，为什么还要 Gateway？

这个问题的本质不是让你说谁替代谁，而是看你能不能讲清楚两者定位。

## 2. Nginx 更擅长什么

`Nginx` 更偏基础设施层的反向代理和负载均衡。

常见作用包括：

- 静态资源服务
- 反向代理
- 外部入口负载均衡
- HTTPS 证书卸载
- 简单路由转发
- 高性能连接处理

可以理解为：

> Nginx 更靠近流量入口和基础网络转发。

## 3. Gateway 更擅长什么

`Spring Cloud Gateway` 更偏微服务网关和业务入口治理。

常见作用包括：

- 根据服务名动态路由
- 和注册中心整合
- 统一鉴权
- 统一限流
- 灰度发布
- 请求头改写
- 链路追踪
- 和 Java 微服务体系整合

可以理解为：

> Gateway 更靠近微服务内部治理和业务请求入口。

## 4. 两者是不是替代关系

不是简单替代关系。

实际架构里经常是：

```text
客户端
  |
Nginx / SLB
  |
Gateway
  |
后端微服务
```

Nginx 负责更前置的入口流量和基础负载均衡。

Gateway 负责微服务路由和统一治理。

当然，小项目也可能只用其中一个。

## 5. 对比表

| 对比点 | Nginx | Gateway |
|---|---|---|
| 定位 | 反向代理、负载均衡、静态资源 | 微服务 API 网关 |
| 位置 | 更靠近外部流量入口 | 更靠近微服务内部 |
| 路由方式 | 配置文件为主 | 可结合服务发现动态路由 |
| 业务扩展 | 扩展相对不如 Java 体系方便 | 适合写 Java 过滤器扩展 |
| 鉴权限流 | 可以做，但业务整合成本较高 | 更适合和微服务体系整合 |
| 性能 | 很强 | 通常不以极限转发性能为核心卖点 |

## 6. 为什么不把所有逻辑都放 Nginx

因为很多逻辑和微服务体系强相关。

比如：

- 根据 Nacos 服务名动态路由
- 解析 token 后透传用户 ID
- 按用户或租户限流
- 根据请求头做灰度
- 记录业务链路追踪 ID

这些能力放在 Gateway 里，通常更容易和 Java 微服务体系结合。

## 7. 为什么不只用 Gateway

因为 Gateway 也不是万能入口。

很多基础能力 Nginx 或 SLB 更适合：

- 公网入口流量接入
- HTTPS 证书处理
- 静态资源
- 基础四层或七层负载均衡
- 更高性能的连接转发

所以两者可以分层配合。

## 8. 面试怎么说

可以这样说：

Nginx 和 Gateway 不是简单替代关系。Nginx 更偏基础设施层，适合做外部入口流量接入、反向代理、静态资源和基础负载均衡；Spring Cloud Gateway 更偏微服务网关，适合和注册中心、鉴权、限流、灰度、链路追踪这些微服务治理能力结合。实际项目中可以是客户端请求先到 Nginx 或 SLB，再进入 Gateway，由 Gateway 根据路由规则转发到后端服务。简单说，Nginx 更靠近网络入口，Gateway 更靠近微服务治理。

## 9. 一句话总结

Nginx 更偏基础流量入口，Gateway 更偏微服务业务网关，两者常常是分层配合而不是互相替代。
