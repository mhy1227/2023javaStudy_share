# Gateway 限流深挖问题

## 1. 这份文档补什么

前面的 Gateway 文档已经讲过：

- 网关为什么需要限流
- 可以按 IP、用户、接口、租户限流
- 网关适合做入口层保护

这份文档补面试里更容易追问的细节：

- 怎么做 IP 限流
- 怎么做用户限流
- 单机限流和分布式限流区别
- 限流怎么防止误伤
- 限流后返回什么
- 限流和 Sentinel / Redis 的关系

## 2. 网关限流一般按哪些维度

常见维度包括：

- IP
- 用户 ID
- 接口路径
- 请求方法
- 租户
- AppKey
- 设备 ID

面试里可以这样说：

> 网关限流不是只有 IP 维度，实际会根据业务选择 IP、用户、接口、租户、AppKey 等维度。IP 更适合防刷和基础保护，用户维度更适合登录后行为限制，接口维度更适合保护高成本接口。

## 3. Gateway 怎么做 IP 限流

基本思路：

1. 请求进入 Gateway
2. 获取客户端 IP
3. 拼接限流 key
4. 判断单位时间内请求次数
5. 超过阈值直接拒绝

限流 key 可以类似：

```text
rate_limit:ip:192.168.1.10
```

面试回答：

> IP 限流一般是在网关过滤器里获取客户端 IP，然后以 IP 为 key 统计单位时间内的请求次数。如果超过阈值，就在网关层直接返回限流响应，不再转发到后端服务。

## 4. 获取真实 IP 要注意什么

如果请求经过 Nginx、SLB 或 CDN，直接取 remote address 可能拿到的是代理 IP。

常见会看：

- `X-Forwarded-For`
- `X-Real-IP`

但也要注意：

> 外部请求头可能被伪造，不能无脑信任客户端传来的 IP 头。

更稳妥的做法：

- 只信任可信代理写入的请求头
- 在入口代理层清理外部伪造头
- 网关从可信头中取真实 IP

## 5. 怎么做用户限流

用户限流通常基于登录态。

流程：

1. 网关先校验 token
2. 解析用户 ID
3. 以用户 ID 拼限流 key
4. 判断访问频率

key 类似：

```text
rate_limit:user:10001
```

面试回答：

> 用户限流一般要先完成鉴权，解析出用户 ID，再按用户维度统计访问次数。相比 IP 限流，用户限流更适合登录后接口，因为同一个 IP 后面可能有多个正常用户。

## 6. IP 限流有什么问题

常见问题：

- NAT 场景下多个用户共用一个出口 IP，容易误伤
- 攻击者可以切换 IP 绕过
- 代理层可能导致取到的不是用户真实 IP
- 只按 IP 无法识别登录用户行为

所以 IP 限流通常只是基础防护。

更完整的限流会结合：

- IP
- 用户
- 接口
- 设备
- AppKey

## 7. 单机限流和分布式限流有什么区别

单机限流：

- 每个 Gateway 实例自己计数
- 实现简单
- 多实例下总流量不准确

分布式限流：

- 多个 Gateway 实例共享计数
- 通常借助 Redis 等外部存储
- 规则更统一
- 有额外网络开销

面试回答：

> 如果 Gateway 只有单实例，内存计数就能做简单限流。但生产一般多实例部署，如果每台都单独限流，总阈值就会失控，所以需要 Redis 或 Sentinel 这类方案做分布式限流。

## 8. 限流算法有哪些

常见算法：

- 固定窗口
- 滑动窗口
- 漏桶
- 令牌桶

面试不用把算法推导太深。

可以这样答：

> 固定窗口简单但边界容易突刺，滑动窗口更平滑，漏桶偏平滑出流，令牌桶允许一定突发流量。实际项目里常用令牌桶或滑动窗口思路。

## 9. 限流后返回什么

常见做法：

- HTTP 状态码 `429 Too Many Requests`
- 业务统一错误码
- 提示“请求过于频繁，请稍后再试”

注意：

> 限流触发后应该在网关直接返回，不要继续转发到后端。

## 10. 限流怎么避免误伤

可以从几个方面：

- 区分接口重要性
- 区分用户等级或租户
- 不同接口配置不同阈值
- 核心接口更谨慎
- 结合监控观察效果
- 灰度调整限流规则

面试回答：

> 限流阈值不能拍脑袋，要结合压测、历史流量和接口成本。上线时最好先灰度观察，避免阈值太低误伤正常用户。

## 11. Gateway 限流和 Sentinel 限流怎么分工

可以这样分：

- Gateway：入口层粗粒度限流
- Sentinel：服务内部更细粒度限流、熔断、降级

比如：

- Gateway 按 IP、用户、接口做入口防刷
- Sentinel 在具体服务内部按资源、热点参数、慢调用做保护

一句话：

> Gateway 先挡入口流量，Sentinel 再保护服务内部资源。

## 12. 面试怎么完整回答“网关怎么限流”

可以这样说：

网关限流一般是在请求进入后端服务之前做入口保护。常见维度包括 IP、用户、接口、租户或 AppKey。比如做 IP 限流时，网关会获取客户端真实 IP，拼成限流 key，统计单位时间内请求次数，超过阈值就直接返回 429 或业务错误码，不再转发到后端。生产环境 Gateway 通常是多实例部署，所以如果要全局限流，一般不能只用本地内存计数，而要结合 Redis、Sentinel 或网关限流组件做分布式限流。同时要注意真实 IP 获取、防请求头伪造、NAT 误伤和阈值灰度调整。

## 13. 一句话总结

Gateway 限流不是“按 IP 拦一下”这么简单，重点是维度选择、真实 IP、分布式计数、误伤控制和入口保护。
