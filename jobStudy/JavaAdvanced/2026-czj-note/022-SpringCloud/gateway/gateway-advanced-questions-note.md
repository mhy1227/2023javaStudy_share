## 2. 什么是动态路由

动态路由指的是：

> 网关路由规则不完全写死在配置文件里，而是可以从配置中心、数据库或注册中心动态加载和刷新。

静态路由适合服务少、规则稳定的场景。

动态路由适合：

- 服务数量较多
- 路由规则经常调整
- 需要灰度发布
- 需要按租户或版本切流
- 不希望每次改路由都重启网关

面试可以这样说：

> Gateway 可以配置静态路由，也可以结合配置中心或数据库做动态路由。动态路由的价值是让路由规则调整不依赖重新发布网关，但也要注意路由变更的权限控制、回滚和一致性。

## 3. Gateway 为什么经常处理跨域

跨域问题通常发生在浏览器访问后端接口时。

微服务拆分后，如果每个服务都单独处理跨域，会导致规则不统一。

Gateway 作为统一入口，更适合集中处理：

- 允许哪些域名访问
- 允许哪些请求方法
- 允许哪些请求头
- 是否允许携带凭证

但要注意：

- 不要生产环境无脑放开所有来源
- 不要随便允许所有请求头
- 和鉴权逻辑要配合

## 4. 请求头透传有什么安全问题

网关经常会把用户信息透传给下游。

比如：

- `userId`
- `tenantId`
- `role`
- `traceId`

风险在于：

> 如果外部请求本身就伪造了这些内部请求头，而网关没有清理，下游可能会误信伪造身份。

所以安全做法是：

1. 网关先删除外部传入的内部身份头
2. 网关校验 token
3. 网关根据校验结果重新写入可信身份头
4. 下游只信任来自网关的内部头

面试里这是一个很加分的点。

## 5. 网关过滤器里适合做什么

适合做：

- token 校验
- 基础权限判断
- 请求日志
- 链路追踪
- 限流
- 黑白名单
- 请求头改写
- 路径重写

不适合做：

- 大量数据库查询
- 复杂业务规则
- 大文件内容处理
- 长时间阻塞操作
- 强业务状态变更

原因是 Gateway 是入口层，所有请求都会经过它。

如果过滤器逻辑过重，会让网关变成瓶颈。

## 6. Gateway 高可用怎么做

Gateway 本身也需要高可用。

常见做法：

- 多实例部署 Gateway
- 前面挂 Nginx 或 SLB
- 开启健康检查
- 异常实例自动摘除
- 做监控告警
- 对关键路由做限流保护

可以这样回答：

> Gateway 是统一入口，如果单实例部署就会成为单点。实际项目里一般会多实例部署，并由上层负载均衡分发流量，同时配合健康检查、监控、限流和日志排查。

## 7. Gateway 出问题怎么排查

可以按这条链路排查：

1. 请求有没有到达 Gateway
2. 路由断言是否匹配
3. 过滤器是否拦截
4. 鉴权是否通过
5. 限流是否触发
6. 目标服务是否存在可用实例
7. 后端服务是否超时或报错
8. 响应是否被网关过滤器改写

常见问题包括：

- 路径匹配错误
- 服务名写错
- 注册中心没有实例
- token 校验失败
- 跨域配置错误
- 请求头被过滤
- 后端服务超时

## 8. Gateway 和业务服务权限怎么分工

网关适合做粗粒度权限。

比如：

- 是否登录
- token 是否有效
- 是否黑名单
- 是否允许访问某类接口

业务服务适合做细粒度权限。

比如：

- 当前用户能否查看这笔订单
- 当前用户是否能删除这个文件
- 当前角色是否能修改某个业务状态

面试可以这样说：

> 网关做入口层通用鉴权，业务服务做业务级细粒度校验。不能把所有权限都塞到网关，否则网关会变重，也很难理解具体业务规则。

## 9. 一句话总结

Gateway 进阶重点是：路由要能动态治理，过滤器要控制边界，请求头要防伪造，网关本身也要高可用和可排查。
