# Gateway 基础

## 1. Gateway 是什么

`Spring Cloud Gateway` 是 Spring Cloud 体系里的 API 网关组件。

它通常位于外部请求和后端微服务之间，作为系统的统一入口。

可以先这样理解：

> 客户端请求不直接访问各个微服务，而是先进入 Gateway，再由 Gateway 根据路由规则转发到对应服务。

## 2. 为什么微服务需要网关

单体项目里，前端通常只访问一个后端应用。

微服务拆分之后，后端会变成多个服务：

- 用户服务
- 订单服务
- 商品服务
- 支付服务
- 文件服务

如果前端直接调用这些服务，会带来很多问题：

1. 服务地址暴露给客户端
2. 前端需要关心多个服务地址
3. 鉴权、限流、日志等逻辑重复分散
4. 服务拆分调整会影响前端
5. 无法统一做入口治理

所以需要一个统一入口，把通用能力集中在入口层处理。

## 3. Gateway 解决什么问题

Gateway 主要解决这些问题：

- 统一入口
- 路由转发
- 统一鉴权
- 统一限流
- 请求日志
- 灰度发布
- 跨域处理
- 请求头改写
- 服务屏蔽

面试里不要只说“网关负责转发请求”。

更完整的说法是：

> Gateway 不只是转发层，它还承接微服务入口处的通用治理能力。

## 4. Gateway 在架构里的位置

典型链路可以这样理解：

```text
客户端
  |
Nginx / SLB
  |
Spring Cloud Gateway
  |
后端微服务
```

说明：

- `Nginx / SLB` 更偏外部流量接入和负载均衡
- `Gateway` 更偏微服务内部路由和业务网关治理
- 后端微服务只专注业务逻辑

实际项目里不一定每层都有，但面试时要知道它们的职责边界。

## 5. Gateway 的核心概念

Gateway 常见核心概念有三个：

- 路由
- 断言
- 过滤器

### 5.1 路由

路由决定请求要转发到哪里。

比如：

- `/api/user/**` 转发到用户服务
- `/api/order/**` 转发到订单服务
- `/api/file/**` 转发到文件服务

### 5.2 断言

断言用于判断请求是否匹配某条路由。

可以根据这些条件判断：

- 路径
- 请求方法
- 请求头
- 参数
- Host
- 时间

### 5.3 过滤器

过滤器用于在请求转发前后做处理。

常见处理包括：

- 鉴权
- 打日志
- 添加请求头
- 修改路径
- 限流
- 统一响应处理

## 6. Gateway 和注册中心是什么关系

Gateway 可以和注册中心配合。

比如服务注册到 Nacos 后，Gateway 可以根据服务名把请求转发给目标服务。

这时就不需要在网关里写死具体 IP 和端口。

可以理解为：

> 注册中心提供服务实例，Gateway 根据路由规则把请求转发到对应服务。

## 7. Gateway 是不是单点

如果只部署一个 Gateway，它确实会有单点风险。

实际项目中一般会：

- 多实例部署 Gateway
- 前面挂 Nginx 或 SLB
- 配合健康检查
- 做合理限流和监控

所以 Gateway 本身也要高可用部署，不能只部署一台。

## 8. 面试怎么说

可以这样说：

`Spring Cloud Gateway` 是微服务体系里的 API 网关组件，通常作为后端服务的统一入口。微服务拆分后，如果客户端直接访问多个服务，会导致服务地址暴露、前端调用复杂、鉴权和限流等逻辑重复分散。所以 Gateway 会统一接收请求，根据路由规则转发到对应服务，并在入口层统一做鉴权、限流、日志、跨域、灰度等通用治理能力。它不只是转发请求，还承担微服务入口治理的职责。

## 9. 一句话总结

Gateway 的核心作用，是给微服务系统提供统一入口，并在入口层承接路由转发和通用治理能力。
