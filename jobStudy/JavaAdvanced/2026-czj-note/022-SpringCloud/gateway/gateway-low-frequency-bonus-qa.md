# Gateway 低频但加分问题

## 1. 这份文档适合什么时候看

如果基础问题已经能回答，可以再看这份。

这些问题不一定每场都会问，但答得出来会显得你不是只背概念。

重点是：

- 知道网关的边界
- 知道网关的风险
- 知道项目里怎么落地

## 2. Gateway 为什么不适合做太重的业务逻辑？

回答方向：

- Gateway 是统一入口，所有请求都会经过它
- 逻辑太重会影响入口吞吐
- 业务规则经常变化，放在网关会增加维护成本
- 网关应该做通用入口治理
- 复杂业务判断应该放在业务服务

可以这样答：

> 网关适合做鉴权、限流、日志、路由这类通用入口逻辑，不适合做复杂业务逻辑。因为网关是统一入口，如果在网关里做大量数据库查询或复杂业务判断，会影响所有请求的入口性能，也会让网关和业务强耦合。

## 3. Gateway 可以直接查数据库做鉴权吗？

回答方向：

- 可以做，但不建议高频复杂查询都放网关
- 基础权限可以缓存或解析 token
- 复杂权限交给业务服务
- 如果必须查，要考虑缓存、超时和降级

可以这样答：

> 网关可以做基础鉴权，但不建议每个请求都在网关里查数据库做复杂权限判断。更常见的是网关校验 token、解析基础身份信息，复杂资源权限由业务服务判断。如果网关确实需要查权限，也要配合缓存、超时和降级，避免拖慢入口。

## 4. Gateway 和服务内部拦截器有什么区别？

回答方向：

- Gateway 是系统入口层
- 服务拦截器是单个服务内部
- Gateway 适合做统一入口逻辑
- 服务拦截器适合做服务内部上下文和业务校验

可以这样答：

> Gateway 是请求进入系统的统一入口，适合做登录校验、限流、跨域、路由等通用处理。服务内部拦截器只对当前服务生效，更适合做用户上下文构建、接口权限、业务校验等服务内部逻辑。

## 5. Gateway 怎么避免请求头伪造？

回答方向：

- 外部请求可能伪造内部用户头
- 网关先删除敏感内部头
- token 校验后重新写入可信头
- 下游只信任网关写入的信息

可以这样答：

> 网关透传用户信息时不能直接相信外部请求头。安全做法是网关先清理外部传入的内部身份头，然后校验 token，再把解析出来的用户 ID、租户 ID 等信息重新写入内部请求头。下游服务只信任经过网关写入的身份信息。

## 6. Gateway 怎么做接口版本控制？

回答方向：

- 路径版本
- 请求头版本
- 灰度标识
- 不同版本路由到不同服务或不同实例

可以这样答：

> 可以通过路径或者请求头做版本控制，比如 `/api/v1/**` 和 `/api/v2/**` 路由到不同服务，或者根据请求头里的版本标识转发到不同版本实例。这样可以支持接口升级和灰度发布，但版本规则要统一，否则路由会越来越乱。

## 7. Gateway 怎么处理 WebSocket？

回答方向：

- Gateway 可以代理 WebSocket
- 要确保路由协议和连接升级配置正确
- WebSocket 是长连接，要关注连接数和资源占用

可以这样答：

> Gateway 可以代理 WebSocket，但它和普通 HTTP 请求不完全一样，因为 WebSocket 是长连接，需要关注连接升级、连接数、超时和资源占用。如果长连接规模比较大，还要单独评估网关承载能力。

## 8. Gateway 日志应该记录什么？

回答方向：

- 请求路径
- 方法
- traceId
- 用户标识
- 路由目标
- 响应状态
- 耗时
- 异常原因

注意不要记录：

- 明文 token
- 密码
- 敏感个人信息
- 大请求体

可以这样答：

> 网关日志适合记录请求入口信息，比如路径、方法、traceId、用户 ID、路由目标、状态码和耗时，用于排查链路问题。但不要记录明文 token、密码、大请求体等敏感或高成本信息。

## 9. Gateway 限流失败时返回什么？

回答方向：

- 返回明确的限流状态
- 通常可以返回 429
- 响应体给出统一错误码
- 不要让请求继续打到后端

可以这样答：

> 限流触发后应该在网关直接返回统一错误响应，比如 429 或业务统一错误码，提示请求过于频繁，而不是继续转发到后端。这样才能达到保护后端服务的目的。

## 10. Gateway 配置变更有什么风险？

回答方向：

- 路由错误会影响请求转发
- 鉴权配置错误可能放开敏感接口
- 限流配置错误可能误伤正常用户
- 动态路由需要回滚机制

可以这样答：

> 网关配置属于入口层配置，风险比较高。路由配错会导致请求打错服务，鉴权规则配错可能放开敏感接口，限流规则配错可能误伤正常用户。所以动态配置要有审核、灰度、回滚和监控。

## 11. Gateway 和 BFF 是一回事吗？

回答方向：

- 不是一回事
- Gateway 更偏统一入口和治理
- BFF 更偏面向特定前端聚合接口
- 小项目可能混在一起，大项目要区分

可以这样答：

> Gateway 更偏统一入口、路由、鉴权、限流这些通用治理；BFF 更偏为某一类前端聚合和裁剪接口，比如移动端 BFF、管理后台 BFF。两者可以共存，但不应该把 Gateway 过度做成复杂 BFF。

## 12. 一句话总结

Gateway 低频加分题的核心，是讲清楚边界、安全、高可用、配置风险和可排查性。
