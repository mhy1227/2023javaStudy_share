# Gateway 0-3 分钟回答包

## 1. 请你介绍一下 Spring Cloud Gateway

`Spring Cloud Gateway` 是 Spring Cloud 体系里的 API 网关组件，通常作为微服务系统的统一入口。

微服务拆分后，后端不再是一个单体应用，而是拆成用户、订单、商品、支付、文件等多个服务。如果客户端直接访问这些服务，会导致服务地址暴露、前端需要维护多个后端地址、鉴权和限流逻辑重复分散，而且后端服务拆分或迁移时也会影响客户端。

Gateway 的作用就是把这些请求先统一接进来，再根据路由规则转发到具体微服务。同时它还可以在入口层统一做一些通用能力，比如鉴权、限流、日志、跨域、灰度发布、请求头改写等。所以 Gateway 不只是转发层，更是微服务入口治理的一部分。

## 2. Gateway 的核心概念是什么？

Gateway 里最核心的概念是路由、断言和过滤器。

路由决定请求要转发到哪个服务，比如 `/api/user/**` 转发到用户服务，`/api/order/**` 转发到订单服务。

断言用于判断请求是否满足某条路由规则，比如根据路径、请求方法、请求头、参数、Host 等条件判断。只有断言匹配成功，请求才会进入这条路由。

过滤器用于在请求转发前后插入通用处理逻辑，比如鉴权、限流、日志、请求头改写、路径重写、统计耗时等。一次请求进入 Gateway 后，会先根据断言匹配路由，然后执行过滤器，再转发到后端服务，响应返回后还可以继续执行响应阶段的过滤逻辑。

可以总结为：路由决定去哪里，断言决定是否匹配，过滤器决定前后做什么。

## 3. Gateway 和 Nginx 有什么区别？

Nginx 和 Gateway 不是简单替代关系，它们定位不一样。

Nginx 更偏基础设施层，常用于外部入口流量接入、反向代理、静态资源、HTTPS 证书处理和基础负载均衡。它更靠近网络入口。

Gateway 更偏微服务网关，适合和注册中心、鉴权、限流、灰度、链路追踪这些微服务治理能力结合。比如 Gateway 可以根据服务名动态路由到注册中心里的服务实例，也可以在 Java 体系里更方便地写过滤器处理 token、用户信息、灰度标识等。

实际项目里两者可以分层配合：客户端请求先到 Nginx 或 SLB，再进入 Gateway，最后由 Gateway 转发到后端微服务。简单说，Nginx 更偏外部流量入口，Gateway 更偏微服务内部治理。

## 4. Gateway 怎么做鉴权和用户信息传递？

一般请求进入 Gateway 后，会先通过全局过滤器做基础鉴权。网关从请求头或 Cookie 中获取 token，校验 token 是否存在、是否过期、是否合法。如果 token 无效，网关直接返回未登录或无权限，不再把请求打到后端服务。

如果校验通过，网关可以解析出用户 ID、租户 ID、角色等信息，再通过内部请求头传给下游服务。下游服务拿到这些信息后，可以构建用户上下文，避免每个服务都重复解析 token。

但这里要注意边界：网关适合做登录校验、基础权限和入口拦截，复杂业务权限不能全部放在网关。比如订单是否属于当前用户、文件目录是否有访问权限，这类强业务规则仍然应该由业务服务自己校验。

另外还要避免请求头伪造。网关应该清理外部传入的内部用户头，再写入自己校验后的可信用户信息。

## 5. Gateway 怎么做限流和灰度？

限流的目标是保护后端系统。因为 Gateway 是统一入口，所以适合在请求进入后端服务之前先做流量控制。

限流可以按不同维度做，比如 IP、用户、接口、服务、租户等。常见限流算法包括令牌桶、漏桶、固定窗口、滑动窗口。面试里不一定要展开所有算法，但要讲清楚限流不是为了拒绝用户，而是在高流量或异常流量下保护后端服务不被打垮。

灰度发布则是把一部分流量先导到新版本服务，而不是一次性让所有用户访问新版本。Gateway 可以根据请求头、用户 ID、IP、租户、灰度标识等条件选择不同路由。比如带有 `gray=true` 请求头的请求转发到新版本，普通请求仍然访问旧版本。这样可以降低发布风险。

## 6. Gateway 会不会成为系统瓶颈？

会有这个风险，因为 Gateway 是统一入口，很多请求都会经过它。

所以实际项目里 Gateway 不能单实例部署，一般会多实例部署，前面再挂 Nginx 或 SLB 做负载均衡。同时要配合健康检查、监控、限流和日志，及时发现入口层的延迟、错误率和流量异常。

另外也不能把大量复杂业务逻辑都塞到 Gateway，否则网关会越来越重，影响入口性能和可维护性。Gateway 更适合做通用入口治理，复杂业务逻辑应该放在对应业务服务里。

## 7. 项目里怎么介绍 Gateway？

可以这样说：

在项目里，Gateway 主要作为后端微服务的统一入口。前端请求先进入 Gateway，再根据路径路由到不同服务，比如用户、订单、文件等服务。网关层会统一处理跨域、token 校验、请求日志和基础限流，校验通过后把用户 ID 等必要信息通过请求头传给下游服务。

对于复杂业务权限，比如文件目录权限、订单归属校验，不会完全放在网关，而是由业务服务继续判断。Gateway 本身也会多实例部署，前面通过 Nginx 或负载均衡入口分发流量，避免单点问题。

## 8. 一段完整口述版

`Spring Cloud Gateway` 是微服务体系里的 API 网关组件，主要作为后端服务的统一入口。微服务拆分后，如果客户端直接访问多个服务，会导致服务地址暴露、前端调用复杂、鉴权限流等逻辑重复分散，所以需要 Gateway 在入口层统一接收请求，再根据路由规则转发到具体服务。

Gateway 的核心概念是路由、断言和过滤器。路由决定请求转发到哪里，断言判断请求是否匹配某条路由，过滤器负责在请求转发前后做通用处理，比如鉴权、限流、日志、跨域、请求头改写等。

它和 Nginx 不是简单替代关系。Nginx 更偏基础设施入口，适合做反向代理、静态资源和基础负载均衡；Gateway 更偏微服务网关，适合和注册中心、鉴权、限流、灰度、链路追踪等治理能力结合。实际项目里可以是请求先经过 Nginx 或 SLB，再进入 Gateway，最后转发到后端服务。

同时 Gateway 作为统一入口也可能成为瓶颈，所以一般要多实例部署，并配合监控、限流和健康检查。网关适合放通用入口逻辑，但不适合承载复杂业务逻辑。
