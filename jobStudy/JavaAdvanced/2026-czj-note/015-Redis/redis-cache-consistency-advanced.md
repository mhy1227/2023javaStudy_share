# Redis 缓存一致性进阶

## 1. 这篇文档解决什么问题

前面的 `redis-cache-problem-note.md` 已经提到了缓存一致性，但还是偏总览。

这一篇专门补：

- 为什么更新数据库和缓存天然容易不一致
- 先删缓存还是先更新数据库
- 为什么常说“更新数据库，再删缓存”
- 延时双删怎么理解
- 删缓存失败怎么办
- 为什么有人会引入 MQ 或 binlog

## 2. 先记住一个前提

MySQL 和 Redis 不是同一个原子事务系统。

所以你要先接受一个现实：

> 缓存一致性很多时候不是追求绝对零偏差，而是尽量缩小不一致窗口，并在关键链路上做兜底。

## 3. 为什么双写天然容易出问题

因为一次业务更新往往涉及两套系统：

- 更新数据库
- 更新或删除缓存

只要不是一个原子动作，就可能出现：

- 一步成功，一步失败
- 或者并发窗口下顺序交叉

这就是缓存一致性的根源。

## 4. 为什么不推荐“先删缓存，再更新数据库”

这个顺序不是完全不能用，但风险更大。

典型问题是：

1. 线程 A 先删缓存
2. 线程 B 读请求进来，缓存 miss
3. 线程 B 从数据库读到旧值，并把旧值重新写回缓存
4. 线程 A 这时才更新数据库新值

最后结果就变成：

- 数据库是新值
- 缓存却被旧值回填了

这就是经典并发窗口问题。

## 5. 为什么更常说“先更新数据库，再删缓存”

因为这个顺序更容易把不一致窗口压小。

常见流程是：

1. 先把数据库更新成新值
2. 再删除缓存

这样后续读请求如果 miss：

- 回源数据库时读到的就是新值
- 再重建缓存时也更容易得到正确结果

### 面试怎么说

> 相比先删缓存再更新数据库，更常见的方案是先更新数据库再删除缓存，因为这样后续缓存重建时更容易读到数据库中的最新值。

## 6. 这是不是就完全一致了

也不是。

因为仍然存在场景：

- 数据库更新成功
- 删除缓存失败

这样缓存里还是旧数据。

所以“更新数据库，再删缓存”是更优实践，不是绝对完美答案。

## 7. 延时双删怎么理解

延时双删的思路可以理解成：

1. 更新数据库
2. 删一次缓存
3. 过一小段时间再删一次缓存

它想解决的是：

- 第一次删缓存之后
- 在并发读请求下，旧值可能又被短暂回填

于是通过第二次删除，再把这类脏缓存清掉。

### 7.1 它的本质

它不是强一致方案，而是：

- 通过“补一刀”缩小并发窗口带来的旧值残留风险

### 7.2 它的问题

延时多久并没有绝对标准。

如果延时太短：

- 可能来不及覆盖并发窗口

如果延时太长：

- 不一致窗口仍然存在较久

所以它更像经验型折中方案。

## 8. 删缓存失败怎么办

这是高频工程题。

思路一般不是“祈祷别失败”，而是做补偿。

常见做法包括：

- 重试
- 异步重试队列
- 记录失败日志后补偿
- 借助 MQ 做异步删除

面试里你只要讲出：

> 删除缓存失败要有补偿机制，而不是只做一次 best effort。

通常就已经比纯八股更稳了。

## 9. 为什么有人会引入 MQ

MQ 的核心价值不是“更高级”，而是：

- 把删缓存动作异步化
- 让失败可以重试
- 把瞬时失败和核心写流程适当解耦

比如：

- 主流程先更新数据库
- 发消息通知缓存删除
- 消费者负责异步删缓存，失败可重试

这更适合复杂系统。

## 10. 为什么有人会提到 binlog

这是再往上一层的工程化方案。

可以粗略理解成：

- 数据库变更已经是事实
- 那就从数据库变更日志出发
- 再驱动缓存失效或异步同步动作

这种思路的优点是：

- 更贴近“以数据库为准”
- 对复杂链路的补偿和解耦更自然

但它也会带来更高的系统复杂度。

## 11. 读多写少和写多读多，一致性设计一样吗

不一样。

### 11.1 读多写少

更容易接受：

- 短暂不一致
- 缓存延迟更新

### 11.2 写频繁、状态敏感

就要更谨慎：

- 更关注删除失败补偿
- 更关注热点并发更新
- 更关注是否需要读主、旁路校验或状态兜底

所以缓存一致性不是死记模板，而是和业务敏感度绑定的。

## 12. 什么时候干脆不该强依赖 Redis 缓存结果

如果是这种场景：

- 订单支付状态
- 库存扣减结果
- 核心资金状态

那你不能把缓存值当成绝对最终真相。

更稳的思路通常是：

- Redis 承担高性能读取
- 数据库承担最终权威状态
- 关键链路做回源校验或幂等兜底

## 13. 高频追问小抄

### 13.1 为什么不推荐先删缓存再更新数据库

因为并发读请求可能把旧值重新写回缓存。

### 13.2 为什么更推荐先更新数据库再删缓存

因为后续缓存重建时更容易读到新值。

### 13.3 更新数据库后删缓存失败怎么办

要有补偿机制，比如重试、异步任务、MQ。

### 13.4 延时双删是不是银弹

不是，只是经验性地缩小并发窗口风险。

### 13.5 缓存一致性能不能做到绝对强一致

理论上代价很高，实际工程里更多是接受短暂不一致并做补偿。

## 14. 一个比较稳的口语版回答

> Redis 和 MySQL 的缓存一致性我一般不会讲成绝对强一致，因为两者天然不是同一个原子事务系统。工程上更常见的做法是先更新数据库，再删除缓存，这样后续缓存 miss 重建时更容易拿到数据库最新值。它也不是完全没有风险，比如删缓存失败或者并发读把旧值重新回填，所以有时会配合延时双删、异步重试、MQ 甚至基于 binlog 的补偿方案去进一步缩小不一致窗口。核心不是追求理论完美，而是根据业务对一致性的敏感程度设计合适的补偿和兜底。

## 15. 一句话收尾

> 缓存一致性题最关键的，不是背一句“先更新数据库再删缓存”，而是你能不能说明白为什么这么做、它还有什么漏洞、你准备怎么补。
