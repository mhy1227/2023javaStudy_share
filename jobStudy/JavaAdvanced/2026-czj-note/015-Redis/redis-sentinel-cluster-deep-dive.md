# Redis 哨兵与 Cluster 深挖

## 1. 这篇文档解决什么问题

前面的高可用文档已经把主从、哨兵、集群的主线拉顺了。

这一篇继续往下补的是面试里更容易被追问的点：

- 哨兵到底监控什么
- 主节点故障后怎么切主
- 脑裂为什么会出现
- Cluster 和主从、哨兵到底差在哪
- `16384` 槽位怎么理解
- 扩容迁移时大概发生了什么

这篇不追求源码级细节，而是追求“你能不能把高可用架构讲得像真的理解过”。

## 2. 哨兵的定位先记一句话

哨兵不是存数据的，它是做管理和故障转移的。

你可以先记住：

> 哨兵的核心作用，是监控 Redis 主从节点状态，在主节点故障时完成故障判定、选主和故障转移。

## 3. 哨兵主要做哪几件事

### 3.1 监控

哨兵会持续关注：

- 主节点是否存活
- 从节点是否正常
- 复制关系是否还健康

### 3.2 通知

一旦发现异常，哨兵会把状态变化通知给其他哨兵或客户端侧感知模块。

### 3.3 自动故障转移

如果主节点真的不可用了，哨兵会推动完成：

- 从节点里挑一个出来
- 把它提升成新的主节点
- 让其他从节点重新指向新的主节点

## 4. 为什么哨兵不能只有一个

如果只有一个哨兵，那么：

- 它自己挂了就没人监控
- 它自己误判也没人纠正

所以哨兵通常是多个节点一起工作，本质上是想降低单点误判风险。

面试里你不用把投票过程说得特别细，但要知道：

> 哨兵模式强调的是多哨兵协同判断，而不是某一个哨兵单独拍板。

## 5. 主观下线和客观下线怎么理解

这是哨兵里很高频的术语题。

### 5.1 主观下线

某个哨兵自己觉得：

- 这个 Redis 节点联系不上了
- 心跳超时了

这时它会先认为对方“可能挂了”。

### 5.2 客观下线

如果不只是一个哨兵这么觉得，而是足够多的哨兵达成类似判断，才会进入更正式的“这个主节点真的不可用”的判定阶段。

面试里可以这么说：

> 主观下线是单个哨兵的判断，客观下线是多个哨兵在一定条件下形成的更可信结论。

## 6. 哨兵切主的大体流程

不要一上来背术语，先按流程说：

1. 发现主节点疑似故障
2. 多个哨兵协同确认故障
3. 在从节点里选一个相对更合适的节点
4. 把它提升为新的主节点
5. 让其他从节点改为复制新主节点
6. 客户端后续流量逐步切到新的主节点

这个流程说清楚，面试官通常就知道你不是只会说“哨兵能自动切换”。

## 7. 选新主节点时一般会看什么

不需要把细节背成源码，但你要知道不会“随便挑一个从节点”。

通常会综合考虑：

- 从节点和主节点的数据同步情况
- 复制偏移量谁更接近主节点
- 网络状态是否稳定
- 节点自身是否健康

一句话记：

> 新主节点一般会优先选择数据更新、状态更稳定、复制进度更靠前的从节点。

## 8. 为什么说哨兵不是零风险

很多人把哨兵讲成“有了它就万事大吉”，这不对。

哨兵解决的是：

- 自动发现问题
- 自动故障转移

但它没有消灭：

- 网络分区
- 误判
- 切主窗口内的数据不一致
- 切主过程中可能的少量数据丢失

所以你要把它理解成：

> 哨兵是在降低故障恢复成本，不是在承诺绝对零损失和绝对零中断。

## 9. 脑裂怎么理解

脑裂是 Redis 高可用题里的经典追问。

你可以这样理解：

- 原主节点没有完全坏掉
- 但因为网络分区或通信异常，哨兵以为它挂了
- 于是又选出了一个新的主节点
- 短时间内像是出现了两个主节点

这就可能带来：

- 客户端写到了不同节点
- 数据版本分叉
- 恢复时要以新的主链路为准

### 9.1 面试怎么说

> 脑裂本质上是网络分区或故障判定窗口导致的双主风险，高可用方案能降低风险，但很难彻底消除这种边界场景。

## 10. 为什么 Cluster 不等于“主从 + 哨兵”

这是很多人的混淆点。

### 10.1 主从 + 哨兵解决什么

- 数据副本
- 自动切主

### 10.2 Cluster 还额外解决什么

- 数据分片
- 单机容量瓶颈
- 更大规模下的横向扩展

所以 Cluster 的关键不只是高可用，而是：

> 它把数据拆到多个节点上去存和处理，而不是只围着一个主节点做主从复制。

## 11. `16384` 槽位怎么理解

Redis Cluster 不是随便“按机器分数据”，它有槽位概念。

你可以把它理解成：

- 整个集群先抽象出 `16384` 个槽位
- 每个 key 通过哈希计算后，归属于其中某个槽位
- 不同节点负责不同槽位范围

### 11.1 为什么面试会问 `16384`

因为这是 Cluster 最常见的辨识度问题。

你不用死抠“为什么不是别的数字”的底层历史，只要知道：

- 它是 Redis Cluster 用来做分片映射的固定槽位空间
- key 先映射到槽位，再由槽位映射到节点

### 11.2 面试怎么说

> Redis Cluster 通过 `16384` 个槽位把数据分片，key 先算出属于哪个槽，再由对应节点负责该槽的数据。

## 12. Cluster 和一致性哈希是不是一回事

严格说不完全一样。

面试里更稳的说法是：

- 它们都在解决分布式场景下的数据分布问题
- 但 Redis Cluster 更常直接按槽位机制去理解

也就是说别上来就把 Cluster 简化成“就是一致性哈希”，这样容易被继续追问卡住。

## 13. Cluster 扩容迁移大概怎么理解

扩容时不是把全量数据一股脑重来一遍，而是：

- 把部分槽位从旧节点迁移到新节点
- 对应槽位的数据逐步跟着迁过去

所以 Cluster 的横向扩展能力，本质上来自：

- 槽位是可重新分配的
- 数据跟着槽位迁移

这也是为什么它比单纯主从结构更适合大规模扩容。

## 14. MOVED 和 ASK 可以怎么粗略理解

这两个属于 Cluster 追问里的加分项。

你不用抠协议细节，先记住这层意思：

- 客户端访问的节点不一定就是目标 key 所在槽位的节点
- 节点会告诉客户端应该去哪里找

可以先粗略理解成：

- `MOVED` 更像“这个槽已经稳定归别的节点了，你以后去那边”
- `ASK` 更像“这个槽位迁移中，这次你先去那边处理”

## 15. 哨兵和 Cluster 分别适合什么场景

### 15.1 更偏哨兵

适合：

- 数据量还没大到必须分片
- 更关注高可用
- 单主架构还能承载容量和写入压力

### 15.2 更偏 Cluster

适合：

- 数据量明显增大
- 单机内存不够
- 单节点压力过大
- 需要水平扩容

一句话记忆：

> 哨兵更像单主体系的高可用增强，Cluster 更像分片 + 高可用的一体化方案。

## 16. 高频追问小抄

### 16.1 哨兵能保证零丢失吗

不能。

它主要保证更快恢复，而不是绝对不丢数据。

### 16.2 哨兵模式能解决容量问题吗

不能从根上解决。

它主要解决高可用，不解决单机容量上限。

### 16.3 Cluster 还需要主从吗

需要。

Cluster 里的每个分片通常也会有副本，用来承担高可用能力。

### 16.4 Cluster 为什么更复杂

因为它不只要解决故障切换，还要解决：

- 数据分片
- 槽位分配
- 槽位迁移
- 客户端路由

## 17. 一个比较稳的口语版回答

> Redis 高可用如果继续往下讲，我会分成两层。第一层是哨兵，它建立在主从复制之上，核心作用是监控节点状态、完成故障判定、自动切主和通知从节点重新跟随，所以它解决的是单主体系下的高可用问题。第二层是 Cluster，Cluster 不只是高可用，还解决了数据分片和横向扩容问题。它通过 `16384` 个槽位把 key 映射到不同节点，每个节点负责一部分槽位，这样就能突破单机容量和性能瓶颈。需要注意的是，不管是哨兵还是 Cluster，都不能理解成绝对零风险方案，像网络分区、脑裂、切换窗口内不一致这些问题仍然要结合业务去兜底。

## 18. 一句话收尾

> Redis 高可用深挖题的关键，不是背“主从、哨兵、集群”三个名词，而是把“谁解决复制、谁解决切主、谁解决分片扩容”讲顺。
