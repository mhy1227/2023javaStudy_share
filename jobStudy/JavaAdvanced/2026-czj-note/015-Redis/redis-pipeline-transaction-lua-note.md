# Redis Pipeline、事务与 Lua

## 1. 这篇文档解决什么问题

Redis 面试里有一类题很容易被答乱：

- `pipeline` 是什么
- Redis 事务是什么
- `Lua` 脚本又是干什么的

很多人会把这三者混在一起，甚至都说成“保证原子性”的工具。

这一篇就是专门把这条线拉顺：

- `pipeline` 解决什么
- Redis 事务解决什么
- `Lua` 解决什么
- 三者分别适合什么场景
- 为什么分布式锁和 Lua 经常连在一起问

## 2. 先记一句总纲

这三个东西看起来都像“把多个操作放一起”，但它们的重点完全不一样：

- `pipeline` 重点是减少网络往返，提高吞吐
- Redis 事务重点是把一组命令按顺序提交执行
- `Lua` 重点是把多步逻辑放到 Redis 端一次原子完成

面试里你先把这句说出来，后面基本就不会乱。

## 3. `pipeline` 是什么

`pipeline` 可以理解成：

- 一次性把多条 Redis 命令连续发出去
- 不必每发一条都等返回再发下一条

它的核心价值是：

- 减少网络来回次数
- 提高批量操作吞吐

### 3.1 它解决的核心问题

很多 Redis 操作本身非常快，真正慢的地方往往在：

- 网络往返
- 大量请求来回传输

如果你要连续执行很多条命令，每条都单独请求和响应，就会有很多 RTT 开销。

这时 `pipeline` 的价值就出来了。

### 3.2 面试怎么说

> `pipeline` 的核心不是保证原子性，而是通过批量发送命令减少网络往返，提高吞吐量。

## 4. `pipeline` 不解决什么问题

这是最容易翻车的点。

`pipeline` 不等于事务，也不等于原子执行。

它不保证：

- 多条命令像一个整体那样原子完成
- 中间不会被别人插入
- 一条失败就整体回滚

所以你一定不要把它答成“批量事务”。

## 5. 什么场景适合 `pipeline`

比较常见的是：

- 批量写缓存
- 批量读多个 key
- 批量更新统计值

只要你的关注点是：

- 性能
- 吞吐
- 降低网络交互成本

就可以考虑 `pipeline`。

## 6. Redis 事务是什么

Redis 事务通常会和：

- `MULTI`
- `EXEC`
- `WATCH`

这些关键词连在一起问。

你可以先粗略理解成：

- 先把一组命令放进队列
- 最后统一提交执行

### 6.1 它解决的核心问题

Redis 事务更像是：

- 把一组命令按顺序组织起来
- 在提交时连续执行

它不是数据库事务那种完整 ACID 语义，也不是你想象中的“出错就自动回滚”。

### 6.2 面试怎么说

> Redis 事务本质上是把多条命令先入队，再统一提交执行，它更强调命令组织和顺序提交，不等同于关系型数据库那种强事务模型。

## 7. Redis 事务和数据库事务最大的不同

这个问题很高频。

你至少要知道：

- Redis 事务一般不提供数据库那种完整回滚语义
- 它不是典型意义上的强 ACID 事务

所以你不要说：

> Redis 事务和 MySQL 事务差不多

这是非常容易被追问翻车的。

更稳的说法是：

> Redis 事务更轻量，重点在于命令的批量提交和有条件执行，不是关系型数据库那种完整事务保证。

## 8. `WATCH` 怎么粗略理解

`WATCH` 常被拿来问“乐观锁”。

可以把它理解成：

- 先监听某些 key
- 如果在提交事务前，这些 key 被别人改过
- 那这次事务提交就可能失败

所以它更像：

- 乐观并发控制

### 面试怎么说

> `WATCH` 可以理解成 Redis 提供的一种乐观锁思路，用来在提交事务前检测 key 是否被别人改动。

## 9. Redis 事务适合什么场景

适合：

- 想把多条命令作为一个顺序提交单元
- 想做轻量的乐观并发控制

但如果你真正想解决的是：

- 多步逻辑的原子性
- 判断 + 修改 + 删除要一口气完成

那很多时候更自然会引到 `Lua`。

## 10. Lua 在 Redis 里是干什么的

Lua 的核心不是“写脚本很高级”，而是：

- 把多步 Redis 操作放到服务端一次执行
- 减少客户端和 Redis 的多次交互
- 把相关逻辑做成原子过程

这也是为什么分布式锁里经常会问到 Lua。

## 11. 为什么 Lua 经常和分布式锁连在一起

因为分布式锁释放时经常要做这样的逻辑：

1. 先判断锁的 value 是不是自己
2. 如果是自己，再删 key

如果你在客户端分两步做：

- `GET`
- `DEL`

中间会有并发窗口。

Lua 的价值就在这里：

- 把判断和删除放到 Redis 端一次做完
- 让整个过程更像一个原子动作

### 面试怎么说

> Lua 在 Redis 里最典型的价值，就是把多步相关操作做成服务端原子逻辑，比如分布式锁里的“校验 value 再删锁”。

## 12. Lua 解决的核心问题是什么

可以总结成两点：

### 12.1 原子性

多步逻辑一次执行，避免客户端多次请求中间被打断。

### 12.2 降低交互成本

原本要来回发多条命令，现在一次脚本就够了。

所以 Lua 既能提升正确性，也能提升一定效率。

## 13. 什么时候更适合 Lua，而不是 Redis 事务

如果你的需求是：

- 判断一个值
- 再决定删不删
- 或者读一个值后决定改成什么

这种“逻辑相关性很强”的场景，一般更适合 Lua。

因为 Redis 事务更像：

- 组织命令
- 顺序提交

而 Lua 更像：

- 把逻辑本身放到 Redis 端执行

## 14. 三者最核心的区别

### 14.1 `pipeline`

重点：

- 批量发命令
- 提高吞吐
- 减少 RTT

不重点解决：

- 原子性

### 14.2 Redis 事务

重点：

- 一组命令统一提交
- 有条件执行
- 可以配合 `WATCH`

不等同于：

- 数据库事务
- 完整回滚模型

### 14.3 `Lua`

重点：

- 多步逻辑放到 Redis 端
- 做成一个原子过程
- 适合判断 + 修改 + 删除这类场景

## 15. 高频追问小抄

### 15.1 `pipeline` 能保证原子性吗

不能。

它主要解决的是性能和吞吐问题。

### 15.2 Redis 事务会自动回滚吗

不要按 MySQL 事务去理解，一般不强调完整回滚语义。

### 15.3 为什么分布式锁删锁常用 Lua

因为要把“校验是不是自己的锁”和“删除锁”做成 Redis 端一次原子执行。

### 15.4 `WATCH` 更像什么

更像乐观锁。

### 15.5 如果只是批量 set / get，优先想到什么

优先想到 `pipeline`。

## 16. 一个比较稳的口语版回答

> Redis 里 `pipeline`、事务和 Lua 很容易混。我的理解是，`pipeline` 重点解决的是批量命令的网络往返成本，通过一次性发送多条命令提高吞吐；Redis 事务更像把一组命令先入队再统一提交执行，可以配合 `WATCH` 做轻量乐观并发控制，但它不等同于 MySQL 那种完整事务；Lua 则更适合把多步相关逻辑直接放到 Redis 端一次原子执行，比如分布式锁里常见的“先校验 value 再删锁”。所以如果我只是想提高批量操作效率，我会先想到 `pipeline`；如果我想做多步逻辑原子性，我会优先想到 Lua。

## 17. 一句话收尾

> 这三个点真正的区分关键，不是都算“批量操作”，而是你要分清它们分别偏性能、偏命令组织，还是偏原子逻辑。
