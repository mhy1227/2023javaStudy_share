# Redis 分布式锁进阶

## 1. 这篇文档解决什么问题

前面的 `redis-distributed-lock-note.md` 已经讲了：

- 为什么需要分布式锁
- 锁 key 的基本思路
- 为什么要设置过期时间
- 为什么释放锁不能乱删

这一篇继续补面试里更容易被追问的点：

- 为什么 `setnx` 和 `expire` 不能拆开
- 为什么 value 要唯一
- 为什么删除锁通常用 Lua
- 锁过期了但业务没执行完怎么办
- Redisson 的看门狗怎么理解
- Redis 主从切换时锁有没有风险

## 2. 先记住一个基本目标

分布式锁不只是“抢到一个 key”这么简单。

它至少想同时做到：

- 互斥
- 死锁可恢复
- 尽量避免误删别人的锁

面试里你只要把这三个目标讲出来，后面很多追问就顺了。

## 3. 为什么 `setnx` 和 `expire` 不能拆成两步

这是非常高频的坑点题。

如果你这样写：

1. 先 `setnx`
2. 再 `expire`

中间一旦出异常，比如：

- 进程挂了
- 网络断了
- 第二步没执行到

就可能出现：

- 锁加上了
- 但没有过期时间

最后锁长期不释放，形成死锁风险。

### 面试怎么说

> `setnx` 和 `expire` 分开执行不是原子操作，中间如果失败，可能造成锁没有过期时间，所以通常要用带过期时间的原子加锁方式。

## 4. 为什么 value 不能随便写

很多人只记得“key 代表锁”，忽略了 value 也很重要。

value 一般要放一个唯一标识，常见理解是：

- 当前线程 / 当前请求 / 当前实例的唯一身份

为什么要这样做？

因为释放锁时要先判断：

- 这把锁是不是自己加的

如果不是，就不能删。

## 5. 为什么不能直接 `del` 释放锁

这是另一个高频坑。

场景是这样的：

1. 线程 A 拿到锁
2. 线程 A 执行太久，锁过期了
3. 线程 B 拿到新的同名锁
4. 线程 A 这时才走到 `del`

如果线程 A 直接删 key，就把线程 B 的锁删掉了。

这就是经典的误删问题。

## 6. 为什么删除锁通常要用 Lua

因为我们希望“判断是不是自己的锁”和“删除锁”这两个动作一起完成。

如果分两步做：

1. 先 `get`
2. 再 `del`

中间仍然会有并发窗口。

Lua 的核心价值是：

- 把判断和删除放在 Redis 端一起执行
- 保证这两个动作具备原子性

### 面试怎么说

> Redis 分布式锁释放时常用 Lua，是因为要把“校验 value”和“删除 key”做成一个原子动作，避免并发窗口导致误删。

## 7. 锁过期了但业务还没执行完怎么办

这是分布式锁题里最容易继续深问的点。

因为设置过期时间只是避免死锁，但也引出了新问题：

- 业务执行时间超过锁 TTL
- 锁先过期
- 其他线程又拿到了锁
- 原线程还在执行

这样互斥就失效了。

## 8. 这个问题一般怎么处理

### 8.1 预估一个更合理的过期时间

这是最简单但不够稳的做法。

问题在于：

- 业务耗时有波动
- 很难永远估准

### 8.2 自动续期

这是更常见的工程方案。

也就是：

- 线程拿到锁后
- 在业务还没执行完时，自动延长锁的过期时间

这就是很多人说的“看门狗”思路。

## 9. Redisson 看门狗怎么理解

你不用去背实现源码，只要抓住本质：

> 看门狗机制的核心作用，是在业务还持有锁且线程还活着时，自动帮锁续期，避免锁提前过期。

它的价值在于：

- 不用死写一个非常长的 TTL
- 可以在业务正常执行期间动态续命

当然它也不是绝对无敌的，如果进程直接挂掉，看门狗也就没法继续续期了，这时还是依赖锁本身的过期机制做最终兜底。

## 10. Redis 主从切换下分布式锁有没有风险

有，这也是高级一点的追问。

一个典型风险场景是：

1. 客户端在主节点加锁成功
2. 锁还没来得及同步到从节点
3. 主节点挂了
4. 从节点晋升成新主节点
5. 新主节点上其实没有刚才那把锁

这时另一个客户端可能又成功加锁，导致同一时间两个客户端都认为自己拿到了锁。

### 面试怎么说

> Redis 主从复制本身不是强一致，所以在主从切换窗口内，分布式锁也存在丢失或双持有风险，这也是 Redis 锁的边界之一。

## 11. 这是不是说明 Redis 锁不能用

不是。

更准确的说法是：

- Redis 锁能用
- 但它适合有边界的互斥场景
- 不要把它理解成金融级绝对事务锁

适合的场景：

- 防重复提交
- 防重复消费
- 控制同一任务并发执行
- 业务级别的轻量互斥

不适合期待它解决一切强一致事务问题。

## 12. RedLock 要不要背

通常校招和一般社招一面，不需要背太深。

你只要知道：

- 有人提出过多节点协同加锁的思路
- 它是为了进一步降低单节点故障带来的锁风险

但更重要的是你要知道：

- 真实业务里很多团队更常落到成熟客户端方案
- 或者结合业务幂等、状态机、数据库约束一起兜底

不要把自己卡死在“必须把 RedLock 协议细节背全”。

## 13. Redis 分布式锁和数据库锁怎么选

### 13.1 更偏 Redis 锁

适合：

- 性能要求更高
- 持锁时间相对短
- 业务级互斥控制

### 13.2 更偏数据库约束或数据库锁

适合：

- 数据最终落库强约束更重要
- 和事务要强绑定
- 并发控制天然依赖数据库状态

面试里比较稳的说法是：

> Redis 锁更多是高性能协调手段，数据库锁或唯一约束更适合做最终一致性兜底，实际项目里常常是组合使用。

## 14. 高频追问小抄

### 14.1 为什么加锁要原子设置过期时间

避免加锁成功但没设置 TTL，形成死锁。

### 14.2 为什么 value 必须唯一

为了释放锁时识别“这把锁是不是自己加的”。

### 14.3 为什么删锁要 Lua

为了把“校验 value”和“删除 key”做成原子操作。

### 14.4 锁过期了业务还没结束怎么办

需要更合理的过期策略，或使用自动续期方案。

### 14.5 Redis 锁是绝对安全吗

不是，尤其在主从复制和故障切换场景下存在边界风险。

## 15. 一个比较稳的口语版回答

> Redis 分布式锁我一般会从三个点回答。第一是加锁要原子完成，而且要带过期时间，不能把 `setnx` 和 `expire` 分开写，否则中间失败会留下死锁风险。第二是释放锁不能直接 `del`，通常会给锁存一个唯一 value，并通过 Lua 把“校验是不是自己的锁”和“删除锁”做成原子操作，避免误删。第三是 Redis 锁并不是绝对无风险，如果业务执行时间超过 TTL，需要续期机制，比如 Redisson 看门狗；如果遇到主从切换，锁还可能因为复制延迟出现双持有风险。所以我会把 Redis 锁理解成高性能互斥工具，而不是绝对强一致事务锁。

## 16. 一句话收尾

> Redis 分布式锁题真正拉开差距的地方，不是会说“setnx”，而是能不能把原子性、过期、误删、续期和主从切换风险讲顺。
