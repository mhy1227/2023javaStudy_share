## 2. String vs Hash

### 2.1 核心区别

`String` 更适合存：

- 简单值
- 数字
- token
- 验证码
- 整体序列化对象

`Hash` 更适合存：

- 字段比较明确的对象
- 需要按字段读取或更新的数据

### 2.2 怎么理解

如果你把一个完整对象序列化之后塞进 `String`，写入和读取都很简单，但问题是：

- 你每次更新一个字段，可能都要整体读出来再整体写回去

而 `Hash` 的优势就在于：

- 更像对象字段结构
- 更适合做局部更新

### 2.3 项目里怎么选

如果是：

- 登录 token
- 验证码
- 简单计数器
- 一个整体缓存对象，读取多、局部修改少

通常 `String` 更自然。

如果是：

- 用户资料
- 商品对象
- 配置对象

而且有按字段更新的需求，`Hash` 会更合适。

### 2.4 面试展开答法

> `String` 和 `Hash` 我一般按“是否需要字段化操作”来区分。`String` 更适合简单值和整体对象缓存，优点是读写直接；`Hash` 更适合结构化对象，尤其是在对象字段比较多、并且存在按字段更新需求时会更灵活。所以项目里如果是 token、验证码、计数器这类，我更容易想到 `String`；如果是用户或商品对象这类结构化数据，我会更倾向 `Hash`。

## 3. RDB vs AOF

### 3.1 核心区别

- `RDB` 偏快照
- `AOF` 偏写命令日志

### 3.2 怎么理解

`RDB` 可以理解成：

- 在某些时间点，把当前内存状态整体快照到磁盘

它的特点是：

- 文件通常更紧凑
- 恢复速度通常更快
- 但两次快照之间如果宕机，可能会丢一段数据

`AOF` 可以理解成：

- 把写命令不断记录下来

它的特点是：

- 数据安全性通常更好一些
- 丢失窗口通常更小
- 但文件体积和恢复成本一般更高

### 3.3 项目里怎么选

如果业务更关心：

- 恢复速度
- 定期备份

会更偏 `RDB`。

如果业务更关心：

- 数据不要丢太多

会更偏 `AOF`。

更稳的项目表达一般不是绝对二选一，而是：

> 会结合业务数据重要程度、恢复速度要求和运维策略综合决定。

### 3.4 面试展开答法

> `RDB` 和 `AOF` 的核心区别在于持久化思路不同。`RDB` 更像阶段性全量快照，优点是文件紧凑、恢复通常更快，但数据丢失窗口相对更大；`AOF` 更像持续记录写操作，优点是数据安全性通常更好，但文件体积和恢复成本一般更高。项目里怎么选，主要还是看业务更在意恢复速度还是数据可靠性。

## 4. 主从 vs 哨兵 vs 集群

### 4.1 核心区别

- 主从：复制
- 哨兵：自动切换
- 集群：分片 + 高可用

### 4.2 怎么理解

主从复制首先解决的是：

- 数据副本
- 读扩展基础

它更多是在说：

- 一主多从的数据同步关系

哨兵是在主从基础上继续补：

- 故障检测
- 自动故障转移

它的重点不在于存数据，而在于：

- 监控
- 通知
- 选主

集群则更进一步，除了高可用之外，还要解决：

- 单机容量不够
- 单机压力太大

所以集群和哨兵最大的差别在于：

- 集群还解决了分片扩展问题

### 4.3 面试展开答法

> Redis 高可用我一般分三层理解。主从复制解决的是副本和读扩展基础；哨兵是在主从之上补故障检测和自动切主能力；集群则不只是高可用，还进一步解决了数据分片和横向扩展问题。所以这三者不是并列替代关系，而是解决的问题层次不同。

## 5. 热点 key vs 大 key

### 5.1 核心区别

- 热点 key 看访问量是否过于集中
- 大 key 看单个 key 的数据量是否过大

### 5.2 怎么理解

热点 key 的问题本质是：

- 某个 key 太热了
- 访问集中度太高

它更偏：

- 并发流量压力
- 局部瓶颈

大 key 的问题本质是：

- 单个 key 存的东西太多

它更偏：

- 传输开销
- 操作耗时
- 对 Redis 整体性能的拖累

### 5.3 面试展开答法

> 热点 key 和大 key 容易一起出现，但本质不是一个问题。热点 key 的核心是访问过于集中，风险在于局部高并发压力和击穿；大 key 的核心是单个 key 数据量过大，风险在于网络传输、操作耗时和整体性能拖累。所以热点 key 更偏流量问题，大 key 更偏数据规模问题。

## 6. 本地锁 vs Redis 分布式锁

### 6.1 核心区别

- 本地锁只控制当前 JVM
- Redis 分布式锁控制跨 JVM、跨节点互斥

### 6.2 怎么理解

如果你的服务只有一个实例，本地锁通常就够了。

但只要变成：

- 多实例部署
- 多节点部署

本地锁就有明显边界，因为它锁不住其他机器上的线程。

这时候才会引出 Redis 分布式锁。

### 6.3 面试展开答法

> 本地锁解决的是单机并发问题，比如当前 JVM 内多个线程对共享资源的互斥；Redis 分布式锁解决的是多实例、多节点场景下的互斥控制。所以不是说 Redis 锁一定更高级，而是它解决的是本地锁覆盖不到的边界问题。

## 7. Redis vs MySQL

### 7.1 核心区别

- MySQL 更偏持久化、关系型、事务型存储
- Redis 更偏热点数据、高频访问、低延迟读取

### 7.2 怎么理解

MySQL 更像：

- 最终数据落库点
- 核心业务数据持久化存储

Redis 更像：

- 高并发缓冲层
- 热点数据承接层

### 7.3 面试展开答法

> MySQL 和 Redis 在项目里通常是配合关系。MySQL 更负责最终持久化和事务型数据，Redis 更负责高频访问、热点流量和低延迟读取，所以一般不是谁替代谁，而是谁负责主存、谁负责加速。
