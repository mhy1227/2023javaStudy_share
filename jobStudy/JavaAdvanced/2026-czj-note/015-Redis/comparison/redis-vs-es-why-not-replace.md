# 为什么不用 Elasticsearch，非要 Redis，或者反过来

## 1. 这份文档怎么用

这一篇专门回答一个不算最高频，但很容易在项目题里被追问的问题：

- 为什么不用 ES，非要 Redis？
- 为什么有 Redis 了，还要 ES？

这个问题本质上是在问：

- 热点访问层和搜索层的边界
- 快速取值和复杂检索的差别

## 2. 先记住最短回答

> Redis 更适合热点数据承接、缓存和快速按 key 取值；Elasticsearch 更适合全文检索、复杂搜索和聚合分析。所以两者不是替代关系，而是访问层和搜索层的不同分工。

## 3. Redis 更擅长什么

Redis 更适合：

- 热点数据缓存
- 高频读
- 低延迟取值
- 排行榜
- 防重状态

它最强的是：

- 快速按 key 访问

## 4. ES 更擅长什么

ES 更适合：

- 全文检索
- 模糊搜索
- 多条件检索
- 聚合分析

它最强的是：

- 搜索能力

## 5. 为什么 Redis 不能简单替代 ES

因为 Redis 本质上不是搜索引擎。

它更像：

- 高性能 key-value 访问层

如果你要做：

- 关键词搜索
- 模糊查询
- 全文匹配

ES 会更自然。

## 6. 为什么 ES 也不能简单替代 Redis

因为 ES 的重点不是：

- 极低延迟的热点缓存承接
- 高频按 key 读写

如果你只是要：

- 商品详情缓存
- 用户信息热点访问
- 防重 key

那 Redis 更直接。

## 7. 面试里怎么讲更稳

### 简洁版

> Redis 更适合做缓存和热点数据承接，ES 更适合做搜索和聚合分析，所以一个偏访问层，一个偏检索层，通常是配合关系。

### 稍微展开一点

> 如果业务重点是热点读、低延迟取值和缓存承接，我会优先想到 Redis；如果重点是全文检索、模糊搜索和复杂条件查询，我会想到 ES。两者解决的问题维度不同，所以真实项目里经常一起出现，而不是替代关系。

## 8. 一句话收尾

> Redis 解决“快取”，ES 解决“快搜”，它们经常同时存在，是因为系统里访问问题和搜索问题本来就不是一回事。
