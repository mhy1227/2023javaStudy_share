# 为什么不用 JVM 里的 Map，非要 Redis

## 1. 这份文档怎么用

这一篇和“本地缓存 vs Redis”很近，但更口语化一点，也更贴近真实追问：

- 为什么不用 JVM 里的 `Map`
- 为什么还要多加一个 Redis

这个问题本质上是在问：

- 进程内存和分布式缓存的边界
- 单机共享和跨节点共享的区别

## 2. 先记住最短回答

> JVM 里的 `Map` 只在当前进程内有效，进程一重启就没了，多实例之间也不共享；Redis 更适合作为跨节点共享的统一缓存层。所以如果系统已经不是单机玩具程序，而是多实例服务，单靠 JVM 里的 Map 往往不够。

## 3. JVM Map 的优势

JVM 里的 `Map` 优势很明显：

- 访问快
- 没有网络开销
- 写起来简单

适合：

- 单机临时缓存
- 小范围、低复杂度场景

## 4. JVM Map 的问题

### 4.1 只在当前进程内有效

如果是多实例服务：

- A 机器放进去的数据
- B 机器看不到

### 4.2 重启就丢

进程一重启：

- 里面的数据就没了

### 4.3 更难统一控制

你要考虑：

- 多机器怎么同步
- 什么时候失效
- 怎么统一更新

## 5. Redis 的优势

Redis 更适合作为：

- 跨节点共享缓存
- 热点流量统一承接层
- 多实例系统里的公共状态层

它的价值在于：

- 所有节点都能访问
- 缓存规则更统一
- 失效和更新更容易集中控制

## 6. 项目里怎么讲更稳

### 简洁版

> JVM 里的 `Map` 更像进程内临时缓存，Redis 更像跨节点共享缓存层。单机时 `Map` 可以很方便，但多实例部署下，它不共享、重启就丢，所以项目里通常会引入 Redis 作为统一缓存层。

### 稍微展开一点

> 如果只是单机程序或者很轻量的局部缓存，JVM 里的 `Map` 会很直接；但只要进入多实例部署，`Map` 就会遇到不共享、难统一更新、难统一失效的问题。Redis 虽然多一跳网络，但它能把缓存层从单个进程提升到整个系统级别，所以更适合作为公共缓存和热点数据承接层。

## 7. 一句话收尾

> JVM 里的 `Map` 解决“当前进程快存取”，Redis 解决“整个系统共享缓存”，它们的边界从一开始就不一样。
