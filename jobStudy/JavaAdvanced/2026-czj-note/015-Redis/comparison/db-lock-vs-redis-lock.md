# 为什么不用数据库锁，非要 Redis 分布式锁

## 1. 这份文档怎么用

这一篇专门回答分布式锁场景里一个很常见的追问：

- 既然数据库也能加锁，为什么还要 Redis 分布式锁？

这个问题本质上是在问：

- 并发控制放在哪里更合适
- 数据库锁和 Redis 锁的边界
- 项目里为什么要把“协调”和“存储”分开

## 2. 先记住最短回答

> 数据库锁当然也能做并发控制，但它更适合围绕数据库记录本身的一致性保护；Redis 分布式锁更适合做跨节点、轻量级的全局互斥协调。简单说，数据库锁更偏“数据层一致性”，Redis 锁更偏“分布式协调层互斥”。

## 3. 数据库锁更适合什么

数据库锁更适合：

- 围绕数据库记录本身做并发保护
- 事务范围内的一致性控制
- 更新数据库时顺手完成锁控制

比如：

- 某条订单记录更新
- 某条库存记录扣减

这种场景数据库锁是天然语境。

## 4. Redis 锁更适合什么

Redis 分布式锁更适合：

- 多实例部署下的跨节点互斥
- 和数据库记录不完全绑定的全局资源控制
- 某些轻量级协调场景

比如：

- 防止任务重复执行
- 防止重复下单
- 控制某个全局资源同一时间只被一个节点处理

## 5. 为什么不总是用数据库锁

### 5.1 它会把数据库拖进并发协调

数据库本来更应该聚焦：

- 持久化
- 事务
- 查询

如果很多“协调型锁”也全压到数据库，会把数据库压力变大。

### 5.2 适用边界更窄

数据库锁更适合围绕数据库记录本身。

如果锁的对象不是某条具体记录，而是某个全局任务、全局资源，数据库锁就没那么自然。

## 6. 为什么 Redis 锁也不是完美方案

这也是面试里要主动补的一句。

Redis 锁常见风险包括：

- 锁过期但业务没执行完
- 释放锁误删别人的锁
- Redis 自身高可用问题

所以更稳的说法不是：

> Redis 锁一定更好

而是：

> Redis 锁更适合某些分布式协调场景，但它也有自己的风险边界。

## 7. 面试里怎么讲更稳

### 简洁版

> 数据库锁更适合围绕数据库记录本身做事务一致性保护，Redis 分布式锁更适合多实例部署下做跨节点互斥控制。所以不是说谁绝对更高级，而是解决的问题边界不一样。

### 稍微展开一点

> 如果锁的对象本来就是数据库记录，而且事务边界也在数据库里，那数据库锁会很自然；但如果是多实例系统里的全局协调，比如任务去重、跨节点互斥，这时候 Redis 锁会更适合。因为它把协调问题从数据库里抽出来了，不过也要注意锁过期、误删和高可用这些边界问题。

## 8. 一句话收尾

> 数据库锁更偏“数据一致性”，Redis 锁更偏“分布式协调”，项目里关键不是选更高级的锁，而是把锁放在更合适的层。
