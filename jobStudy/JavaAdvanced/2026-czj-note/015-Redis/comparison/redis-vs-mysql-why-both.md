# 为什么 Redis 很好，项目里还是要用 MySQL

## 1. 这份文档怎么用

这一篇专门回答一个很常见、但也很容易答浅的问题：

- 既然 Redis 那么快，为什么项目里不直接都用 Redis，还要用 MySQL？

这个问题本质上不是在问 Redis 好不好，而是在问你有没有：

- 存储分层意识
- 主存和缓存的边界意识
- 项目里的技术选型意识

## 2. 先记住最短回答

> 因为 Redis 和 MySQL 解决的问题不一样。Redis 更适合做缓存、热点数据承接和低延迟访问；MySQL 更适合做最终持久化、事务控制和结构化业务数据管理。所以项目里通常是 MySQL 做主存，Redis 做加速层，而不是 Redis 替代 MySQL。

## 3. 为什么不能只用 Redis

### 3.1 Redis 更适合热点数据，不适合所有数据都长期存

Redis 的核心优势是：

- 基于内存
- 访问快
- 适合热点数据

但也正因为它以内存为主，所以如果你把大量长期业务数据都塞进去：

- 成本会很高
- 数据分层会变乱
- 运维和容量压力也会明显上升

所以 Redis 更适合：

- 热点数据
- 高频读数据
- 临时状态数据

而不是所有业务数据的总仓库。

### 3.2 MySQL 更适合做最终落库

MySQL 更像系统里的主存。

它更适合存：

- 用户主数据
- 订单
- 支付记录
- 库存事实
- 履约数据

因为这些数据通常有几个特点：

- 要长期保存
- 要结构化管理
- 要能追溯
- 要能做事务控制

这些事情 MySQL 更擅长。

### 3.3 事务和一致性要求更适合 MySQL

很多核心业务数据不是“存进去就行”，而是还要满足：

- 一致性
- 事务控制
- 约束管理

比如：

- 下单
- 支付
- 库存扣减
- 订单状态流转

这种数据如果全交给 Redis 来承担，就会把很多原本该由关系型数据库解决的问题，强行转移到缓存层。

这不是 Redis 擅长的方向。

### 3.4 查询能力不一样

Redis 更适合：

- 按 key 快速取值
- 简单结构操作
- 高并发低延迟访问

MySQL 更适合：

- 条件过滤
- 排序分页
- 关联查询
- 统计分析

所以如果业务场景里存在比较强的结构化查询需求，MySQL 仍然是更自然的选择。

## 4. Redis 和 MySQL 在项目里的典型分工

### 4.1 适合 Redis 的数据

- 商品详情缓存
- 首页热点数据
- 验证码
- token
- 排行榜
- 防重 key
- 热点配置

这些数据的共同点一般是：

- 访问频率高
- 对延迟敏感
- 可以接受缓存层设计

### 4.2 适合 MySQL 的数据

- 用户表
- 订单表
- 支付流水
- 库存数据
- 履约数据

这些数据的共同点一般是：

- 要长期保存
- 要保证结构化和可追溯
- 往往带有事务和一致性要求

## 5. 面试里怎么讲更稳

### 5.1 简洁版

> Redis 很好，但它更像数据库前面的一层高性能缓存和热点流量承接层，不适合直接替代 MySQL。MySQL 更适合做最终持久化、事务控制和结构化数据管理，所以两者通常是配合关系。

### 5.2 稍微完整一点的版本

> 我理解 Redis 和 MySQL 在项目里不是替代关系，而是分工关系。Redis 的优势是内存级访问速度快，适合承接热点流量、高频读和临时状态；MySQL 的优势是持久化、事务和结构化数据管理，更适合做主存。所以真实项目里一般是 MySQL 负责最终落库，Redis 负责加速和减压。

## 6. 面试官继续追问时怎么接

### 6.1 问：那为什么不把所有数据都缓存到 Redis

比较稳的答法：

> 因为缓存的本质是为热点数据和高频访问服务，不是把主存职责整体搬走。如果把所有数据都放到 Redis，不仅成本高，还会把缓存和主存边界混乱化。

### 6.2 问：那 Redis 会不会丢数据

比较稳的答法：

> Redis 会通过持久化和高可用机制尽量降低数据丢失风险，但它本质上还是以内存为主，所以不能简单把它理解成主存数据库的完全替代品。

### 6.3 问：那排行榜为什么你们反而更倾向 Redis

比较稳的答法：

> 因为排行榜这种场景更偏高频读和低延迟排序，Redis 的 `ZSet` 更适合承接这种热点访问，而 MySQL 更适合做最终业务数据存储。

## 7. 一句话收尾

> Redis 的价值在于“加速”和“分流”，MySQL 的价值在于“主存”和“落库”，项目里真正成熟的做法通常是两者分层配合，而不是二选一。
