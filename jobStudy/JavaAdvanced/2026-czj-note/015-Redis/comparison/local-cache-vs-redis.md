# 为什么不用本地缓存，非要上 Redis

## 1. 这份文档怎么用

这一篇专门回答一个很常见的项目题：

- 既然本地缓存更快，为什么还要上 Redis？

这个问题本质上是在问：

- 单机缓存和分布式缓存的边界
- 多实例部署下的数据一致性问题
- 项目里缓存分层怎么设计

## 2. 先记住最短回答

> 本地缓存确实更快，但它只在当前进程内生效，多实例部署下不共享，也更难统一失效和更新；Redis 虽然比本地缓存多一跳网络，但更适合作为跨节点共享的统一缓存层。所以真实项目里很多时候不是本地缓存和 Redis 二选一，而是两者配合使用。

## 3. 本地缓存的优势

本地缓存的最大优点很直接：

- 没有网络开销
- 访问速度更快
- 使用简单

比如：

- JVM 里的 `Map`
- Caffeine
- Guava Cache

这种缓存很适合：

- 单机应用
- 进程内热点数据
- 极低延迟要求的小范围缓存

## 4. 本地缓存的问题

### 4.1 只在当前进程内有效

这意味着：

- 当前机器能命中
- 其他机器不一定有

只要系统是多实例部署，本地缓存就天然不共享。

### 4.2 数据一致性更难控制

如果数据变了，你要考虑：

- 当前机器怎么更新
- 其他机器怎么更新

节点一多，这件事就会变复杂。

### 4.3 重启就没了

本地缓存一般随进程生命周期存在。

进程一重启，缓存就没了。

## 5. Redis 的优势

Redis 更适合作为：

- 跨节点共享缓存层
- 热点数据统一承接层
- 多实例系统里的公共缓存

它的价值在于：

- 多实例可共享
- 缓存数据集中管理
- 热点流量更容易统一承接
- 失效和更新更容易统一控制

## 6. 为什么很多项目两者一起用

最稳的项目答法通常不是：

- 只用本地缓存
- 或者只用 Redis

而是：

- 本地缓存做第一层
- Redis 做第二层

也就是常见的多级缓存思路。

### 怎么理解

- 本地缓存解决“当前机器超快访问”
- Redis 解决“跨机器共享缓存”

## 7. 面试里怎么讲更稳

### 简洁版

> 本地缓存更快，但只在当前进程内有效；Redis 虽然多一跳网络，但能作为多实例共享的统一缓存层。所以真实项目里很多时候会用本地缓存加 Redis 做多级缓存，而不是简单二选一。

### 稍微展开一点

> 如果系统是单机、数据变化少，本地缓存会很高效；但只要进入多实例部署，本地缓存就会遇到数据不共享、更新不一致和失效难统一的问题。Redis 更适合承接跨节点共享缓存和热点流量，所以项目里通常会把本地缓存当成第一层，把 Redis 当成第二层。

## 8. 一句话收尾

> 本地缓存解决“更快”，Redis 解决“共享和统一”，项目里真正成熟的做法通常是分层配合，而不是非此即彼。
