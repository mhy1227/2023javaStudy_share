# Redis 对比题易踩坑问法

## 1. 这份文档怎么用

这一篇不是补新知识，而是专门防止你在 Redis 对比题里翻车。

很多 Redis 对比题的问题不是不会，而是：

- 只会背一句
- 说得太绝对
- 把几个概念混成一团

## 2. 坑 1：String 和 Hash 只是“两个不同数据结构”

这个说法不能算错，但太浅。

面试官更想听的是：

- 你怎么选
- 什么时候用 `String`
- 什么时候用 `Hash`

更稳的说法：

> `String` 更适合简单值和整体对象，`Hash` 更适合结构化对象和字段级更新。

## 3. 坑 2：RDB 一定更好，或者 AOF 一定更好

这类答法太绝对。

更稳的说法：

> `RDB` 和 `AOF` 没有绝对谁更好，关键是业务更看重恢复速度还是数据可靠性。

## 4. 坑 3：主从、哨兵、集群都是一种“Redis 集群”

这会显得你没有分层理解。

更稳的记法是：

- 主从：复制
- 哨兵：切主
- 集群：分片 + 高可用

如果你把这三个混成一句“都是高可用”，面试官很容易继续追问把你问散。

## 5. 坑 4：热点 key 和大 key 是同一个问题

这也是非常常见的误区。

要记住：

- 热点 key 看访问是否集中
- 大 key 看单个 key 是否过大

一个偏流量维度，一个偏数据规模维度。

## 6. 坑 5：本地锁和 Redis 分布式锁只是“范围不一样”

这句话太薄。

更完整一点应该是：

> 本地锁解决的是单 JVM 内的互斥，Redis 分布式锁解决的是多实例、多节点场景下的跨机器互斥，它们的适用边界本来就不同。

## 7. 坑 6：Redis 能替代 MySQL

如果你这么说，基本很容易出问题。

更稳的说法：

> Redis 和 MySQL 在项目里通常是配合关系，MySQL 负责最终持久化，Redis 负责热点流量和低延迟访问。

## 8. 坑 7：穿透、击穿、雪崩都差不多，反正都是缓存出问题

这是典型的“知道名词但不会区分”。

要直接分清：

- 穿透：数据本来不存在
- 击穿：热点 key 失效
- 雪崩：大面积失效或 Redis 整体崩

## 9. 坑 8：解决办法乱配

比如：

- 用布隆过滤器去讲击穿
- 用互斥锁去讲雪崩
- 用随机过期去讲穿透

这类错法很容易暴露你没有真正理解问题对象。

更稳的记法是：

- 穿透：挡无效请求
- 击穿：保热点 key
- 雪崩：防大面积一起崩

## 10. 一句话收尾

> Redis 对比题最容易翻车的地方，不是完全不会，而是几个概念都知道一点，但一比较就开始乱。
