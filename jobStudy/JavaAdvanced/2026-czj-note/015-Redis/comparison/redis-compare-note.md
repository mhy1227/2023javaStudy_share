# Redis 常见对比

## 1. 这份文档怎么用

这一篇专门把 Redis 里高频、易混的几个对比放在一起。

它不是替代主线文档，而是帮助你在面试里答得更清楚。

## 2. String vs Hash

### 2.1 最核心的区别

- `String` 更适合简单值、数字、序列化后的整体对象
- `Hash` 更适合结构化字段较多、希望按字段更新的对象

### 2.2 怎么选

如果是：

- 验证码
- token
- 计数器

通常更容易想到 `String`。

如果是：

- 用户对象
- 商品对象
- 配置对象

而且有按字段更新需求，就更容易想到 `Hash`。

### 2.3 面试口语版

> `String` 更适合简单值和整体对象缓存，`Hash` 更适合字段化对象存储，尤其是在需要按字段修改时会更灵活。

## 3. RDB vs AOF

### 3.1 最核心的区别

- `RDB` 偏快照
- `AOF` 偏写命令日志

### 3.2 怎么理解

- `RDB` 恢复通常更快，但数据丢失窗口相对更大
- `AOF` 数据安全性通常更好，但文件体积和恢复成本一般更高

### 3.3 面试口语版

> `RDB` 更像定期全量快照，恢复通常更快；`AOF` 更像持续记录写操作，数据丢失风险通常更小。

## 4. 主从 vs 哨兵 vs 集群

### 4.1 最核心的区别

- 主从：复制
- 哨兵：自动切换
- 集群：分片 + 高可用

### 4.2 怎么理解

主从复制先解决副本和读扩展基础；

哨兵是在主从之上再补故障检测和切主能力；

集群不只是高可用，还要解决容量和流量横向扩展问题。

### 4.3 面试口语版

> 我一般把 Redis 高可用分三层记：主从负责复制，哨兵负责切主，集群负责分片扩展。

## 5. 热点 key vs 大 key

### 5.1 最核心的区别

- 热点 key 看访问量是否过于集中
- 大 key 看单个 key 的数据规模是否过大

### 5.2 怎么理解

热点 key 更偏高并发访问压力问题；

大 key 更偏单次操作开销和性能风险问题。

### 5.3 面试口语版

> 热点 key 的问题是访问过于集中，大 key 的问题是单个 key 太大导致传输和操作成本过高，这两个不是一个维度。

## 6. 本地锁 vs Redis 分布式锁

### 6.1 最核心的区别

- 本地锁只能锁住当前 JVM
- Redis 分布式锁可以做跨节点互斥

### 6.2 怎么理解

如果应用是单实例，本地锁通常就够；

如果已经多实例部署，要控制共享资源的互斥访问，本地锁就不够了。

### 6.3 面试口语版

> 本地锁解决的是单机并发问题，Redis 分布式锁解决的是多实例、多节点下的互斥问题。

## 7. 缓存 vs MySQL

### 7.1 最核心的区别

- MySQL 更偏持久化、事务型、最终数据落库
- Redis 更偏高频访问、热点数据、低延迟读取

### 7.2 面试口语版

> MySQL 负责最终持久化数据，Redis 负责承接高频读和热点流量，它们更多是配合关系，不是替代关系。

## 8. 一句话收尾

> Redis 对比题最怕的不是不会，而是几个概念都知道一点，但一比较就混。
