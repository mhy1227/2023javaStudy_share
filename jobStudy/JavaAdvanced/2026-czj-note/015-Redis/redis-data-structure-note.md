# Redis 常见数据结构

## 1. 这份文档怎么用

Redis 数据结构不是为了背 API。

这一篇更重要的是帮你回答：

- Redis 常用数据结构有哪些
- 各自适合什么场景
- 为什么排行榜常提 `ZSet`

## 2. 常见数据结构总览

Redis 面试里最常提的有：

- `String`
- `Hash`
- `List`
- `Set`
- `ZSet`

如果是基础面，先把这五个讲顺就够用了。

## 3. String

### 3.1 特点

- 最基础
- 使用最广
- 可以存字符串、数字、序列化后的对象等

### 3.2 场景

- 缓存对象
- 计数器
- 验证码
- Token

### 3.3 面试怎么说

> `String` 是 Redis 里最常用的数据类型，适合存简单值、数字计数和序列化对象。

## 4. Hash

### 4.1 特点

- 更适合表达对象结构
- 一个 key 下可以挂多个 field-value

### 4.2 场景

- 用户信息
- 商品信息
- 配置信息

### 4.3 和 String 的区别

如果整个对象都序列化进 `String`，每次更新一个字段都可能要整体读写。

而 `Hash` 更适合按字段修改。

### 4.4 面试怎么说

> `Hash` 更适合存结构化对象，尤其是对象字段较多、又希望按字段更新时，比直接序列化成一个 `String` 更灵活。

## 5. List

### 5.1 特点

- 有序
- 可重复
- 适合头尾操作

### 5.2 场景

- 消息队列的简单实现
- 最新动态列表
- 评论流、任务队列的轻量场景

### 5.3 注意点

现在真实项目里，如果是比较正式的消息队列需求，更多还是 MQ。

所以 List 更适合拿来讲“轻量队列场景”。

## 6. Set

### 6.1 特点

- 无序
- 去重

### 6.2 场景

- 标签集合
- 好友关系集合
- 去重统计

### 6.3 面试怎么说

> `Set` 适合做去重类场景，比如共同好友、共同关注、标签去重。

## 7. ZSet

### 7.1 特点

- 有序
- 可按 score 排序

### 7.2 场景

- 排行榜
- 热度排序
- 延迟任务的轻量实现

### 7.3 为什么排行榜喜欢用它

因为它天然就能把成员和分数绑定起来，并按分数排序。

### 7.4 面试怎么说

> `ZSet` 是面试里非常高频的结构，因为它特别适合做排行榜、热度榜这类“既要存元素又要排序”的场景。

## 8. 面试里怎么做数据结构选型

一个很稳的答法是：

> 我会先看业务数据有没有结构化字段，再看是否需要有序、去重、计数或者排序。如果只是简单值，优先 `String`；如果是对象字段，优先 `Hash`；如果是去重集合，用 `Set`；如果需要排序，优先考虑 `ZSet`。

## 9. 一句话收尾

> Redis 数据结构题的关键，不是背名字，而是把“结构特点”和“业务场景”对应起来。
