# Redis 基础总览

## 1. 这份文档怎么用

这一篇先把 Redis 的整体轮廓拉起来。

你看完之后，至少要能回答：

- Redis 是什么
- Redis 为什么快
- Redis 常见使用场景有哪些
- Redis 和 MySQL 分别适合做什么

## 2. Redis 是什么

Redis 本质上是一个：

- 基于内存的
- 高性能的
- Key-Value 存储系统

它经常被拿来做：

- 缓存
- 计数器
- 排行榜
- 分布式锁
- 消息队列的轻量场景
- Session 或临时数据存储

面试里不要只答“它是数据库”。

更稳一点的说法是：

> Redis 是一个以内存为主的高性能 Key-Value 存储系统，经常用来做缓存和高并发场景下的热点数据访问。

## 3. Redis 为什么快

这是 Redis 里最常见的问题之一。

### 3.1 核心原因

- 数据主要放在内存里
- 数据结构设计比较高效
- 单线程模型避免了很多线程切换和锁竞争成本
- 网络模型和 IO 多路复用让它能高效处理大量连接

### 3.2 面试里怎么答

> Redis 快，首先是因为它主要基于内存读写；其次它对字符串、哈希、列表、集合、有序集合这些数据结构做了专门优化；另外它采用单线程处理命令执行，避免了锁竞争和上下文切换；再加上 IO 多路复用，整体吞吐能力会很强。

## 4. 单线程怎么理解

很多人会被这个点绕住。

比较稳的理解是：

- Redis 说自己是单线程，主要是说命令执行这一层是单线程串行处理
- 不是说整个 Redis 进程只有一个线程
- 它的一些后台任务，比如持久化相关工作，不一定完全是单线程

面试里别说成：

> Redis 就只有一个线程

这样太死。

## 5. Redis 适合什么场景

### 5.1 缓存

这是最常见的场景。

比如：

- 商品详情缓存
- 用户信息缓存
- 首页热点数据缓存

### 5.2 计数器

比如：

- 点赞数
- 浏览量
- 接口访问次数

### 5.3 排行榜

这个场景非常经典，一般会联想到 `ZSet`。

比如：

- 积分榜
- 热门文章排行
- 游戏分数排行

### 5.4 分布式锁

多个服务实例下，要控制同一资源同一时刻只被一个线程或节点处理时，就可能会用 Redis 做分布式锁。

### 5.5 临时状态或会话数据

比如：

- 登录验证码
- 短时 token
- 分布式 Session

## 6. Redis 和 MySQL 的分工

这是项目题里很常见的联动问法。

可以这样理解：

- MySQL 更偏持久化、关系型、事务型存储
- Redis 更偏高性能访问、热点数据、临时数据和缓存

面试里比较稳的讲法是：

> MySQL 负责最终落库和持久化数据，Redis 负责高频访问、低延迟读取和热点数据承接，它们通常是配合关系，不是互相替代关系。

## 7. Redis 常见问题的主线

Redis 这一章后面最常被继续追问的，其实就是三条线：

- 数据结构怎么选
- 持久化怎么做
- 缓存问题怎么处理

再往后就会自然引到：

- 分布式锁
- 热点 key
- 大 key
- 缓存一致性

## 8. 一句话收尾

> Redis 最重要的不是记住多少命令，而是知道它为什么快、适合扛什么场景、不能替代什么。
