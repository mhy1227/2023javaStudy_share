# Redis 主从复制进阶

## 1. 这篇文档解决什么问题

前面的 `redis-high-availability-note.md` 已经把主从、哨兵、集群的主线拉顺了。

这一篇继续往下补的是面试里更容易被追问的细节：

- 主从复制到底怎么开始
- 全量复制和增量复制分别是什么
- `psync` 是干什么的
- `repl_backlog_buffer` 有什么作用
- 为什么主从会有延迟
- 读写分离为什么会有一致性风险
- 脑裂大概怎么理解

如果你面试里已经说到“Redis 做了主从 + 哨兵”，那后面十有八九就会追到这些点。

## 2. 先记住主从复制的核心定位

Redis 主从复制的核心不是为了让多个节点一起写，而是为了：

- 做数据副本
- 做读扩展
- 给哨兵故障转移提供基础

一句话概括就是：

> 主节点负责接收核心写请求，从节点负责同步主节点数据，承担备份和部分读流量。

## 3. 主从复制的大体流程

可以先用最粗的流程去记：

1. 从节点向主节点发起复制请求
2. 主节点判断是做全量复制还是增量复制
3. 主节点把数据同步给从节点
4. 后续主节点的写命令继续传播给从节点

面试里先别一上来就抖术语，先把流程说清楚，再补 `psync`、偏移量、积压缓冲区这些关键词，层次会更稳。

## 4. 什么是全量复制

全量复制一般发生在这些场景：

- 从节点第一次跟主节点建立复制关系
- 从节点断线太久，主节点没法只补丢失的那一段数据
- 复制条件不满足，退化成重新全量同步

全量复制可以粗略理解成：

- 主节点生成当时的一份数据快照
- 把这份快照发给从节点
- 从节点加载快照
- 再继续接收快照之后发生的写命令

### 4.1 面试怎么说

> 全量复制就是主节点把某个时刻的数据全集同步给从节点，从节点先拿到一份完整副本，再继续追后续的命令流。

### 4.2 为什么全量复制成本高

因为它通常意味着：

- 主节点要生成快照
- 要走一次比较重的数据传输
- 从节点要重新加载整份数据

所以如果频繁发生全量复制，通常不是好现象，说明网络抖动严重、从节点频繁重连，或者复制链路设计不稳定。

## 5. 什么是增量复制

增量复制是更理想的状态。

它的意思不是“再来一份完整数据”，而是：

- 从节点告诉主节点自己已经同步到哪里了
- 主节点只把后续缺失的那一段命令补给从节点

这样同步成本会小很多，恢复速度也更快。

### 5.1 面试怎么说

> 增量复制本质上是按偏移量补数据，从节点不是重拿全量快照，而是把自己没追上的那一段命令继续补齐。

## 6. `psync` 是什么

`psync` 可以理解成 Redis 用来协商复制方式的机制。

你可以把它记成一句话：

> `psync` 的作用，是让主从节点判断这次同步能不能走增量复制，如果不行，再退回全量复制。

它背后通常会涉及几个关键词：

- `replication id`
- `offset`
- `repl_backlog_buffer`

## 7. `replication id` 和 `offset` 怎么理解

### 7.1 `replication id`

你可以把它理解成“这一条复制历史链路的身份标识”。

主节点会维护自己的复制标识，从节点会记住自己跟随的是哪条复制历史。

### 7.2 `offset`

`offset` 可以理解成复制进度。

它表示：

- 主节点的命令流已经推进到了哪里
- 从节点已经同步到了哪里

面试里不用扣得太死，可以直接说：

> `offset` 就像同步进度游标，主从节点靠它判断从节点差了多少数据。

## 8. `repl_backlog_buffer` 是什么

这是主从复制里很高频的追问点。

`repl_backlog_buffer` 可以理解成主节点为复制准备的一块环形缓冲区，里面会暂时保留最近传播过的复制命令。

它的价值在于：

- 从节点短暂掉线后重新连上
- 主节点不一定要重新做全量复制
- 只要掉线期间缺失的数据还在 backlog 里，就可以直接增量补发

### 8.1 为什么说它很重要

因为它直接影响“断线重连后能不能增量复制”。

如果 backlog 太小，而从节点掉线时间稍长，缺失的数据已经被覆盖掉了，那么主节点就只能退回全量复制。

### 8.2 面试怎么说

> `repl_backlog_buffer` 是主节点为复制保留的一段最近命令缓冲区，主要作用是支持从节点断线重连后的增量复制，避免频繁全量同步。

## 9. 为什么会出现复制延迟

主从复制不是强一致，它天然会有同步延迟窗口。

常见原因包括：

- 网络传输有延迟
- 主节点写入压力大
- 从节点处理能力不够
- 全量复制或重同步期间负担较重

这意味着一个很现实的问题：

- 你刚写入主节点成功
- 紧接着去读从节点
- 可能还没同步过去

这就是面试里常说的“主从不一致窗口”。

## 10. 读写分离为什么会有风险

很多人只记得“读写分离能抗流量”，忘了它也会带来一致性问题。

典型场景是：

1. 用户刚下单，写入主节点成功
2. 业务立即去从节点查询订单
3. 结果从节点还没同步到，看到的是旧数据

所以读写分离不是不能用，而是要分场景。

### 10.1 什么场景更适合读从库

- 实时一致性要求不高的查询
- 报表类、统计类查询
- 允许短暂延迟的数据读取

### 10.2 什么场景更适合读主库

- 刚写完立刻读的场景
- 强依赖最新状态的核心流程
- 订单、支付、库存这类一致性敏感业务

### 10.3 面试怎么说

> Redis 主从读写分离的核心风险是主从复制有延迟，所以写后立刻读从节点可能读到旧数据。对于一致性敏感的链路，一般会优先读主节点，或者做延迟容忍与兜底设计。

## 11. 为什么说 Redis 主从不是强一致

这个问题其实不复杂。

因为主节点写成功，不代表所有从节点在同一时刻都已经同步完成。

所以 Redis 主从更接近：

- 最终一致
- 弱一致

而不是强一致。

如果面试官问“那还能不能用”，你可以这样答：

> 缓存系统很多时候更关注吞吐、响应速度和高可用，所以接受一定时间窗口内的不一致是常见工程取舍，但关键链路要配合业务兜底。

## 12. 脑裂怎么理解

脑裂可以先用很朴素的话去讲：

> 原来的主节点其实没有完全坏，但因为网络分区或心跳异常，被系统误判下线，于是又选出了一个新的主节点，导致一段时间里像是出现了两个主节点。

这样会带来几个问题：

- 客户端可能写到了不同节点
- 数据版本可能分叉
- 后续恢复时需要以新的主节点为准

### 12.1 为什么面试会问脑裂

因为这能体现你是否真的理解“高可用不是免费午餐”。

高可用方案解决了自动切换问题，但也引入了：

- 误判风险
- 短时不一致风险
- 切换期间的数据丢失窗口

### 12.2 哨兵在这里的作用

哨兵并不能让系统变成绝对零风险。

它做的是：

- 监控主节点状态
- 达成故障判定共识
- 选择新的主节点
- 通知从节点重新跟随

所以更准确地说，哨兵是“降低故障恢复成本”，不是“彻底消灭一致性问题”。

## 13. 主从复制、哨兵、集群这一条线怎么串

这几个点很容易被分开背，面试里反而容易乱。

更顺的串法是：

### 13.1 第一层：主从复制

解决：

- 数据副本
- 读扩展

### 13.2 第二层：哨兵

解决：

- 主节点故障检测
- 自动故障转移

### 13.3 第三层：集群

解决：

- 分片
- 横向扩容
- 更大规模下的高可用

你可以把它理解成：

- 主从解决“复制”
- 哨兵解决“切主”
- 集群解决“分片 + 扩容 + 高可用”

## 14. 高频追问小抄

### 14.1 为什么从节点重连后不一定全量复制

因为如果主节点 backlog 里还保留着从节点缺失的命令，就可以直接增量补发。

### 14.2 backlog 越大越好吗

也不是。

它太小会导致更容易退化成全量复制，但太大也会占用更多内存。

所以本质上还是容量和恢复能力之间的权衡。

### 14.3 为什么主从复制适合缓存，不太适合拿来保证严格事务一致

因为主从复制关注的是高性能和高可用，不是强一致事务语义。

### 14.4 主从复制能解决容量问题吗

只能部分缓解读压力，不能从根上解决单机容量瓶颈。

真正要解决容量和水平扩展问题，通常还是要靠集群分片。

## 15. 一个比较稳的口语版回答

> Redis 主从复制本质上是主节点负责写，从节点同步主节点数据，用来做副本、读扩展和故障恢复基础。复制过程中有全量复制和增量复制两种方式，第一次同步或者补不上缺失数据时会走全量复制，正常断线重连更希望走增量复制。Redis 会通过 `psync`、复制偏移量和 `repl_backlog_buffer` 来判断能不能只补缺失的那一段命令。需要注意的是，主从复制不是强一致，会存在复制延迟，所以读写分离虽然能抗流量，但在写后立刻读这类场景下可能读到旧数据。再往上就是哨兵负责故障转移，集群负责分片和更大规模扩展。

## 16. 一句话收尾

> Redis 主从复制题不要只会说“一个主多个从”，真正能拉开差距的是你能不能把全量复制、增量复制、`psync`、backlog、复制延迟和读写分离风险讲顺。
