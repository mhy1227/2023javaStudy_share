# Redis 持久化

## 1. 这份文档怎么用

Redis 虽然以内存为主，但面试里一定会问持久化。

这一篇重点要回答：

- Redis 为什么还需要持久化
- `RDB` 和 `AOF` 是什么
- 两者怎么选

## 2. 为什么 Redis 还需要持久化

因为 Redis 数据主要在内存里。

如果服务重启、宕机或者机器出问题，内存数据可能会丢。

所以持久化的意义就是：

- 降低数据丢失风险
- 支持重启恢复
- 给高可用和数据恢复提供基础

## 3. RDB 是什么

`RDB` 可以理解成：

- 在某个时间点
- 把当前内存数据快照
- 落到磁盘

### 3.1 优点

- 文件更紧凑
- 恢复速度通常比较快
- 更适合做冷备或定期备份

### 3.2 缺点

- 不是实时持久化
- 两次快照之间如果宕机，可能会丢这段时间的数据

## 4. AOF 是什么

`AOF` 可以理解成：

- 把写命令追加记录下来
- 重启时再根据这些命令恢复数据

### 4.1 优点

- 数据更不容易丢太多
- 持久化粒度更细

### 4.2 缺点

- 文件可能更大
- 恢复速度可能比 `RDB` 慢
- 需要考虑重写和文件膨胀问题

## 5. RDB 和 AOF 怎么对比

可以这样记：

- `RDB` 偏快照
- `AOF` 偏操作日志

面试里比较稳的表达：

> `RDB` 更像阶段性全量快照，恢复速度快，但数据丢失窗口相对大；`AOF` 更像把写操作日志化，数据安全性更好，但文件体积和恢复成本通常更高。

## 6. 该怎么选

### 6.1 如果更关心恢复速度和备份

会更偏 `RDB`。

### 6.2 如果更关心数据丢失更少

会更偏 `AOF`。

### 6.3 真实项目怎么说

更稳的回答不是绝对站队，而是说：

> 实际项目里一般会结合业务对数据可靠性和恢复速度的要求来选，有的场景会同时考虑两者的配合使用。

## 7. 面试里怎么答“Redis 会不会丢数据”

不要答成：

> 开了 Redis 就不会丢

更稳的答法是：

> Redis 本质上以内存为主，所以天然会有数据丢失风险。持久化、高可用、主从和集群这些机制，都是为了尽量降低风险，但不能简单理解成绝对不丢。

## 8. 一句话收尾

> Redis 持久化题最重要的是把 `RDB` 和 `AOF` 的核心差异讲清楚，再顺手讲业务选型。
