# Redis 分布式锁

## 1. 这份文档怎么用

Redis 分布式锁是高频项目题。

这一篇主要回答：

- 为什么需要分布式锁
- Redis 分布式锁怎么实现
- 它有哪些风险

## 2. 为什么需要分布式锁

单机下你可能用：

- `synchronized`
- `ReentrantLock`

但如果服务已经是多实例部署，这些锁只能锁住当前进程，锁不住其他机器上的线程。

这时候才会引出分布式锁。

### 面试怎么说

> 分布式锁的核心场景是多实例部署下对共享资源做互斥控制，本地锁只能管住当前 JVM，跨节点时就需要借助像 Redis 这样的外部协调组件。

## 3. Redis 分布式锁的核心思路

最常见的理解方式是：

- 某个线程尝试往 Redis 里写一个锁 key
- 写成功，说明拿到锁
- 写失败，说明锁已经被别人占用

为了避免死锁，一般还会配过期时间。

## 4. 为什么要带过期时间

如果拿到锁的线程挂了，或者业务执行异常，没有主动释放锁，那锁可能一直不释放。

所以过期时间的作用是：

- 防止死锁
- 给锁一个自动兜底释放机制

## 5. 释放锁为什么也有讲究

面试里经常会继续追问：

- 是不是直接 `del` 就行

这里要注意：

- 不能简单删
- 要确认这个锁是不是自己加的

所以一般会配一个唯一标识，确保“谁加的锁，谁来删”。

## 6. Redis 分布式锁的常见风险

### 6.1 锁过期但业务没执行完

这时候其他线程可能又拿到锁，导致并发问题。

### 6.2 锁误删

如果不校验唯一标识，可能删掉别人的锁。

### 6.3 Redis 自身高可用问题

如果锁组件本身异常，也会影响整个锁机制可靠性。

## 7. 面试里怎么答

> Redis 分布式锁的基本思路是通过写锁 key 实现互斥，并给锁设置过期时间防止死锁。为了安全释放锁，一般还会给锁带唯一标识，释放时校验是自己持有的锁再删除。它能解决跨 JVM、跨机器的互斥问题，但也要关注锁过期、误删和高可用等风险。

## 8. 它适合什么场景

比如：

- 防止重复下单
- 控制同一任务重复执行
- 某些全局资源同一时间只允许一个节点处理

## 9. 一句话收尾

> Redis 分布式锁题最重要的不是背命令，而是回答清楚“为什么本地锁不够、为什么要过期时间、为什么释放锁不能乱删”。
