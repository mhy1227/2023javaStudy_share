# Redis 特殊结构与场景题

## 1. 这篇文档解决什么问题

前面的 `redis-data-structure-note.md` 讲的是更常见的数据结构主线。

这一篇补的是面试里偶尔会出现、但回答好了很加分的结构和场景：

- `Bitmap`
- `HyperLogLog`
- `GEO`
- `Stream`
- `Lua`

这些点不一定每场都问，但一旦问到，很多人容易答空。

## 2. 先记住：面试官通常不是想考命令

大多数时候，他真正想问的是：

- 你知不知道 Redis 不只有 String / Hash / List / Set / ZSet
- 你能不能把结构和业务场景挂钩

所以答法重点不是命令大全，而是“这个结构适合拿来做什么”。

## 3. Bitmap 适合什么场景

Bitmap 很适合表达“某个位置是不是 0/1”。

最常见的口径是：

- 签到
- 打卡
- 活跃标记
- 某用户某天是否完成过某动作

### 3.1 为什么签到常想到 Bitmap

因为签到问题通常长这样：

- 某用户一个月每天是否签到
- 只需要是 / 否

这种状态非常适合用 bit 位表达。

它的优势在于：

- 节省空间
- 按位统计方便

### 3.2 面试怎么说

> Bitmap 适合做布尔状态压缩存储，比如签到、是否登录、是否完成某动作这类按位标记场景。

## 4. HyperLogLog 适合什么场景

HyperLogLog 最常见的场景是：

- UV 统计
- 去重人数估算

它的关键特点不是“绝对精确”，而是：

- 占用空间很小
- 能快速做基数估算

### 4.1 为什么 UV 场景会想到它

因为如果你只是想知道：

- 今天大概有多少独立访客

那很多时候你不一定需要绝对精准到最后一个人。

如果直接把所有用户都放进 Set：

- 会更占空间

所以这时 HyperLogLog 的价值就出来了。

### 4.2 面试怎么说

> HyperLogLog 适合做大规模去重计数的近似统计，比如 UV，它的优势是空间开销小，但代价是结果不是绝对精确。

## 5. GEO 适合什么场景

GEO 适合位置相关业务。

最常见问法包括：

- 附近的人
- 附近的门店
- 附近的骑手

它的本质是：

- 存地理坐标
- 按距离做检索和排序

### 面试怎么说

> GEO 更适合处理经纬度和附近范围查询这类位置场景，比如附近门店、附近骑手、附近用户。

## 6. Stream 适合什么场景

Stream 是 Redis 在消息流场景下的重要补充。

你可以先粗略理解成：

- 它是 Redis 用来承载消息流的一种结构
- 比单纯 `List` 更适合拿来讲“消息队列”场景

### 6.1 为什么它比 List 更容易被拿来回答 MQ 场景

因为真实面试里如果问：

- Redis 能不能当消息队列

你如果只答 `List`，通常显得偏老旧。

而 `Stream` 更容易让你把回答往：

- 消费组
- 多消费者
- 消息积压

这些方向上带。

### 6.2 但它是不是就能替代专业 MQ

通常不能直接这么说。

更稳的回答是：

- Redis Stream 可以承接轻量消息流场景
- 但如果是高可靠、强顺序、复杂重试、海量堆积等场景，通常还是专业 MQ 更稳

## 7. Lua 为什么经常和 Redis 一起出现

因为 Redis 很多场景都在追求：

- 减少多次网络往返
- 把多个动作组合成原子操作

Lua 正好适合做这件事。

最常见的面试口径：

- 分布式锁删锁
- 库存扣减
- 一次执行多个相关命令

### 7.1 它的本质价值

不是“会写脚本很酷”，而是：

- 把多个相关操作在 Redis 端一次执行完
- 避免并发窗口问题

### 面试怎么说

> Redis 里 Lua 最常见的价值是把多个相关操作做成原子执行，减少并发窗口和多次网络交互。

## 8. 常见场景怎么快速挂结构

### 8.1 签到、活跃标记

优先想到：

- `Bitmap`

### 8.2 UV、大规模去重人数

优先想到：

- `HyperLogLog`

### 8.3 附近门店、附近骑手

优先想到：

- `GEO`

### 8.4 轻量消息流、消费组

优先想到：

- `Stream`

### 8.5 扣库存、删锁、原子组合操作

优先想到：

- `Lua`

## 9. 高频追问小抄

### 9.1 Bitmap 为什么适合签到

因为签到本质上是大量 0/1 状态标记，按位存储更节省空间。

### 9.2 HyperLogLog 为什么适合 UV

因为它适合做大规模基数估算，空间开销小，但结果是近似值。

### 9.3 GEO 是不是只会存经纬度

更重要的是它还能支持“附近范围”这类检索。

### 9.4 Redis 能不能做消息队列

能做轻量场景，但不要直接说它能完全替代专业 MQ。

### 9.5 Lua 最大的价值是什么

把多步 Redis 操作合并成原子执行，减少并发问题。

## 10. 一个比较稳的口语版回答

> Redis 除了常见的五大结构，还有一些面试里很加分的特殊结构。比如 Bitmap 很适合签到、活跃标记这种 0/1 状态压缩；HyperLogLog 适合做 UV 这类大规模去重计数的近似统计；GEO 适合附近门店、附近骑手这类位置场景；Stream 更适合轻量消息流和消费组场景；Lua 则常用于把多步操作做成 Redis 端原子执行，比如删锁、扣库存这类逻辑。面试时重点不是背命令，而是把结构和业务场景对上。

## 11. 一句话收尾

> Redis 特殊结构题的关键，不是背缩写，而是知道“什么业务场景会自然想到它”。
