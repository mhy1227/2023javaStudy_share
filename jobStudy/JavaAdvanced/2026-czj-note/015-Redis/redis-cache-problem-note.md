# Redis 缓存常见问题

## 1. 这份文档怎么用

这一篇是 Redis 面试里最容易连续追问的一块。

你至少要能讲清楚：

- 缓存穿透
- 缓存击穿
- 缓存雪崩
- 缓存和数据库一致性

## 2. 缓存穿透

### 2.1 是什么

查询一个根本不存在的数据。

因为缓存里没有，数据库里也没有，所以每次请求都会打到数据库。

### 2.2 风险

- 无效请求直接穿过缓存
- 数据库压力上升

### 2.3 常见解决方案

- 缓存空对象
- 布隆过滤器
- 接口层参数校验

### 2.4 面试怎么说

> 缓存穿透指的是请求查询的数据本身不存在，导致缓存不命中后每次都打到数据库。常见处理方式有缓存空值、布隆过滤器和前置参数校验。

## 3. 缓存击穿

### 3.1 是什么

某个热点 key 突然失效了。

此时大量并发请求同时去查数据库。

### 3.2 风险

- 热点请求瞬间压到数据库

### 3.3 常见解决方案

- 热点 key 永不过期或逻辑过期
- 互斥锁
- 预热热点数据

### 3.4 面试怎么说

> 缓存击穿通常发生在热点 key 失效瞬间，大量并发请求同时回源数据库，所以常见思路是对热点数据做保护，比如互斥重建、逻辑过期或者热点预热。

## 4. 缓存雪崩

### 4.1 是什么

大量缓存 key 在同一时间集中失效，或者 Redis 整体不可用。

### 4.2 风险

- 大量请求同时打到数据库
- 整体系统可能被拖垮

### 4.3 常见解决方案

- 过期时间加随机值
- 多级缓存
- 降级限流
- Redis 高可用

### 4.4 面试怎么说

> 缓存雪崩强调的是“大面积失效”或“缓存整体不可用”，常见手段是错峰过期、限流降级和提升缓存高可用能力。

## 5. 缓存一致性

这是 Redis 和 MySQL 最常见的联动题。

### 5.1 本质

数据库和缓存不是强一致天然系统。

所以真正要回答的是：

- 你能接受多大程度的不一致
- 你怎么把不一致窗口尽量缩小

### 5.2 常见思路

比较常见的口径是：

- 先更新数据库，再删除缓存

这个比“先删缓存再改数据库”通常更常见一些。

### 5.3 面试里怎么答

> 缓存一致性通常不是追求绝对实时强一致，而是尽量缩小不一致窗口。比较常见的做法是先更新数据库，再删除缓存；如果业务更复杂，还可能结合延迟双删、MQ 或异步补偿去进一步降低风险。

## 6. 面试里一个很稳的总答法

> Redis 缓存常见问题一般可以分成四类：穿透、击穿、雪崩和一致性。穿透是查不存在的数据，击穿是热点 key 失效，雪崩是大面积失效，一致性则更多是缓存和数据库怎么配合。不同问题的重点不一样，但本质上都是为了避免数据库被异常流量直接打穿。

## 7. 一句话收尾

> Redis 缓存题最值钱的，不是背术语，而是你能不能把“问题本质 + 常见解决思路”讲顺。
