# Redis 项目场景题

## 1. 这份文档怎么用

这一篇不是基础定义题，而是更像项目面里的问法。

面试官更可能这样问：

- 你项目里 Redis 怎么用
- 缓存为什么这么设计
- 为什么要加 Redis
- 热点 key 怎么处理
- 分布式锁为什么用 Redis

## 2. 场景题 1

### 问题

你项目里 Redis 主要用来做什么？

### 回答方向

- 先说最核心的 1 到 2 个场景
- 再说为什么不用数据库直接扛

### 稳一点的答法

> 我项目里 Redis 主要还是承担缓存层的作用，比如商品详情、用户信息、首页热点数据这类高频读场景。另外也会用在一些临时状态数据上，比如验证码、登录态、短时防重 key。核心目的还是降低数据库压力、提升读取响应速度。

## 3. 场景题 2

### 问题

为什么这里要加 Redis，不直接查 MySQL？

### 回答方向

- 高频读
- 低延迟
- 削峰

### 稳一点的答法

> 如果是高频读场景，所有请求都直接打 MySQL，数据库很容易成为瓶颈。Redis 作为内存型存储，能更好承接热点流量，让数据库负责最终落库和持久化，Redis 负责高频访问和低延迟读取。

## 4. 场景题 3

### 问题

缓存和数据库怎么保证一致？

### 回答方向

- 不要说成绝对强一致
- 强调缩小不一致窗口

### 稳一点的答法

> 我一般会把它理解成尽量缩小不一致窗口，而不是追求绝对实时强一致。常见做法是先更新数据库，再删除缓存；如果业务更敏感，还可以结合延迟双删、MQ 或异步补偿机制进一步降低风险。

## 5. 场景题 4

### 问题

如果某个商品特别火，热点 key 怎么处理？

### 回答方向

- 不轻易过期
- 多级缓存
- 预热
- 限流

### 稳一点的答法

> 热点 key 我会优先考虑不让它轻易失效，比如逻辑过期或者提前预热。同时如果流量特别高，可以再结合本地缓存或多级缓存，把一部分流量挡在 Redis 之前。再往后还可以配限流和降级兜底。

## 6. 场景题 5

### 问题

如果 Redis 挂了怎么办？

### 回答方向

- 高可用
- 降级
- 数据库保护

### 稳一点的答法

> 一方面架构上要靠 Redis 的高可用方案，比如主从、哨兵、集群去降低整体不可用概率；另一方面业务上还要有降级和限流方案，避免 Redis 一挂数据库被直接打穿。

## 7. 场景题 6

### 问题

为什么这里用 Redis 分布式锁？

### 回答方向

- 多实例部署
- 本地锁不够

### 稳一点的答法

> 因为服务已经是多实例部署，本地锁只能锁住当前 JVM，管不住其他节点，所以要借助 Redis 这种外部协调组件做跨节点互斥控制。

## 8. 场景题 7

### 问题

Redis 分布式锁有什么风险？

### 回答方向

- 锁过期
- 误删
- 组件高可用

### 稳一点的答法

> 风险主要有三类：第一是业务没执行完但锁过期了；第二是释放锁时误删别人的锁；第三是 Redis 自身高可用问题。所以通常要给锁带唯一标识，并关注锁时长和续期策略。

## 9. 场景题 8

### 问题

你们为什么要给缓存设置过期时间？

### 回答方向

- 控制数据生命周期
- 避免脏数据长期存在
- 结合随机过期

### 稳一点的答法

> 给缓存设置过期时间，一方面是控制数据生命周期，避免脏数据长期存在；另一方面也让系统有机会自动刷新缓存。当然为了避免集中失效，过期时间一般还会做随机打散。

## 10. 场景题 9

### 问题

你遇到过大 key 吗？怎么处理？

### 回答方向

- 先说风险
- 再说拆分思路

### 稳一点的答法

> 大 key 的问题主要在于传输开销大、操作耗时长，甚至会影响 Redis 整体性能。处理上一般优先考虑拆分、分页读取或者按业务维度分片，而不是把过多数据塞进一个 key。

## 11. 场景题 10

### 问题

如果面试官问“Redis 和 MQ 的边界怎么理解”，你怎么接？

### 回答方向

- List 只能做轻量队列
- 正式消息链路优先 MQ

### 稳一点的答法

> Redis 可以在一些轻量场景里做简单队列，但如果是正式的消息可靠投递、重试、顺序性和消费确认这些要求，还是更适合交给 MQ。Redis 更偏缓存和高频访问承接，不适合把所有消息职责都扛下来。

## 12. 一句话收尾

> Redis 项目题最重要的是让面试官听出来：你不是只知道它快，而是知道它在系统里扛什么、风险在哪、怎么兜底。
