# Redis 模拟口语脚本

## 1. 这份脚本怎么用

这不是题库。

这是按真实一面对话的感觉，把 Redis 常见追问串成一条线。

你可以自己练，也可以找人按这份脚本追问你。

## 2. 第一轮：先看你会不会讲整体

### 面试官

你项目里用 Redis 吗，主要怎么用？

### 参考口语版

> 用。我们项目里 Redis 主要还是放在数据库前面承接热点读和高频重复访问。像商品详情、首页聚合数据、活动页展示信息这类读多写少的数据，我会优先考虑走 Redis；另外像验证码、登录态、短期状态控制、幂等 key 这类天然带过期时间的数据，也比较适合放 Redis。它在项目里更多是高性能缓存层和轻量协调层，不是拿来替代 MySQL 的。

## 3. 第二轮：继续追“为什么一定要用 Redis”

### 面试官

那为什么要加 Redis，不直接查 MySQL 不行吗？

### 参考口语版

> 如果这些高频请求都直接打 MySQL，数据库会比较容易成为瓶颈，尤其是热点数据被频繁重复读取的时候。Redis 的价值不只是快，而是能把重复读先挡在前面，降低数据库压力，同时把接口响应时间拉下来。我的理解是 Redis 更像一个高性能缓冲层，让数据库专注做最终持久化和事务。

## 4. 第三轮：问你为什么快

### 面试官

那 Redis 为什么快？

### 参考口语版

> 我一般会从几个点回答。第一是它基于内存，天然比磁盘访问快；第二是它的单线程命令执行模型减少了很多线程切换和锁竞争成本；第三是它配合 IO 多路复用处理网络连接；第四是它本身很多数据结构就是为高性能读写设计的。所以它的快不是单靠“内存”一个点，而是模型和数据结构一起起作用。

## 5. 第四轮：开始问缓存问题

### 面试官

那你说说缓存穿透、击穿、雪崩。

### 参考口语版

> 这三个我会先区分本质。穿透是查一个根本不存在的数据，请求直接一路打到数据库；击穿是某个热点 key 失效瞬间，大量请求同时回源；雪崩则是大面积 key 集中失效，或者 Redis 整体不可用，导致很多请求一起压到数据库。三者结果都可能把数据库打爆，但成因不一样，所以处理思路也不一样。

## 6. 第五轮：继续问缓存一致性

### 面试官

那缓存和数据库怎么保持一致？

### 参考口语版

> 我一般不会把它讲成绝对强一致，因为 Redis 和 MySQL 不是一个原子事务系统。工程上更常见的做法是先更新数据库，再删除缓存，这样后续缓存 miss 重建时更容易读到数据库里的新值。当然这也不是完美方案，因为还可能遇到删缓存失败或者并发读回填旧值的问题，所以如果业务更敏感，还要考虑延时双删、异步重试、MQ 或补偿机制。

## 7. 第六轮：开始追高可用

### 面试官

如果 Redis 挂了怎么办？

### 参考口语版

> 这个我一般分两层说。第一层是 Redis 自身高可用，比如主从、哨兵、集群，尽量降低缓存层整体挂掉的概率；第二层是业务兜底，比如热点保护、限流、降级、必要时部分请求回源数据库但控制节奏。因为真正可怕的不是 Redis 挂本身，而是它挂了以后数据库和下游被一起打穿。

## 8. 第七轮：继续深问主从和哨兵

### 面试官

那你讲讲主从、哨兵、集群分别是干什么的。

### 参考口语版

> 主从是高可用基础，主要解决数据副本和读扩展；哨兵是在主从基础上补故障检测和自动切主能力；集群则不只是高可用，还进一步解决数据分片和横向扩容问题。也就是主从解决复制，哨兵解决切主，集群解决分片加高可用。

## 9. 第八轮：开始追主从复制细节

### 面试官

主从复制是强一致吗？

### 参考口语版

> 不是。主从复制天然会有同步延迟，所以更接近最终一致。比如刚写完主节点就立刻去读从节点，是有可能读到旧数据的。所以读写分离不是不能做，而是要分业务场景，一致性敏感链路通常还是更适合读主。

## 10. 第九轮：开始追分布式锁

### 面试官

你们 Redis 还用来做过分布式锁吗？

### 参考口语版

> 会考虑。因为服务一旦是多实例部署，本地锁只能管住当前 JVM，跨实例互斥就需要外部协调点。Redis 锁比较适合做轻量业务互斥，比如防重复提交、控制某类任务并发执行。但我不会把它讲成绝对安全锁，因为它还要考虑过期时间、误删、业务超时、主从切换这些边界问题。

## 11. 第十轮：继续追锁的坑

### 面试官

为什么释放锁不能直接 `del`？

### 参考口语版

> 因为锁可能已经过期并被别人重新拿到了。如果我不校验 value，直接删 key，就有可能把别人的锁删掉。所以更稳的做法通常是给锁一个唯一 value，然后通过 Lua 把“校验是不是自己的锁”和“删除锁”做成 Redis 端原子操作。

## 12. 第十一轮：问特殊结构

### 面试官

除了常见五大结构，你还用过别的吗？

### 参考口语版

> 如果按场景说，像签到这种 0/1 状态压缩我会想到 Bitmap，做 UV 近似统计会想到 HyperLogLog，做附近门店或附近骑手会想到 GEO，如果是轻量消息流场景会想到 Stream。我的理解不是背这些结构本身，而是知道什么业务会自然想到它们。

## 13. 第十二轮：开始问边界

### 面试官

那 Redis 这么好，为什么不什么都用 Redis 做？

### 参考口语版

> 因为 Redis 不是银弹。它非常适合热点读、临时状态、轻量协调和低延迟场景，但它不适合承担所有最终权威数据。持久化、事务、一致性、复杂查询这些能力，MySQL 仍然更合适。项目里更常见的是两者分工合作，而不是让 Redis 把所有事情都接管。

## 14. 第十三轮：最后压缩成总结

### 面试官

那你总结一下 Redis 在系统里的价值和风险。

### 参考口语版

> 我会把 Redis 理解成系统里的高性能缓冲层和轻量协调层。它的价值是承接热点读、降低数据库压力、提升响应速度，并在部分场景下承担分布式协调能力；它的风险则在于缓存一致性、热点失效、分布式锁边界、高可用切换窗口这些问题。所以真正做项目时，不能只看 Redis 带来的性能收益，还要同时设计限流、降级、补偿和数据库兜底。

## 15. 使用建议

练这份脚本时，重点看三件事：

- 你是不是总在背定义，没有往业务和系统里挂
- 你是不是总把 Redis 说成绝对安全、绝对一致、绝对高可用
- 你是不是能在被追问两三轮后还能接得住

## 16. 一句话收尾

> 模拟口语脚本的目标，不是让你逐字背，而是帮你把 Redis 这一章练成“先能讲，再能接，再能收”。
