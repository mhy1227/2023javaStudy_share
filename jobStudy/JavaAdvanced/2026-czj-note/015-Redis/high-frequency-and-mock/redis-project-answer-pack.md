# Redis 项目答法包

## 1. 这份文档怎么用

这一篇专门解决一个问题：

- 你知道 Redis 的知识点，但一到“你项目里怎么用”的题就答得很空

所以这里的重点不是解释概念，而是给你“项目口语模板”。

## 2. 题 1：你项目里 Redis 主要怎么用

### 口语版

> 我项目里 Redis 主要还是承担缓存层的作用，比如商品详情、用户信息、首页热点数据这类高频读场景。另外也会用在一些临时状态数据上，比如验证码、token、防重 key。核心目的还是承接热点流量，降低数据库压力。

## 3. 题 2：为什么一定要加 Redis

### 口语版

> 如果这些高频读请求都直接打 MySQL，数据库会比较容易成为瓶颈。Redis 更适合承接热点数据和低延迟访问，所以我会把它理解成数据库前面的一层加速和缓冲。

## 4. 题 3：Redis 和 MySQL 在项目里怎么分工

### 口语版

> MySQL 更负责最终持久化数据和事务型数据，Redis 更负责高频访问和热点流量承接。两者在项目里更多是配合关系，不是互相替代。

## 5. 题 4：缓存一致性怎么讲更稳

### 口语版

> 我一般不会把它讲成绝对强一致，而是尽量缩小不一致窗口。比较常见的做法是先更新数据库，再删除缓存；如果业务更敏感，还可以结合延迟双删、MQ 或补偿机制进一步降低风险。

## 6. 题 5：热点 key 怎么处理

### 口语版

> 热点 key 我会优先考虑不让它轻易失效，比如逻辑过期、预热，或者在活动开始前把热点数据提前准备好。如果流量特别高，还可以结合本地缓存、多级缓存、限流和降级一起兜底。

## 7. 题 6：Redis 分布式锁怎么放进项目里讲

### 口语版

> 如果服务是多实例部署，本地锁只能锁住当前 JVM，所以跨节点互斥场景下我会考虑 Redis 分布式锁。它比较适合做一些全局资源互斥控制，比如防止重复下单、控制任务重复执行，但同时也要注意锁过期和误删风险。

## 8. 题 7：Redis 挂了怎么办

### 口语版

> 一方面架构层面要依赖 Redis 的高可用方案，比如主从、哨兵、集群去降低整体不可用概率；另一方面业务层面还要有降级和限流方案，避免 Redis 一挂数据库被直接打穿。

## 9. 题 8：为什么排行榜喜欢用 Redis

### 口语版

> 排行榜这种场景典型特点是读多、排序频繁、延迟敏感，所以更适合用 Redis 的 `ZSet` 承接。MySQL 当然也能做排序，但在高频访问场景下，Redis 的体验会更好。

## 10. 题 9：如果面试官问“你遇到过 Redis 的问题吗”

### 口语版

> 我会从几个维度讲：一是缓存一致性，二是热点 key 和击穿风险，三是 Redis 不可用时数据库承压问题，四是分布式锁边界。这样面试官会更容易感觉到我不是只会背 Redis 的优点，也知道它的失败路径。

## 11. 题 10：最稳的一段项目总答法

### 口语版

> 我理解 Redis 在项目里最主要的价值，不是“它快”这么简单，而是它能作为数据库前面的一层高性能缓存和热点流量承接层。像高频读、热点数据、临时状态、分布式锁、排行榜这类场景都比较适合交给 Redis。但同时它也带来一致性、热点 key、分布式锁边界和高可用这些问题，所以项目里通常要把它和 MySQL、高并发保护、降级限流一起考虑。

## 12. 一句话收尾

> Redis 项目题最重要的，不是背功能点，而是让面试官听出来你知道它在系统里扛什么、风险在哪、怎么兜底。
