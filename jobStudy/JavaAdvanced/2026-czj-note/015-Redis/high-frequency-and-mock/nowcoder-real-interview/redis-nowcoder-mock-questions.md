# Redis 牛客面经模拟追问题

## 1. 这份文档怎么用

这一篇不是知识点总结，而是更像一面现场的连续追问脚本。

适合：

- 自己口头顺一遍 Redis 问答节奏
- 找同学陪练，一问一答模拟一面
- 看看自己是卡在基础、场景，还是卡在继续追问

## 2. 第一轮：基础起手题

### 面试官

你先整体讲一下 Redis 是什么，项目里一般拿它干什么？

### 候选人作答方向

- 它是基于内存的高性能键值数据库
- 常见用途是缓存、分布式锁、计数器、排行榜、会话、热点数据
- 项目里通常是 MySQL 负责“数据准”，Redis 负责“访问快”

### 面试官继续追问

那 Redis 为什么快？

### 候选人作答方向

- 基于内存
- 数据结构做过专门优化
- 早期核心命令执行模型单线程，减少锁竞争
- IO 多路复用提高网络层效率

## 3. 第二轮：数据结构题

### 面试官

Redis 常见数据类型有哪些？你项目里各自怎么用？

### 候选人作答方向

- String：缓存对象、计数器、分布式锁
- Hash：适合存对象属性
- List：消息队列、时间线
- Set：去重、共同关注
- Sorted Set：排行榜、按分数排序

### 面试官继续追问

那为什么对象经常用 Hash 存，而不是直接 String 存 JSON？

### 候选人作答方向

- String 存 JSON 简单，序列化方便
- Hash 适合部分字段读写
- 最终要看对象大小、字段更新频率、业务读写方式

## 4. 第三轮：持久化和高可用

### 面试官

Redis 的持久化方案有哪些？RDB 和 AOF 有什么区别？

### 候选人作答方向

- RDB：快照，恢复快，但可能丢最后一段数据
- AOF：记录写命令，数据更完整，但文件更大，恢复更慢
- 实际生产里可以组合使用

### 面试官继续追问

那主从、哨兵、集群呢？你怎么讲它们的区别？

### 候选人作答方向

- 主从：读写分离、数据副本
- 哨兵：在主从基础上做故障监控和自动切换
- 集群：在高可用基础上进一步解决数据分片和扩容

## 5. 第四轮：缓存问题

### 面试官

缓存穿透、击穿、雪崩分别是什么？

### 候选人作答方向

- 穿透：查不存在的数据，打到数据库
- 击穿：热点 key 失效瞬间，大量请求同时回源
- 雪崩：大量 key 同时失效，或 Redis 整体不可用

### 面试官继续追问

那你分别怎么解决？

### 候选人作答方向

- 穿透：缓存空值、布隆过滤器
- 击穿：互斥锁、逻辑过期、热点 key 常驻
- 雪崩：过期时间打散、多级缓存、限流降级、集群高可用

## 6. 第五轮：一致性

### 面试官

缓存和数据库一致性怎么保证？

### 候选人作答方向

- 先说大多数场景追求的是最终一致，不是绝对强一致
- 常见策略：更新数据库后删除缓存
- 可继续补：延迟双删、消息队列异步通知、重试机制

### 面试官继续追问

为什么很多人推荐删除缓存，而不是更新缓存？

### 候选人作答方向

- 删除缓存逻辑更简单
- 更新缓存容易漏字段、漏逻辑
- 下次读取时可以按最新数据库结果重建缓存

## 7. 第六轮：分布式锁

### 面试官

Redis 分布式锁怎么实现？

### 候选人作答方向

- `SET key value NX EX`
- 必须带唯一 value
- 释放锁时用 Lua 脚本做 compare-and-delete

### 面试官继续追问

锁过期了但业务还没执行完怎么办？

### 候选人作答方向

- 可引入续期机制
- 业务超时要有兜底
- 也要考虑锁误删和锁重入问题

### 面试官再追问

RedLock 怎么看？

### 候选人作答方向

- 可以知道它是多节点分布式锁方案
- 面试里不必吹得太满
- 更稳的答法是：它解决的是更复杂的分布式场景，但实现复杂、争议也大，很多业务场景单实例 Redis 锁配合高可用已经够用

## 8. 第七轮：性能和故障题

### 面试官

Redis 变慢了，你先怀疑什么？

### 候选人作答方向

- 是否出现 hotkey / bigkey
- 网络抖动、连接数暴涨
- 慢查询命令或阻塞命令
- 持久化压力
- 内存打满、淘汰过多

### 面试官继续追问

hotkey 和 bigkey 怎么处理？

### 候选人作答方向

- hotkey：本地缓存、多副本分摊、热点拆分、限流
- bigkey：拆 key、异步删除、优化结构、控制单 key 大小

## 9. 第八轮：项目题

### 面试官

你项目里 Redis 真正用来做什么？

### 候选人作答方向

- 不要只说“做缓存”
- 要具体说缓存了什么数据
- 为什么这些数据适合进 Redis
- 更新时怎么处理一致性
- 有没有遇到热点 key、缓存失效、锁问题

### 面试官继续追问

为什么不用 MySQL 顶着？

### 候选人作答方向

- MySQL 适合持久化和复杂查询
- Redis 适合高频读、热点数据、瞬时并发
- 两者不是替代关系，而是分工不同

## 10. 最后一轮：总收口

### 面试官

如果让你用两分钟总结 Redis 在面试里最重要的部分，你会怎么讲？

### 候选人作答方向

- Redis 是高性能内存数据库
- 高频问题集中在数据结构、持久化、缓存问题、一致性、为什么快、分布式锁、主从哨兵集群、项目场景
- 重点不是背命令，而是能把原理、场景、问题和项目落地串起来

## 11. 一句话收尾

> Redis 的模拟追问题，最关键的不是把每个小点都背满，而是能不能顺着“基础 -> 缓存 -> 一致性 -> 锁 -> 高可用 -> 项目”这条线稳定往下讲。
