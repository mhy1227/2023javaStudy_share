# Redis 牛客高频题口语答法

## 1. 这份文档怎么用

这一篇不是把所有题都展开成长文，而是把牛客上最常见的 Redis 问法压成更适合面试现场的答法骨架。

建议使用方式：

- 先自己尝试回答
- 再对照这里的答法框架补短板
- 回答时优先讲“问题是什么、为什么、怎么做、项目里怎么落”

## 2. Redis 为什么这么快

### 口语版答法

> Redis 快的核心原因有几层。第一，它的绝大部分操作基于内存，不像 MySQL 那样大量依赖磁盘。第二，它的数据结构做了针对性优化，不是简单把数据扔进内存。第三，早期核心命令执行模型是单线程，避免了大量锁竞争和上下文切换。第四，它使用了 IO 多路复用，所以网络层处理效率也比较高。面试里如果继续追问，我会再补 Redis 6 之后引入多线程主要是为了网络读写和协议解析，不是把命令执行彻底多线程化。

## 3. RDB 和 AOF 有什么区别

### 口语版答法

> RDB 更像定时做快照，恢复速度快，但可能丢最后一次快照之后的数据；AOF 是把写命令追加记录下来，数据更完整，但文件会更大，恢复时也更慢一些。面试里更稳的答法不是只背区别，而是顺手补一句：如果更看重恢复速度和文件紧凑，偏 RDB；如果更看重数据完整性，偏 AOF；实际生产里也常常会组合使用。

## 4. 缓存穿透、击穿、雪崩怎么区分

### 口语版答法

> 穿透是查一个本来就不存在的数据，请求直接打到数据库；击穿是某个热点 key 失效瞬间，大量请求同时打到数据库；雪崩是大量 key 在同一时间段失效，或者 Redis 整体不可用，导致请求成片回源。  
> 解决思路也要一起说：穿透常见是布隆过滤器或者缓存空值；击穿常见是互斥锁、逻辑过期、热点永不过期；雪崩常见是过期时间打散、多级缓存、降级限流、主从哨兵集群。

## 5. 缓存和数据库一致性怎么保证

### 口语版答法

> 这个问题面试里通常不要求你说成绝对强一致，而是要讲清常见策略。我一般会先说 Redis 作为缓存，大多数业务场景做的是最终一致。比较常见的是先更新数据库，再删除缓存，让下次读取时回源重建缓存。如果再往下问，我会补延迟双删、重试机制、消息队列异步通知、热点数据更新策略这些点。

## 6. Redis 分布式锁怎么实现

### 口语版答法

> 最基本的思路是 `SET key value NX EX`，也就是只有 key 不存在时才能加锁，并且一定要带过期时间避免死锁。释放锁时不能直接 `del`，更稳的做法是比较 value 再删除，通常会用 Lua 脚本保证原子性。面试里如果继续追问，我会再补锁续期、业务执行时间超过过期时间怎么办、RedLock 有什么争议这些问题。

## 7. Redis 主从、哨兵、集群怎么讲

### 口语版答法

> 主从主要解决读写分离和数据副本问题；哨兵是在主从基础上再加故障监控和自动故障转移；集群则更进一步，不只是高可用，还解决数据分片和横向扩容问题。面试时我一般会先按“能力层次”去讲，这样比死记定义更顺。

## 8. Redis 的淘汰策略和过期策略怎么讲

### 口语版答法

> 这两个问题容易混。过期策略是在 key 到达过期时间后，Redis 怎么清理它，通常是惰性删除配合定期删除；淘汰策略是在内存打满以后，Redis 该淘汰哪些 key，比如 LRU、LFU、随机淘汰、只淘汰设置了过期时间的 key 等。一个是“到期怎么删”，一个是“内存满了删谁”。

## 9. hotkey / bigkey 怎么讲

### 口语版答法

> hotkey 是流量过于集中打到某个 key，容易把单机 Redis 打爆；bigkey 是某个 key 本身体积太大，可能带来网络阻塞、删除阻塞、主从同步压力。面试里讲这类题，除了定义，最好顺手补怎么排查和治理，比如热点拆分、本地缓存、多副本分摊、大 key 拆分、异步删除等。

## 10. 项目里为什么 Redis 和 MySQL 要一起用

### 口语版答法

> Redis 和 MySQL 不是替代关系，而是分工不同。MySQL 更适合做持久化、强约束、复杂查询和核心数据存储；Redis 更适合做高频读缓存、热点数据、分布式锁、计数器、排行榜这类对性能要求更高的场景。项目里通常是 MySQL 负责“数据准”，Redis 负责“访问快”。

## 11. 项目里怎么讲 Redis 用法更真实

### 口语版答法

> 我一般不会只说“项目里用了 Redis 做缓存”，而是会具体说缓存了什么数据、为什么这些数据适合进 Redis、更新时怎么保证一致性、有没有遇到热 key 或缓存失效问题、以及最终为什么这个方案比只查 MySQL 更稳。

## 12. 一句话收尾

> Redis 相关题最稳的答法，不是只背定义，而是把“为什么、怎么做、有什么坑、项目里怎么落”四件事一起讲出来。
