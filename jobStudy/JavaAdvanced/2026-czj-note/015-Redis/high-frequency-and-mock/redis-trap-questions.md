# Redis 易踩坑问法

## 1. 这份文档怎么用

这一篇不是补新知识，而是帮你避开 Redis 面试里最常见的翻车点。

很多 Redis 题不是不会，而是：

- 说得太绝对
- 把几个概念混了
- 只会背结论，不会结合场景

## 2. 坑 1：Redis 快就是因为“单线程”

这句话不算全错，但太浅。

如果你只答：

> Redis 快是因为单线程

面试官很容易继续追问你：

- 那单线程为什么不慢
- 多线程是不是就一定更快

更稳的答法应该是：

> Redis 快的核心还是内存和高效数据结构，单线程更多是避免锁竞争和线程切换，不是唯一原因。

## 3. 坑 2：把 Redis 说成“不会丢数据”

这是一个明显错误。

Redis 本质上以内存为主，所以天然就存在数据丢失风险。

更稳的说法：

> Redis 会通过持久化和高可用机制尽量降低数据丢失风险，但不能简单理解成绝对不丢。

## 4. 坑 3：穿透、击穿、雪崩混着说

这是最经典的翻车点之一。

你只要记住：

- 穿透：查不存在的数据
- 击穿：热点 key 失效
- 雪崩：大面积失效或 Redis 整体不可用

不要为了显得会讲，把三个问题说成一大段但区分不清。

## 5. 坑 4：缓存一致性答成“绝对强一致”

很多人一紧张就会说：

> 我们会保证缓存和数据库完全一致

这就很容易被追问穿。

更稳的回答：

> Redis 缓存一致性一般不是追求绝对实时强一致，而是尽量缩小不一致窗口。

## 6. 坑 5：Redis 分布式锁直接删就行

如果你答：

> 业务执行完直接 `del` 锁 key

面试官大概率会继续追：

- 如果删掉别人的锁怎么办

所以要记住：

- 锁要带唯一标识
- 释放时要确认是不是自己持有的锁

## 7. 坑 6：热点 key 和大 key 是一回事

这两个很容易混。

要分清：

- 热点 key 看访问是否过于集中
- 大 key 看单个 key 的数据规模是否过大

它们可能同时出现，但不是同一个维度的问题。

## 8. 坑 7：主从、哨兵、集群不分

这也是很常见的问题。

只要记住三句话就够了：

- 主从：复制
- 哨兵：切主
- 集群：分片 + 高可用

## 9. 坑 8：项目里为什么用 Redis，只会答“因为快”

这个回答太学生气了。

更像项目表达的说法是：

> 因为 Redis 能承接高频读和热点流量，降低数据库压力，同时在某些场景下还能承担计数、排行榜、分布式锁和临时状态数据的职责。

## 10. 一句话收尾

> Redis 题最容易翻车的地方，不是不会，而是答得太绝对、太泛、太像背诵。
