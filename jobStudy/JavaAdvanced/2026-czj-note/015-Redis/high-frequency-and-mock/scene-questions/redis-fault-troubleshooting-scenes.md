# Redis 故障排查场景题

## 1. 这份文档怎么用

这一篇偏“故障处理”和“线上排查”。

面试官如果想往更实战一点追，经常会这样问：

- Redis 挂了怎么办
- 命中率突然下降怎么办
- 延迟突然变高怎么办

## 2. 场景题 1

### 问题

如果 Redis 整体不可用了，你第一反应会怎么做？

### 回答方向

- 先止血
- 再排查
- 最后恢复

### 稳一点的答法

> 我第一反应会先止血，也就是保护数据库和主链路，比如限流、降级、本地兜底缓存，避免 Redis 不可用后所有请求同时打到 MySQL。然后再去排查是节点问题、主从切换问题还是网络问题，最后再看恢复和补偿。

## 3. 场景题 2

### 问题

如果 Redis 命中率突然下降，你会怎么分析？

### 回答方向

- 先看是否是大面积过期
- 再看是否是业务 key 变了
- 再看是否有异常流量

### 稳一点的答法

> 命中率突然下降，我会先看是不是一批 key 集中过期了，或者缓存重建策略失效；然后再看是否有业务代码改动导致 key 规则变化；最后看有没有异常流量、穿透请求或者 Redis 本身状态异常。

## 4. 场景题 3

### 问题

如果 Redis 延迟突然升高，你会优先怀疑什么？

### 回答方向

- 大 key
- 热点 key
- 网络 / 连接问题
- 阻塞性操作

### 稳一点的答法

> Redis 延迟升高，我会优先看是不是大 key 或热点 key 带来的局部压力，再看是否有慢命令、阻塞性操作、网络抖动或者连接数异常。因为 Redis 本身很快，延迟明显升高往往意味着某个点已经不正常了。

## 5. 场景题 4

### 问题

如果某个热点 key 被打得特别厉害，你会怎么止损？

### 回答方向

- 本地缓存
- 多级缓存
- 限流
- 不轻易过期

### 稳一点的答法

> 我会先把这个热点 key 保护起来，比如不要轻易过期，或者改成逻辑过期；如果流量继续升高，再加本地缓存或多级缓存分流；必要时做限流和降级，避免 Redis 单点被持续打爆。

## 6. 场景题 5

### 问题

如果面试官问“你怎么发现 Redis 有大 key”，你怎么答？

### 回答方向

- 监控
- 慢查询 / 操作延迟
- 网络流量异常

### 稳一点的答法

> 大 key 往往不是靠猜出来的，一般会从监控里发现，比如某些命令耗时变长、网络流量异常、某些 key 的读写明显偏慢。再结合 key 分布和数据规模去确认是不是单个 key 数据量过大。

## 7. 场景题 6

### 问题

如果缓存一致性被投诉了，你会怎么排查？

### 回答方向

- 先确认数据链路
- 再看删缓存是否失败
- 再看异步补偿是否失效

### 稳一点的答法

> 我会先还原数据链路，看这条业务是先更数据库再删缓存，还是别的策略；然后排查删缓存有没有失败、重试有没有丢、补偿或异步链路有没有失效；最后确认是不是缓存过期策略或者热点 key 保护策略导致的旧数据残留。

## 8. 场景题 7

### 问题

如果 Redis 分布式锁出现了误删或者锁过期问题，你怎么解释？

### 回答方向

- 唯一标识
- 业务执行时间超过锁时长
- 边界控制

### 稳一点的答法

> 这类问题一般会从两个方向解释：一个是释放锁时没有严格校验唯一标识，导致误删别人的锁；另一个是业务执行时间超过锁时长，锁已经自动过期又被别人拿走。核心还是锁时长、唯一标识和续期策略没控制好。

## 9. 一句话收尾

> Redis 故障排查题最重要的，不是说得多，而是先止血、再定位、再恢复，这个顺序要稳。
