# Redis 缓存一致性与热点场景题

## 1. 这份文档怎么用

这一篇专门把 Redis 面试里最常和项目题绑定的两条线放在一起：

- 缓存一致性
- 热点 key / 热点流量

这两块是最容易被深问的。

## 2. 场景题 1

### 问题

为什么缓存一致性通常不讲成绝对强一致？

### 回答方向

- Redis 是缓存层
- 数据链路有窗口
- 重点是缩小不一致时间

### 稳一点的答法

> 因为 Redis 缓存本身就不是强一致主存，它更像加速层。只要数据库更新和缓存更新不是一个天然原子操作，就一定会有时间窗口。所以项目里更常讲的是尽量缩小不一致窗口，而不是承诺绝对实时强一致。

## 3. 场景题 2

### 问题

如果让你设计“先更新数据库还是先更新缓存”的流程，你会怎么讲？

### 回答方向

- 常见口径：先更数据库，再删缓存
- 说明原因

### 稳一点的答法

> 比较常见的口径是先更新数据库，再删除缓存。因为缓存本质上是数据库结果的派生层，先把数据库更新成功，再让缓存失效去重建，会更符合主存和缓存的关系。当然这也不是说完全没有风险，所以还要考虑失败补偿。

## 4. 场景题 3

### 问题

如果删缓存失败了怎么办？

### 回答方向

- 重试
- 延迟双删
- MQ / 补偿任务

### 稳一点的答法

> 删缓存失败是缓存一致性里很典型的失败路径，所以一般不能只写一条删除逻辑就算了。更稳的方式是配重试、延迟双删，或者通过 MQ、补偿任务把这条失败路径接住。

## 5. 场景题 4

### 问题

如果某个热点 key 失效了，大量请求同时回源数据库，你怎么处理？

### 回答方向

- 这是击穿
- 热点 key 保护
- 互斥重建 / 逻辑过期 / 预热

### 稳一点的答法

> 这类场景本质上是缓存击穿，重点不是一般缓存策略，而是热点 key 保护。我会优先考虑逻辑过期、互斥重建和预热，尽量不让热点 key 一失效就把数据库暴露出来。

## 6. 场景题 5

### 问题

如果一个大促活动开始前，你会怎么利用 Redis 提前做热点保护？

### 回答方向

- 热点数据预热
- 多级缓存
- 限流
- key 生命周期设计

### 稳一点的答法

> 我会先把活动相关热点数据提前预热到 Redis，避免首波请求全部回源数据库；如果访问特别集中，再考虑本地缓存或多级缓存分流；同时对热点 key 的过期时间和重建策略单独设计，必要时再加限流兜底。

## 7. 场景题 6

### 问题

为什么热点 key 的处理不能只靠“加机器”？

### 回答方向

- 热点集中度问题
- 单 key 局部瓶颈
- 需要从数据和流量两侧处理

### 稳一点的答法

> 热点 key 的问题本质上是访问太集中，不是简单的总流量变大。所以只加机器不一定能解决单 key 局部瓶颈，还是要从热点数据生命周期、多级缓存、流量分流和限流策略去处理。

## 8. 场景题 7

### 问题

如果面试官说“那你们为什么不直接让热点 key 永不过期”，你怎么接？

### 回答方向

- 永不过期有价值
- 但也要考虑数据更新
- 逻辑过期更稳

### 稳一点的答法

> 热点 key 永不过期确实是一种思路，但前提是这类数据允许这么做。如果数据还会变化，就不能只追求稳定命中，还要考虑更新时机和旧数据问题。所以很多时候逻辑过期会比简单永不过期更稳。

## 9. 一句话收尾

> Redis 里最容易被深问的场景题，往往不是“它快不快”，而是“缓存一致性和热点流量出问题时你怎么接得住”。
