# Redis 高频面试场景题

## 1. 这份文档怎么用

这一篇专门整理 Redis 最常被问到的项目场景题。

它不是考你会不会背定义，而是看你能不能把 Redis 放回系统里讲清楚。

## 2. 场景题 1

### 问题

如果商品详情页访问量很高，你为什么会考虑在这里加 Redis？

### 回答方向

- 高频读
- 热点流量
- 数据库减压

### 稳一点的答法

> 这种场景最典型的特点就是读多写少、访问集中，如果所有请求都直接打 MySQL，数据库很容易成为瓶颈。Redis 更适合承接热点数据和高频访问，所以我会把它作为数据库前面的一层缓存，让数据库负责最终持久化，Redis 负责低延迟读取和流量分流。

## 3. 场景题 2

### 问题

如果一个活动开始后流量突然暴涨，你会怎么利用 Redis 做削峰？

### 回答方向

- 热点数据预热
- 限流计数
- 本地缓存 / 多级缓存

### 稳一点的答法

> 如果是活动流量暴涨，我会先考虑把热点配置和热点商品信息提前预热到 Redis，避免首波流量直接打数据库。另外 Redis 还可以配合做限流计数和热点流量承接，如果单个 key 特别热，再结合本地缓存或多级缓存去进一步削峰。

## 4. 场景题 3

### 问题

项目里什么时候会用到 Redis 分布式锁？

### 回答方向

- 多实例部署
- 本地锁不够
- 互斥控制

### 稳一点的答法

> 只要服务已经是多实例部署，本地锁就只能锁住当前 JVM，这时候如果需要做跨节点互斥，比如防重复下单、控制某个任务只能执行一次，就会考虑 Redis 分布式锁。

## 5. 场景题 4

### 问题

如果让你用 Redis 做排行榜，你会怎么解释这个选型？

### 回答方向

- `ZSet`
- 高并发读
- 排序需求

### 稳一点的答法

> 排行榜这种场景既要存成员，又要基于分数排序，所以很自然会想到 Redis 的 `ZSet`。再加上排行榜通常是高频读场景，用 Redis 承接会比直接用 MySQL 做排序查询更适合。

## 6. 场景题 5

### 问题

项目里为什么要给缓存设置过期时间，而不是永久放着？

### 回答方向

- 控制数据生命周期
- 避免脏数据长期存在
- 系统自动刷新机会

### 稳一点的答法

> 给缓存设置过期时间，一方面是为了控制数据生命周期，避免脏数据长期存在；另一方面也让系统有机会自动刷新缓存。当然如果是热点 key，还要考虑逻辑过期或者随机过期，避免集中失效。

## 7. 场景题 6

### 问题

如果项目里有人说“既然 Redis 很快，那能不能把很多数据都塞 Redis，不查数据库了”，你怎么回应？

### 回答方向

- Redis 不是 MySQL 替代品
- 内存成本
- 持久化和事务边界

### 稳一点的答法

> Redis 适合承接热点数据和高频访问，但它不等于 MySQL 的替代品。MySQL 更负责最终持久化和事务型数据，Redis 更像加速层。如果把所有数据都堆到 Redis，不仅成本高，也会把缓存和主存的边界搞乱。

## 8. 场景题 7

### 问题

你项目里 Redis 和 MySQL 一般怎么配合？

### 回答方向

- MySQL 主存
- Redis 缓存层
- 一致性窗口

### 稳一点的答法

> 我一般把它理解成主存和缓存层的配合。MySQL 负责最终落库和持久化，Redis 负责高频读取和热点数据承接。真正麻烦的地方不是“怎么一起用”，而是缓存一致性窗口怎么控制，所以这块通常要配套删除缓存、补偿或异步机制去考虑。

## 9. 场景题 8

### 问题

如果某些接口用了 Redis 之后，面试官继续问“那 Redis 失效的时候怎么办”，你该怎么接？

### 回答方向

- Redis 高可用
- 限流降级
- 数据库保护

### 稳一点的答法

> 我会从两层讲：第一层是架构层面，通过主从、哨兵、集群降低 Redis 整体不可用概率；第二层是业务层面，通过限流、降级和热点数据保护避免 Redis 一失效就把数据库直接打穿。

## 10. 一句话收尾

> Redis 场景题最重要的不是“我会不会用”，而是“我知不知道它在系统里承担什么角色，失败时怎么兜底”。
