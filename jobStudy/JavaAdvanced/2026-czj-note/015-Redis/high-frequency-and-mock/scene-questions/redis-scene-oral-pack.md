# Redis 场景题口语版

## 1. 这份文档怎么用

这一篇专门把 Redis 场景题压成更容易直接说出口的版本。

适合：

- 面试前最后一轮过口语
- 发现自己知道场景，但答得不够像项目表达
- 想把 Redis 放回系统里去讲

## 2. 场景 1：为什么这里要加 Redis

### 口语版

> 这种场景我会优先考虑 Redis，核心原因是它能承接高频读和热点流量，减轻 MySQL 压力。像商品详情、首页热点数据这类读多写少场景，如果所有请求都直接打数据库，数据库很容易成为瓶颈，所以 Redis 更像数据库前面的一层缓存和加速层。

## 3. 场景 2：活动流量暴涨怎么办

### 口语版

> 如果是活动流量暴涨，我会先做热点数据预热，避免首波请求直接打到数据库；如果热点非常集中，再结合本地缓存或多级缓存分流；必要时配合限流、降级一起兜底。Redis 在这里最核心的价值就是扛热点流量，而不是单纯存数据。

## 4. 场景 3：缓存一致性怎么讲

### 口语版

> 缓存一致性我不会讲成绝对强一致，而是尽量缩小不一致窗口。比较常见的项目口径是先更新数据库，再删除缓存；如果业务更敏感，还可以结合延迟双删、MQ 或补偿任务进一步降低风险。

## 5. 场景 4：删缓存失败怎么办

### 口语版

> 删缓存失败是典型失败路径，所以不能只写一条删除逻辑就算了。更稳的方式是配重试、延迟双删，或者通过 MQ、补偿任务把这条链路接住，这样一致性问题不会因为一次失败直接放大。

## 6. 场景 5：热点 key 怎么处理

### 口语版

> 热点 key 我会优先保护它，不让它轻易失效，比如逻辑过期或者活动前预热；如果流量继续升高，再考虑本地缓存、多级缓存和限流。因为热点 key 的问题本质是访问太集中，不是普通缓存策略就能完全解决的。

## 7. 场景 6：Redis 挂了怎么办

### 口语版

> Redis 挂了我会先考虑止血，保护数据库和主链路，比如限流、降级、本地兜底缓存；架构层面再去看主从、哨兵、集群有没有生效，排查是节点故障、切换失败还是网络问题。顺序上一般是先止血，再定位，再恢复。

## 8. 场景 7：为什么这里用 Redis 分布式锁

### 口语版

> 如果服务已经是多实例部署，本地锁只能锁住当前 JVM，跨节点就不够了。这时候 Redis 分布式锁更适合做全局资源互斥，比如防重复下单、控制任务重复执行。不过它也有边界，比如锁过期和误删问题，所以不是用了就万无一失。

## 9. 场景 8：排行榜为什么适合 Redis

### 口语版

> 排行榜这种场景最自然会想到 Redis 的 `ZSet`，因为它既能存成员，又能按 score 排序。再加上排行榜通常是高频读场景，用 Redis 承接会比直接让 MySQL 做排序查询更适合。

## 10. 场景 9：Redis 和 MySQL 在项目里怎么分工

### 口语版

> 我一般把它理解成主存和缓存层的关系。MySQL 更负责最终持久化和事务型数据，Redis 更负责热点流量和低延迟读取，所以两者更多是配合关系，而不是谁替代谁。

## 11. 场景 10：最稳的一段总答法

### 口语版

> 如果从项目视角讲 Redis，我会把它理解成数据库前面的一层高性能缓存和热点流量承接层。它最常用在高频读、热点数据、排行榜、临时状态和分布式锁这些场景里。真正要讲清楚的不只是“它快”，而是为什么用它、它帮系统扛了什么压力、如果它出问题了怎么兜底。

## 12. 一句话收尾

> Redis 场景题讲得像不像项目表达，关键不在术语多不多，而在你能不能把“作用、风险、兜底”三件事讲清楚。
