# Redis 场景题最终模拟卷

## 1. 这份文档怎么用

这一篇适合做 Redis 场景题的最后一轮自测。

建议方式：

1. 从头顺着答
2. 每题控制在 30 秒到 1 分钟
3. 卡住的题回前面的场景题文档补

## 2. 模拟题目

### 1

商品详情页访问量很高，你为什么会在这里加 Redis？

### 2

如果活动流量突然暴涨，你会怎么用 Redis 做削峰？

### 3

如果缓存和数据库不一致了，你会怎么解释这个问题？

### 4

如果删缓存失败了，你会怎么兜底？

### 5

如果一个热点 key 失效后大量请求同时回源数据库，你会怎么处理？

### 6

为什么热点 key 的处理不能只靠“多加机器”？

### 7

如果 Redis 整体不可用了，你第一反应会先做什么？

### 8

如果 Redis 命中率突然下降，你会优先怀疑什么？

### 9

如果 Redis 延迟突然升高，你会优先看哪几个方向？

### 10

如果面试官问你“你怎么发现大 key”，你怎么答？

### 11

如果某个 key 既很热又很大，你会怎么解释它的风险？

### 12

多实例部署下，为什么本地锁不够，而要考虑 Redis 分布式锁？

### 13

Redis 分布式锁最容易出什么问题？你怎么解释？

### 14

如果面试官说“那为什么不直接用数据库锁”，你怎么接？

### 15

如果要做排行榜，你为什么更倾向 Redis，而不是 MySQL 直接排序？

### 16

Redis 和 MySQL 在项目里分别承担什么角色？

### 17

如果 Redis 挂了，数据库会不会被打穿？你怎么保护主链路？

### 18

如果面试官最后让你总结“Redis 在系统里到底解决了什么问题”，你怎么说？

## 3. 这套卷子的重点

这一套题主要检查三件事：

- 你能不能把 Redis 放回业务系统里讲
- 你会不会分析失败路径
- 你有没有兜底和边界意识

如果这 18 题你都能讲顺，Redis 场景题这一组就已经很够面试用了。

## 4. 一句话收尾

> Redis 场景题最后一轮训练，不是看你会不会背定义，而是看你能不能在真实系统语境下稳定输出。
