# Redis 过期与淘汰策略

## 1. 这份文档怎么用

这一篇主要解决 Redis 面试里很常见的两个追问：

- key 过期是怎么处理的
- 内存满了 Redis 怎么办

很多人会把“过期”和“淘汰”混在一起，这一篇就是把它们拉开。

## 2. 先记住最核心的区别

- 过期：某个 key 自己到时间了，该不该失效
- 淘汰：Redis 内存不够时，该删谁

这两个不是一回事。

## 3. 什么是过期时间

Redis 可以给 key 设置过期时间。

常见场景：

- 验证码
- 登录 token
- 短时缓存
- 防重 key

这类数据天然就不需要永久保存，所以很适合设置 TTL。

## 4. 过期是立刻删除吗

不是简单理解成“时间一到马上删掉”。

更稳的理解是：

- Redis 会在访问 key 时检查它有没有过期
- Redis 也会通过定期扫描的方式清理一部分已过期 key

所以你可以把它理解成：

- 被动删除
- 定期清理

面试里别答得太死：

> 到点了 Redis 就马上删

这种说法不严谨。

## 5. 为什么不能只靠一种方式

如果只靠访问时检查：

- 某些过期 key 长时间没人访问，就会一直占内存

如果只靠全量定时扫描：

- 成本太高

所以 Redis 会在效率和内存回收之间做平衡。

## 6. 什么是淘汰策略

淘汰策略是指：

- 当 Redis 内存达到限制
- 又来了新的写请求
- 这时候该删哪些 key

它解决的是“内存不够”的问题，不是“key 到期没到期”的问题。

## 7. 常见淘汰策略怎么理解

面试里不一定要把所有策略名字背到极细，但要知道大方向。

### 7.1 不淘汰

可以理解成：

- 内存满了就不再接受新写入

### 7.2 从设置过期时间的 key 里淘汰

这种策略的核心想法是：

- 优先删掉那些本来就带 TTL 的数据

### 7.3 从所有 key 里淘汰

这种策略的核心想法是：

- 不只看是否设置过期时间，而是在更大范围内选择淘汰对象

## 8. 面试里怎么答

如果被问：

> Redis 的过期策略和淘汰策略有什么区别

比较稳的答法是：

> 过期策略解决的是 key 到时间后怎么失效的问题，淘汰策略解决的是内存不够时该删除谁的问题。前者更偏 TTL 管理，后者更偏内存管理，它们是两个不同层面的机制。

## 9. 项目里怎么理解这一块

实际项目里，这部分通常会和缓存设计连起来问：

- 为什么要给缓存加过期时间
- 为什么过期时间要打散
- Redis 内存满了有什么风险

比较稳的回答是：

> 给缓存加过期时间主要是控制数据生命周期，也能避免脏数据长期存在；而过期时间打散可以减少集中失效风险；如果 Redis 内存打满，还要考虑淘汰策略对业务命中率和稳定性的影响。

## 10. 一句话收尾

> 过期和淘汰最容易混，记住一句话就够了：过期看时间，淘汰看内存。
