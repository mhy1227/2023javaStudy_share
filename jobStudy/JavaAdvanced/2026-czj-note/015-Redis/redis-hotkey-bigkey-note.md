# Redis 热点 Key 与大 Key

## 1. 这份文档怎么用

这一篇是项目题很容易被问到的一块。

尤其是做过：

- 秒杀
- 排行榜
- 热门接口缓存
- 高并发场景

面试官很喜欢顺手问：

- 热点 key 怎么处理
- 大 key 有什么风险

## 2. 什么是热点 key

热点 key 可以理解成：

- 某个 key 的访问量特别高
- 远高于普通 key

比如：

- 爆款商品详情
- 首页推荐数据
- 某个超热门活动配置

## 3. 热点 key 有什么问题

### 3.1 单点压力特别高

即使 Redis 很快，如果某个 key 被极端集中访问，也可能形成局部瓶颈。

### 3.2 一旦失效风险更大

如果它还是热点缓存 key，一旦过期，就容易引发缓存击穿。

## 4. 热点 key 常见处理思路

### 4.1 不让热点 key 轻易失效

比如：

- 永不过期
- 逻辑过期

### 4.2 做本地缓存或多级缓存

让请求不要全部直接打到 Redis。

### 4.3 提前预热

在活动开始前把热点数据先准备好。

### 4.4 限流和降级

如果流量已经明显异常，就要从系统层面兜底。

## 5. 什么是大 key

大 key 可以简单理解成：

- 某个 key 对应的 value 特别大
- 或者集合类型里元素特别多

比如：

- 一个特别大的 Hash
- 一个成员数极多的 Set
- 一个很大的 List

## 6. 大 key 有什么风险

### 6.1 网络传输压力大

一次读写就可能带来很大网络开销。

### 6.2 操作耗时变长

某些删除、遍历、读取操作可能明显变慢。

### 6.3 影响 Redis 整体性能

如果大 key 操作频繁，可能拖慢整体响应。

## 7. 大 key 怎么处理

### 7.1 拆分

不要把太多数据全塞进一个 key。

### 7.2 分页或分片

让一次读取的数据量受控。

### 7.3 避免粗暴删除

大 key 删除本身也可能有压力，要谨慎处理。

## 8. 面试里怎么答

如果被问：

> 热点 key 和大 key 有什么区别

比较稳的答法是：

> 热点 key 的核心问题是访问过于集中，容易形成局部高并发压力；大 key 的核心问题是单个 key 数据量太大，容易带来传输、操作和性能风险。一个 key 可以是热点 key，也可能同时还是大 key，但它们关注的问题不一样。

## 9. 项目口语版

> 如果项目里出现热点 key，我一般会优先考虑它的生命周期控制和多级缓存，尽量避免热点流量全集中在 Redis 上；如果是大 key，我会更关注拆分、分页读取和删除风险，因为它更偏单个 key 数据规模的问题。

## 10. 一句话收尾

> 热点 key 看“访问是否集中”，大 key 看“单个 key 是否过大”，这是两个不同维度的问题。
