# 树 vs 堆 vs 哈希 vs 图

## 1. 为什么这几个值得横向对比

因为它们都属于：

- 不是简单一条线的数据组织方式

而面试里很容易混的正是：

- 树和堆到底什么关系
- 树和哈希为什么不是一类思路
- 图和树又差在哪

如果只一个个单独学，很容易“每个都见过，但一放一起就答不清”。

## 2. 先给一个最短结论

可以先粗记成：

- 树：层级结构
- 堆：优先级结构
- 哈希：快速定位结构
- 图：网状关系结构

这四个的关注点其实完全不同。

## 3. 树最核心在什么地方

树最核心的是：

- 层级
- 分叉
- 父子关系

它适合描述：

- 目录
- 组织架构
- 分类体系
- 家谱

如果题目更像“从根往下分层展开”，就很容易想到树。

## 4. 堆最核心在什么地方

堆虽然和树长得像，也常和完全二叉树联系在一起，但它真正强调的不是“层级关系”，而是：

- 优先级

例如：

- 大顶堆看最大值
- 小顶堆看最小值

所以堆更适合：

- TopK
- 第 K 大
- 优先队列
- 数据流最值

## 5. 哈希最核心在什么地方

哈希最核心的是：

- 快速定位

也就是把：

- “我要快速找到某个 key”

放在最前面。

所以如果题目最关心的是：

- 查得快
- 去重
- 统计频次

通常就会优先想到哈希。

## 6. 图最核心在什么地方

图最核心的是：

- 网状关系

树其实可以理解成一种更特殊的结构，但图更一般。

图适合描述：

- 路网
- 社交关系
- 依赖关系
- 节点之间复杂连接

所以如果题目里不是“单根分层”，而是“节点之间互相连、可能成环”，那就更像图。

## 7. 树和堆为什么最容易混

因为它们都和树形结构有关。

但要抓住一句话：

- 树强调结构关系
- 堆强调堆顶优先级

树里你更关心：

- 左右子树
- 节点关系
- 遍历
- 路径

堆里你更关心：

- 谁最大
- 谁最小
- 如何快速拿堆顶

所以：

- 堆不是“普通树题”的思路
- 普通树也不是“优先队列题”的思路

## 8. 树和哈希为什么经常一起问

因为它们常常都能“查找”，但查找方式完全不同。

### 树

- 更偏有序组织
- 适合范围、最值、前驱后继

### 哈希

- 更偏快速命中
- 不强调顺序

所以如果题目强调：

- 有序
- 范围查询
- 最小值 / 最大值

更自然想到树。

如果题目强调：

- 快速判断存不存在
- 频次统计
- 去重

更自然想到哈希。

## 9. 树和图为什么也容易混

因为很多人第一次看图，会觉得：

- 好像也是很多节点连起来

但树和图的区别非常大。

### 树

- 通常有根
- 强调父子关系
- 一般不讨论“环”

### 图

- 不一定有根
- 不一定有明显父子关系
- 经常要考虑环、路径、连通性

一句话：

- 树更像层级组织
- 图更像网络连接

## 10. 一个面试里更稳的选择思路

### 题目更像层级关系

优先想到：

- 树

### 题目更像优先级 / 最值 / TopK

优先想到：

- 堆

### 题目更像快速查找 / 去重 / 计数

优先想到：

- 哈希

### 题目更像节点连边 / 路径 / 连通性

优先想到：

- 图

## 11. 和 Java 里的常见类怎么对应

可以先这样对应：

- 树 -> `TreeMap` / `TreeSet`
- 堆 -> `PriorityQueue`
- 哈希 -> `HashMap` / `HashSet`
- 图 -> 更常见于算法题里的自定义结构，不直接对应一个基础集合实现

## 12. 面试里怎么答更稳

可以按这条线答：

1. 树适合层级关系
2. 堆适合优先级处理
3. 哈希适合快速定位
4. 图适合复杂连接关系
5. 它们不是“谁替代谁”，而是擅长处理的问题不同

## 13. 一句话总结

这几个结构最容易混，但只要抓住主语义就不难区分：树看层级，堆看堆顶，哈希看定位，图看连接。
