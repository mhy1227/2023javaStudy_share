# 设计模式基础总览

## 1. 这份文档怎么用

这一篇先不急着讲某一个模式，而是先把设计模式这件事拉直。

你看完之后，至少要能回答：

- 什么是设计模式
- 为什么要学设计模式
- 面试里到底该怎么讲

## 2. 什么是设计模式

设计模式可以简单理解成：

- 在软件开发里
- 面对某一类反复出现的问题
- 前人总结出来的一种更稳定的解决思路

注意，是“解决思路”，不是“固定模板”。

所以面试里不要答成：

> 设计模式就是一套固定代码写法

这会显得很死。

更稳一点的讲法是：

> 设计模式是针对常见设计问题的一套可复用思路，它的重点不是背代码模板，而是理解这种模式适合解决什么问题。

## 3. 为什么要学设计模式

### 3.1 让代码更容易扩展

很多业务代码一开始能跑，但后面：

- 新需求一来就要改一大堆
- 条件分支越来越多
- 耦合越来越重

设计模式的价值之一，就是帮助你把变化点提前隔离出来。

### 3.2 让代码结构更清楚

很多模式本质上都在做一件事：

- 把职责拆开
- 把变化隔开
- 把依赖关系理顺

### 3.3 面试里高频

设计模式在 Java 面试里很容易被问到：

- 单例
- 工厂
- 代理
- 策略
- 观察者

再往后还可能会追：

- 责任链
- 模板方法
- 适配器

## 4. 面试里怎么理解设计模式

设计模式题通常不是想考你：

- 能不能默写完整代码

而是更想看：

- 你知不知道这个模式解决什么问题
- 你能不能说出适用场景
- 你能不能结合项目或框架举例

所以更稳的答法通常是：

1. 先说这个模式解决什么问题
2. 再说核心结构
3. 最后说项目或框架里的例子

## 5. 设计模式最常见的误区

### 5.1 只背定义

只背“单例是什么、工厂是什么”，面试官一追问就接不上。

### 5.2 只会写 demo

你知道怎么写一个很小的示例，但说不出为什么项目里要这样设计。

### 5.3 把模式当成目的

设计模式不是为了“显得高级”。

它真正的价值是：

- 让代码更稳
- 更容易维护
- 更容易扩展

## 6. 这一章怎么学更顺

我建议按这个顺序：

- 先学单例：最好入门，但容易被问深
- 再学工厂：很常和 Spring 串起来
- 再学代理：和 Spring AOP 直接相关
- 再学策略：最容易结合业务代码
- 再学观察者：适合和事件驱动、监听器联系起来

## 7. 一句话收尾

> 学设计模式最重要的不是背名字，而是形成一种习惯：看到问题时，能想到有没有更成熟的设计方式。
