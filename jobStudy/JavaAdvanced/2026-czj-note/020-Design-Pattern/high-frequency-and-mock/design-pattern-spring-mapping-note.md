# 设计模式在 Spring 体系里的落地

## 1. 这份文档怎么用

很多面试官并不会孤立地问设计模式，而是会直接问：

- Spring 里用到了哪些设计模式
- Spring MVC 里能看到哪些设计模式
- AOP 为什么总和代理模式放在一起讲

所以这一篇的重点不是讲教科书定义，而是把设计模式和 Spring 体系做映射。

## 2. Spring 里最常被提到的几类模式

### 单例模式

- Spring Bean 默认作用域就是单例
- 容器通常只创建一份实例并统一管理

面试口语版：

> Spring 默认单例 Bean，本质上体现了单例思想，不过它是由容器管理生命周期，不是传统手写单例类的那种写法。

### 工厂模式

- `BeanFactory`、`FactoryBean` 都很典型
- 容器负责对象创建、装配、初始化

面试口语版：

> Spring 最明显的设计模式之一就是工厂思想。开发者更多是声明 Bean，真正的对象创建和管理交给容器来做。

### 代理模式

- Spring AOP 背后核心就是代理
- 事务、日志、权限等横切逻辑常通过代理织入

面试口语版：

> Spring AOP 很多能力本质上就是代理模式。它不直接改业务类源码，而是在目标对象外面生成代理对象，把事务、日志、权限这些公共逻辑统一织进去。

### 观察者模式

- Spring 事件机制里很典型
- 事件发布器和监听器之间体现了一对多通知思想

面试口语版：

> Spring 事件机制很适合从观察者模式角度理解。发布一个事件后，多个监听器可以各自处理自己的逻辑。

### 模板方法模式

- `JdbcTemplate`
- `RedisTemplate`
- `RestTemplate`

面试口语版：

> Spring 里很多 `Template` 类都能看到模板方法思想。整体流程框架已经封装好了，开发者主要补业务相关的局部实现。

## 3. Spring MVC 里常见的模式映射

### 适配器模式

- `HandlerAdapter` 是很典型的适配器思路
- 不同类型的处理器通过适配器统一接入调用流程

面试口语版：

> Spring MVC 里 `HandlerAdapter` 很适合用适配器模式来理解。因为控制器形式可能不同，但通过适配器，DispatcherServlet 可以用统一方式去调。

### 策略模式

- 不同参数解析器
- 不同视图解析器
- 不同返回值处理器

面试口语版：

> Spring MVC 里也能看到很多策略思想，比如不同的参数解析、视图解析、返回值处理，本质上都是一组规则接口加多种实现，然后按条件选择合适策略。

### 责任链模式

- 拦截器链
- 过滤器链
- 参数处理链路里的多节点处理

面试口语版：

> 请求进入 Web 层之后，通常会经过过滤器、拦截器等多个环节，这种一层层往后处理的结构，很容易从责任链模式角度去理解。

## 4. AOP 为什么几乎必问代理模式

### 面试核心点

- AOP 不修改原业务代码
- 通过代理在方法前后织入公共逻辑
- 体现了横切关注点和业务逻辑分离

### 常见追问

- JDK 动态代理和 CGLIB 区别
- 为什么有接口时优先 JDK 动态代理
- 为什么代理比手动在代码里埋日志、事务更稳

### 更稳的答法

> AOP 之所以总和代理模式放在一起，是因为它的核心实现思路就是生成代理对象，在目标方法执行前后统一织入增强逻辑。这样事务、日志、权限这些横切逻辑就不用散落在业务代码里。

## 5. Spring 面试里最常见的一套回答主线

如果面试官问“Spring 中用到了哪些设计模式”，可以按下面顺序讲：

1. 工厂模式：Bean 创建和容器管理
2. 单例思想：默认单例 Bean
3. 代理模式：AOP、事务、权限增强
4. 模板方法：各种 `Template`
5. 适配器模式：`HandlerAdapter`
6. 观察者模式：事件发布与监听
7. 策略模式：解析器、处理器的多实现选择
8. 责任链模式：过滤器链、拦截器链

## 6. 面试时怎么讲得更像真实理解

- 不要一次性背一堆名词
- 优先讲自己最熟的 3 到 4 个
- 每讲一个都带一个 Spring 场景
- 最后补一句“这些是从设计思想角度理解，不一定机械等于教科书模式”

## 7. 一句话收尾

> 设计模式和 Spring 最稳的连接方式，不是背框架源码，而是把容器创建、AOP 增强、模板封装、事件通知、Web 调度这些典型场景串起来。
