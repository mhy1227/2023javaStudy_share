# 设计模式项目场景题

## 1. 这份文档怎么用

这一篇不是基础定义题，而是更像项目面里的问法。

面试官更可能这样问：

- 你项目里有没有用过设计模式
- 为什么这里不用 if else，而是用策略
- Spring 里哪些地方体现了设计模式

## 2. 场景题 1

### 问题

你项目里真的有用过设计模式吗？

### 回答方向

- 不要上来背 23 种
- 先挑 1 到 2 个自己最能讲清楚的

### 稳一点的答法

> 我一般不会说项目里刻意“套模式”，而是说在解决某些问题时，其实自然用了类似策略、模板方法、责任链这些思路。比如不同业务规则拆成不同策略，或者统一流程下把变化步骤下沉到不同实现里。

## 3. 场景题 2

### 问题

为什么这里不用 if else，而要考虑策略模式？

### 回答方向

- 规则会增长
- if else 会变长
- 策略更容易扩展

### 稳一点的答法

> 如果业务规则只有两三个、而且以后基本不变，if else 当然也能写。但如果规则会持续扩展，或者不同规则有独立实现逻辑，那策略模式会更稳，因为它能把不同规则拆开，降低修改时互相影响的风险。

## 4. 场景题 3

### 问题

你怎么判断一个场景更适合责任链？

### 回答方向

- 一串处理步骤
- 每个节点职责单一
- 顺序可扩展

### 稳一点的答法

> 如果一个请求天然要经过多个独立处理节点，比如校验链、审批流、过滤链，我就会更容易想到责任链。因为它特别适合把一大段串行处理拆成多个可组合的节点。

## 5. 场景题 4

### 问题

模板方法模式在项目里一般怎么落地？

### 回答方向

- 流程统一
- 局部差异

### 稳一点的答法

> 如果整体流程基本固定，只是其中几个步骤因场景不同而变化，我会更容易想到模板方法。比如统一导出流程、统一结算流程，但某些细节步骤由不同实现补充。

## 6. 场景题 5

### 问题

你怎么理解 Spring 里用到的设计模式？

### 回答方向

- Bean 创建和工厂
- AOP 和代理
- 事件和观察者

### 稳一点的答法

> Spring 里很容易看到设计模式的影子。比如 Bean 创建和管理里有工厂思想，AOP 背后有代理模式，事件发布和监听器又很适合用观察者模式来理解。所以面试里讲设计模式，最好能顺手和 Spring 串起来。

## 7. 场景题 6

### 问题

如果面试官问“那你项目里用的是设计模式，还是只是写法像”，你怎么接？

### 回答方向

- 不硬装
- 讲设计思路而不是强贴标签

### 稳一点的答法

> 我一般不会硬说“这里我用了某某模式”，而是更倾向说这段设计体现了某种模式思想。因为很多项目代码不一定是按教科书模式精确实现的，但背后的解耦、扩展、职责拆分思路是相通的。

## 8. 一句话收尾

> 设计模式项目题最重要的，不是炫耀自己知道多少模式，而是让面试官听出来你有结构化设计意识。
