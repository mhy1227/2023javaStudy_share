# 设计模式考前最终 Checklist

## 1. 这份文档怎么用

这一篇不是用来学新知识，而是用来考前最后一轮过稿。

你只需要快速检查自己能不能做到这些点：

- 定义能不能讲顺
- 场景能不能讲到位
- 对比题会不会答混
- Spring 和项目能不能串起来

## 2. 最基础的 3 个问题

- 我能不能用自己的话说清什么是设计模式
- 我能不能讲清为什么设计模式的价值不只是“背模板代码”
- 我能不能说出“设计模式解决的本质是重复出现的设计问题”

## 3. 高频模式是否能讲顺

- 单例模式：核心目的、线程安全、常见写法
- 工厂模式：为什么不直接 `new`
- 代理模式：为什么总和 AOP 一起问
- 策略模式：为什么适合替代复杂规则分支
- 模板方法：为什么强调流程固定、步骤可变
- 观察者模式：为什么适合事件通知
- 责任链模式：为什么适合过滤器链、审批流
- 适配器模式：为什么适合老系统接口适配
- 装饰器模式：为什么常和 Java IO 一起举例

## 4. 高频对比题是否稳定

- 我能区分策略模式和模板方法模式
- 我能区分代理模式和装饰器模式
- 我能区分观察者模式和发布订阅
- 我能区分工厂模式和直接 `new`

## 5. Spring 相关是否能接上

- 我能回答 Spring 中用到了哪些设计模式
- 我能解释 AOP 为什么体现代理模式
- 我能解释 `Template` 系列为什么常被说成模板方法
- 我能解释 `HandlerAdapter` 为什么适合从适配器角度理解
- 我能解释过滤器链、拦截器链为什么常和责任链一起讲

## 6. 项目题是否能讲得真实

- 我不会上来背 23 种模式
- 我会先讲问题，再讲设计，再落到模式
- 我能用“规则太多、流程太长、对象创建太耦合、公共逻辑散落”这类真实问题切入
- 我会说“体现了某种模式思想”，而不是机械硬贴标签

## 7. 口语表达是否过关

- 我能 30 秒讲清一个模式最核心的价值
- 我能 1 分钟讲出“问题 + 方案 + 场景”
- 我能 3 分钟讲出“问题 + 设计 + 对比 + 框架或项目落地”
- 我不会只会背定义，面试官一追问就断

## 8. 容易翻车的点

- 把所有拆分类都强行说成设计模式
- 只会说结构，不会说解决了什么问题
- 把模式当万能解法，忽略简单场景下的过度设计
- 把 Spring 里的实现现象和设计模式机械画等号

## 9. 考前最后一轮建议

1. 先过一遍 `design-pattern-1min-3min-answer-pack.md`
2. 再过一遍 `design-pattern-compare-note.md`
3. 再扫一遍 `design-pattern-spring-mapping-note.md`
4. 最后用这份 checklist 逐条自问

## 10. 一句话收尾

> 考前最后看设计模式，不是追求背更多，而是确认自己能不能把“问题、模式、场景、Spring、项目”这五件事讲顺。
