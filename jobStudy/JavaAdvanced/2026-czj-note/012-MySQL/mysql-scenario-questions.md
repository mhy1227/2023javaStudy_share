# MySQL 场景题笔记

## 1. 这份文档怎么用

前面的文档更多是在补知识主线和标准回答，这一份更偏面试里的场景题。

它的目标不是讲概念，而是训练你在被追问时，能把排查思路、分析路径和落地做法讲顺。

建议用法：

1. 先自己口头回答每一道题。
2. 再对照下面的回答思路补齐结构。
3. 最后把这些场景和你自己的项目经历挂起来。

## 2. 场景题一：项目里一条 SQL 很慢，你怎么排查

回答思路：

1. 先确认是不是慢 SQL，而不是接口其他环节慢。
2. 通过慢 SQL 日志、APM、接口日志定位具体语句。
3. 用 `EXPLAIN` 看执行计划。
4. 重点看 `type`、`key`、`rows`、`Extra`。
5. 看有没有命中合适索引、扫描行数是否过大。
6. 看有没有回表、临时表、额外排序。
7. 再决定是改 SQL、补联合索引，还是改表结构。

面试可直接说：

> 我一般不会一上来就说加索引，而是先定位具体 SQL，再结合 EXPLAIN 看执行计划，判断问题到底出在索引命中、扫描范围、回表、排序，还是 SQL 写法本身。最后再根据原因决定是改 SQL、补索引，还是做更大的结构调整。

## 3. 场景题二：为什么明明有索引，SQL 还是慢

回答思路：

- 索引区分度低。
- 扫描范围太大。
- 虽然走索引，但回表很多。
- 排序、分组没用上索引。
- 联合索引没满足最左前缀。
- 隐式转换或函数操作导致索引用得不理想。
- 优化器判断全表扫描更划算。

面试可直接说：

> 有索引不代表一定快，关键看整体执行成本是否真的下降。比如索引区分度低、扫描行数仍然很大、回表很多、排序没法复用索引，这些都可能让 SQL 依然很慢。

## 4. 场景题三：项目里 update 很慢，你先看什么

回答思路：

1. 先看更新条件有没有命中索引。
2. 看锁等待情况，是不是被其他事务阻塞。
3. 看事务是不是太长。
4. 看是否批量更新太大，导致锁时间过长。
5. 看 binlog / redo log 压力是否明显。

面试可直接说：

> update 慢我会优先看两类问题，一类是定位问题，比如条件没走索引导致扫描太大；另一类是并发问题，比如锁等待、事务持锁太久。尤其在高并发场景下，update 慢很多时候不只是索引问题，还可能是锁冲突问题。

## 5. 场景题四：高并发扣库存时，MySQL 层面要关注什么

回答思路：

- 更新条件是否精确命中索引。
- 是否会出现超卖。
- 是否需要乐观锁字段，比如版本号。
- 是否需要悲观锁，比如 `select ... for update`。
- 事务是否足够短。
- 是否会有热点行竞争。

面试可直接说：

> 高并发扣库存时，MySQL 层面主要关注两件事：一是正确性，不能超卖；二是并发性能，不能因为热点行竞争把吞吐打崩。实现上常见会结合条件更新、版本号乐观锁，或者在必要时使用悲观锁，但不管哪种方式，前提都是索引要准、事务要短。

## 6. 场景题五：分页查询到后面越来越慢，为什么

回答思路：

- 深分页时 `offset` 很大。
- MySQL 可能需要先扫描大量数据，再丢弃前面的结果。
- 如果排序字段没命中索引，开销会更大。

面试可直接说：

> 深分页慢的本质是扫描和丢弃成本高。比如 `limit 100000,20`，数据库往往要先找到前面 100000 条再丢掉，只返回最后 20 条。如果排序还没命中索引，成本会更高。常见优化是基于主键或游标做翻页，而不是一直用大 offset。

## 7. 场景题六：order by 很慢，你怎么分析

回答思路：

- 看排序字段有没有利用索引。
- 看 `Extra` 里是否有 `Using filesort`。
- 看是不是先过滤后排序，导致候选集太大。
- 看能不能通过联合索引复用排序。

面试可直接说：

> order by 慢时，我会先看排序字段有没有命中索引，以及 EXPLAIN 里有没有 Using filesort。如果排序不能复用索引，数据量一大排序成本就会上来。优化方向一般是调整联合索引，让过滤和排序尽量一起被索引利用。

## 8. 场景题七：group by 很慢，你怎么分析

回答思路：

- 看是否产生临时表。
- 看是否有 `Using temporary`。
- 看分组字段有没有配合索引。
- 看是不是先筛得不够精确，导致分组数据量太大。

面试可直接说：

> group by 慢时，我会重点关注临时表和扫描范围。如果分组字段没能很好利用索引，而且前置过滤不够强，MySQL 就可能需要把大量数据先拉出来再分组，性能会比较差。

## 9. 场景题八：为什么项目里查询走缓存了，还要关注 MySQL

回答思路：

- Redis 只能扛热点，不替代 MySQL。
- 缓存失效、穿透、回源时还是要落到 MySQL。
- MySQL 是最终数据源。
- 一旦 MySQL 设计不好，缓存也救不住底层结构问题。

面试可直接说：

> Redis 更多是做热点访问加速，但 MySQL 依然是最终存储和回源基础。缓存命中高时压力会小很多，但只要出现缓存失效、回源或批量查询，MySQL 设计不合理的问题还是会暴露出来，所以底层 SQL、索引、事务设计不能因为用了缓存就忽视。

## 10. 场景题九：如果面试官追问主从复制和事务提交，你怎么接

回答思路：

- 先把 binlog 作用讲清楚。
- 再提 redo log 和 binlog 分属不同层。
- 最后引出两阶段提交。

面试可直接说：

> 如果继续追问主从复制，我会先说明 binlog 是 Server 层日志，主从复制主要依赖它；再补充 redo log 属于 InnoDB，负责崩溃恢复。因为这两份日志不在同一层，所以事务提交时需要通过两阶段提交保证它们一致，否则主库本地恢复和从库复制看到的结果可能对不上。

## 11. 场景题十：如果出现死锁，你怎么讲

回答思路：

- 先解释死锁本质：互相等待。
- 再说常见原因：访问顺序不同、锁范围过大、事务过长。
- 最后说避免思路：固定顺序、缩短事务、命中索引、分批更新。

面试可直接说：

> 死锁本质上是多个事务互相等待对方释放锁。常见原因包括事务访问资源顺序不一致、更新没走索引导致锁范围扩大、事务持锁时间太长。优化思路一般是固定访问顺序、缩短事务、让更新条件命中索引，以及把大事务拆小。

## 12. 场景题十一：如果表数据量越来越大，你会怎么考虑

回答思路：

- 先优化 SQL 和索引。
- 再考虑冷热数据分离。
- 再考虑归档历史数据。
- 最后才是分库分表。

面试可直接说：

> 如果表越来越大，我不会一上来就分库分表。通常先看 SQL 和索引有没有优化空间，再看冷热数据能不能拆、历史数据能不能归档。如果业务增长已经明显超过单表承载能力，再评估分库分表。顺序上应该先做低成本优化，再做高复杂度改造。

## 13. 当前阶段最值得重点练的场景

优先练这几题：

1. 一条 SQL 很慢，怎么排查。
2. 有索引为什么还慢。
3. 高并发扣库存时 MySQL 层面关注什么。
4. 深分页为什么慢。
5. 主从复制和两阶段提交怎么衔接着讲。

## 14. 怎么把这些场景挂到项目里

建议你最后把每道题都改成项目版表达：

- 把“某条 SQL 很慢”换成你自己的订单表、课程表、题库表。
- 把“库存扣减”换成你自己的业务资源扣减。
- 把“分页很慢”换成你自己的列表页或后台管理页。

这样面试时会更像你真的做过，而不是只会背模板。
