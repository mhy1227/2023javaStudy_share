# MySQL 锁 SQL 题型笔记

## 1. 这份文档是干什么的

这一篇专门整理面试里那种“直接给你一条 SQL，让你判断会不会加锁、锁多大范围、会不会阻塞”的题型。

这种题和背定义不一样，更像现场判断题。

最常见的问法就是：

- 这条 SQL 会不会加锁？
- 是锁表还是锁行？
- `where id = ?` 是不是就只锁一行？
- 如果没索引会怎样？
- 如果是范围查询会怎样？
- 两个事务会不会互相阻塞？

## 2. 先记一个最稳的判断口诀

你以后拿到一条 SQL，先不要急着说“锁一行”或“不会加锁”。

先按这个顺序判断：

1. 先看是不是当前读。
2. 再看条件字段有没有索引。
3. 再看执行计划是否真的走了索引。
4. 最后看是等值条件还是范围条件。

一句话记忆：

> **先看当前读，再看索引命中，再看执行计划，最后看等值还是范围。**

## 3. 题型一：普通 `select` 会不会加这种锁

题目例子：

```sql
select * from user where id = 10;
```

更稳的回答：

> 普通 `select` 很多时候更偏快照读，不是这里讨论的那种“加排他锁把记录锁住”的语义。所以不能把普通 select 和 `select ... for update` 混为一谈。

这道题面试里常常是为了引出：

- 快照读
- 当前读
- MVCC

## 4. 题型二：`select ... for update` 会不会加锁

题目例子：

```sql
select * from user where id = 10 for update;
```

更稳的回答：

> `select ... for update` 属于当前读，它会读取最新版本，并且对读取到的记录加排他锁，防止其他事务同时修改。

这是最经典的一道入门题。

## 5. 题型三：`where id = ?` 是不是就只锁一行

题目例子：

```sql
select * from user where id = 10 for update;
```

不要直接答“是”。

更稳的回答：

> 不能只看 `where id = ?` 就判断只锁一行，还要看这条 SQL 是不是当前读、`id` 有没有索引、执行计划是否真的走索引。如果它是当前读，又精准命中主键或唯一索引，通常锁会更精确；但如果不走索引，锁范围可能扩大。

## 6. 题型四：如果 `id` 是主键索引，会怎样

题目例子：

```sql
select * from user where id = 10 for update;
```

前提：

- `id` 是主键
- 执行计划命中主键索引

更稳的回答：

> 这种情况下通常能更精确地定位到目标记录，所以锁范围更小，更接近“锁住这一条记录”。

注意：

> 这里也不要机械说成“100% 只锁一行”，更稳的表达是“通常会更精确”。

## 7. 题型五：如果没有索引，会怎样

题目例子：

```sql
select * from user where age = 20 for update;
```

前提：

- `age` 没有索引

更稳的回答：

> 如果条件字段没有索引，MySQL 可能要扫描更多记录，InnoDB 在扫描过程中锁范围也可能扩大，并发性能会明显下降。严重时看起来会像锁住了很多行，甚至表现得接近锁表效果。

## 8. 题型六：有索引但执行计划没走索引，会怎样

题目例子：

- 字段上有索引
- 但因为函数、隐式转换、优化器选择等原因没有真正走索引

更稳的回答：

> 不能只看“字段上有没有索引”，还要看执行计划是否真的使用索引。如果没真正走索引，本质效果和不命中索引差不多，扫描范围会变大，锁范围也可能跟着扩大。

## 9. 题型七：范围查询会怎样

题目例子：

```sql
select * from user where id > 10 and id < 20 for update;
```

更稳的回答：

> 范围查询通常比等值查询更容易涉及多条记录，也更容易引出间隙锁和临键锁。所以这类 SQL 不应该简单理解成锁某一行，而应该理解成可能锁一个范围。

## 10. 题型八：`update` 和 `select ... for update` 的共同点是什么

题目例子：

```sql
update user set name = 'A' where id = 10;
```

更稳的回答：

> 它们都属于当前读这条线，都会涉及锁。区别在于 `select ... for update` 更像先查并锁住，后面准备改；而 update 是直接走修改路径，但两者都很依赖索引是否命中，因为这会影响锁范围。

## 11. 题型九：两个事务会不会阻塞

典型问法：

事务 A：

```sql
select * from user where id = 10 for update;
```

事务 B：

```sql
update user set name = 'B' where id = 10;
```

更稳的回答：

> 这类题要先看事务 A 是否已经持有锁，如果 A 已经通过 `for update` 锁住对应记录，那么事务 B 对同一记录的 update 通常会被阻塞，直到 A 提交或回滚。

## 12. 题型十：为什么这类题容易继续追到死锁

因为一旦面试官开始问：

- 会不会加锁
- 锁多大范围
- 两个事务会不会互相等

后面就很容易继续追到：

- 会不会阻塞
- 会不会死锁
- 怎么减少死锁

所以你答这类 SQL 题时，不要只停在“锁几行”，最好顺手补一句：

> 如果锁范围扩大，在高并发场景下就更容易带来锁等待和死锁风险。

## 13. 面试里最稳的一段总答

可以直接背：

> 这种带 SQL 的锁题，我一般会先判断它是不是当前读，比如是不是 `select ... for update`、update 或 delete；如果只是普通 select，很多时候不是这里说的加锁语义。确认是当前读之后，再看条件字段有没有索引，以及执行计划是否真的走了索引；最后再看它是等值查询还是范围查询。只有把这几步结合起来，才能更准确判断它是锁一行、锁多行，还是锁范围扩大。

## 14. 当前阶段最值得练的 SQL 问法

优先练这几种：

1. `select * from t where id = ? for update`
2. `select * from t where 非索引字段 = ? for update`
3. `select * from t where id > ? and id < ? for update`
4. 普通 `select` 和 `select ... for update` 对比
5. 两个事务是否阻塞的判断题
