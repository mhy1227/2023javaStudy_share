# MySQL 事务、隔离级别与 MVCC 笔记

## 1. 这份文档是干什么的

事务和 MVCC 是 MySQL 面试里第二条特别高频的主线。

通常面试官会这样连续追问：

1. 什么是事务。
2. ACID 是什么。
3. 四种隔离级别是什么。
4. 脏读、不可重复读、幻读分别是什么。
5. MySQL 默认隔离级别是什么。
6. MVCC 是什么。
7. MVCC 和锁是什么关系。
8. 快照读和当前读怎么区分。

所以这一篇的目标就是把这整条线先讲顺，而不是一开始就陷进特别细的实现细节里。

## 2. 先记最核心的结论

先记这几句：

1. **事务就是一组操作要么都成功，要么都失败。**
2. **ACID 是事务的四大特性：原子性、一致性、隔离性、持久性。**
3. **MySQL InnoDB 默认隔离级别是可重复读。**
4. **MVCC 的核心目标是尽量让读写不互相阻塞。**
5. **快照读主要依赖 MVCC，当前读主要依赖锁。**

如果先把这五句记住，后面的内容会更容易挂进去。

## 3. 什么是事务

可以先这样回答：

> 事务就是把一组相关操作当成一个整体来执行，要么全部成功，要么全部失败，不会只执行一半。

更口语一点可以说：

> 比如转账时，A 扣钱和 B 加钱必须一起成功。如果只扣钱不加钱，或者只加钱不扣钱，业务数据就错了，所以这种操作通常要放进一个事务里。

## 4. ACID 是什么

ACID 是事务最经典的四大特性。

### 4.1 原子性 Atomicity

原子性强调：

- 事务中的操作是一个整体。
- 要么全部完成。
- 要么全部不完成。

你可以理解成“不允许只做一半”。

### 4.2 一致性 Consistency

一致性强调：

- 事务执行前后，数据都应该处于合法、正确的状态。
- 不会从一个正常状态变成一个错误状态。

例如：

- 转账前两边余额总和正确。
- 转账后两边余额总和仍然正确。

### 4.3 隔离性 Isolation

隔离性强调：

- 多个事务并发执行时，彼此之间要尽量互不干扰。
- 一个事务不应该随便看到另一个事务中间未提交的结果。

### 4.4 持久性 Durability

持久性强调：

- 一旦事务提交，结果就应该被可靠保存下来。
- 即使数据库崩溃，提交过的数据也不能轻易丢失。

## 5. 为什么会有隔离级别

因为数据库里往往不是只有一个事务在跑，而是多个事务并发执行。

如果完全不控制并发读写，就会出现各种数据问题。

最常见的三个异常现象是：

- 脏读
- 不可重复读
- 幻读

为了在“正确性”和“性能”之间做平衡，数据库就设计了不同的隔离级别。

## 6. 脏读、不可重复读、幻读是什么

### 6.1 脏读

脏读指的是：

- 一个事务读到了另一个事务还没提交的数据。
- 如果对方后来回滚了，那你刚才读到的其实就是脏数据。

一句话记忆：

> **读到了别人未提交的数据。**

### 6.2 不可重复读

不可重复读指的是：

- 同一个事务里，对同一条记录读两次。
- 两次结果不一样。
- 原因通常是中间有别的事务修改并提交了这条记录。

一句话记忆：

> **同一条记录前后两次读出来不一样。**

### 6.3 幻读

幻读指的是：

- 同一个事务里，按相同条件查两次。
- 第二次发现多了几条或者少了几条“新记录”。
- 原因通常是别的事务插入或删除了符合条件的数据。

一句话记忆：

> **同一批范围数据前后两次查，结果集变了。**

## 7. 四种隔离级别是什么

MySQL 常说的四种隔离级别，从低到高是：

1. 读未提交 Read Uncommitted
2. 读已提交 Read Committed
3. 可重复读 Repeatable Read
4. 串行化 Serializable

### 7.1 读未提交

特点：

- 别的事务还没提交的数据也可能读到。
- 隔离性最弱。
- 性能可能高，但问题也最多。

会出现：

- 脏读
- 不可重复读
- 幻读

### 7.2 读已提交

特点：

- 只能读到别人已经提交的数据。
- 脏读被解决。

仍可能出现：

- 不可重复读
- 幻读

### 7.3 可重复读

特点：

- 同一个事务里，多次读取同一条记录，通常结果一致。
- MySQL InnoDB 默认就是这个级别。

通常重点记：

- 脏读解决了。
- 不可重复读解决了。
- 在 InnoDB 里，结合 MVCC 和间隙锁等机制，对幻读也做了较强控制。

### 7.4 串行化

特点：

- 隔离性最强。
- 并发能力最差。
- 相当于很多操作要排队执行。

通常只在特别强调绝对正确性的场景下才考虑，日常业务里用得不多。

## 8. MySQL 默认隔离级别是什么

这个要直接背：

> **MySQL InnoDB 默认隔离级别是可重复读 Repeatable Read。**

这是特别高频的一句。

## 9. 什么是 MVCC

MVCC 全称是 Multi-Version Concurrency Control，多版本并发控制。

可以先这样回答：

> MVCC 的核心思想是让数据保留多个可见版本，读操作尽量去读合适的历史版本，而不是一上来就和写操作互相阻塞。

更口语一点可以说：

> 你可以把 MVCC 理解成“不给读和写硬碰硬”。写在更新新版本，读则去看自己当前事务能看到的那个版本。

所以 MVCC 的目标不是“完全不用锁”，而是：

> **尽量减少读写冲突，提高并发性能。**

## 10. MVCC 依赖什么

初学阶段先记住三个关键词：

- 隐藏字段
- undo log
- Read View

### 10.1 隐藏字段

InnoDB 每行记录里通常会维护一些隐藏信息，例如：

- 这条记录由哪个事务修改
- 这条记录的回滚指针

这些信息用于追踪版本关系。

### 10.2 undo log

undo log 可以帮助数据库回到旧版本。

对于 MVCC 来说，它很重要，因为旧版本数据并不是凭空来的，而是可以沿着版本链回溯。

### 10.3 Read View

Read View 可以理解成：

> 当前事务看数据时采用的一套“可见性规则”。

也就是说，并不是所有版本你都能看到，而是要根据当前事务的 Read View 决定哪一个版本对你可见。

## 11. 快照读和当前读是什么

这是 MVCC 里很常见的追问。

### 11.1 快照读

快照读强调：

- 读的是某个历史一致性版本。
- 主要依赖 MVCC。
- 典型的普通 `select` 常常属于快照读。

一句话记：

> **快照读更偏“读版本”。**

### 11.2 当前读

当前读强调：

- 读的是最新版本。
- 往往还要对读取到的记录加锁。
- 例如 `select ... for update`、`update`、`delete` 这类操作更接近当前读。

一句话记：

> **当前读更偏“读最新并控制并发”。**

## 12. MVCC 和锁是什么关系

这个问题特别容易问。

可以先记这个回答：

> MVCC 和锁不是互斥关系，而是配合关系。MVCC 主要用于提升读写并发能力，减少普通读和写之间的阻塞；锁主要用于保证当前读、更新、删除等操作的并发正确性。

所以不要把它们理解成二选一。

更准确地说：

- 普通一致性读，很多时候靠 MVCC。
- 更新、删除、加锁读取，很多时候靠锁。

## 13. 为什么说 MVCC 能提升并发

因为如果没有 MVCC，很多读操作可能也得等写操作完成。

有了 MVCC 之后：

- 写事务可以去生成新的版本。
- 读事务不一定非要等写事务结束。
- 读事务可以根据自己的可见性规则去读适合自己的版本。

这样读写冲突就减少了，并发性能会更好。

## 14. 面试里怎么回答“事务和 MVCC 的关系”

可以直接说：

> 事务解决的是一组操作整体上的一致性问题，而 MVCC 主要是事务隔离实现中的一种并发控制手段。它通过多版本机制让普通读尽量不阻塞写，从而在保证一定隔离性的前提下提升并发能力。

## 15. 当前阶段最值得先背的版本

> 事务就是一组操作要么都成功要么都失败，ACID 分别对应原子性、一致性、隔离性、持久性。MySQL InnoDB 默认隔离级别是可重复读。隔离级别是为了解决并发事务中的脏读、不可重复读、幻读问题。MVCC 是多版本并发控制，它通过隐藏字段、undo log、Read View 等机制，让普通读尽量读取合适的历史版本，从而减少读写阻塞。快照读主要依赖 MVCC，当前读主要依赖锁，它们共同配合保证并发场景下的数据正确性和性能。
