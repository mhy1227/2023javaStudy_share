# MySQL 锁并发判断题笔记

## 1. 这份文档是干什么的

这一篇专门整理那种面试里最常见的“双事务锁判断题”。

它和普通锁定义题不一样，通常问法更像这样：

- 事务 A 先执行了某条 SQL。
- 事务 B 再执行另一条 SQL。
- 问你会不会阻塞。
- 问你能不能修改当前 id。
- 问你能不能修改其他 id。
- 问你能不能插入某个范围内的新数据。

所以这类题本质上是在考：

> 锁住的是当前一行、一个范围，还是更大范围的扫描结果。

## 2. 先记一个最稳的判断顺序

判断这类题，建议统一按这个顺序走：

1. 先看事务 A 执行的是不是当前读。
2. 再看条件字段有没有索引。
3. 再看执行计划是否真的走索引。
4. 最后看是等值查询还是范围查询。

一句话记忆：

> **先看当前读，再看索引命中，再看执行计划，最后看等值还是范围。**

## 3. 场景一：主键等值命中索引

事务 A：

```sql
begin;
select * from user where id = 10 for update;
```

假设：

- `id` 是主键索引
- 执行计划也确实走了主键索引

常见追问：

- 事务 A 能不能修改 `id = 10`？
- 事务 B 能不能修改 `id = 10`？
- 事务 B 能不能修改 `id = 11`？
- 事务 B 能不能查询 `id = 10`？

更稳的回答：

- 事务 A 后续当然可以继续修改 `id = 10`
- 事务 B 如果也要改 `id = 10`，通常会被阻塞
- 事务 B 修改 `id = 11`，一般不受这条锁影响
- 事务 B 普通查询 `id = 10`，很多时候仍然可以读到自己的可见版本

一句话理解：

> 这是最接近“精准锁住目标记录”的情况。

## 4. 场景二：等值查询但字段没索引

事务 A：

```sql
begin;
select * from user where age = 20 for update;
```

假设：

- `age` 没有索引

常见追问：

- 事务 B 能不能改另一条 `age = 20` 的记录？
- 事务 B 能不能改 `age = 21` 的记录？
- 为什么这类语句比想象中更危险？

更稳的回答：

> 因为没有索引，MySQL 可能要扫描很多记录，InnoDB 在扫描过程中锁范围也可能扩大。所以这时不能简单按条件字面理解成“只影响 age = 20 的某几条”，更稳的说法是：锁冲突范围可能明显变大，很多扫描到的记录都可能被波及。

一句话理解：

> 不走索引的 `for update`，问题往往不只是慢，还会放大锁冲突。

## 5. 场景三：范围查询

事务 A：

```sql
begin;
select * from user where id > 10 and id < 20 for update;
```

常见追问：

- 事务 B 能不能改 `id = 9`？
- 事务 B 能不能改 `id = 15`？
- 事务 B 能不能插入 `id = 12`？
- 事务 B 能不能插入 `id = 25`？

更稳的回答：

> 范围查询一般不能再按“锁一行”去理解了，它更容易涉及记录锁、间隙锁和临键锁。通常范围内已有记录更容易被影响，范围内插入也更可能被拦住，而范围外数据通常更不容易受影响。

一句话理解：

> 范围查询更像“锁一段区间”，而不是锁某个点。

## 6. 场景四：普通 select 和 for update 对比

事务 A：

```sql
begin;
select * from user where id = 10;
```

事务 B：

```sql
update user set name = 'B' where id = 10;
```

常见追问：

- 为什么这时候未必会像 `for update` 一样阻塞？

更稳的回答：

> 因为普通 select 很多时候更偏快照读，不是这里说的加排他锁锁住记录的语义。它更常见的意义是读取自己当前事务可见的版本，而不是把记录锁住不让别人改。

## 7. 场景五：事务 A 先锁，事务 B 再改

事务 A：

```sql
begin;
select * from user where id = 10 for update;
```

事务 B：

```sql
update user set name = 'B' where id = 10;
```

更稳的回答：

> 如果事务 A 已经通过 `for update` 精确锁住了目标记录，那么事务 B 再更新同一条记录，通常会进入等待，直到事务 A 提交或回滚。

这类题经常继续追问：

- 那如果事务 B 改的是 `id = 11` 呢？
- 那如果事务 A 没走索引呢？

## 8. 场景六：事务 A 和事务 B 会不会死锁

典型题目：

事务 A：

```sql
begin;
select * from user where id = 10 for update;
select * from user where id = 11 for update;
```

事务 B：

```sql
begin;
select * from user where id = 11 for update;
select * from user where id = 10 for update;
```

更稳的回答：

> 这类场景很容易形成死锁，因为两个事务先锁住不同记录，再去拿对方已经持有的锁，就可能互相等待。

一句话记忆：

> **顺序不一致，是死锁题最常见的坑。**

## 9. 场景七：为什么面试官喜欢问“能不能改其他 id”

因为这类追问不是在考你会不会背“行锁”“表锁”，而是在考你：

- 你是否真的理解锁范围
- 你是否知道索引和执行计划的重要性
- 你是否知道等值条件和范围条件完全不同

所以当面试官问：

- 能不能改 `id = 10`
- 能不能改 `id = 11`
- 能不能插入 `id = 12`

本质上都是在问：

> 这条 SQL 到底锁住了什么。

## 10. 面试里最稳的一段总答

可以直接这样说：

> 这类并发判断题我一般不会直接说“锁一行”或者“不能改”，而是先判断事务 A 的 SQL 是不是当前读，比如是不是 `select ... for update`、update 或 delete；然后再看条件字段有没有索引，以及执行计划是否真的走了索引；最后再看它是等值查询还是范围查询。如果是主键等值并命中索引，通常锁范围会比较精确；如果不走索引，或者是范围查询，锁范围就可能扩大，进而影响其他记录甚至某些插入操作。

## 11. 当前阶段最值得先练的题

优先练这几类：

1. `select * from t where id = ? for update`，问能不能改当前 id
2. `select * from t where id = ? for update`，问能不能改其他 id
3. `select * from t where 非索引字段 = ? for update`，问锁范围
4. `select * from t where id between ? and ? for update`，问能不能插入范围内数据
5. 两个事务不同顺序加锁，问会不会死锁
