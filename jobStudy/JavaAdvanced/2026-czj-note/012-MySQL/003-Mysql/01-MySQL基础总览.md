# MySQL 基础总览

## 1. 这份文档是干什么的
这是一份给面试准备用的 MySQL 基础总览稿。

目标不是讲得特别深，而是先把最常见的主线理顺：

1. MySQL 是干什么的
2. 索引为什么快
3. 事务是什么
4. 锁和 MVCC 是什么
5. redo log、undo log、binlog 分别干什么
6. 一条 SQL 大概怎么执行

如果你先把这份吃透，后面再补细节会轻松很多。

---

## 2. MySQL 是什么
可以这样回答：

> MySQL 是一个关系型数据库，主要用来做业务数据的持久化存储，支持表结构、SQL 查询、事务、一致性控制和复杂条件检索。

更口语一点：

> 简单说，MySQL 是项目里最常见的核心数据存储，用来存订单、用户、课程、试卷、答题记录这类业务数据。

---

## 3. 为什么项目里既用 MySQL，又用 Redis
这个问题经常和 MySQL 一起被问。

你可以这样说：

> MySQL 更适合做持久化存储、事务控制和复杂查询；Redis 更适合做缓存、热点数据访问和高并发场景优化。实际项目里一般不是二选一，而是 MySQL 负责落库，Redis 负责加速。

一句话版：

> MySQL 保证数据可靠，Redis 负责性能优化。

---

## 4. 什么是索引
标准回答：

> 索引本质上是帮助数据库快速定位数据的一种数据结构。

口语版：

> 索引就像书的目录，不用把整张表从头翻到尾，就能更快找到目标数据。

要再补一句：

> 索引能提高查询效率，但会占空间，也会增加写入维护成本。

---

## 5. MySQL 为什么常用 B+ 树
你不用一上来讲很深，先记住这版：

> MySQL 常用 B+ 树，是因为它更适合磁盘场景。它的层级比较低，查找时磁盘 IO 次数更少，而且也更适合范围查询和排序。

如果继续追问：

### 为什么不是红黑树
> 红黑树层级太高，磁盘 IO 次数会更多。

### 为什么不是 Hash
> Hash 适合等值查询，但不适合范围查询和排序。

---

## 6. 什么是聚簇索引和二级索引
这是高频题。

### 聚簇索引
> InnoDB 里主键索引一般是聚簇索引，它的叶子节点直接存整行数据。

### 二级索引
> 普通索引一般是二级索引，它的叶子节点存的是主键值，不是整行数据。

---

## 7. 什么是回表
面试回答：

> 回表就是先通过二级索引查到主键，再根据主键回到聚簇索引里查整行数据。

更口语一点：

> 先查目录，再回正文，这个过程就叫回表。

---

## 8. 什么是覆盖索引
面试回答：

> 覆盖索引就是查询需要的字段，索引里已经都包含了，不需要再回表。

最重要的一句：

> 覆盖索引的核心价值就是少一次回表。

---

## 9. 事务是什么
最稳的回答：

> 事务就是把一组操作当成一个整体，要么都成功，要么都失败。

比如：
- 扣库存
- 扣余额
- 生成订单

这三个操作如果只成功一部分，数据就会不一致，所以要放在事务里。

---

## 10. ACID 是什么
事务最常配套问这个。

### A：Atomicity 原子性
> 要么都成功，要么都失败。

### C：Consistency 一致性
> 事务前后，数据要保持一致的业务状态。

### I：Isolation 隔离性
> 多个事务之间不能互相乱影响。

### D：Durability 持久性
> 事务一旦提交，结果不能丢。

你可以直接说：

> ACID 就是事务的四大特性，实际面试里最常往下追的是隔离性和持久性。

---

## 11. 事务隔离级别有哪些
四个标准答案：

1. 读未提交
2. 读已提交
3. 可重复读
4. 串行化

你一定要记住：

> MySQL InnoDB 默认隔离级别是可重复读。

---

## 12. MVCC 是什么
这是 MySQL 面试很容易被问到的点。

你可以这样答：

> MVCC 是多版本并发控制，核心作用是让读写尽量不互相阻塞。

再补一句：

> 它会让查询去读某个合适版本的数据快照，而不是每次都强行读当前正在修改的数据。

简单理解：

> MVCC 主要是为了解决并发场景下的读性能和一致性问题。

---

## 13. undo log 是什么
你可以先记简单版：

> undo log 记录的是数据修改前的旧值，主要有两个作用，一个是事务回滚，一个是给 MVCC 提供历史版本。

这句话很重要，因为很多人只会说回滚，不会说 MVCC。

---

## 14. redo log 是什么
面试回答：

> redo log 是 InnoDB 层的重做日志，主要是为了保证事务提交后的持久性。

更容易记的版本：

> 可以理解为，数据页还没来得及真正刷盘，但事务已经提交了，这时候 redo log 可以保证崩溃恢复后数据不丢。

---

## 15. binlog 是什么
面试回答：

> binlog 是 MySQL Server 层的归档日志，主要用于主从复制和数据恢复。

你要区分清楚：

- `redo log` 偏底层存储引擎恢复
- `binlog` 偏复制和归档

---

## 16. redo log、undo log、binlog 的区别
这是特别高频的整理题。

你可以这样讲：

> undo log 记录修改前的数据，主要用于回滚和 MVCC；redo log 记录的是为了保证事务提交后的持久性；binlog 是 Server 层的归档日志，主要用于主从复制和恢复。

一句话压缩版：

> undo 管回滚和版本，redo 管持久性，binlog 管复制和归档。

---

## 17. 锁是什么
面试里可以先答得朴素一点：

> 锁是数据库为了保证并发情况下数据正确性而引入的控制机制。

常见的你先记这几个：

- 表锁
- 行锁
- 间隙锁

---

## 18. 行锁、表锁、间隙锁分别是什么
### 表锁
> 锁整张表，粒度大，并发低。

### 行锁
> 锁某一行或某几行，粒度小，并发高。

### 间隙锁
> 锁的是一个范围，不是具体某一行，主要是为了防止幻读。

---

## 19. 死锁是什么
面试回答：

> 死锁就是多个事务互相等待对方释放锁，最后谁都走不下去。

如果继续追问怎么处理：

> 一般先看死锁日志，再看 SQL 执行顺序、是否走索引、锁范围是否过大，优化方向通常是固定顺序、缩小锁范围、减少长事务。

---

## 20. 一条 SQL 大概怎么执行
这是后面很值得展开的主线。

你现在先记基础版：

1. 客户端发送 SQL
2. MySQL 进行连接和权限校验
3. 解析 SQL
4. 优化器选择执行方案
5. 执行器调用存储引擎
6. 返回结果

一句话回答：

> 一条 SQL 进入 MySQL 后，一般会经过连接、解析、优化、执行这几个核心阶段。

---

## 21. SQL 的常见逻辑执行顺序
这个也很容易被问。

你先记这一版：

1. `from`
2. `where`
3. `group by`
4. `having`
5. `select`
6. `order by`
7. `limit`

注意：

> 这是逻辑执行顺序，不是你写 SQL 的书写顺序。

---

## 22. 慢 SQL 怎么排查
这道题很常见。

你可以这样答：

> 我一般会先看慢 SQL 日志，再用 EXPLAIN 看执行计划，重点看有没有走索引、扫描行数是否过多、有没有回表、有没有临时表和 filesort，然后再决定是改 SQL、补索引还是调表结构。

---

## 23. 面试时可以怎么整体讲
如果面试官问你“你对 MySQL 怎么理解”，你可以按这个骨架答：

> MySQL 这块我一般从四个层次去理解。第一层是索引和查询优化，比如 B+ 树、回表、覆盖索引、慢 SQL 排查；第二层是事务，包括 ACID、隔离级别、锁和死锁；第三层是并发控制，也就是 MVCC 和 undo log；第四层是日志体系，比如 redo log、binlog 以及事务提交流程。这样从查询、事务到底层机制，整个链路就比较完整。

---

5. **事务就是一组操作要么都成功，要么都失败。**
6. **MySQL InnoDB 默认隔离级别是可重复读。**
7. **MVCC 的核心作用是让读写尽量不互相阻塞。**
8. **undo log 主要用于回滚和给 MVCC 提供历史版本。**
