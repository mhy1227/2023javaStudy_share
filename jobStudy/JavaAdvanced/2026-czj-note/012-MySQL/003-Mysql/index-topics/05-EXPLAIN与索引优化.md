# EXPLAIN 与索引优化

## 1. 这份文档是干什么的
这份文档主要解决索引专题里最后一块非常关键的内容：

1. `EXPLAIN` 是干什么的
2. `EXPLAIN` 里重点看哪些字段
3. 怎么根据执行计划判断索引有没有真正用好
4. 怎么结合执行计划做索引优化

很多人会说：

> 我会看执行计划

但一旦继续问：

- 你重点看什么
- `key` 和 `possible_keys` 有什么区别
- `rows` 大说明什么
- `extra` 里你最关注哪些信息

就容易卡住。

这份就是专门补这个缺口。

---

## 2. 先说最重要的结论
你先记住这三句：

1. **`EXPLAIN` 的核心作用，是帮助我们看 MySQL 打算怎么执行这条 SQL。**
2. **看执行计划不能只看“有没有索引”，还要看扫描行数、访问方式和额外开销。**
3. **索引优化的本质，不是让 SQL 看起来用了索引，而是让整体执行成本更低。**

---

## 3. `EXPLAIN` 是干什么的
标准回答：

> `EXPLAIN` 用来查看 SQL 的执行计划，也就是 MySQL 准备如何执行这条语句。

更口语一点：

> 它相当于提前告诉你，这条 SQL 打算怎么跑。

---

## 4. 为什么不能只看“有没有走索引”
因为：

> 走索引不等于一定快。

比如：

- 索引区分度很低
- 扫描行数还是很多
- 回表次数很多
- 排序无法复用索引

这些情况就算用了索引，执行效果也不一定好。

所以：

> 判断 SQL 优不优化，不能只看用了没有，还要看用得好不好。

---

## 5. `EXPLAIN` 里你最该关注哪些字段
面试里最常问的就是这四个：

1. `key`
2. `type`
3. `rows`
4. `extra`

这四个你答顺了，基本就够一面用了。

---

## 6. `key` 看什么
你可以这样答：

> `key` 表示这条 SQL 最终实际使用了哪个索引。

最重要的一句：

> 如果 `possible_keys` 里有索引，但 `key` 最终没选上，说明优化器认为那个索引未必划算。

这句话很像真实排查经验。

---

## 7. `type` 看什么
你不用把所有类型都背死，先会答核心意思。

你可以这样说：

> `type` 反映的是访问方式，通常可以帮助判断 SQL 访问数据的效率高不高。

面试里更稳的答法：

> 一般来说，访问方式越接近精确定位，通常越好；如果已经接近大范围扫描甚至全表扫描，那就需要警惕了。

---

## 8. `rows` 看什么
这是非常重要的一列。

你可以这样答：

> `rows` 表示 MySQL 预估需要扫描的行数。这个值越大，通常说明这条 SQL 代价越高。

一句话版：

> `rows` 大，往往意味着扫描范围大。

---

## 9. `extra` 看什么
这是最容易出细节追问的地方。

你不用一开始背太多，先抓几类最常见的：

- 是否出现额外排序
- 是否用了临时表
- 是否形成覆盖索引

更稳的回答：

> `extra` 主要帮助我们看有没有额外开销，比如额外排序、临时表，以及是否出现覆盖索引等信息。

---

## 10. 为什么 `rows` 和 `extra` 很重要
因为这两列很能说明：

> 这条 SQL 到底是真的高效，还是只是“表面上用了索引”。

比如：

- `key` 看起来用了索引
- 但 `rows` 特别大
- `extra` 里还有额外排序或临时表

那这条 SQL 的执行效果就未必好。

---

## 11. 怎么根据执行计划判断索引有没有真正用好
你可以这样答：

> 我一般不会只看有没有命中 `key`，还会结合 `rows` 和 `extra` 一起看。如果虽然用了索引，但扫描行数还是很大，或者有大量回表、额外排序和临时表，那说明索引未必真正用好了。

这句话很重要，建议直接背。

---

## 12. 索引优化一般怎么做
你可以按这个顺序答：

1. 先看 SQL 是否能改写
2. 再看索引有没有命中
3. 看联合索引顺序合不合理
4. 看是否存在大量回表
5. 看排序和分组能不能复用索引
6. 再决定是否补索引或调整索引

这就是一个比较完整的思路。

---

## 13. 常见优化方向有哪些

### 13.1 调整联合索引顺序
如果高频查询条件和索引顺序不匹配，效果就可能不理想。

---

### 13.2 减少回表
比如让查询字段尽量被索引覆盖。

---

### 13.3 避免无效排序和临时表
如果 `order by`、`group by` 能和索引顺序匹配，通常会更好。

---

### 13.4 改写 SQL
有时问题不在索引，而在 SQL 写法本身，比如：

- 取字段太多
- 条件写法不利于索引
- 分页太深

---

## 14. 一个很实用的面试回答结构
如果面试官问你“你怎么看 `EXPLAIN`”，你可以这样答：

> 我一般重点看四类信息。第一是 `key`，看最终用了哪个索引；第二是 `type`，看访问方式是不是高效；第三是 `rows`，看预计扫描行数大不大；第四是 `extra`，看有没有额外排序、临时表或者覆盖索引。判断一条 SQL 是否真的优化好，不能只看用了没用索引，还要看扫描成本和额外开销。 

---

## 15. 一个更像真实排查的回答
如果面试官继续问“你怎么根据 `EXPLAIN` 优化 SQL”，你可以这样答：

> 我通常会先看 `EXPLAIN` 的 `key`、`rows` 和 `extra`。如果没命中合适索引，就先考虑 SQL 条件和索引设计是否匹配；如果虽然命中了索引，但扫描行数很大，就要看索引区分度和查询范围；如果 `extra` 里出现额外排序或临时表，就会看排序、分组能不能复用索引；如果回表过多，就考虑是否能通过联合索引或覆盖索引优化。整体思路是先判断瓶颈在哪，再决定改 SQL 还是调索引。 

---

## 16. 一段标准面试回答
如果面试官问你“`EXPLAIN` 里你重点看什么”，你可以直接这样答：

> 我一般重点看 `key`、`type`、`rows` 和 `extra`。`key` 看实际使用了哪个索引，`type` 看访问方式是不是足够高效，`rows` 看预计扫描行数大不大，`extra` 看有没有额外排序、临时表或者覆盖索引。真正判断索引有没有用好，不能只看有没有命中索引，还要结合扫描行数和额外开销一起判断。 

---

## 17. 更口语一点的版本
你也可以这样讲：

> 我看 `EXPLAIN` 不会只看有没有索引。我会先看用的是哪个索引，再看大概要扫多少行，然后看有没有额外排序、临时表这些开销。因为有时候表面上用了索引，但实际上扫描很多、回表很多，SQL 还是会慢。 

---

## 18. 最值得先背的 8 句
1. **`EXPLAIN` 的核心作用是看 SQL 准备怎么执行。**
2. **判断 SQL 优不优，不能只看有没有走索引。**
3. **我一般重点看 `key`、`type`、`rows` 和 `extra`。**
4. **`key` 看实际用了哪个索引。**
5. **`rows` 大，通常说明扫描范围大。**
6. **`extra` 主要看额外排序、临时表和覆盖索引等信息。**
7. **索引优化的本质是降低整体执行成本。**
8. **最终要判断是改 SQL，还是调索引。**
