# MySQL 执行流程与慢 SQL 排查

## 1. 这份文档是干什么的
这份文档主要解决两个非常高频的问题：

1. 一条 SQL 到 MySQL 之后大概怎么执行
2. 线上遇到慢 SQL 时怎么排查

这两道题在面试里很像“收尾题”。

因为前面你可能已经讲了：

- 索引
- 事务
- MVCC
- redo log
- binlog

但面试官最后常常会落到：

> 你真实工作里怎么判断 SQL 慢？怎么排查？

所以这份很值得补。

---

## 2. 先说最重要的结论
你先记住这两句：

1. **一条 SQL 进入 MySQL 后，通常会经过连接、解析、优化、执行这几个核心阶段。**
2. **慢 SQL 排查不能只盯着“有没有索引”，而是要结合执行计划、扫描行数、回表、排序、临时表一起看。**

这两句讲顺了，后面就好接。

---

## 3. 一条 SQL 大概怎么执行
先记最基础版本：

1. 客户端发送 SQL
2. MySQL 建立连接并做权限校验
3. 解析 SQL
4. 优化器选择执行方案
5. 执行器调用存储引擎
6. 返回结果

这已经够面试里先开头用了。

---

## 4. 连接阶段是干什么的
这一步你不用讲太深，够面试就行。

你可以这样答：

> SQL 先通过连接器进入 MySQL，连接器负责身份认证、权限校验和连接管理。

一句话版：

> 先连上，再确认你有没有权限执行这条 SQL。

---

## 5. 解析阶段是干什么的
你可以这样理解：

> 解析器会先对 SQL 做词法和语法分析，确认这条 SQL 写得合不合法，识别表名、字段、关键字这些信息。

一句话版：

> 解析阶段主要是“看懂你写了什么”。

---

## 6. 优化阶段是干什么的
这是很重要的一步。

你可以这样答：

> 优化器会根据 SQL、表结构和索引情况，选择一个它认为成本更合适的执行方案，比如决定用哪个索引、从哪张表先查、是否要走排序或临时表。

更口语一点：

> 优化器本质上是在决定“这条 SQL 该怎么跑更划算”。

---

## 7. 执行阶段是干什么的
你可以这样说：

> 执行器会按照优化器选定的执行计划，真正去调用存储引擎读取或修改数据，最后再把结果返回给客户端。

这时候你也可以顺手补一句：

> 如果是查询语句，重点就是怎么取数据；如果是更新语句，还会进一步涉及事务、锁和日志。

---

## 8. `select` 和 `update` 的面试区别怎么讲
这题有时会被继续问。

### `select`
你可以说：

> `select` 更偏向查询路径，重点在于是否走索引、扫描多少行、有没有回表、有没有额外排序和临时表。

### `update`
你可以说：

> `update` 除了查询定位数据之外，还会涉及事务、锁、undo log、redo log、binlog 这些内容。

---

## 9. 什么是慢 SQL
这题不要答太虚。

更稳的说法是：

> 慢 SQL 指的是执行时间明显过长、影响接口响应或者数据库负载的 SQL。它不一定只是“没走索引”，也可能是扫描行太多、回表太多、排序开销大、临时表过多或者 SQL 本身写法不合理。

一句话版：

> 慢 SQL 的本质是执行成本高，不只是索引问题。

---

## 10. 慢 SQL 一般怎么排查
这是最核心的题。

你可以按这个顺序回答：

1. 先确认是不是慢 SQL
2. 看慢 SQL 日志
3. 用 `EXPLAIN` 看执行计划
4. 看有没有走索引
5. 看扫描行数多不多
6. 看有没有回表、临时表、排序开销
7. 再决定是改 SQL、补索引还是调表结构

这就是一个很完整的排查思路。

---

## 11. 面试时最稳的一段排查回答
你可以直接背这段：

> 我一般会先通过慢 SQL 日志确认是哪条语句慢，然后用 `EXPLAIN` 看执行计划，重点看有没有命中合适的索引、扫描行数是否过多、有没有回表、有没有临时表和额外排序。如果发现是索引没用好，就会考虑调整 SQL 写法或补联合索引；如果是扫描范围太大或者回表太多，就会进一步评估索引设计和查询字段；如果表结构本身不合理，才会考虑调表结构或者做数据拆分。 

这段很适合一面。

---

## 12. 排查慢 SQL 时你重点看什么
你可以这样答：

> 我重点会看三类东西。第一是执行计划，看索引命中和扫描行数；第二是 SQL 本身，看 where 条件、排序、分组、分页写法是否合理；第三是数据特征，比如表数据量、字段区分度、是否存在大量回表或者热点范围扫描。

---

## 13. 为什么有索引，SQL 还是会慢
这题几乎一定会被问到。

你可以这样答：

> 因为走索引不等于一定快。比如索引区分度太低、扫描行数仍然很多、回表次数太多、范围查询过大、排序无法复用索引，或者优化器判断全表扫描成本更低，这些都可能导致 SQL 依然很慢。

一句话版：

> 索引只是一个手段，关键还是看整体执行成本。

---

## 14. 什么情况下慢 SQL 不只是索引问题
这题很像真实经验题。

常见情况有：

- `select *` 取了太多不必要字段
- 分页太深
- 排序字段和索引顺序不匹配
- `group by`、`order by` 引入额外排序
- 业务条件本身范围太大
- 表设计不合理

所以你可以这样说：

> 慢 SQL 排查不能只盯着索引，还要看 SQL 写法、业务条件和表结构设计。

---

## 15. 怎么区分是“改 SQL”还是“补索引”
这题答好了会很像真实做过优化。

你可以这样说：

> 如果问题是条件写法不合理、查询字段太多、分页太深、排序方式不匹配，那优先改 SQL；如果问题是高频查询缺少合适索引，或者联合索引顺序设计不合理，那再考虑补索引；如果数据模型本身有问题，最后才考虑调表结构。

---

## 16. 一段标准面试回答
如果面试官问你“一条 SQL 怎么执行，慢 SQL 怎么排查”，你可以这样答：

> 一条 SQL 进入 MySQL 后，通常会先经过连接和权限校验，然后由解析器做语法分析，再由优化器选择执行计划，最后执行器调用存储引擎完成数据读取或修改。慢 SQL 排查时，我一般不会只看有没有索引，而是先通过慢日志定位语句，再用 `EXPLAIN` 看执行计划，重点关注索引命中、扫描行数、回表、临时表和额外排序。如果是 SQL 写法问题就优先改 SQL，如果是索引设计问题再补索引，必要时才考虑调整表结构。 

---

## 17. 更口语一点的版本
你也可以这样讲：

> SQL 到 MySQL 里，先是连接和权限校验，然后解析 SQL，接着优化器决定怎么跑，最后执行器去调存储引擎。慢 SQL 排查我一般先看慢日志，再看 `EXPLAIN`，重点看有没有走对索引、扫了多少行、有没有回表和额外排序。因为很多慢 SQL 不只是“没索引”，也可能是 SQL 写法本身有问题。 

---

## 18. 最值得先背的 8 句
1. **一条 SQL 通常会经过连接、解析、优化、执行这几个核心阶段。**
2. **优化器本质上是在决定这条 SQL 怎么跑更划算。**
3. **慢 SQL 的本质是执行成本高，不只是索引问题。**
4. **排查慢 SQL 先看慢日志，再看 `EXPLAIN`。**
5. **重点要看索引命中、扫描行数、回表、临时表和额外排序。**
6. **有索引不等于一定快。**
7. **慢 SQL 可能是 SQL 写法问题，也可能是索引设计问题。**
8. **最终要判断是改 SQL、补索引，还是调表结构。**
