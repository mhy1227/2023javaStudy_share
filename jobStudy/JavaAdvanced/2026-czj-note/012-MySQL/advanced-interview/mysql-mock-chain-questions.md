# MySQL 连环追问模拟题

## 1. 这份文档怎么用

这一篇不是单纯的问题清单，也不是标准答案合集，而是更接近真实面试的一种整理方式。

它的重点是：

- 面试官不会只问一个点
- 很多题都是从一个基础问题顺着往下追
- 你单点会答，不代表整条链都能接住

建议你这样练：

1. 先只看第一个问题，自己口头回答。
2. 再顺着后面的追问继续答。
3. 哪一环断掉了，就回对应主线文档补。

一句话理解：

> 这一篇练的不是“会不会某一题”，而是“能不能顺着一条线一直答下去”。

## 2. 连环追问一：从索引追到执行计划

### 起手题

- 什么是索引？

### 常见追问

- 为什么索引能加快查询？
- 为什么 MySQL 里经常说 B+ 树，而不是红黑树或者哈希？
- 聚簇索引和二级索引有什么区别？
- 什么是回表？
- 什么是覆盖索引？
- 联合索引的最左前缀原则是什么？
- 什么情况下索引会失效？
- 为什么有索引，SQL 还是会慢？
- 你平时怎么看一条 SQL 有没有真正走索引？

### 这一链真正考什么

- 你是不是只会背“索引加快查询”
- 你能不能把索引结构、查询过程、执行计划串起来
- 你知不知道“建了索引”和“真正跑得快”不是一回事

### 练习时最稳的主线

可以按这个顺序组织：

> 索引是数据结构 -> B+ 树适合磁盘和范围查询 -> InnoDB 有聚簇索引和二级索引 -> 二级索引可能回表 -> 覆盖索引可以减少回表 -> 联合索引受最左前缀影响 -> 索引也可能失效 -> 最后用 `explain` 看执行计划

## 3. 连环追问二：从事务追到隔离级别和 MVCC

### 起手题

- 什么是事务？

### 常见追问

- ACID 是什么？
- 四种隔离级别是什么？
- MySQL 默认隔离级别是什么？
- 什么是脏读、不可重复读、幻读？
- 可重复读为什么还要引出 MVCC？
- MVCC 是什么？
- MVCC 想解决什么问题？
- MVCC 依赖什么机制？
- 什么是 Read View？
- 快照读和当前读有什么区别？

### 这一链真正考什么

- 你是不是把事务只理解成 `commit` 和 `rollback`
- 你能不能把事务和并发读写问题联系起来
- 你是不是知道 MySQL 不是所有一致性问题都靠锁硬抗

### 练习时最稳的主线

可以按这个顺序组织：

> 事务保证一组操作整体性 -> ACID 是事务基本特性 -> 并发下会有脏读、不可重复读、幻读 -> 所以需要隔离级别 -> MySQL 默认是 RR -> RR 下普通读很多时候借助 MVCC -> MVCC 通过 undo log、隐藏字段、Read View 让读写更少阻塞 -> 当前读则更依赖锁

## 4. 连环追问三：从 MVCC 追到锁

### 起手题

- 锁和 MVCC 是什么关系？

### 常见追问

- 为什么普通 `select` 很多时候不加这种锁？
- 什么是当前读？
- `select ... for update` 是什么？
- update、delete 属于什么读？
- 为什么说 InnoDB 行锁和索引强相关？
- `where id = ? for update` 是不是就只锁一行？
- 记录锁、间隙锁、临键锁是什么？
- Gap Lock 和 Next-Key Lock 在解决什么问题？

### 这一链真正考什么

- 你有没有把“读”和“当前读”区分开
- 你知不知道锁和索引、隔离级别是绑在一起的
- 你会不会把 `where id = ?` 这种 SQL 直接想当然

### 练习时最稳的主线

可以按这个顺序组织：

> 普通一致性读偏 MVCC -> 当前读更依赖锁 -> `for update` 是锁定读 -> update/delete 也会加锁 -> InnoDB 行锁通常依赖索引定位 -> 所以不能只看 SQL 长相，还要看索引和执行计划 -> 范围场景再继续聊 gap lock、next-key lock

## 5. 连环追问四：从锁追到并发场景

### 起手题

- 为什么数据库需要锁？

### 常见追问

- 行锁和表锁有什么区别？
- 共享锁和排他锁是什么？
- 什么是意向锁？
- 为什么说锁是为了保证正确性，不只是为了“挡住别人”？
- 扣库存场景为什么总和锁一起问？
- 事务 A 先 `select ... for update`，事务 B 能不能改当前 id？
- 能不能改别的 id？
- 能不能在锁定范围里插入新数据？

### 这一链真正考什么

- 你能不能把抽象锁概念落到业务场景
- 你是不是知道“锁一行”和“锁范围”差别很大
- 你能不能对双事务题做出稳定判断

### 练习时最稳的主线

可以按这个顺序组织：

> 锁是为了控制并发顺序 -> 行锁粒度小并发更强 -> 不同锁语义不一样 -> 真正落到业务里常见的是库存、状态变更、抢单 -> 判断阻塞时统一按“当前读、事务、索引、执行计划、范围、隔离级别”来分析

## 6. 连环追问五：从锁追到死锁和排查

### 起手题

- 什么是死锁？

### 常见追问

- 死锁和普通锁等待有什么区别？
- 为什么会发生死锁？
- 你线上遇到过死锁吗？
- 没遇到过也行，那你知道怎么排查吗？
- `show engine innodb status` 看什么？
- `performance_schema.data_locks`、`data_lock_waits`、`metadata_locks` 分别能看什么？
- 怎么减少死锁发生概率？

### 这一链真正考什么

- 你是不是只会背定义
- 你是否具备一点线上排障意识
- 你能不能从“现象”回到“锁顺序”和“资源竞争”

### 练习时最稳的主线

可以按这个顺序组织：

> 先区分死锁和普通锁等待 -> 再说死锁本质是循环等待 -> 接着说排查入口 -> 最后补减少死锁的方法，比如固定加锁顺序、缩短事务、避免大范围扫描

## 7. 连环追问六：从日志追到两阶段提交

### 起手题

- redo log、undo log、binlog 分别是什么？

### 常见追问

- redo log 主要解决什么问题？
- undo log 为什么和 MVCC 有关系？
- binlog 为什么和主从复制有关？
- redo log 和 binlog 有什么区别？
- 为什么 MySQL 既要 redo log 又要 binlog？
- 什么是两阶段提交？
- 为什么两阶段提交很重要？
- 如果没有两阶段提交，会出现什么问题？

### 这一链真正考什么

- 你是不是只会背三个日志名字
- 你能不能把“崩溃恢复”和“主从复制”区分开
- 你知不知道两阶段提交解决的是一致性问题

### 练习时最稳的主线

可以按这个顺序组织：

> redo log 偏崩溃恢复 -> undo log 偏回滚和 MVCC -> binlog 偏归档和复制 -> redo/binlog 各自职责不同 -> 更新流程里两者都重要 -> 为了避免一边成功一边失败，需要两阶段提交保证一致性

## 8. 连环追问七：从日志追到主从复制和读写分离

### 起手题

- binlog 为什么经常和主从复制放在一起问？

### 常见追问

- MySQL 主从复制大概流程是什么？
- 读写分离为什么能提高性能？
- 主库写完，从库一定立刻能读到吗？
- 主从延迟会带来什么问题？
- 如果用户刚写完就去查，结果查不到，怎么解释？
- 业务上怎么缓解主从延迟带来的一致性问题？

### 这一链真正考什么

- 你是不是只会背“读写分离提高性能”
- 你能不能意识到读写分离会引入一致性问题
- 你知不知道系统设计里的 trade-off

### 练习时最稳的主线

可以按这个顺序组织：

> binlog 为复制提供基础 -> 主从复制让读请求分摊 -> 读写分离提高读吞吐 -> 但复制不是绝对实时 -> 所以会有主从延迟 -> 再往下聊一致性策略，比如关键读走主库、延迟容忍、重试或缓存兜底

## 9. 连环追问八：从慢 SQL 追到调优方法

### 起手题

- 一条 SQL 很慢，你一般怎么排查？

### 常见追问

- 先看什么？
- `explain` 看哪些关键字段？
- `type`、`key`、`rows`、`extra` 怎么理解？
- 什么叫回表太多？
- 什么叫排序没走索引？
- 深分页为什么慢？
- count 为什么有时也慢？
- 你最后会落到哪些优化动作上？

### 这一链真正考什么

- 你是不是只有“加索引”这一招
- 你能不能从现象走到执行计划
- 你知不知道调优是 SQL、索引、数据分布、业务写法一起看

### 练习时最稳的主线

可以按这个顺序组织：

> 先确认慢在哪里 -> 再看执行计划 -> 判断有没有走索引、扫描行数多不多、有没有回表和文件排序 -> 再结合 SQL 写法、索引设计、分页方式、count 方式一起优化

## 10. 连环追问九：从单机 MySQL 追到分库分表

### 起手题

- 单机 MySQL 在什么情况下会考虑分库分表？

### 常见追问

- 是表大了就一定要分吗？
- 分库分表主要解决什么问题？
- 水平拆分和垂直拆分有什么区别？
- 分了以后会带来什么问题？
- 跨库分页、排序、聚合怎么处理？
- 主键怎么设计？
- 为什么很多系统喜欢用分布式 ID？

### 这一链真正考什么

- 你是不是把分库分表当成“银弹”
- 你是否知道拆分带来的复杂度
- 你能不能说出拆分前后的收益和代价

### 练习时最稳的主线

可以按这个顺序组织：

> 单机有容量、QPS、写入热点瓶颈 -> 分库分表是扩展手段 -> 但会带来路由、事务、分页、聚合、主键设计复杂度 -> 所以不是越早拆越好，而是瓶颈明显后再做

## 11. 最后一轮常见总收口

如果面试官一路追完，最后通常会回到一句比较大的问题：

- 你怎么理解 MySQL 面试主线？

可以这样收口：

> 我理解 MySQL 这块真正的主线其实就几条。第一条是索引和执行计划，也就是一条 SQL 为什么快、为什么慢；第二条是事务、隔离级别、MVCC、锁，也就是并发下一致性怎么保证；第三条是 redo、undo、binlog，也就是恢复、回滚、复制怎么做；最后再往工程上走，就是慢 SQL 排查、主从复制、读写分离、分库分表这些场景。面试里很多题看起来分散，其实都是沿着这几条线在追问。
