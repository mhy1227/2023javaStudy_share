# MySQL 分库分表与大表优化

## 1. 这份文档是干什么的

这一篇主要补的是另一个非常典型的进阶追问方向：

- 表越来越大怎么办
- 什么时候考虑分库分表
- 分库分表解决什么问题
- 分库分表会带来什么问题

很多面试官会通过这条线判断你是不是只会背八股，还是知道工程上该怎么取舍。

## 2. 先记最核心的结论

先记住这几句：

1. **分库分表不是默认方案，而是大数据量和高并发下的后置方案。**
2. **大表优化通常先做 SQL 优化、索引优化、冷热分离、归档，再考虑分库分表。**
3. **分库分表解决的是单库单表容量和吞吐瓶颈，但会显著增加系统复杂度。**
4. **分库分表最难的不是拆，而是拆完之后的跨库查询、分页排序、事务和扩容。**

## 3. 为什么要考虑分库分表

本质原因通常有两个：

- 数据量太大，单表撑不住。
- 并发量太大，单库吞吐成为瓶颈。

也就是说，分库分表不是“看起来高级就上”，而是当单库单表已经明显接近瓶颈时才会考虑。

## 4. 什么情况下会考虑分库分表

常见信号：

- 单表数据量持续膨胀，查询和维护成本都很高。
- 索引已经优化得比较充分，但性能还是不够。
- 热点业务写入量太大，主库压力持续过高。
- 大表导致备份、迁移、DDL 维护成本很高。

面试里可以这样说：

> 一般不会一上来就分库分表，通常是在 SQL、索引、缓存、归档这些手段都做过之后，单库单表还是扛不住，才会评估分库分表。

## 5. 什么是垂直拆分

垂直拆分可以从两个角度理解：

- 按业务拆库
- 按字段拆表

例如：

- 用户库、订单库、课程库分开
- 把不常用的大字段拆出去

它解决的问题更偏：

- 职责解耦
- 降低单表宽度
- 降低单库耦合

## 6. 什么是水平拆分

水平拆分指的是：

- 表结构不变
- 但数据按规则拆到多张表、多套库里

例如：

- 按用户 ID 取模
- 按时间分表
- 按区域分库

它解决的问题更偏：

- 数据量过大
- 单表扫描过大
- 单库并发过高

## 7. 分库分表解决什么问题

它主要解决：

- 单表数据量太大。
- 单库写入和读取瓶颈。
- 索引规模过大。
- 维护和扩展困难。

但要记住：

> 它解决的是容量和吞吐问题，不是所有问题的通用解法。

## 8. 分库分表会带来什么问题

这是面试最爱追问的地方。

常见副作用：

- 跨库 join 复杂。
- 跨库事务困难。
- 全局唯一 ID 设计复杂。
- 跨分片分页和排序困难。
- 扩容迁移麻烦。
- 运维成本显著上升。

所以你必须能说出：

> 分库分表不是“做了就更高级”，而是用系统复杂度换容量和吞吐。

## 9. 为什么不建议一上来就分库分表

因为在很多场景下，前置优化还远远没做完。

更合理的顺序通常是：

1. 优化 SQL 写法。
2. 优化索引设计。
3. 做缓存。
4. 做冷热分离。
5. 归档历史数据。
6. 再评估分库分表。

面试里可以这样说：

> 分库分表是重型方案，会带来很多副作用，所以通常应该作为后置方案，而不是默认起手式。

## 10. 大表优化通常先做什么

这是很常见的项目化问题。

你可以按这个顺序答：

- 先查 SQL 和索引有没有问题。
- 再看冷热数据能不能拆开。
- 再看历史数据能不能归档。
- 再看能不能通过缓存降低数据库压力。
- 最后才考虑分库分表。

## 11. 深分页为什么也常和大表一起问

因为大表场景里，很多性能问题最终会落到：

- 扫描太多
- 丢弃太多
- 排序成本太高

比如：

```sql
limit 100000, 20
```

这种查询，数据库往往要先扫出大量记录，再丢掉前面的结果，所以越往后越慢。

## 12. 面试里怎么回答“什么时候考虑分库分表”

可以直接说：

> 一般不会因为表数据一大就马上做分库分表，而是会先看 SQL、索引、缓存、冷热分离、历史归档这些优化手段是不是已经做到位。如果这些都做过之后，单库单表的容量、吞吐、维护成本依然明显成为瓶颈，才会考虑分库分表。因为分库分表虽然能解决容量和并发问题，但会带来跨库查询、事务、分页排序和扩容迁移等一系列复杂度。

## 13. 当前阶段最值得先背的一版

> 分库分表不是默认方案，而是后置方案。通常先做 SQL 优化、索引优化、冷热分离、归档和缓存，确认单库单表确实已经接近瓶颈，再考虑分库分表。分库分表主要解决数据量和并发量过大的问题，但也会带来跨库 join、跨库事务、分页排序、全局 ID 和扩容迁移等复杂度，所以本质上是用更高的系统复杂度去换更大的容量和吞吐。
