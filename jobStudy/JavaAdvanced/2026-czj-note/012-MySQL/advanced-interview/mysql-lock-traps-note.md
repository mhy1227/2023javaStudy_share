# MySQL 锁易错追问题笔记

## 1. 这份文档是干什么的

这一篇不是重复讲 MySQL 锁的基础定义，而是专门整理那些面试里很容易被追问、也很容易答偏的“变体题”“陷阱题”。

这类题的特点通常是：

- 题面看起来很简单
- 你如果只会背“行锁、表锁、间隙锁”就容易答错
- 面试官喜欢在 `select ... for update`、索引、事务、隔离级别、DDL 阻塞之间来回追问

所以这篇的目标不是背概念，而是让你遇到追问时不慌。

## 2. 先记一个通用判断框架

遇到锁相关题，不要一眼就说“锁一行”或者“会阻塞”。

更稳的判断顺序是：

1. 先看是不是当前读。
2. 再看是不是在事务里。
3. 再看条件字段有没有索引。
4. 再看执行计划是否真的走索引。
5. 最后看是等值查询还是范围查询。
6. 再补充隔离级别是不是会影响锁范围。

一句话记忆：

> **先看当前读，再看事务和索引，再看执行计划，最后看范围和隔离级别。**

## 3. 追问一：写了 `for update` 就一定能锁住吗

常见问法：

- 我已经写了 `select ... for update`，为什么别人还是能改？
- `for update` 是不是写上就一定生效？

更稳的回答：

> 不要把 `for update` 理解成“写了这几个字就自动生效”。它本质上是锁定读，通常要放在事务里理解才有意义。如果没有正确的事务边界，锁的效果就可能和你想象的不一样。

你在面试里可以补一句：

> 所以判断一条 `for update` 是否真的起到了并发控制作用，不能只看 SQL 文本，还要看它是不是处在事务中。

## 4. 追问二：外层 `for update` 会不会把子查询里的表也锁住

常见问法：

```sql
select *
from orders
where user_id in (
    select id from user where status = 1
)
for update;
```

面试官会问：

- 这里 `orders` 被锁了，那 `user` 会不会也被锁？

更稳的回答：

> 不能想当然地说“整条 SQL 里出现的表都会被锁”。锁定读主要作用在它真正加锁读取的对象上。面试里如果涉及子查询，应该特别小心，不能默认外层的 `for update` 会自动把子查询里的表也一起锁住。

这类题的核心考点是：

- 不要把“SQL 看起来引用了某张表”和“那张表一定被锁住”画等号
- 锁语义要结合具体读法理解

## 5. 追问三：`where id = ? for update` 为什么有时像锁一行，有时像锁一片

常见问法：

```sql
select * from user where id = 10 for update;
```

面试官继续追：

- `id` 是主键时呢？
- `id` 不是索引时呢？
- 有索引但执行计划没走索引时呢？

更稳的回答：

> 不能只看 `where id = ?` 这个形式。真正决定锁范围的，不只是条件长得像不像“点查”，还要看这个字段是不是索引列、是不是唯一索引，以及执行计划是否真的按索引去扫描。

一句话记忆：

> **长得像精准查询，不代表执行时就是精准加锁。**

## 6. 追问四：锁的是最终命中的行，还是扫描到的范围

这类题很容易让人掉坑里。

常见问法：

- `where` 最后只返回 1 行，是不是就只锁 1 行？
- 只查到几条数据，是不是就只锁这几条？

更稳的回答：

> 面试里不要把“结果集只有几条”直接等价成“只锁这几条”。很多时候更关键的是执行过程中扫描了哪些索引记录、扫描了多大范围。锁往往和扫描路径、索引范围联系更紧，而不是只和最后返回结果的条数绑定。

这句话在面试里很加分：

> 判断锁范围时，要更多站在“执行器怎么扫描”的角度，而不是只站在“结果返回了几行”的角度。

## 7. 追问五：`RR` 和 `RC` 下为什么现象不一样

常见问法：

- 为什么这条 SQL 在可重复读下会挡住插入，在读已提交下可能现象不同？
- Gap Lock、Next-Key Lock 为什么老和隔离级别一起问？

更稳的回答：

> 因为锁的表现不只取决于 SQL 本身，还和事务隔离级别有关。尤其是涉及范围查询、幻读防护的时候，`REPEATABLE READ` 下更容易谈到间隙锁、临键锁；而隔离级别变化后，很多并发现象也会随之变化。

面试里不要死背细节表格，先把主线说清楚：

- 点查更关注记录本身
- 范围查更容易和间隙、临键联系起来
- 隔离级别变化会影响锁策略和并发现象

## 8. 追问六：普通 `select` 不加这种锁，为什么 DDL 还是会被卡住

这是很多人会忽略的一类题。

常见问法：

- 业务一直只是查数据，为什么 `alter table` 卡住了？
- 线上为什么会出现“没改数据也把建索引卡住”的情况？

更稳的回答：

> 不能只盯着行锁。MySQL 里除了你平时常说的行锁、表锁，还有元数据锁这类机制。很多 DDL 被阻塞，不一定是因为某个事务拿着行锁不放，也可能是元数据层面的锁冲突。

这类题的价值在于：

- 说明你知道锁不只有“改数据时的那种锁”
- 说明你能把线上 DDL 阻塞和 MDL 联系起来

## 9. 追问七：插入为什么有时能成功，有时会被挡住

常见问法：

- 事务 A 先做了范围 `for update`
- 事务 B 再往这个范围里 `insert`
- 问你能不能插进去

更稳的回答：

> 这类题不要只看“插的是不是同一行”，而要看事务 A 锁住的是不是一个范围。如果前一个事务锁住的是范围，而不是单条记录，那么新的插入就可能落在这个受保护范围里，从而被阻塞。

一句话记忆：

> **插入能不能成功，不只看有没有同 id 的旧数据，还要看有没有撞进别人的锁范围。**

## 10. 追问八：`NOWAIT`、`SKIP LOCKED` 是什么，为什么现在更常被问

常见问法：

- 如果我不想一直等锁怎么办？
- 多个线程抢任务时，怎么避免大家都堵住？

更稳的回答：

> 这类问法已经不只是基础锁概念，而是把锁拿到业务里用了。`NOWAIT` 更像“拿不到锁就立即失败”，`SKIP LOCKED` 更像“跳过已经被别人锁住的数据，继续处理别的”。

面试里你可以把它们往这些场景上靠：

- 抢单
- 消费任务表
- 批量处理待办任务
- 库存预占

如果你能顺手说一句：

> 这类能力本质上是在平衡一致性、吞吐量和等待时间。

通常会比单纯背概念更像真实做过业务的人。

## 11. 追问九：死锁和普通锁等待，怎么区分

常见问法：

- 两个事务都卡住了，这是死锁吗？
- 遇到死锁你怎么排查？
- 线上怎么看是谁锁住了谁？

更稳的回答：

> 不能把“卡住了”直接等同于“死锁”。普通锁等待可能只是前一个事务还没提交；死锁则是多个事务互相等待，形成循环依赖。排查时要区分这两类问题，再去看锁信息和事务信息。

可以顺手提到的排查方向：

- `show engine innodb status`
- `performance_schema.data_locks`
- `performance_schema.data_lock_waits`
- `performance_schema.metadata_locks`

这类回答的重点不是把命令背全，而是让面试官知道：

> 你知道要看哪里，不会只停留在“重启试试”。

## 12. 面试里最容易答偏的几个点

下面这些坑最常见：

- 只看 `where id = ?` 就断言“只锁一行”
- 只看 SQL 长相，不看是否真的走索引
- 只背行锁、表锁，不知道 MDL
- 只知道 `for update` 会加锁，不知道事务边界也重要
- 只看最终返回结果，不看扫描路径和扫描范围
- 只会说“会阻塞”，但说不清是锁一条记录还是锁一个范围
- 只会背死锁定义，不知道怎么排查

## 13. 一版适合面试现场的总回答

如果面试官问：

> MySQL 锁这块还有什么容易忽略的点？

你可以这样回答：

> 我觉得这块最容易答错的，不是锁分类本身，而是把 SQL 的表面形式和真实加锁行为直接画等号。比如很多人看到 `where id = ? for update` 就说只锁一行，但实际上还得结合事务、索引、执行计划、等值还是范围查询、隔离级别一起判断。另外线上场景里还要注意 MDL，不然会解释不清为什么普通查询也能把 DDL 卡住。再往业务里延伸，像 `NOWAIT`、`SKIP LOCKED`、死锁排查这些，也很适合当作锁知识是否真正落地的分界点。
