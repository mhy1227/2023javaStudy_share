# MySQL 自增主键与自增锁笔记

## 1. 为什么这题高频

这类题很常见，因为它会同时牵出：

- 为什么很多表喜欢用自增主键
- 自增主键对 B+ 树有什么好处
- 自增 ID 为什么不一定连续
- 自增锁和并发插入有什么关系

所以它表面在问主键，实际上会顺手追到索引结构和并发行为。

## 2. 为什么很多场景喜欢自增主键

更稳的回答：

> 自增主键更符合 B+ 树顺序插入的特点，新数据通常会往索引尾部追加，页分裂和页移动压力相对更小，所以整体更稳定，也更利于插入性能。

一句话记忆：

> **顺序主键更友好，随机主键更容易打乱页结构。**

## 3. 自增主键一定比 UUID 好吗

不能答死。

更稳的说法是：

> 如果从 InnoDB 聚簇索引和顺序插入角度看，自增主键通常更友好；但主键设计最终还是要结合业务约束、分布式场景和全局唯一性要求一起看。

这类题重点是：

- 不要把自增主键说成银弹
- 但要知道它对 B+ 树插入路径更友好

## 4. 自增 ID 一定连续吗

这是很高频的坑题。

更稳的回答：

> 不一定。很多人看到自增就以为一定连续，其实更准确的理解应该是“通常递增，但不保证绝对连续”。事务回滚、并发插入、预分配等情况，都可能导致中间有空洞。

一句话记忆：

> **自增更偏保证递增趋势，不保证绝对无缝连续。**

## 5. 为什么会不连续

面试里可以从这个角度回答：

- 有的值已经分配了，但事务后来回滚了
- 并发插入时，申请顺序和最终提交顺序不完全一致
- 底层实现不会为了“连续美观”而牺牲并发和性能

你不需要把实现细节讲得特别深，但要能说明：

> 数据库更关心唯一性和可用性，不会为了让编号好看而强行保证连续。

## 6. 什么是自增锁

更稳的理解是：

> 自增锁可以理解成数据库在分配自增值时，为了保证这件事本身可控而使用的一种并发控制手段。

面试里更关键的不是背定义，而是知道它和这些问题相关：

- 并发插入
- 自增值分配
- 为什么可能不连续
- 为什么大批量插入和单条插入表现可能不同

你在面试里可以补一句：

> 它本质上是“分配自增值时的并发控制问题”，不是普通业务行锁那种语义。

## 7. 为什么大批量插入和单条插入常被一起问

因为这能继续往下追到：

- 自增值分配策略
- 插入并发能力
- 为什么某些场景下锁竞争更明显

面试里不一定非要把内部模式背到特别细，但要知道：

> 不同插入方式下，对自增值分配和并发行为的影响并不完全一样。

## 8. 为什么不建议把自增 ID 当业务含义

这是个很实用的项目题。

更稳的回答：

> 因为自增 ID 更适合作为技术主键，而不是业务语义字段。它不保证绝对连续，也不应该拿来推断真实业务顺序、用户规模或者订单量。把它当业务含义用，后续很容易踩坑。

## 9. 自增主键和分布式 ID 怎么选

可以这样答：

> 如果是单库单表、本地写入为主，自增主键通常简单稳定；但如果系统已经走到分库分表或者分布式写入场景，就不能只依赖单机自增主键，这时往往要引入分布式 ID 方案。

这题的关键不是站队，而是表达出：

- 自增主键很适合单机 InnoDB
- 分布式场景下还要看全局唯一和路由需求

## 10. 面试官为什么爱顺手问“为什么推荐用 bigint 自增主键”

因为这题能一起看出你是否理解：

- 主键是聚簇索引
- 顺序插入对 B+ 树更友好
- 主键类型会影响二级索引体积

可以这样答：

> InnoDB 里主键索引通常就是聚簇索引，而且二级索引叶子节点里还会带主键值，所以主键越大，二级索引也会更胖。很多场景喜欢用整型、自增、较短的主键，核心就是希望聚簇索引和二级索引都更稳定、更紧凑。

## 11. 最容易答错的点

- 说自增 ID 一定连续
- 只会说“插入快”，但说不出和 B+ 树顺序插入的关系
- 不知道主键值大小也会影响二级索引体积
- 把技术主键直接当成业务语义字段

## 12. 一版适合面试的回答

> 我一般会把这题拆成三层。第一层，自增主键更符合 InnoDB B+ 树顺序插入的特点，所以通常比随机主键更稳定。第二层，自增 ID 不保证绝对连续，因为事务回滚和并发分配都可能造成空洞。第三层，自增主键不只是“方便编号”，它还关系到聚簇索引写入路径和二级索引大小，所以很多业务表会优先选整型自增主键。
