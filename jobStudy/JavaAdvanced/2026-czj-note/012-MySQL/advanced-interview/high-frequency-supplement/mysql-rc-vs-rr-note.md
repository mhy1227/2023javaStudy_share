# MySQL RC 与 RR 取舍笔记

## 1. 为什么这题越来越常见

很多人只会背：

- MySQL 默认隔离级别是 `REPEATABLE READ`

但真实面试会继续问：

- 生产环境为什么有些团队更喜欢 `READ COMMITTED`
- 默认是 RR，为什么还要改
- RC 和 RR 在并发行为上怎么取舍

所以这题本质上是在考“默认配置”和“线上取舍”之间的差别。

## 2. RR 和 RC 最基础的区别

最基础先记：

- `RR` 是可重复读
- `RC` 是读已提交

更口语一点说：

> RR 更强调同一事务里前后读的一致性；RC 更强调只读到别人已经提交的数据。

## 3. 为什么默认是 RR

更稳的回答：

> 因为 RR 在一致性保证上更强，尤其在 MySQL InnoDB 体系下，它配合 MVCC、间隙锁等机制，可以更好地处理并发读写场景和幻读问题。

## 4. 最常见的直观对比怎么记

你可以先粗记成：

- `RR`：事务里前后读更强调一致
- `RC`：每次读都更贴近“别人已经提交的最新结果”

如果面试官继续追，你再补一句：

> 这不只是“读出来的内容”不一样，后面还会继续牵到锁冲突、间隙锁、死锁概率和并发表现。

## 5. 那为什么很多线上场景又会谈 RC

因为很多真实业务更关心的是：

- 更高并发
- 更少锁冲突
- 更低死锁概率

面试里你可以这样答：

> 默认是 RR，不代表所有业务场景都最适合 RR。很多线上系统会根据业务一致性要求和并发压力，在 RR 和 RC 之间做权衡。如果更担心锁冲突、间隙锁影响和并发吞吐，有些场景会更偏向 RC。

## 6. 这题真正想让你说什么

不是让你死背定义，而是让你表达出：

- 一致性更强，不一定总是更合适
- 并发更高，不一定总是更安全
- 隔离级别是业务取舍，不是单纯背配置

一句话记忆：

> **RR 偏更强一致性，RC 偏更高并发友好。**

## 7. 这题为什么经常和锁一起问

因为很多业务并不是只关心“读出来是不是一致”，还关心：

- 会不会锁太多
- 会不会死锁变多
- 会不会影响吞吐

也就是说，面试官问 RC 和 RR，很多时候是在顺着问：

> 你知不知道隔离级别会影响并发成本，而不是只会背定义。

## 8. 面试里怎么回答更像做过项目的人

可以这样答：

> 我会先说明 MySQL InnoDB 默认是 RR，然后再补一句，生产环境并不是所有系统都机械沿用默认值。因为隔离级别本质上是在一致性和并发性之间取舍。像一些更看重吞吐和降低锁冲突的业务，可能会更偏向 RC；但如果业务对事务内前后读取一致性要求更高，那 RR 会更稳。真正的选择逻辑不是背默认值，而是看业务更怕一致性问题，还是更怕并发成本过高。

## 9. 如果面试官问“那你会怎么选”

更稳的回答不是直接站队，而是这样说：

> 我不会脱离业务场景直接说 RC 或 RR 谁更好。我会先看业务对事务内前后读取一致性的要求，再看系统是不是很在意锁冲突、死锁概率和并发吞吐。如果一致性要求更强，我会更偏向 RR；如果更希望并发友好、减少锁冲突，一些场景会更偏向 RC。

## 10. 这题最容易和哪些内容连起来

- 幻读
- MVCC
- 间隙锁
- 死锁
- `select ... for update`

也就是说，面试官问 RC 和 RR，往往不是孤立问，而是准备顺着往锁和并发行为上追。

## 11. 最容易答错的点

- 只会背默认隔离级别
- 把 RR 说成绝对更好
- 把 RC 说成“更差的版本”
- 完全不提业务取舍

## 12. 一句话收尾

> RC 和 RR 的核心不是谁绝对更高级，而是业务到底更偏一致性还是更偏并发友好。默认值只是起点，不是所有系统的终点。
