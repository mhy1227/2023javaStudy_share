# MySQL 字符集与排序规则笔记

## 1. 为什么这题也会被问

这类题不是最核心主线，但在真实项目和面经里出现频率并不低。

常见问法：

- 为什么建议用 `utf8mb4`
- MySQL 的 `utf8` 和 `utf8mb4` 有什么区别
- 什么是 `collation`
- 排序规则为什么会影响查询结果

## 2. 为什么很多人强调 `utf8mb4`

更稳的回答：

> 因为 MySQL 里常说的 `utf8` 不是完整意义上的 UTF-8，而 `utf8mb4` 对更完整的 Unicode 字符支持更好，所以很多现代业务表更推荐直接用 `utf8mb4`。

一句话记忆：

> **现在项目里谈 UTF-8，很多时候实际更推荐直接落到 `utf8mb4`。**

## 3. `utf8` 和 `utf8mb4` 最值得先记的区别

你可以先这样记：

- `utf8` 在 MySQL 语境里历史上不是完整 Unicode 支持
- `utf8mb4` 对更完整字符支持更好

所以很多时候更稳的习惯是：

> 新项目默认优先考虑 `utf8mb4`，而不是继续用历史遗留的 `utf8`。

## 4. 什么是 collation

可以这样理解：

> 字符集更像“能存什么字符”，排序规则更像“这些字符怎么比较、怎么排序、怎么判断相等”。

这是最核心的主线。

## 5. 为什么排序规则重要

因为它会影响：

- 排序结果
- 大小写敏感与否
- 字符串比较结果
- 唯一索引冲突判断

也就是说，它不是一个纯展示层问题，而会真实影响数据库行为。

## 6. 为什么它还会影响索引和唯一约束

因为字符串比较规则一旦变化，数据库对“相等”与“顺序”的判断也会变化。

你在面试里可以这样说：

> 排序规则不只是 `order by` 的展示问题，它还会影响字符串比较逻辑，而比较逻辑变化后，唯一索引冲突判断、范围比较结果、甚至某些查询命中表现看起来都会变得不一样。

## 7. 这题最容易落到什么项目场景

最常见的是：

- 线上出现乱码
- 表和库字符集不一致
- 明明看起来字符串不同，却碰到唯一约束冲突
- 排序结果和预期不一致
- 大小写敏感与否和业务理解不一致

这类题一旦被问，回答最好别只停在概念层。

## 8. 面试里怎么回答更稳

可以这样答：

> 我会把字符集和排序规则分开讲。字符集决定能存什么字符，比如为什么很多场景更推荐 `utf8mb4`；排序规则决定字符怎么比较和排序，比如是否区分大小写、排序顺序怎么定。真实项目里如果这两个设置不合理，可能会带来乱码、比较异常、排序不符合预期，甚至索引和唯一约束行为看起来“怪怪的”。所以这类问题不能只当成显示问题，而要当成数据库行为配置问题来看。

## 9. 这题容易和什么场景绑在一起

- 表字段乱码
- 大小写敏感问题
- 唯一索引冲突问题
- `order by` 排序结果不符合预期

## 10. 最容易答错的点

- 只会背 `utf8mb4`，说不清为什么
- 分不清字符集和排序规则
- 不知道排序规则会真实影响数据库比较行为

## 11. 一句话收尾

> 字符集解决的是“能不能存”，排序规则解决的是“怎么比、怎么排”。面试里只要把这两层分清，基本就稳了。
