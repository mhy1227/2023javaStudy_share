# MySQL 事务边界与隐式提交笔记

## 1. 为什么这题会被问

很多人会背：

- 事务要么都成功，要么都失败
- 用 `begin`、`commit`、`rollback`

但真实面试会继续问：

- 为什么我以为在事务里，结果事务没按预期工作
- DDL 为什么会影响事务边界
- `autocommit` 有什么影响
- 为什么 `for update` 写了却感觉没锁住

所以这题本质上是在考：

> 你是不是只会背事务定义，还是知道事务真正的边界在哪里。

## 2. 什么叫事务边界

可以先这样理解：

> 事务边界就是事务从哪里开始，到哪里结束，中间哪些语句真的属于同一个事务。

一句话记忆：

> **不是你心里觉得“这些 SQL 算一组”，数据库就一定按你心里想的来。**

## 3. `autocommit` 为什么重要

更稳的回答：

> `autocommit` 会直接影响每条 SQL 是不是自动单独提交。如果你没有明确开启一个完整事务，那很多语句执行完可能就已经提交了，后面再谈“统一回滚”就不成立了。

这类题最适合和下面这些内容一起记：

- `begin`
- `commit`
- `rollback`
- `select ... for update`

## 4. 为什么“我以为在事务里”是高频坑

因为现实里很常见的误区就是：

- 以为几条 SQL 连着写就是一个事务
- 以为只要写了 `for update` 就一定一直锁着
- 以为遇到异常就一定会整体回滚

数据库其实更关心：

- 有没有真的开启事务
- 有没有提前提交
- 中间有没有改变事务边界的语句

## 5. 什么是隐式提交

更稳的理解是：

> 隐式提交可以理解成数据库在某些语句执行前后，自动帮你结束了当前事务边界。也就是说，你原本以为事务还在持续，但实际上边界已经变了。

你不一定要把所有语句类型都背全，但要知道：

> 这类问题常常和 DDL 一起出现。

## 6. 为什么 DDL 总和隐式提交绑在一起问

因为 DDL 不像普通 DML 那样只处理数据，它会影响表结构和事务边界。

面试里更稳的回答是：

> DDL 往往要单独看，不能把它简单等同于普通增删改查。很多时候它不仅影响表结构，还会影响事务边界，所以面试官喜欢顺着 DDL 去追问隐式提交。

## 7. 为什么这题经常和 `for update` 一起问

因为很多人写了：

```sql
select * from user where id = 10 for update;
```

就以为一定能稳定锁住。

更稳的理解是：

> 不能只看 SQL 文本，还要看它是不是处在真正的事务边界里。如果事务边界没按你想的那样存在，那锁的持续时间和效果也可能和预期不一致。

## 8. 哪些现象说明你可能把事务边界想错了

比如：

- 明明以为能回滚，结果一部分已经生效
- 明明以为锁还在，结果别人已经改了
- 明明几条 SQL 想放一起，结果数据库是分开处理的

这类题目本质上都在提醒你：

> 事务不是“脑补分组”，而是数据库实际认可的边界。

## 9. 面试里怎么回答更稳

可以这样答：

> 我理解事务问题最容易出错的地方，不是 ACID 定义，而是事务边界判断。比如有没有显式开启事务、`autocommit` 是什么状态、中间有没有 DDL 这种会影响事务边界的语句，这些都会直接影响“到底是不是同一个事务”。所以我不会只看 SQL 长相，还会先确认事务边界是不是成立。

## 10. 这题最容易和什么内容连起来

- DDL / MDL
- `autocommit`
- `select ... for update`
- 锁持续时间
- 回滚为什么没按预期发生

所以它很适合放在：

> 事务 -> 锁 -> DDL 这条链路里一起复习。

## 11. 最容易答错的点

- 把“连着执行几条 SQL”直接等同于一个事务
- 完全忽略 `autocommit`
- 知道 DDL 会影响事务，但说不清为什么要单独看
- 只会背“隐式提交”四个字，不会解释它对边界的影响

## 12. 一版适合面试的回答

> 事务题里我觉得最容易答错的不是 ACID，而是事务边界。也就是说，不能只看几条 SQL 写在一起，就认为它们一定属于同一个事务。还要看有没有显式开启事务，`autocommit` 是什么状态，以及中间有没有 DDL 这类可能影响事务边界的语句。隐式提交的核心不是背规则，而是知道数据库可能在你没意识到的时候已经把当前事务结束了，这会直接影响回滚效果、锁持续时间和业务预期。

## 13. 一句话收尾

> 事务真正难的不是“会不会写 begin/commit”，而是你能不能判断事务边界到底有没有按你以为的方式成立。
