# MySQL 进阶高频题整理

## 1. 这份文档是干什么的

这一篇专门整理 MySQL 在中高级面试里更容易被追问的内容。

这些题通常不会出现在最开头，但一旦你前面的基础题答顺了，面试官很可能继续往下挖。

## 2. 主从复制方向

高频问题：

- MySQL 主从复制大概怎么做？
- binlog 在主从复制里起什么作用？
- 主从延迟是什么？
- 如果从库延迟明显，你会怎么分析？
- 主从复制为什么会出现一致性问题？

回答方向：

- 先讲主库写 binlog，从库拉取并重放。
- 再讲主从延迟本质是主库写入速度和从库应用速度不一致。
- 再补一致性影响，比如读写分离下“刚写完查不到”。

## 3. 读写分离方向

高频问题：

- 为什么要做读写分离？
- 读写分离的收益是什么？
- 读写分离会带来什么问题？
- 如何处理“写完立刻读不到”的问题？

回答方向：

- 先讲主库负责写，从库负责读，降低主库压力。
- 再讲读写分离最大问题是主从延迟带来的一致性体验问题。
- 再讲常见处理手段，比如关键读走主库、延迟敏感读走主库、业务层兜底。

## 4. 分库分表方向

高频问题：

- 为什么要分库分表？
- 什么情况下会考虑分库分表？
- 分库分表解决什么问题？
- 分库分表会带来什么问题？
- 水平拆分和垂直拆分怎么理解？

回答方向：

- 先讲单库单表容量和吞吐瓶颈。
- 再讲分库分表主要是为了解决数据量和并发量过大的问题。
- 再讲副作用，比如跨库 join、跨库事务、分页排序复杂、扩容迁移麻烦。

## 5. 深分页方向

高频问题：

- 为什么深分页慢？
- `limit 100000,20` 为什么慢？
- 深分页怎么优化？

回答方向：

- 先讲 offset 很大时扫描和丢弃成本高。
- 再讲如果排序还没命中索引，成本会更大。
- 再讲优化思路：基于主键/游标翻页，而不是一直用大 offset。

## 6. count 优化方向

高频问题：

- `count(*)`、`count(1)`、`count(字段)` 有什么区别？
- 为什么 count 有时会慢？
- 大表统计总数怎么优化？

回答方向：

- 先讲语义区别，尤其 `count(字段)` 会忽略 `null`。
- 再讲大表 count 慢的本质是扫描成本高。
- 再讲优化思路：缓存计数、异步统计、近似值、分段统计。

## 7. Buffer Pool 方向

高频问题：

- Buffer Pool 是什么？
- 为什么 Buffer Pool 重要？
- Buffer Pool 命中率低意味着什么？
- 脏页是什么？

回答方向：

- 先讲 Buffer Pool 是 InnoDB 的内存缓冲区，主要缓存数据页和索引页。
- 再讲命中率低意味着很多读要落到磁盘，性能会下降。
- 再讲脏页是内存中被修改但还没刷盘的页。

## 8. 大表优化方向

高频问题：

- 表越来越大怎么办？
- 大表优化从哪些方向考虑？
- 为什么不建议一上来就分库分表？

回答方向：

- 先讲 SQL 和索引优化。
- 再讲冷热分离、历史归档。
- 再讲缓存、异步化。
- 最后才是分库分表。

## 9. 高并发热点更新方向

高频问题：

- 为什么同一条记录高并发更新会很慢？
- 热点行问题是什么？
- 高并发更新库存时为什么数据库会成为瓶颈？

回答方向：

- 先讲热点行会导致锁竞争严重。
- 再讲事务持锁时间一长，吞吐会明显下降。
- 再讲业务上常见的削峰、异步、分段、预扣等思路。

## 10. 一致性与高可用方向

高频问题：

- 主从复制下一致性怎么保证？
- 为什么说数据库高可用不等于强一致？
- 数据库层面常见的高可用思路有哪些？

回答方向：

- 先讲主从和故障切换提升可用性。
- 再讲复制延迟带来的一致性问题。
- 再讲业务上往往需要在一致性、性能、可用性之间做平衡。

## 11. 进阶面试里最值得先背的题

如果时间不多，优先准备：

1. MySQL 主从复制大概怎么做。
2. 读写分离的收益和问题。
3. 深分页为什么慢，怎么优化。
4. 为什么不建议一上来就分库分表。
5. Buffer Pool 是什么，为什么重要。
6. 热点行更新为什么会慢。
7. 大表优化通常先做什么，后做什么。

## 12. 当前阶段最值得先背的一版

> MySQL 面试除了基础的索引、事务、锁、日志之外，进阶部分还经常问主从复制、读写分离、分库分表、深分页、大表优化、Buffer Pool 和热点更新。主从复制和读写分离主要解决读压力和可用性问题，但会带来主从延迟和一致性问题；深分页慢的本质是扫描和丢弃成本高；大表优化通常先做 SQL 和索引优化、冷热分离、历史归档，最后才考虑分库分表；Buffer Pool 是 InnoDB 很重要的内存缓冲区，命中率低往往意味着 IO 压力大；而热点更新则常常导致锁竞争和吞吐下降。
