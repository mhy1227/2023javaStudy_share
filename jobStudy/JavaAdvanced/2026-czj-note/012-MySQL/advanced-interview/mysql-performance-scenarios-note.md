# MySQL 性能场景题笔记

## 1. 这份文档是干什么的

这一篇补的是 MySQL 进阶面试里非常常见的一类“性能场景题”：

- 深分页为什么慢
- `count(*)` 为什么有时慢
- 热点更新为什么慢
- `order by` / `group by` 为什么容易出性能问题

这些题比基础题更像真实面试，因为它们和项目性能问题直接相关。

## 2. 先记最核心的结论

先记住这几句：

1. **深分页慢的本质是扫描和丢弃成本高。**
2. **count 慢的本质通常是扫描成本高。**
3. **热点更新慢的本质通常是锁竞争严重。**
4. **排序和分组慢，很多时候是因为没能很好复用索引。**

## 3. 深分页为什么慢

典型 SQL：

```sql
select * from user order by id limit 100000, 20;
```

为什么慢：

- 不是只取 20 条就只做 20 条的工作。
- 数据库往往要先扫描前面大量记录。
- 再丢掉前面的结果。
- 最后只返回后面的少量数据。

一句话记忆：

> **深分页慢，不是返回慢，而是前面被丢掉的数据太多。**

## 4. 深分页怎么优化

常见思路：

- 基于主键或游标翻页。
- 减少大 offset。
- 尽量让排序字段命中索引。
- 避免深分页直接查大宽表。

面试里可以这样答：

> 深分页的本质问题是扫描和丢弃成本高，所以通常不会一直用大 offset 分页，而是更推荐基于主键或游标做翻页，让数据库从上次位置继续查。

## 5. count(*) 为什么有时慢

很多人觉得 `count(*)` 就是一个很轻的操作，但大表里不一定。

本质原因通常是：

- 需要扫描大量数据
- 没有更便宜的统计路径
- 大表下统计本身就有成本

所以不要把它想成“数据库立刻就知道总数”。

## 6. count(*)、count(1)、count(字段) 怎么答

面试里建议这样说：

> 语义上，`count(*)` 和 `count(1)` 都是统计行数，而 `count(字段)` 会忽略字段值为 `null` 的行。性能上在很多场景下不需要过度神化差异，真正影响性能的往往还是扫描成本和执行路径，而不是写成 `*` 还是 `1`。

## 7. 大表 count 怎么优化

常见思路：

- 接受近似值，而不是每次实时精确统计。
- 对热点统计做缓存。
- 异步汇总。
- 分段统计，再聚合结果。

面试里可以这样答：

> 如果表很大，实时 count 成本太高，很多业务不会每次都强求实时精确值，而是会通过缓存、异步统计或者近似值的方式降低数据库压力。

## 8. 热点更新为什么慢

热点更新一般指：

- 很多请求同时更新同一条或同一小批记录。
- 例如库存、计数器、热门状态位。

为什么慢：

- 大量事务争抢同一行锁。
- 持锁时间一长，后面的请求都在排队。
- 吞吐会明显下降。

一句话记忆：

> **热点更新慢，本质上是高并发下的锁竞争问题。**

## 9. 热点更新怎么优化

常见思路：

- 缩短事务。
- 让更新条件精确命中索引。
- 业务上削峰。
- 把热点拆散。
- 用缓存、异步、预扣等思路分摊数据库压力。

## 10. order by 为什么会慢

常见原因：

- 排序字段没复用索引。
- 数据量太大。
- 先筛选后排序的候选集仍然很大。
- `Extra` 出现 `Using filesort`。

面试里可以这样答：

> order by 慢，很多时候不是语法本身的问题，而是排序没有借助索引完成，导致 MySQL 需要额外排序，数据量一大开销就明显上来了。

## 11. group by 为什么会慢

常见原因：

- 分组字段没用好索引。
- 分组前的过滤不够强。
- 候选数据太多。
- `Extra` 里出现 `Using temporary`。

一句话记忆：

> **group by 慢，很多时候是因为大量数据先被拉出来，再做分组。**

## 12. 面试里怎么回答这些性能场景题

可以直接这样说：

> 这类 MySQL 性能问题我一般会先抓本质。深分页慢，本质是扫描和丢弃成本高；count 慢，本质通常是大表扫描成本高；热点更新慢，本质是锁竞争严重；order by 和 group by 慢，则很多时候是没有很好复用索引，导致额外排序或临时表开销。判断问题时我会结合执行计划、索引设计、数据量和并发情况一起看，而不会只凭一个点下结论。

## 13. 当前阶段最值得先背的一版

> MySQL 常见性能场景里，深分页慢是因为大 offset 会带来大量扫描和丢弃，count 慢通常是因为大表扫描成本高，热点更新慢本质是高并发下的锁竞争，order by 和 group by 慢则往往和索引复用不好、额外排序或临时表有关。优化方向通常是减少扫描范围、让索引尽量同时服务过滤和排序、避免热点集中，以及结合缓存和异步手段分担数据库压力。
