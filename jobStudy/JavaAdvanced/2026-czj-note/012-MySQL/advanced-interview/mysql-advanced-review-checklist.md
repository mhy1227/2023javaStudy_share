# MySQL 进阶考前复盘 Checklist

## 1. 这份文档怎么用

这一份是给进阶 MySQL 面试做最后速扫用的，不是详细笔记。

建议：

- 面试前 20 分钟：按模块过一遍。
- 面试前 5 分钟：只看“最值得先背”。
- 如果某块说不顺，就回对应专题文档。

## 2. 主从复制与读写分离 Checklist

- 我能说清主从复制大概怎么做。
- 我能说清 binlog 在主从复制中的作用。
- 我能解释主从延迟是什么。
- 我能说清读写分离的收益。
- 我能说清读写分离最大的问题是主从延迟带来的一致性问题。
- 我能回答“刚写完查不到怎么办”。

最值得先背：

> 主从复制的核心是主库写 binlog，从库拉取并重放；读写分离是主库负责写，从库负责读。它能提升吞吐，但最大问题是主从延迟带来的一致性风险，所以关键读、写后立即读通常会优先走主库。

## 3. 分库分表 Checklist

- 我能说清为什么要分库分表。
- 我能说清什么时候才会考虑分库分表。
- 我能区分水平拆分和垂直拆分。
- 我能说清分库分表解决什么问题。
- 我能说清分库分表带来的副作用。
- 我能说清为什么不建议一上来就分库分表。

最值得先背：

> 分库分表不是默认方案，而是后置方案。通常先做 SQL、索引、缓存、冷热分离、归档等优化，只有单库单表容量和吞吐确实成为瓶颈时才考虑分库分表。它解决的是容量和并发问题，但会带来跨库 join、跨库事务、分页排序和扩容迁移等复杂度。

## 4. Buffer Pool Checklist

- 我能说清 Buffer Pool 是什么。
- 我能说清它为什么重要。
- 我能说清命中率低意味着什么。
- 我能解释脏页是什么。
- 我知道 Buffer Pool 和 redo log 是配合关系。

最值得先背：

> Buffer Pool 是 InnoDB 的核心内存缓冲区，主要缓存数据页和索引页，本质上是用内存换磁盘 IO。命中率高说明更多访问能直接走内存；脏页则是已经在内存中改过但还没刷盘的页，它会和 redo log 配合，一起平衡性能和可靠性。

## 5. 性能场景 Checklist

- 我能解释深分页为什么慢。
- 我能说清深分页怎么优化。
- 我能解释 count 为什么有时慢。
- 我能说清热点更新为什么慢。
- 我能说清 order by / group by 慢的常见原因。

最值得先背：

> 深分页慢的本质是扫描和丢弃成本高；count 慢通常是大表扫描成本高；热点更新慢本质是锁竞争严重；order by 和 group by 慢则很多时候是因为没能很好复用索引，产生了额外排序或临时表开销。

## 6. 如果时间只剩 5 分钟

只看这 5 个点：

1. 主从复制和读写分离。
2. 主从延迟和写后立即读问题。
3. 为什么不建议一上来就分库分表。
4. Buffer Pool 是什么、为什么重要。
5. 深分页、热点更新、count 的本质原因。

## 7. 如果时间只剩 2 分钟

只背这段：

> MySQL 进阶面试最常问的是主从复制、读写分离、分库分表、深分页、大表优化、Buffer Pool 和热点更新。主从复制的核心是主库写 binlog、从库拉取并重放，读写分离虽然能提升吞吐，但会带来主从延迟和一致性问题；分库分表解决容量和并发瓶颈，但会显著增加系统复杂度；Buffer Pool 是 InnoDB 的核心内存缓冲区，本质上是用内存换磁盘 IO；而深分页、count、热点更新这类题，本质上都在考你对数据库性能瓶颈和成本来源的理解。
