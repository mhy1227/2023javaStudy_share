# MySQL 索引笔记

## 1. 这份文档是干什么的

索引是 MySQL 面试里最容易被连续追问的一条线。

面试官通常不会只问一句“什么是索引”，而是会顺着往下追：

1. 为什么索引能加快查询。
2. 为什么 MySQL 常说 B+ 树。
3. 聚簇索引和二级索引怎么区分。
4. 什么是回表。
5. 什么是覆盖索引。
6. 联合索引和最左前缀原则怎么理解。
7. 哪些情况会导致索引失效。
8. 什么情况下该建索引，什么情况下不该建。

所以这份文档的目标是把索引这一整条主线先讲顺。

## 2. 先记最核心的结论

先记住这几句：

1. **索引的本质是帮助 MySQL 更快定位数据的数据结构。**
2. **MySQL InnoDB 常说的索引底层是 B+ 树。**
3. **主键索引通常就是聚簇索引，二级索引保存的是主键值。**
4. **通过二级索引查到主键后，再去聚簇索引查整行数据，这一步就叫回表。**
5. **联合索引是否生效，通常要看最左前缀原则。**
6. **索引不是越多越好，索引会占空间，也会拖慢插入、删除、更新。**

## 3. 什么是索引

可以先这样回答：

> 索引本质上是一种辅助查询的数据结构，作用是让数据库在查询时不用每次都全表扫描，而是能更快定位到目标数据。

更口语一点可以说：

> 你可以把索引理解成书的目录。没有目录时，你只能一页一页翻；有目录时，你能先定位章节，再快速找到具体内容。

## 4. 为什么索引能让查询更快

因为没有索引时，MySQL 很多时候只能全表扫描。

全表扫描的意思是：

- 从第一条数据开始，一条一条比对。
- 数据量小时还好。
- 数据量一大，性能就会明显下降。

有索引之后，数据库就不需要盲目从头扫到尾，而是借助索引结构快速缩小查找范围。

所以索引解决的核心问题就是：

> **减少扫描范围，降低查找成本。**

## 5. 为什么 MySQL 常说 B+ 树

这是高频中的高频。

可以先记这个版本：

> 因为数据库索引既要支持高效单点查询，也要支持范围查询和顺序遍历，B+ 树在磁盘 IO、范围查找、有序性方面更适合数据库索引场景。

你可以从这几个角度理解：

### 5.1 B+ 树层级低

B+ 树一个节点可以存很多键值，所以整棵树通常不会太高。

树越矮，查找时需要经过的层数越少，磁盘 IO 通常也更少。

### 5.2 叶子节点有序

B+ 树的叶子节点通常按顺序连接。

这意味着：

- 范围查询很方便。
- 排序查询也更友好。
- 例如 `between`、`>`、`<` 这类查询比较适合。

### 5.3 非叶子节点只做导航

B+ 树非叶子节点主要存键，不存完整数据行。

这样一个页里可以放更多索引项，更适合数据库页式存储和查询定位。

## 6. InnoDB 中常见的索引类型

### 6.1 聚簇索引

可以先记：

> 聚簇索引就是数据和索引放在一起，叶子节点里直接存整行数据。

在 InnoDB 里，主键索引通常就是聚簇索引。

也就是说：

- 表数据本身是按聚簇索引组织的。
- 如果有主键，通常按主键组织。
- 如果没有显式主键，InnoDB 会选一个合适的非空唯一索引。
- 如果都没有，会生成隐藏主键。

### 6.2 二级索引

可以先记：

> 二级索引的叶子节点不直接存整行数据，通常存的是索引列值加主键值。

这就引出一个高频点：回表。

## 7. 什么是回表

如果通过二级索引已经找到了主键值，但还需要查整行完整数据，就还要再去聚簇索引里查一次。

这个过程就叫回表。

可以这样回答：

> 回表就是先通过二级索引定位到主键，再根据主键去聚簇索引查询完整数据。

所以回表本质上是“查了两次”。

## 8. 什么是覆盖索引

覆盖索引是和回表相对的概念。

如果你要查询的字段，刚好都已经在索引里，那么数据库就没必要再回到聚簇索引查整行数据。

可以这样回答：

> 覆盖索引就是查询所需字段都能从索引里直接拿到，不需要回表。

这通常会更快，因为少了一次额外的数据访问。

## 9. 索引分类怎么理解

常见分类先记这几个：

- 主键索引
- 唯一索引
- 普通索引
- 联合索引

### 9.1 主键索引

主键列上的索引，通常也是 InnoDB 的聚簇索引。

特点：

- 不能重复。
- 不能为 `null`。
- 经常用于唯一标识一条记录。

### 9.2 唯一索引

唯一索引要求索引列值不能重复，但不一定就是主键。

例如：

- 手机号
- 身份证号
- 业务唯一编码

### 9.3 普通索引

最普通的索引类型，主要用来提升查询性能，不要求唯一。

### 9.4 联合索引

联合索引是把多个列放在同一个索引里。

例如：

- `(user_id, status)`
- `(course_id, create_time)`

联合索引是实际项目里特别常用的一类索引，因为很多查询条件本来就是多个字段一起出现。

## 10. 最左前缀原则是什么

这是联合索引里的高频考点。

如果索引是 `(a, b, c)`，那通常更容易命中的情况是：

- `a`
- `a, b`
- `a, b, c`
- 以 `a` 开头的连续条件

而如果直接跳过最左边，比如只用：

- `b`
- `c`
- `b, c`

通常就不能很好利用这个联合索引。

可以先这样回答：

> 最左前缀原则指的是联合索引在匹配时，通常要从最左边的列开始连续使用，才能更好命中索引。

## 11. 哪些情况容易导致索引失效

这是面试里特别喜欢问的实战题。

常见情况：

- 在索引列上做函数运算。
- 在索引列上做计算或表达式操作。
- 隐式类型转换。
- `like` 以 `%` 开头。
- 联合索引没有遵守最左前缀。
- 过滤性太差，优化器判断全表扫描更划算。
- 使用 `or` 时部分条件无法有效走索引。

可以先记一句：

> 索引失效的本质，不是“语法错了”，而是数据库没法有效利用现有索引快速缩小扫描范围，或者优化器判断走索引反而不划算。

## 12. 什么情况下适合建索引

一般更适合建索引的字段：

- 经常出现在 `where` 条件中的字段。
- 经常用于排序、分组的字段。
- 区分度较高的字段。
- 经常作为联合查询条件一部分的字段。

典型例子：

- 用户 ID
- 订单号
- 课程 ID
- 创建时间
- 状态字段和业务主键组合字段

## 13. 什么情况下不适合建索引

这些情况一般要慎重：

- 表数据量本来就很小。
- 字段区分度太低。
- 更新特别频繁的字段。
- 索引建得过多，写入成本明显升高。

所以索引不是越多越好。

你要记住：

> 索引提升的是读性能，但会增加写成本和存储成本。

## 14. 面试里怎么回答“索引为什么不是越多越好”

可以直接说：

> 因为索引虽然能提升查询效率，但也会占用额外存储空间，而且插入、删除、更新数据时，索引也要同步维护。索引太多会导致写入变慢、维护成本上升，所以索引设计应该结合查询模式和业务场景，而不是盲目多建。

## 15. 当前阶段最值得先背的版本

> MySQL 索引本质上是帮助数据库快速定位数据的数据结构。InnoDB 常见索引底层是 B+ 树，因为它适合磁盘 IO、范围查询和有序遍历。主键索引通常是聚簇索引，二级索引叶子节点一般存主键值，所以通过二级索引查完整行数据时可能发生回表。如果查询字段都在索引里，就能形成覆盖索引。联合索引常见考点是最左前缀原则，而函数运算、隐式转换、前导模糊匹配等情况可能导致索引失效。
