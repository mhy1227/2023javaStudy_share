# MySQL 社区高频题回答版

## 1. 这份文档怎么用

这一份不是教材版答案，而是结合外部面经里最常出现的问题，整理成更适合一面、二面复述的版本。

## 2. 为什么 MySQL 索引底层常说 B+ 树

可以这样答：

> 因为数据库索引既要支持高效单点查询，也要支持范围查询和顺序遍历，B+ 树在磁盘 IO、范围扫描和有序性方面更适合数据库场景，所以 InnoDB 常见索引底层会说是 B+ 树。

## 3. 聚簇索引和二级索引有什么区别

可以这样答：

> InnoDB 里主键索引通常就是聚簇索引，叶子节点直接存整行数据。二级索引叶子节点一般存索引列值和主键值，所以通过二级索引查完整行数据时，通常还要再根据主键回到聚簇索引查一次，这就是回表。

## 4. 什么是覆盖索引

可以这样答：

> 覆盖索引就是查询需要的字段都能直接从索引里拿到，不需要回表。通常这样会更快，因为少了一次访问聚簇索引的过程。

## 5. 为什么有索引，SQL 还是慢

可以这样答：

> 因为走索引不等于一定快。索引可能区分度低，扫描范围仍然很大；也可能虽然走了索引，但回表很多；或者排序、分组没法复用索引，产生额外开销。所以最终还是要看整体执行成本，而不只是看有没有用索引。

## 6. 什么是事务，ACID 是什么

可以这样答：

> 事务就是把一组相关操作作为一个整体执行，要么全部成功，要么全部失败。ACID 分别是原子性、一致性、隔离性、持久性。原子性强调不做一半，一致性强调事务前后数据状态正确，隔离性强调并发事务尽量互不干扰，持久性强调提交后的结果要可靠保存。

## 7. MySQL 默认隔离级别是什么

可以这样答：

> MySQL InnoDB 默认隔离级别是可重复读，也就是 Repeatable Read。

## 8. MVCC 是什么

可以这样答：

> MVCC 是多版本并发控制，核心目的是让普通读尽量不阻塞写。它通过多版本机制，让事务去读取自己可见的那个版本，从而在保证一定隔离性的同时提升并发性能。

## 9. 锁和 MVCC 是什么关系

可以这样答：

> MVCC 和锁不是互相替代，而是配合关系。MVCC 更多用于普通一致性读，减少读写阻塞；锁更多用于当前读、更新、删除等需要强一致性的场景。

## 10. 行锁为什么和索引有关

可以这样答：

> InnoDB 的行锁通常是加在索引记录上的。如果 SQL 没有命中索引，就可能扫描和锁住更多记录，所以更新类 SQL 一般很看重索引是否命中。

## 11. 什么是间隙锁和临键锁

可以这样答：

> 记录锁锁的是具体索引记录，间隙锁锁的是索引记录之间的空隙，临键锁可以理解成记录锁加间隙锁。间隙锁和临键锁常用于可重复读级别下控制范围并发插入带来的问题。

## 12. redo log、undo log、binlog 分别是什么

可以这样答：

> redo log 属于 InnoDB，主要用于崩溃恢复；undo log 主要用于事务回滚，也为 MVCC 提供旧版本；binlog 属于 MySQL Server 层，主要用于记录变更、支持主从复制和归档。

## 13. 为什么既有 redo log，又有 binlog

可以这样答：

> 因为它们解决的问题不同。redo log 更偏事务持久性和崩溃恢复，binlog 更偏数据变更记录、归档和主从复制，所以不能简单互相替代。

## 14. 什么是两阶段提交

可以这样答：

> 两阶段提交的核心目的是保证 redo log 和 binlog 的一致性。因为 redo log 属于 InnoDB，binlog 属于 Server 层，不在同一层，所以事务提交时要通过两阶段提交保证两者状态对得上。

## 15. 一条 SQL 怎么执行

可以这样答：

> 一条 SQL 进入 MySQL 后，大致会经过连接器、解析器、优化器、执行器，然后由执行器调用存储引擎完成真正的数据访问。查询语句重点看索引和执行计划，更新语句还会涉及锁、undo log、redo log、binlog 和事务提交。

## 16. EXPLAIN 重点看什么

可以这样答：

> 我一般会重点看 `type`、`key`、`rows`、`Extra`。type 反映访问方式，key 看实际用了哪个索引，rows 看扫描行数大不大，Extra 看有没有临时表、额外排序、覆盖索引等信息。

## 17. 慢 SQL 怎么排查

可以这样答：

> 我一般会先通过慢 SQL 日志定位语句，再用 EXPLAIN 看执行计划，重点看索引命中、扫描行数、回表、临时表和排序情况。然后再判断问题是出在 SQL 写法、索引设计、扫描范围，还是需要更大的结构调整，比如缓存或数据拆分。

## 18. 当前阶段最值得先背的一版

> MySQL 面试最核心的就是索引、事务、MVCC、锁、日志、执行流程和慢 SQL 排查。索引本质是帮助数据库快速定位数据，InnoDB 常见索引底层是 B+ 树；事务强调 ACID，默认隔离级别是可重复读，MVCC 让普通读尽量不阻塞写；锁用于保证当前读和写操作正确性；redo log、undo log、binlog 分别解决崩溃恢复、回滚与 MVCC、复制与归档问题，两阶段提交保证 redo log 和 binlog 一致；而慢 SQL 排查则要结合 EXPLAIN、索引命中、扫描行数、回表、排序和临时表一起看。
