# MySQL 高频面试题回答版

## 1. 这份文档怎么用

这份文档不是标准教材答案，而是更适合一面、二面口语化复述的版本。

建议：

- 先自己答一遍。
- 再对照这里修正表达。
- 不要求逐字背，但要求主线顺。

## 2. 什么是索引

可以这样答：

> 索引本质上是帮助 MySQL 快速定位数据的数据结构。没有索引时很多查询只能全表扫描，有索引后可以缩小扫描范围，提高查询效率。InnoDB 里常说的索引底层是 B+ 树，因为它比较适合磁盘 IO、范围查询和有序遍历。

## 3. 聚簇索引和二级索引有什么区别

可以这样答：

> InnoDB 里主键索引通常就是聚簇索引，叶子节点里直接存整行数据。二级索引叶子节点里一般存的是索引列值和主键值，所以通过二级索引查完整行数据时，往往还要再根据主键去聚簇索引查一次，这就是回表。

## 4. 联合索引最左前缀原则是什么

可以这样答：

> 如果联合索引是 `(a,b,c)`，一般要从最左边开始连续使用，像 `a`、`a,b`、`a,b,c` 这类条件更容易命中索引；如果直接跳过最左边，比如只查 `b` 或 `c`，通常就不能很好利用这个联合索引。

## 5. 为什么有索引，SQL 还是会慢

可以这样答：

> 因为走索引不等于一定快。比如索引区分度太低、扫描行数仍然很多、回表次数太多、排序没法复用索引、范围查询过大，或者优化器判断全表扫描更划算，这些情况都可能导致 SQL 依然很慢。

## 6. 什么是事务

可以这样答：

> 事务就是把一组相关操作当成一个整体来执行，要么全部成功，要么全部失败，不会只执行一半。像转账这种场景，扣钱和加钱必须一起成功，否则数据就不一致，所以通常要放进事务里。

## 7. ACID 是什么

可以这样答：

> ACID 是事务的四大特性。原子性表示要么全做要么全不做，一致性表示事务前后数据都处于正确状态，隔离性表示并发事务之间尽量互不干扰，持久性表示事务一旦提交，结果应该能可靠保存下来。

## 8. MySQL 默认隔离级别是什么

可以这样答：

> MySQL InnoDB 默认隔离级别是可重复读，也就是 Repeatable Read。

## 9. MVCC 是什么

可以这样答：

> MVCC 是多版本并发控制，核心目标是尽量让普通读和写不互相阻塞。它通过多版本机制，让读操作去读自己当前事务可见的那个版本，而不是一上来就和写操作硬碰硬。它通常依赖隐藏字段、undo log、Read View 等机制。

## 10. 锁和 MVCC 是什么关系

可以这样答：

> MVCC 和锁不是互相替代，而是配合关系。MVCC 主要用于普通一致性读，减少读写阻塞；锁主要用于当前读、更新、删除这类需要强一致性的场景。所以普通 select 很多时候偏向 MVCC，而 `select ... for update`、update、delete 则更依赖锁。

## 11. 为什么数据库需要锁

可以这样答：

> 因为数据库里会有大量并发请求同时读写同一批数据，如果没有锁，就可能出现丢失更新、并发写乱序、库存超卖等问题。锁的本质是控制并发访问顺序，保证数据正确性。

## 12. 行锁和表锁有什么区别

可以这样答：

> 表锁锁的是整张表，粒度大，并发能力相对差；行锁锁的是具体记录，粒度小，并发能力更强。InnoDB 支持行级锁，所以更适合高并发业务场景。但行锁通常依赖索引实现，如果 SQL 没走索引，锁范围可能会扩大。

## 13. 间隙锁和临键锁是什么

可以这样答：

> 记录锁锁的是具体索引记录，间隙锁锁的是索引记录之间的空隙，临键锁可以理解成记录锁加间隙锁。间隙锁和临键锁常用于可重复读隔离级别下控制范围并发插入，从而降低幻读问题。

## 14. 什么是死锁，怎么避免

可以这样答：

> 死锁就是多个事务互相等待对方释放锁，最后谁也执行不下去。减少死锁一般从几个方向做：让事务按固定顺序访问资源、尽量缩短事务时间、让更新条件走索引避免锁范围扩大、避免在事务里做耗时操作，大批量更新尽量分批。

## 15. redo log、undo log、binlog 分别是什么

可以这样答：

> redo log 是 InnoDB 的重做日志，主要用于崩溃恢复；undo log 是回滚日志，主要用于事务回滚，也为 MVCC 提供旧版本；binlog 是 MySQL Server 层的二进制日志，主要用于记录变更、支持主从复制和归档。

## 16. 为什么既有 redo log，又有 binlog

可以这样答：

> 因为两者解决的问题不同。redo log 主要是为了崩溃恢复和事务持久性，binlog 主要是为了归档和主从复制。redo log 属于 InnoDB，binlog 属于 MySQL Server 层，所以不能简单互相替代。

## 17. 什么是两阶段提交

可以这样答：

> 两阶段提交的核心目的，是保证 redo log 和 binlog 的一致性。简单理解就是事务提交时，先让 redo log 进入 prepare 状态，再写 binlog，最后再把 redo log 标记为 commit。这样可以避免一份日志成功另一份失败，导致本地恢复结果和主从复制结果对不上。

## 18. 一条 SQL 怎么执行

可以这样答：

> 一条 SQL 进入 MySQL 后，大致会经过连接器、解析器、优化器、执行器，然后由执行器调用存储引擎完成真正的数据访问。连接器负责连接和权限，解析器负责词法语法分析，优化器负责选择执行计划，执行器负责调用存储引擎，存储引擎负责真正的索引访问、数据读写、锁和日志处理。

## 19. EXPLAIN 是干什么的

可以这样答：

> EXPLAIN 用来查看 SQL 的执行计划，也就是 MySQL 准备怎么执行这条 SQL。通常我会重点看访问类型 type、实际使用索引 key、预估扫描行数 rows，以及 Extra 里的额外信息，比如有没有临时表、额外排序、覆盖索引。

## 20. 慢 SQL 怎么排查

可以这样答：

> 我一般会先通过慢 SQL 日志定位具体是哪条语句慢，然后用 EXPLAIN 看执行计划，重点看有没有命中合适索引、扫描行数是不是过多、有没有回表、有没有 Using temporary 或 Using filesort。再结合 SQL 写法、联合索引设计、排序和分页方式分析原因。如果是索引用得不好，就调整 SQL 或补联合索引；如果问题不只是索引，还会考虑控制扫描范围、调整表结构，必要时再考虑缓存或数据拆分。
