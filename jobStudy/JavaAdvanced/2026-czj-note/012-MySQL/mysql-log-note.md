# MySQL 日志笔记

## 1. 这份文档是干什么的

日志是 MySQL 面试里很容易从事务继续追问下去的一条主线。

面试官常见追问路径：

1. MySQL 里有哪些重要日志。
2. redo log、undo log、binlog 分别是干什么的。
3. 为什么既有 redo log，又有 binlog。
4. 为什么 undo log 和 MVCC 有关系。
5. 两阶段提交是什么。
6. 为什么要保证 redo log 和 binlog 一致。

所以这一篇的目标就是把 MySQL 日志体系的主线先讲顺。

## 2. 先记最核心的结论

先记住这几句：

1. **redo log 主要解决已提交事务的崩溃恢复问题。**
2. **undo log 主要解决事务回滚和多版本读取问题。**
3. **binlog 主要用于 MySQL Server 层的归档和主从复制。**
4. **redo log 属于 InnoDB，binlog 属于 MySQL Server，它们不是同一层的日志。**
5. **两阶段提交的核心目的是保证 redo log 和 binlog 的一致性。**

## 3. 为什么 MySQL 需要日志

数据库不是把数据写进磁盘就结束了。

真实场景里会遇到这些问题：

- 事务执行到一半数据库宕机怎么办。
- 数据改了以后想回滚怎么办。
- 主库的数据怎么同步给从库。
- 已提交的事务怎么保证重启后不丢。

日志就是为了解决这些问题而存在的。

可以先这样理解：

> MySQL 的日志不是“为了记流水”这么简单，而是为了支持恢复、回滚、复制、一致性控制等核心能力。

## 4. MySQL 里最常被问的三种日志

初学阶段先重点抓这三个：

- redo log
- undo log
- binlog

这三个几乎是 MySQL 日志面试的核心。

## 5. redo log 是什么

可以先这样回答：

> redo log 是 InnoDB 的重做日志，核心作用是保证已提交事务在数据库异常宕机后也能恢复出来。

它解决的问题是：

> 数据已经提交了，但页面还没来得及真正刷到磁盘，如果这时宕机了怎么办？

redo log 的思路是：

- 先把这次修改对应的日志记下来。
- 就算数据页还没真正落盘，崩溃恢复时也可以根据 redo log 把改动重新做出来。

一句话记忆：

> **redo log 负责把“已经提交但还没完全写盘”的数据补回来。**

## 6. 为什么 redo log 很重要

因为磁盘随机写很慢，而数据库又不能每改一条数据就立刻把整页数据同步刷盘。

所以 InnoDB 通常会：

- 先修改内存中的页。
- 先把 redo log 记下来。
- 再选择合适时机把数据页刷盘。

这样能平衡性能和可靠性。

所以 redo log 常和这几个词一起出现：

- WAL（Write-Ahead Logging，预写式日志）
- 崩溃恢复
- 持久性

## 7. undo log 是什么

可以先这样回答：

> undo log 是回滚日志，主要作用是支持事务回滚，并为 MVCC 提供旧版本数据。

它主要解决两个问题：

### 7.1 事务回滚

如果事务执行失败或者主动回滚，就需要把之前做过的修改撤销掉。

undo log 就记录了“如何撤销这次修改”。

### 7.2 MVCC 读旧版本

前面事务/MVCC 那篇讲过，多版本并发控制依赖旧版本链。

undo log 就是旧版本的重要来源之一。

所以一句话记忆：

> **undo log 既服务于回滚，也服务于 MVCC。**

## 8. binlog 是什么

可以先这样回答：

> binlog 是 MySQL Server 层的二进制日志，主要用于记录数据库变更，常见用途是主从复制和数据归档。

和 redo log 不同，它不是 InnoDB 自己的日志，而是 MySQL Server 层统一记录的。

binlog 更常和这些关键词绑定：

- 主从复制
- 数据同步
- 归档
- 恢复到某个时间点

一句话记忆：

> **binlog 更偏“把数据库改动记下来，供复制和审计使用”。**

## 9. redo log、undo log、binlog 分别解决什么问题

这个问题很高频，最好能直接回答。

可以这样说：

> redo log 主要负责崩溃恢复，undo log 主要负责事务回滚和 MVCC 旧版本读取，binlog 主要负责归档和主从复制。

如果要再补一句：

> redo log 和 undo log 更偏 InnoDB 内部事务能力，binlog 更偏 MySQL Server 层的数据变更记录和外部同步能力。

## 10. redo log 和 binlog 有什么区别

这是特别常问的一道题。

可以从这几个角度记：

### 10.1 所属层次不同

- redo log：InnoDB 层
- binlog：MySQL Server 层

### 10.2 作用不同

- redo log：崩溃恢复
- binlog：归档、复制

### 10.3 记录内容不同

- redo log 更偏物理层修改信息。
- binlog 更偏逻辑层变更记录。

初学阶段不必死扣“物理日志”和“逻辑日志”的实现细节，但要知道它们侧重点不同。

## 11. 为什么既有 redo log，又有 binlog

可以先这样回答：

> 因为两者解决的问题不同，redo log 主要为了崩溃恢复和事务持久性，binlog 主要为了归档和主从复制，所以不能简单互相替代。

这道题回答时别说成“只是历史包袱”。

更准确的理解是：

- redo log 关注存储引擎层的数据可靠落地。
- binlog 关注 Server 层的数据变更记录和外部同步。

## 12. 什么是两阶段提交

两阶段提交是日志体系里最经典的追问。

可以先记这一句：

> 两阶段提交的核心目的是保证 redo log 和 binlog 的一致性。

为什么会有这个问题？

因为：

- redo log 在 InnoDB 层
- binlog 在 MySQL Server 层

也就是说，一个事务提交时，实际上涉及两套不同层次的日志系统。

如果没有协调机制，就可能出现：

- redo log 写成功了，binlog 没写成功
- binlog 写成功了，redo log 没写成功

这样一来，数据库恢复和主从复制看到的状态就可能不一致。

## 13. 两阶段提交大概怎么理解

初学阶段先记流程，不用死抠源码：

1. 先写 redo log，进入 prepare 状态。
2. 再写 binlog。
3. 最后把 redo log 标记为 commit。

你可以把它理解成：

> 先让 InnoDB 说“我准备好了”，再让 Server 层把 binlog 记下来，最后再真正确认事务完成。

## 14. 为什么两阶段提交重要

因为如果 redo log 和 binlog 不一致，会出这类问题：

- 主库本地恢复时觉得事务成功了。
- 但从库复制时没拿到对应的 binlog。
- 结果主从数据不一致。

或者反过来：

- binlog 里记录了这次改动。
- 但 redo log 没保证本地真正可恢复。
- 数据状态也会对不上。

所以两阶段提交的本质是：

> **保证本地恢复能力和外部复制能力看到的是同一个事务结果。**

## 15. 面试里怎么回答“redo log、undo log、binlog”

可以直接这样说：

> redo log 是 InnoDB 的重做日志，主要用于崩溃恢复；undo log 是回滚日志，主要用于事务回滚和 MVCC；binlog 是 MySQL Server 层的二进制日志，主要用于归档和主从复制。redo log 和 binlog 分属不同层次，所以事务提交时需要通过两阶段提交来保证两者一致性。

## 16. 当前阶段最值得先背的版本

> MySQL 常见高频日志主要有 redo log、undo log 和 binlog。redo log 属于 InnoDB，核心作用是保证已提交事务在宕机后可恢复；undo log 主要用于事务回滚，也为 MVCC 提供旧版本；binlog 属于 MySQL Server 层，主要用于记录变更、支持主从复制和归档。因为 redo log 和 binlog 不属于同一层，所以事务提交时需要通过两阶段提交保证它们的一致性，避免主库恢复结果和主从复制结果对不上。
