# MySQL 考前复盘 Checklist

## 1. 这份文档怎么用

这一份不是详细笔记，而是考前快速扫一遍的版本。

建议用法：

- 面试前 30 分钟：按模块快速过一遍。
- 面试前 10 分钟：只看“最值得先背”的部分。
- 如果某一块卡住，就回跳到对应主线文档补一遍。

## 2. 基础认知 Checklist

- 我能说清 MySQL 是什么，以及它在 Java 后端项目里的角色。
- 我能说清 MySQL 和 Redis 的分工。
- 我知道 MySQL 大体分为 Server 层和存储引擎层。
- 我知道 InnoDB 为什么更常用。

最值得先背：

> MySQL 是 Java 后端项目里最常见的关系型数据库，负责核心业务数据持久化；Redis 更多负责缓存和高并发场景加速。MySQL 大体分为 Server 层和存储引擎层，InnoDB 是最常用的存储引擎，因为它支持事务、行锁、MVCC 和崩溃恢复。

## 3. 索引 Checklist

- 我能解释什么是索引。
- 我能解释为什么 MySQL 常说 B+ 树。
- 我能区分聚簇索引和二级索引。
- 我知道什么是回表、覆盖索引。
- 我知道联合索引和最左前缀原则。
- 我知道常见索引失效场景。
- 我知道为什么有索引 SQL 还是会慢。
- 我知道为什么索引不是越多越好。

最值得先背：

> 索引本质上是帮助 MySQL 快速定位数据的数据结构。InnoDB 常说的索引底层是 B+ 树，因为适合磁盘 IO、范围查询和有序遍历。主键索引通常是聚簇索引，二级索引叶子节点一般存主键值，所以会有回表。联合索引常见考点是最左前缀原则，而函数运算、隐式转换、前导模糊匹配等都可能导致索引失效。

## 4. 事务与 MVCC Checklist

- 我能解释什么是事务。
- 我能解释 ACID。
- 我知道四种隔离级别。
- 我知道 MySQL 默认隔离级别是可重复读。
- 我能区分脏读、不可重复读、幻读。
- 我能解释 MVCC 想解决什么问题。
- 我知道 MVCC 和 undo log、Read View 的关系。
- 我能区分快照读和当前读。
- 我知道 MVCC 和锁不是替代关系，而是配合关系。

最值得先背：

> 事务就是一组操作要么都成功要么都失败，ACID 分别对应原子性、一致性、隔离性、持久性。MySQL InnoDB 默认隔离级别是可重复读。MVCC 是多版本并发控制，核心目的是让普通读尽量不阻塞写，它通常依赖隐藏字段、undo log、Read View 等机制。快照读主要依赖 MVCC，当前读主要依赖锁。

## 5. 锁 Checklist

- 我能解释为什么数据库需要锁。
- 我能区分表锁和行锁。
- 我知道 InnoDB 行锁为什么和索引有关。
- 我能解释共享锁和排他锁。
- 我知道意向锁的大概作用。
- 我能解释记录锁、间隙锁、临键锁。
- 我知道死锁是什么，以及大概怎么避免。

最值得先背：

> 锁的核心作用是控制并发读写，保证数据正确性。InnoDB 支持行锁，但行锁通常依赖索引实现，如果 SQL 没走索引，锁范围可能扩大。常见锁包括共享锁、排他锁，以及记录锁、间隙锁、临键锁。间隙锁和临键锁常用于控制范围查询下的并发插入，死锁本质上是多个事务互相等待对方释放锁。

## 6. 日志 Checklist

- 我能说清 redo log、undo log、binlog 分别干什么。
- 我知道 redo log 为什么和崩溃恢复有关。
- 我知道 undo log 为什么和回滚、MVCC 有关。
- 我知道 binlog 为什么和主从复制有关。
- 我能解释 redo log 和 binlog 的区别。
- 我能解释为什么要两阶段提交。

最值得先背：

> redo log 属于 InnoDB，主要用于崩溃恢复；undo log 主要用于事务回滚，也为 MVCC 提供旧版本；binlog 属于 MySQL Server 层，主要用于归档和主从复制。因为 redo log 和 binlog 分属不同层，所以事务提交时需要通过两阶段提交保证它们一致。

## 7. 执行流程 Checklist

- 我能说清一条 SQL 进入 MySQL 后的大致执行流程。
- 我能解释连接器、解析器、优化器、执行器分别做什么。
- 我能区分查询 SQL 和更新 SQL 的执行重点。
- 我知道执行流程为什么会和索引、锁、日志串起来。

最值得先背：

> 一条 SQL 进入 MySQL 后，大致会经过连接器、解析器、优化器、执行器，然后由执行器调用存储引擎完成真正的数据访问。查询语句重点关注索引和执行计划，更新语句还会涉及锁、undo log、redo log、binlog 和事务提交。

## 8. EXPLAIN 与慢 SQL Checklist

- 我能解释 EXPLAIN 是干什么的。
- 我知道重点看 `type`、`key`、`rows`、`Extra`。
- 我知道 `possible_keys` 和 `key` 的区别。
- 我知道 `Using index`、`Using temporary`、`Using filesort` 大概意味着什么。
- 我能说出慢 SQL 的排查顺序。
- 我知道为什么慢 SQL 排查不能只回答“加索引”。

最值得先背：

> 慢 SQL 排查一般会先通过慢日志定位 SQL，再用 EXPLAIN 看执行计划，重点看访问类型、实际使用索引、扫描行数和额外开销，比如临时表、文件排序、回表等。然后再判断是改 SQL、补索引、缩小扫描范围，还是进一步调整表结构、加缓存、做数据拆分。

## 11. 如果时间只剩 3 分钟

只背这段：

> MySQL 面试最核心的主线就是索引、事务、MVCC、锁、日志、执行流程和慢 SQL 排查。索引本质是帮助数据库快速定位数据，InnoDB 常见索引底层是 B+ 树；事务强调 ACID，默认隔离级别是可重复读，MVCC 让普通读尽量不阻塞写；锁用于保证当前读和写操作的正确性；redo log、undo log、binlog 分别对应崩溃恢复、回滚与 MVCC、复制与归档，两阶段提交保证 redo log 和 binlog 一致；一条 SQL 大致经过连接器、解析器、优化器、执行器和存储引擎；慢 SQL 排查则要结合慢日志、EXPLAIN、索引命中、扫描行数、回表、临时表和排序一起看。
