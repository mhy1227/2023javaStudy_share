# MySQL EXPLAIN 与慢 SQL 排查笔记

## 1. 这份文档是干什么的

前面已经补了 SQL 执行流程，这一篇继续补面试里非常常见的落地问题：

> 线上 SQL 慢了，你怎么排查？

这类问题很容易从索引、执行流程继续追问过来。

常见问法：

1. `EXPLAIN` 是干什么的。
2. `EXPLAIN` 里重点看哪些字段。
3. 怎么判断索引用得好不好。
4. 慢 SQL 一般怎么排查。
5. 为什么有索引 SQL 还是慢。
6. 什么情况下需要改 SQL、补索引、调表结构。

## 2. 先记最核心的结论

先记住这几句：

1. **EXPLAIN 用来查看 SQL 的执行计划，也就是 MySQL 准备怎么执行这条 SQL。**
2. **看执行计划不能只看有没有索引，还要看访问类型、实际使用的索引、扫描行数和额外开销。**
3. **慢 SQL 排查要结合慢日志、执行计划、索引、扫描行数、回表、排序、临时表一起看。**
4. **有索引不代表一定快，关键要看整体执行成本是否真的下降。**

## 3. 什么是慢 SQL

慢 SQL 不只是“执行超过某个固定秒数”的 SQL。

更适合面试的说法是：

> 慢 SQL 指的是执行时间明显过长，影响接口响应、用户体验或数据库负载的 SQL。

它可能来自很多原因：

- 没有合适索引。
- 索引用得不好。
- 扫描行数太多。
- 回表太多。
- 排序或分组开销大。
- 临时表太多。
- 深分页。
- SQL 写法不合理。
- 表数据量过大。

## 4. 慢 SQL 一般怎么排查

面试里可以按这个顺序说：

1. 先确认是不是慢 SQL。
2. 看慢 SQL 日志，定位具体 SQL。
3. 用 `EXPLAIN` 看执行计划。
4. 看有没有命中合适索引。
5. 看扫描行数是否过多。
6. 看是否存在回表、临时表、文件排序。
7. 分析 SQL 写法、索引设计和数据分布。
8. 决定是改 SQL、补索引、调整表结构，还是做缓存/分库分表等更大改造。

注意：

> 不要一上来就说“加索引”，这会显得排查思路太单薄。

## 5. EXPLAIN 是什么

可以先这样回答：

> EXPLAIN 用来查看 SQL 的执行计划，可以帮助我们判断 MySQL 准备怎么执行这条 SQL，比如是否走索引、走哪个索引、扫描多少行、有没有额外排序或临时表。

更口语一点：

> EXPLAIN 相当于提前告诉你，这条 SQL 打算怎么跑。

## 6. EXPLAIN 重点看哪些字段

初学和面试阶段，优先记这几个：

- `type`
- `possible_keys`
- `key`
- `rows`
- `Extra`

### 6.1 type

`type` 表示访问类型，大致能反映 SQL 查找数据的方式。

常见从好到差可以粗略记：

- `system`
- `const`
- `eq_ref`
- `ref`
- `range`
- `index`
- `ALL`

面试里重点记：

> 如果出现 `ALL`，通常意味着全表扫描，需要重点关注。

但也不要绝对化，小表全表扫描不一定有问题。

### 6.2 possible_keys

`possible_keys` 表示 MySQL 认为这条 SQL 可能用到哪些索引。

它只是“可能用”，不是最终一定用。

### 6.3 key

`key` 表示 MySQL 最终实际选择使用的索引。

所以：

- `possible_keys`：候选索引
- `key`：实际使用索引

如果 `key` 是 `null`，就说明没有使用索引。

### 6.4 rows

`rows` 是 MySQL 估算需要扫描的行数。

一般来说：

- 越小越好。
- 如果 rows 很大，说明扫描范围可能太大。
- 但它是估算值，不是绝对真实值。

### 6.5 Extra

`Extra` 里会出现一些额外信息。

面试里常关注：

- `Using index`
- `Using where`
- `Using temporary`
- `Using filesort`

## 7. Extra 里的常见信息怎么理解

### 7.1 Using index

通常表示使用了覆盖索引。

也就是说，查询字段可以直接从索引里拿到，不需要回表。

### 7.2 Using where

表示 MySQL 还需要根据 `where` 条件进一步过滤。

它本身不一定是坏事，要结合其他字段看。

### 7.3 Using temporary

表示可能使用了临时表。

常见于某些 `group by`、`order by`、复杂查询。

如果数据量大，可能会比较耗性能。

### 7.4 Using filesort

表示 MySQL 需要额外排序。

注意：

> `filesort` 不一定真的写文件，但通常说明排序没能很好利用索引。

## 8. 为什么不能只看有没有走索引

因为走索引不等于一定快。

例如：

- 索引区分度很低。
- 范围扫描太大。
- 回表次数太多。
- `order by` 没法利用索引。
- 优化器认为全表扫描更划算。

所以判断 SQL 好不好，不是只看 `key` 有没有值。

更合理的是看：

- `type` 是否合理。
- `key` 是否是预期索引。
- `rows` 是否过大。
- `Extra` 是否有临时表、文件排序。
- 是否存在大量回表。

## 9. 为什么有索引，SQL 还是会慢

可以这样回答：

> 有索引不代表 SQL 一定快。索引可能区分度太低，扫描行数仍然很多；也可能虽然走了索引，但需要大量回表；还可能排序、分组没有用上索引，产生临时表或额外排序；另外如果查询范围太大，优化器也可能认为全表扫描更划算。

一句话版本：

> 索引只是降低成本的手段，关键还是看整体执行成本。

## 10. 常见优化方向

慢 SQL 优化通常从这些角度看：

### 10.1 改 SQL 写法

例如：

- 避免在索引列上做函数运算。
- 避免隐式类型转换。
- 避免前导 `%` 模糊匹配。
- 避免 `select *` 导致不必要回表。

### 10.2 调整索引

例如：

- 给高频查询条件加索引。
- 使用联合索引匹配查询条件。
- 让排序字段尽量复用索引。
- 尽量形成覆盖索引。

### 10.3 控制扫描范围

例如：

- 减少一次查询返回的数据量。
- 避免深分页。
- 增加更精确的过滤条件。

### 10.4 调整表结构或架构

如果单纯 SQL 和索引已经解决不了，才考虑更大的动作：

- 冷热数据拆分。
- 分库分表。
- 引入缓存。
- 异步化或预计算。

## 11. 面试里怎么回答“慢 SQL 怎么排查”

可以直接背这个版本：

> 我一般会先通过慢 SQL 日志定位具体是哪条 SQL 慢，然后用 EXPLAIN 看执行计划，重点看 type、key、rows、Extra 这些字段。先判断有没有命中合适索引、扫描行数是不是过多、有没有大量回表、有没有 Using temporary 或 Using filesort。如果是索引没用好，就看是否存在函数操作、隐式转换、联合索引不满足最左前缀等问题；如果是扫描范围过大，就考虑补充过滤条件、调整联合索引或优化分页；如果 SQL 本身和索引都优化后仍然不够，再考虑缓存、表结构调整或数据拆分。

## 12. 面试里怎么回答“EXPLAIN 你重点看什么”

可以这样答：

> 我会重点看访问类型 type、实际使用索引 key、预估扫描行数 rows 和 Extra 里的额外信息。type 可以判断访问方式是否合理，key 可以确认是否命中预期索引，rows 可以判断扫描范围是否过大，Extra 里如果出现 Using temporary 或 Using filesort，就要进一步关注分组排序是否产生了额外开销。

## 13. 当前阶段最值得先背的版本

> 慢 SQL 排查不能只说加索引。更完整的思路是先通过慢 SQL 日志定位 SQL，再用 EXPLAIN 看执行计划，重点关注 type、key、rows、Extra。看是否命中合适索引、扫描行数是否过多、是否有大量回表、临时表和额外排序。然后再根据原因决定是改 SQL、补联合索引、减少扫描范围，还是进一步做缓存、表结构调整或数据拆分。
