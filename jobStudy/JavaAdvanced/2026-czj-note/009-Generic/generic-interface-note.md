# 泛型接口基础笔记

## 1. 为什么还要单独讲泛型接口

前面已经补了：

- 泛型类
- 泛型方法

但泛型其实还可以出现在：

- 接口

而且这个点在集合、比较器、函数式接口等场景里都很常见。

所以泛型接口也需要单独立住。

## 2. 什么是泛型接口

泛型接口可以先理解成：

- 接口在定义时，把类型参数先留出来

也就是说：

- 接口本身不把类型写死
- 让实现类或使用方后续再具体指定

## 3. 一个最基础的例子

```java
interface Generator<T> {
    T next();
}
```

这里可以先这样理解：

- `Generator` 是一个泛型接口
- `T` 表示它将来要生成的某种类型

## 4. 实现类怎么实现泛型接口

最常见有两种方式。

### 4.1 实现类直接指定具体类型

```java
class StringGenerator implements Generator<String> {
    @Override
    public String next() {
        return "hello";
    }
}
```

这里的感觉是：

- 接口上原来的 `T`
- 到实现类这里被具体指定成了 `String`

### 4.2 实现类自己也保留泛型

```java
class CommonGenerator<T> implements Generator<T> {
    @Override
    public T next() {
        return null;
    }
}
```

这里的感觉是：

- 接口没具体化
- 实现类自己也继续保留泛型能力

## 5. 泛型接口和泛型类的关系怎么理解

这个点容易混。

可以先这样记：

### 泛型类

- 类自己带泛型参数

### 泛型接口

- 接口自己带泛型参数

它们本质上的共同点是：

- 都把类型参数先留出来

区别只是：

- 一个是类
- 一个是接口

## 6. 为什么泛型接口很常见

因为接口本来就强调：

- 规范
- 抽象

如果接口再加上泛型，就能更自然地表达：

- 针对不同类型的数据，使用统一的一套规范

所以后面你看到：

- `Comparable<T>`
- `Comparator<T>`

就不会觉得陌生。

## 7. 一个更贴近实际的例子

例如：

```java
interface Converter<S, T> {
    T convert(S source);
}
```

这里可以先这样理解：

- 输入是一种类型 `S`
- 输出是另一种类型 `T`

这类设计在实际开发里很常见。

## 8. 泛型接口这块面试里容易问什么

常见问题有：

1. 什么是泛型接口
2. 泛型接口和泛型类有什么区别
3. 实现类怎么实现泛型接口
4. 实现类能不能继续保留泛型
5. 为什么很多集合相关接口会和泛型绑在一起

## 9. 当前阶段最容易混的点

1. 泛型接口不是泛型类
2. 实现类不一定非要把类型立刻写死
3. 实现类也可以继续保留泛型参数
4. 接口上的类型参数和实现类上的类型参数要对应着看

## 10. 当前阶段最该记住的结论

1. 泛型接口是在接口定义时把类型参数留出来
2. 实现类既可以指定具体类型，也可以继续保留泛型
3. 泛型接口很适合描述“针对不同类型数据的一套统一规范”

## 11. 一句话总结

泛型接口最核心的意义，就是让“接口规范”不把类型写死，而是把类型参数留给实现类或使用方后续决定。
