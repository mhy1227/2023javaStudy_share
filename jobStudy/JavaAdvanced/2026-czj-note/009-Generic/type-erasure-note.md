# 类型擦除基础笔记

## 1. 为什么类型擦除特别重要

前面的泛型类、泛型方法、泛型接口、通配符，更多还是在说：

- 泛型怎么写

但面试里真正很爱追问的是：

- 泛型到底是编译期的，还是运行期的
- 为什么很多地方会有限制

而这些问题往后大多都会落到一个关键词上：

- 类型擦除

所以这一块必须单独讲。

## 2. 什么是类型擦除

可以先粗理解成：

- Java 泛型的类型信息，主要在编译期起作用

编译完成后，很多具体泛型类型信息不会完整按你写代码时的样子一直保留下去。

所以可以先记：

- 泛型很强地服务于编译期类型检查

## 3. 为什么会有这种感觉

前面你会写：

```java
List<String> list = new ArrayList<>();
```

看起来很像：

- 运行时这个对象永远明确知道自己就是 `List<String>`

但 Java 泛型并不是按这个直觉一直工作到底层的。

所以类型擦除的核心感觉可以先记成：

- 你写代码时看到的泛型信息，很多是给编译器看的

## 4. 类型擦除带来的一个直观影响

最常见的追问就是：

- 为什么不能直接 `new T()`

例如：

```java
class Box<T> {
    public T create() {
        return new T();
    }
}
```

这类写法通常不行。

可以先这样理解：

- 因为运行时并没有按你想象的方式完整保留“这个 T 到底是什么具体类型”

所以编译器不会让你直接这样做。

## 5. 为什么不能直接创建泛型数组

这个点也很常见。

例如：

```java
T[] arr = new T[10];
```

这类写法通常也不行。

当前阶段先有一个印象就够了：

- 泛型和数组在 Java 里不是天然很顺滑的一组组合
- 这里背后也和类型擦除强相关

## 6. 为什么运行时常看到的还是原始类型感觉

比如你写：

```java
List<String> list1 = new ArrayList<>();
List<Integer> list2 = new ArrayList<>();
```

很多时候面试官会继续追问：

- 运行时它们还是不是完全不同的两类对象

这个问题背后的主线其实也是：

- 泛型类型参数更多是在编译期参与检查

所以这里先建立一个感觉就够了：

- 泛型不是“把每个具体类型都重新生成一套完全独立的类”

## 7. 类型擦除为什么会影响方法重载理解

这个点第一轮不一定死磕，但要有印象。

因为如果泛型类型参数在底层不能按源码里的样子一直保留，那么有些你以为“签名不同”的方法，最后可能并不像你想的那样真正不同。

所以面试里才会进一步问：

- 为什么有些泛型重载写法不合法

## 8. 一个更稳的面试答法

可以这样答：

Java 泛型的核心价值主要体现在编译期类型检查上，而不是运行时一直完整保留所有泛型类型参数信息。很多泛型限制，比如不能直接 `new T()`、不能直接创建泛型数组、某些重载不成立，背后都和类型擦除有关。

## 9. 类型擦除这块最容易混的点

1. 泛型不是“运行时永远完整保留”的强类型模板机制
2. 泛型的核心价值更多在编译期
3. `new T()` 这类限制和类型擦除强相关
4. 泛型数组问题也和类型擦除强相关

## 10. 当前阶段最该记住的结论

1. Java 泛型很强地服务于编译期类型检查
2. 类型擦除是泛型很多限制的底层原因
3. 为什么不能 `new T()`、不能直接创建泛型数组，这些都和类型擦除有关

## 11. 一句话总结

类型擦除最核心的意义，就是告诉你：Java 泛型的重点主要在编译期，而不是运行时无限保留所有具体类型信息。
