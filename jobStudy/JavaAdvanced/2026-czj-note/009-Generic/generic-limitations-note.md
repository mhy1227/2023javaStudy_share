# 泛型常见限制

## 1. 为什么泛型还要单独讲限制

因为泛型这一章最容易让人产生一种错觉：

- 好像什么地方都能随便加尖括号

但实际并不是这样。

泛型很好用，可它也有边界。

而这些边界正是面试里很爱问的地方。

## 2. 最常见的限制先给一个总印象

当前阶段最值得先记的限制有这些：

1. 不能直接 `new T()`
2. 不能直接创建泛型数组
3. 静态上下文不能直接使用类上的泛型参数
4. 通配符场景下不能随便 `add`
5. 有些看起来不同的泛型重载实际上不成立

## 3. 为什么不能直接 `new T()`

例如：

```java
class Box<T> {
    public T create() {
        return new T();
    }
}
```

这类写法通常不行。

可以先这样理解：

- 因为编译器没法让你在这里直接按一个不确定的类型去创建对象

背后主线和：

- 类型擦除

强相关。

## 4. 为什么不能直接创建泛型数组

例如：

```java
T[] arr = new T[10];
```

这类写法通常也不行。

当前阶段先不用死抠特别底层，只要先记住：

- 泛型数组是高频限制点

## 5. 为什么静态上下文不能直接使用类上的泛型参数

例如：

```java
class Box<T> {
    private static T value;
}
```

这种写法通常不行。

可以先这样理解：

- 类上的 `T` 是跟具体对象实例使用时的类型相关
- 静态成员属于类本身

所以这两个层级放在一起就会有问题。

## 6. 为什么 `List<?>` 不能随便 add

例如：

```java
List<?> list = new ArrayList<String>();
list.add("hello");
```

这类写法不行。

原因可以先粗记成：

- 你只知道它是某种列表
- 但不知道它具体允许放什么类型

所以这里最核心的是：

- 通配符带来了“读写边界”

## 7. 为什么泛型重载有时会出问题

例如很多人会直觉觉得：

- `List<String>`
- `List<Integer>`

应该能构成完全不同的方法签名

但事情没有这么简单。

当前阶段先有一个印象：

- 泛型方法签名和你源码里看到的直觉不完全一致
- 这背后通常又会回到类型擦除

## 8. 泛型限制这块面试里容易问什么

常见问题有：

1. 为什么不能 `new T()`
2. 为什么不能直接 `new T[]`
3. 为什么静态变量不能直接用类上的泛型参数
4. 为什么 `List<?>` 不能随便添加元素
5. 泛型和重载为什么会有冲突

## 9. 一个更稳的面试答法

可以这样答：

泛型虽然很好用，但它主要服务于编译期类型检查，所以也有一些常见限制。比如不能直接 `new T()`、不能直接创建泛型数组、静态上下文不能直接使用类上的泛型参数、通配符场景下不能随便写入元素，这些限制很多都和类型擦除以及泛型的类型边界有关。

## 10. 当前阶段最容易混的点

1. 泛型限制不是零散规则，很多都和类型擦除相关
2. `new T()` 和 `new T[]` 是两类高频限制
3. 类上的泛型参数和静态成员不是一个层级
4. 通配符限制 add 操作不是语法刁难，而是类型安全要求

## 11. 当前阶段最该记住的结论

1. 泛型有明确边界，不是哪里都能随便用
2. `new T()`、泛型数组、静态泛型、通配符写入，这几类最常考
3. 很多限制背后的主线都和类型擦除、类型安全相关

## 12. 一句话总结

泛型常见限制这块最核心的意义，就是提醒你：泛型很强，但它不是没有边界，很多地方都必须服从类型安全和类型擦除的规则。
