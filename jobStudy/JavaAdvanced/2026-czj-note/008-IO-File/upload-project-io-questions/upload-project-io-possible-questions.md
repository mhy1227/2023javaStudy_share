# 文件上传项目里的 IO 可能追问

## 1. 这一份怎么用

这一份只整理“可能会被问到的问题”，先不写答案。

使用方式很简单：

1. 先看自己项目里到底做了哪些上传相关能力
2. 再对照这里的问题，筛出最可能被追问的点
3. 后续再给这些问题补正式回答

## 2. 第一组：最容易先被问到的基础问题

1. 文件上传这个场景，为什么通常会想到字节流而不是字符流？
2. 图片、视频、压缩包这类文件，为什么不能按字符流处理？
3. `InputStream` 和 `OutputStream` 在上传场景里分别处在什么位置？
4. 上传文件落盘时，最基础的 IO 处理流程是什么？
5. 为什么文件上传不能简单理解成“接收一个字符串然后存起来”？
6. 文件上传时，为什么经常会强调二进制数据而不是文本数据？
7. 如果上传的是一个文本文件，为什么后端保存时很多时候仍然按字节处理？

## 3. 第二组：围绕大文件上传的高频追问

1. 为什么大文件上传不建议一次性读进内存？
2. 如果把一个 1GB 或 10GB 的文件一次性读进内存，可能会有什么问题？
3. 大文件上传为什么更适合分块处理？
4. 分块上传在 IO 视角下到底解决了什么问题？
5. 为什么分块上传比直接整文件上传更适合网络不稳定场景？
6. 文件上传过程中，为什么常会提“边读边写”？
7. 上传大文件时，服务端为什么要尽量避免大对象常驻内存？

## 4. 第三组：围绕分块落盘和文件合并的追问

1. 前端上传的每个分块，后端通常怎么保存？
2. 为什么很多实现会先把分块写到临时目录，而不是直接写成最终文件？
3. 分块文件合并时，后端一般采用什么样的 IO 流程？
4. 合并分块时，为什么要强调“按顺序合并”？
5. 如果分块乱序上传，后端如何保证最终文件内容正确？
6. 为什么合并文件时更适合用字节流而不是字符流？
7. 分块合并完成后，临时分块一般怎么处理？
8. 如果文件合并过程中失败，为什么不能直接把目标文件当成成功文件使用？
9. 合并时如果只合并了一半就中断，应该怎么避免产生脏文件？
10. 为什么很多实现会先生成临时目标文件，合并成功后再改名？

## 5. 第四组：围绕断点续传与幂等性的追问

1. 断点续传和普通重新上传，IO 处理上的核心区别是什么？
2. 为什么断点续传场景里要记录已经上传了哪些分块？
3. 如果某个分块重复上传，后端应该怎么处理才更稳妥？
4. 为什么分块上传场景要特别关注幂等性？
5. 断点续传里，后端为什么不能只靠文件名判断上传进度？
6. 如果上传过程中服务重启，如何避免分块状态和磁盘文件状态不一致？
7. 为什么断点续传除了 Redis 状态，还要关注磁盘上的真实分块文件？

## 6. 第五组：围绕秒传和 MD5 校验的追问

1. 秒传为什么通常会和文件 MD5 绑定在一起？
2. 为什么前端算了 MD5，后端还可能需要再做校验？
3. 只靠前端传来的 MD5 做秒传判断，有什么风险？
4. 为什么秒传本质上不是“真的不用处理”，而是复用已有文件？
5. 秒传成功以后，为什么还需要保存业务记录，而不是直接结束？
6. 如果两个不同用户上传同一个文件，存储层和业务层应该怎么区分？
7. 秒传场景里，文件物理存储和文件业务记录为什么通常要拆开理解？

## 7. 第六组：围绕缓冲流和性能的追问

1. 文件上传或文件合并时，为什么经常会提到缓冲流？
2. `BufferedInputStream` 和 `BufferedOutputStream` 在上传链路里可能用在哪里？
3. 为什么加缓冲通常会比直接逐字节读写更高效？
4. 如果每次只读一个字节、写一个字节，会有什么性能问题？
5. 大文件合并时，读写缓冲区大小为什么可能影响性能？
6. 为什么 IO 优化很多时候不是“换框架”，而是先减少不必要的磁盘读写次数？

## 8. 第七组：围绕 `RandomAccessFile` 的追问

1. `RandomAccessFile` 是什么，它和普通文件流最大的区别是什么？
2. 为什么 `RandomAccessFile` 很容易和断点续传、分块写入联系起来？
3. 如果想把某个分块直接写入目标文件的指定位置，为什么会想到 `RandomAccessFile`？
4. `seek()` 在文件上传场景里能解决什么问题？
5. 你的项目为什么用了普通分块文件合并，而不是直接用 `RandomAccessFile` 定位写入？
6. 如果后续要优化分块合并方案，`RandomAccessFile` 会不会是一个候选方案？

## 9. 第八组：围绕 `NIO` 和 `FileChannel` 的追问

1. 如果面试官问你上传链路有没有考虑 `NIO`，你应该从哪些角度理解这个问题？
2. `BIO` 和 `NIO` 放到文件上传场景里，差异可以怎么理解？
3. `FileChannel` 在文件读写里通常是什么角色？
4. 如果要优化大文件合并，为什么 `FileChannel` 可能会被提到？
5. 你的项目当前如果没用 `NIO`，原因可能是什么？
6. `NIO` 在文件上传项目里更像必须项，还是进阶优化项？
7. 如果面试官继续追问零拷贝，你会被带到哪些话题上？

## 10. 第九组：围绕一致性与异常处理的追问

1. 文件上传到一半失败时，数据库记录、Redis 状态、磁盘分块怎么保证一致？
2. 为什么上传场景经常会有“元数据成功了，但物理文件失败了”的风险？
3. 如果文件已经落盘，但数据库事务失败了，应该怎么理解这个问题？
4. 如果数据库记录写成功了，但文件合并失败了，会留下什么后果？
5. 上传流程里，为什么文件系统操作和数据库事务很难做到完全原子？
6. 面对这种非原子场景，通常会用哪些补偿思路？
7. 为什么文件上传项目里经常要考虑清理临时文件？
8. 临时分块如果长期不清理，会带来哪些问题？

## 11. 第十组：围绕安全与边界的追问

1. 文件上传为什么不能只看前端传来的文件后缀？
2. 为什么要限制文件大小、文件类型和上传路径？
3. 如果用户上传一个恶意脚本文件，后端需要考虑哪些风险？
4. 文件名为什么不能直接拿来当服务器上的真实存储名？
5. 为什么很多上传系统会把业务文件名和物理存储文件名分开？
6. 上传路径如果完全由用户输入，会有什么问题？
7. 为什么要防止目录穿越这类路径问题？

## 12. 第十一组：围绕项目选型和方案对比的追问

1. 你的项目为什么选择本地文件系统存储，而不是对象存储？
2. 如果后续文件量变大，本地文件系统方案会遇到什么瓶颈？
3. 为什么分块上传、断点续传、秒传这几个能力经常一起出现？
4. 你的上传方案更像基础可用版，还是做过性能和一致性优化的工程化版本？
5. 如果让你重新设计一次上传链路，你最先会优化哪一层？
6. 你现在这套文件上传方案里，IO 相关最脆弱的点可能在哪？

## 13. 第十二组：围绕“你真的做过吗”的深挖问题

1. 你在这个文件上传项目里，真正亲手做了哪些和 IO 相关的部分？
2. 你在实现上传或合并时，实际踩过哪些坑？
3. 你当时是怎么验证分块合并出来的文件是正确的？
4. 你为什么这样设计临时目录、分块命名和最终文件命名？

6. 你项目里的 IO 部分，有哪些地方是“先实现可用，后面还能继续优化”的？

如果你时间有限，建议优先准备下面这些：

1. 文件上传为什么用字节流不用字符流？
2. 为什么大文件要分块上传？
3. 分块上传后端怎么落盘和合并？
4. 断点续传为什么要关注幂等性？
5. 秒传为什么和 MD5 有关？
6. 为什么上传场景常会提缓冲流？
7. `RandomAccessFile` 在这个场景里能干什么？
8. 如果合并失败，怎么处理半成品文件？

## 15. 一句话总结

文件上传项目里的 IO 面试题，重点不是背流的定义，而是能不能把“上传、分块、落盘、合并、校验、一致性、优化”这一整条链路讲清楚。
