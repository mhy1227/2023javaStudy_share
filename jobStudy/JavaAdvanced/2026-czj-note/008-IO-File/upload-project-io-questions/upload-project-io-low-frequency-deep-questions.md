# 文件上传项目 IO 低频深挖问题

## 1. 这一份怎么用

这一份不是主流高频题，而是“低频但有区分度”的深挖问题。

它的特点是：

- 不一定每场面试都会问
- 但一旦问到，往往说明面试官想继续判断你有没有工程深度
- 很适合社招、偏中高级后端、或者项目深挖环节提前准备

这一份只整理问题，先不写回答。

## 2. 第一组：实现细节更深一点

1. 分块大小为什么这样定？`5MB`、`10MB`、`20MB` 这种设置背后的取舍是什么？
2. 分块太小会带来什么问题？
3. 分块太大会带来什么问题？
4. 如果临时分块目录里的文件特别多，会不会有目录性能问题？
5. 文件命名为什么不能直接用原文件名？
6. 物理存储 Key 一般会怎么设计？
7. 合并文件时，如果机器宕机在最后一步 rename 前，会留下什么状态？
8. 你有没有考虑过 `fsync`、刷盘时机和真正落盘可靠性？

## 3. 第二组：一致性和状态机更深一点

1. 上传任务状态一般要不要设计成状态机？
2. 如果要做状态机，像 `INIT / UPLOADING / MERGING / SUCCESS / FAILED` 这种状态是否有必要？
3. 如果 Redis 里显示“上传已完成”，但磁盘上实际少了一块分片，最终应该以谁为准？
4. 合并任务重复执行时，怎么保证最终只成功一次？
5. 如果异步合并任务失败很多次，系统怎么告警或人工介入？
6. 秒传命中之后，如果业务记录创建失败了，该怎么处理？
7. 文件记录、上传任务、物理文件状态三者之间，谁应该作为主状态源？

## 4. 第三组：性能和资源治理

1. 上传大文件时，如何避免内存抖动和 Full GC？
2. 为什么上传接口除了权限控制，还要做限流、配额和并发控制？
3. 如果用户一直上传大文件但从不完成，怎么防止临时空间被打爆？
4. 如何做按用户、按租户的存储容量控制？
5. 上传高峰期如果磁盘写满了，系统应该怎么处理？
6. 如果上传请求很多，磁盘写入已经成为瓶颈，你会优先优化哪一层？
7. 分块上传场景里，线程池和磁盘 IO 压力之间是什么关系？

## 5. 第四组：安全方向

1. 文件内容类型怎么校验？为什么不能只信 `Content-Type`？
2. 如果用户上传的是木马、WebShell 或恶意脚本文件，系统应该考虑哪些风险？
3. 如果用户上传的是超大压缩包或压缩炸弹，系统要怎么防？
4. 下载接口如何避免越权访问？
5. 预览接口如何防止把内部真实存储路径暴露给前端？
6. 文件访问链接要不要做时效控制或签名控制？
7. 如果文件上传成功了，但后续被发现有安全风险，应该怎么隔离或下线？

## 6. 第五组：预览和下载链路

1. 下载大文件时，为什么会提到断点下载和 `Range` 请求？
2. 预览和下载是不是应该走同一套链路？
3. 图片、PDF、视频预览为什么处理方式可能不一样？
4. 如果文件放在对象存储里，下载应该走后端代理流式返回，还是前端直连签名 URL？
5. 预览接口和下载接口在权限控制上有什么差别？
6. 文件下载时，文件名编码和响应头处理为什么也可能变成细节坑？

## 7. 第六组：演进和架构方向

1. 如果从本地文件系统迁到 MinIO / OSS / COS，最先应该改哪一层？
2. 现在这套设计如果要支持多机部署，最容易出问题的是哪一块？
3. 如果以后要做 CDN 加速，上传链路和下载链路分别怎么改？
4. 如果以后要做在线预览、转码、病毒扫描，应该挂在上传链路的哪个阶段？
5. 你现在这套方案里，最值得优先重构的一点是什么？
6. 如果业务量继续增大，文件服务有没有必要从主业务系统里独立出来？

## 8. 当前更值得优先准备的低频题

如果时间有限，建议优先准备下面这些低频题：

1. 分块大小为什么这样定？
2. Redis 和磁盘状态不一致时以谁为准？
3. 合并任务重复执行时怎么保证最终只成功一次？
4. 如何避免临时分块把磁盘打爆？
5. 为什么不能只信 `Content-Type`？
6. 下载大文件为什么会提到 `Range` 请求？
7. 从本地文件系统迁到对象存储时最先改哪一层？
8. 多机部署下文件上传最容易出什么问题？

## 9. 一句话总结

这一份低频深挖题，不是为了无限扩题，而是为了让你在上传项目被继续往工程细节、状态一致性、资源治理和系统演进方向追问时，不会突然断掉。
