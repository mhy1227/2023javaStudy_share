## 2. `MultipartFile` 和真正落盘文件之间是什么关系

### 2.1 短答版

`MultipartFile` 更像是后端接收到上传请求后的一个封装对象，代表“当前这次请求里的文件内容和元信息”，它不等于最终落盘文件。真正落盘文件是服务端把它的内容写到磁盘或对象存储之后形成的持久化结果，所以两者一个是请求态，一个是存储态。

### 2.2 展开版

我会把这个问题拆成“传输层对象”和“持久化结果”来看。`MultipartFile` 本质上是 Spring MVC 对 `multipart/form-data` 上传请求的一层封装，它里面带的是当前请求中的文件流、文件名、大小、内容类型这些信息。它的生命周期通常只和当前 HTTP 请求强相关。

而真正落盘文件是服务端读取 `MultipartFile` 的内容后，按自己的存储规则写到磁盘目录、临时分块目录或者对象存储里生成的实际文件。也就是说：

1. `MultipartFile` 代表上传请求中的文件
2. 落盘文件代表服务端最终持久化后的结果

所以我不会把这两个概念混在一起。面试里如果问到这点，其实是在看你有没有把“接收上传”和“存储落地”拆开理解。

## 3. 上传接口为什么要限制并发分块数

### 3.1 短答版

因为分块上传不是并发越高越好，并发太高会把服务端线程、磁盘 IO、网络带宽和 Redis 状态更新压力一起拉高，反而更容易抖动甚至失败。限制并发分块数本质上是在控制资源占用和系统稳定性。

### 3.2 展开版

很多人第一反应会觉得并发越高上传越快，但服务端视角不完全是这样。分块并发过高会带来几个问题：

1. 同一时间大量请求占用线程和连接资源
2. 分块频繁落盘，磁盘 IO 更碎
3. 上传状态频繁更新，Redis 或数据库写压力变大
4. 文件合并前的临时分块积压更多

所以实际系统里通常不会无限放开并发，而是会设置一个相对合适的并发分块上限。这个上限不是纯粹为了“限用户”，而是为了在上传速度和整体稳定性之间做平衡。面试时如果能把这点说出来，会更像你站在服务端角度考虑过资源管理。

## 4. 如果多个线程同时合并同一个文件，怎么防止重复合并

### 4.1 短答版

核心思路就是给“合并动作”做互斥控制，避免同一个文件被重复触发合并。常见做法是基于文件唯一标识加锁，比如 Redis 分布式锁、数据库状态位或者 JVM 内部锁，再配合状态判断，保证只有一个线程真正进入合并流程。

### 4.2 展开版

这个问题本质上是在问“最终动作的幂等和互斥”。因为分块全部上传完成后，多个请求、重试或者异步任务都可能同时判断“可以合并了”，如果没有控制，就可能出现重复合并。

更稳的做法一般是两层：

1. 先判断文件当前状态是不是“待合并”
2. 再对该文件唯一标识加锁，保证同一时刻只有一个执行者

这样就算多个线程同时过来，也只有一个线程能真正开始合并，其他线程要么直接返回，要么发现状态已经变化后退出。这里的关键不是只靠 if 判断，而是要把状态判断和互斥控制结合起来。

## 5. 上传完成后文件预览、下载链路怎么设计

### 5.1 短答版

我会把“上传完成”和“文件访问”拆开设计。上传成功后先保存文件元数据和存储位置，后续预览和下载通过独立接口按权限校验后再读取文件。这样链路更清晰，也更方便做权限控制、文件名处理和访问审计。

### 5.2 展开版

上传链路的终点不应该只是“文件存好了”，后续通常还要支持预览和下载。我会倾向做成三层：

1. 业务层保存文件记录，比如文件名、大小、所属用户、存储路径、状态
2. 访问层提供预览或下载接口
3. 接口层在返回文件前先做权限校验和状态校验

这样做的好处是：文件访问不会直接暴露底层存储路径，权限控制也不会散落在前端。特别是如果以后从本地文件系统迁移到对象存储，访问层还可以比较平滑地切换实现，而不会影响业务层的文件记录设计。

## 6. 文件删除时，为什么业务记录删除和物理文件删除要分开考虑

### 6.1 短答版

因为业务记录和物理文件不是天然一一对应的。业务记录更像“谁拥有什么文件的引用”，物理文件更像“底层真正存着的那份数据”。如果两者不区分，删除一个业务记录时就可能误删仍然被别人使用的物理文件。

### 6.2 展开版

我会把这个问题看成“引用关系”和“真实存储”的分层问题。业务记录通常描述的是：

- 哪个用户上传了这个文件
- 文件显示名是什么
- 文件属于哪个目录
- 文件当前状态是什么

而物理文件关注的是：

- 这份文件真实存在哪里
- 有没有被多个业务记录复用

在秒传和文件复用场景下，这个差别特别重要。因为一个物理文件完全可能被多个用户或多条业务记录共同引用。所以删除时应该先删业务引用，再判断底层物理文件是否还有其他引用，只有确认没有任何引用了，才适合真正删除物理文件。

## 7. 秒传场景下，什么时候可以真正删除物理文件

### 7.1 短答版

只有在确认这个物理文件已经没有任何业务引用时，才适合真正删除。也就是说，不能因为某一个用户删了自己的文件记录，就立刻把底层物理文件删掉，要先判断引用计数或引用关系是否已经归零。

### 7.2 展开版

秒传的本质是复用已有物理文件，所以一个文件的底层存储很可能对应多条业务记录。这样一来，物理删除条件就不能只看“当前这个用户删没删”，而是要看“系统里还有没有别的引用”。

更稳的做法一般是：

1. 先删除当前用户的业务记录
2. 再判断这个物理文件是否还有其他业务记录引用
3. 只有当引用彻底归零时，才做真正的物理删除

如果再往工程上走一步，还可以做成异步删除或延迟删除，避免误删以后无法恢复。面试里如果能把“业务删除”和“物理回收”拆开讲，通常会比较加分。

## 8. 如果两个用户引用同一个物理文件，删除一个用户的文件记录为什么不能直接删物理文件

### 8.1 短答版

因为另一个用户可能还在使用这份物理文件。如果直接删物理文件，就会导致别人的文件记录还在，但实际文件已经没了。所以这类场景必须先删引用，再判断底层文件是不是还能被别人访问。

### 8.2 展开版

这个问题本质上和秒传复用是同一个底层逻辑。两个用户看到的是各自的业务文件记录，但它们背后可能共用同一个物理文件。如果删掉一个用户记录时就直接删物理文件，会立刻破坏另一个用户的访问能力。

所以我会把它回答成：

1. 用户文件记录和物理文件要拆开建模
2. 删除用户记录只代表删除一个引用
3. 真正删物理文件之前，要确认引用是否全部消失

这说明文件系统设计里不能只盯着“删了一条记录”，还要盯着“底层共享存储有没有被别的地方复用”。

## 9. 分块上传过程中，怎么防止用户串改分块序号或上传别人的分块

### 9.1 短答版

核心是不要只信前端传参。服务端需要校验分块序号是否合法、文件标识是不是属于当前上传任务、当前用户是否有操作权限，必要时还要把上传任务和用户绑定起来。这样就算前端串改参数，也不至于把分块写到别人的任务里。

### 9.2 展开版

这个问题本质上是上传接口的越权和参数篡改风险。因为前端传来的文件标识、分块序号、任务 ID 都可能被恶意修改，所以服务端不能直接照单全收。

更稳的做法一般有几层：

1. 上传任务创建时就把任务 ID、文件标识、所属用户绑定起来
2. 上传分块时校验当前请求用户和任务归属是否一致
3. 校验分块序号是否在合法范围内
4. 校验这个分块是不是当前文件应该接收的那一块，而不是别的任务的块

如果再严格一点，还可以对上传会话做签名校验或时效控制。面试时如果你能把问题从“分块怎么传”讲到“分块参数会不会被伪造”，工程感会明显更强。

## 10. 上传接口为什么容易被刷流量、刷磁盘

### 10.1 短答版

因为上传接口天然消耗网络、磁盘和存储空间，而且一个请求的资源占用通常比普通查询接口大很多。如果没有限流、大小限制、权限控制和清理策略，就很容易被恶意刷请求、刷分块、刷临时文件，把系统资源打满。

### 10.2 展开版

上传接口和普通业务接口不一样，它不是只查个数据库或改条记录，而是会持续消耗：

1. 带宽
2. 线程和连接
3. 磁盘写入能力
4. 存储空间
5. 临时分块目录

所以它天然更容易被滥用。比如攻击者可以反复上传大文件、反复发伪造分块、故意不走合并只堆积临时块，这些都会快速吃掉系统资源。

所以比较稳的系统一般要配套：

1. 文件大小限制
2. 用户权限控制
3. 并发限制和限流
4. 非法任务校验
5. 临时文件过期清理

我会把这个问题理解成“上传能力不仅是功能问题，也是资源防护问题”。

## 11. 如果以后改成对象存储，分块、秒传、断点续传哪些逻辑还能复用

### 11.1 短答版

大部分业务层逻辑都还能复用，比如上传任务状态管理、分块状态记录、秒传判重、文件元数据设计、权限控制这些都不依赖具体是本地磁盘还是对象存储。真正变化更大的，主要是底层落盘、合并和访问 URL 这部分实现。

### 11.2 展开版

这个问题我会从“业务逻辑”和“存储实现”拆开回答。很多人以为换成对象存储就要全部重写，其实不一定。

通常还能复用的逻辑包括：

1. 上传任务建模
2. 分块上传状态管理
3. 断点续传的幂等设计
4. 秒传判重逻辑
5. 文件元数据表设计
6. 权限校验和业务访问控制

真正变化大的部分主要是：

1. 分块内容怎么写入底层存储
2. 分块合并是不是交给对象存储侧完成
3. 文件预览和下载是不是改成签名 URL
4. 本地路径是否改成对象 Key

所以我会把这类迁移理解成“业务层尽量稳定，存储适配层替换实现”。如果前期结构分层做得比较清楚，从本地文件系统切到对象存储并不一定意味着推倒重来。
