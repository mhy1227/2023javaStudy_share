# 文件上传项目 IO 低频问题回答样例

## 1. 这一份怎么用

这一份不是把低频题全部写完，而是先抽几道最有区分度的问题，给出可直接用于面试的回答样例。

选题标准是：

- 能体现工程深度
- 和文件上传项目强相关
- 在社招深挖里有区分度

每题仍然拆成两层：

1. `短答版`：适合先快速接住问题
2. `展开版`：适合面试官继续深挖时往下讲

## 2. 分块大小为什么这样定？`5MB`、`10MB`、`20MB` 的取舍是什么？

### 2.1 短答版

分块大小本质上是在做平衡：太小会让请求数变多、状态管理更碎，太大又会让单块失败重传成本上升、单次内存和网络压力变大。所以像 `5MB`、`10MB` 这种设置通常不是绝对标准，而是结合网络情况、文件规模、服务端承载能力做出来的折中值。

### 2.2 展开版

我会把这个问题理解成“吞吐、稳定性、重传成本”三者之间的取舍。

如果分块太小，比如几百 KB 或 1MB，问题是：

1. 分块数量会明显变多
2. 上传请求数更多
3. Redis 或数据库里的分块状态更碎
4. 合并前临时文件数量也更多

这样会让服务端状态管理和磁盘目录压力上升。

如果分块太大，比如几十 MB，问题又会变成：

1. 单次上传失败的重传成本更高
2. 单块请求耗时更长
3. 单次内存占用和网络抖动风险更大
4. 用户感知也可能更差

所以像 `5MB`、`10MB` 往往是一个比较常见的折中区间。面试里我不会把某个值说成唯一正确，而是会强调：这个值应该根据文件类型、典型文件大小、网络环境和服务端承载情况做调优，而不是拍脑袋定死。

## 3. 如果 Redis 里显示“上传已完成”，但磁盘上实际少了一块分片，最终应该以谁为准？

### 3.1 短答版

最终还是应该以磁盘上的真实分块为准，因为 Redis 在这里更多是状态缓存，磁盘文件才是真正的数据落点。缓存可以错，真实数据不能靠猜，所以这种不一致场景下一般要回到磁盘做校验，再修正状态。

### 3.2 展开版

这个问题本质上是在考你有没有把“缓存状态”和“真实存储”区分开。

我会明确说：

1. Redis 更适合做上传进度和状态缓存
2. 磁盘分块文件才是真正决定能不能合并的事实依据

如果 Redis 说已经完成，但磁盘少了一块，那说明状态和真实数据已经不一致了。这时候如果还继续只信 Redis，就可能把一个不完整文件误判成可合并状态，后面合并出来的结果也会有问题。

所以更稳妥的处理思路通常是：

1. 以磁盘分块为准做一次真实校验
2. 判断哪些块真的存在、哪些块缺失
3. 再把 Redis 或数据库状态修正回来

面试里如果能说出“缓存是加速手段，不是最终事实源”，这题就会显得比较稳。

## 4. 合并任务重复执行时，怎么保证最终只成功一次？

### 4.1 短答版

核心是把合并动作做成“有状态 + 有互斥”的流程。也就是先判断文件是不是仍处于待合并状态，再对文件唯一标识加锁，保证只有一个执行者真正进入合并流程，其他重复任务直接退出或发现状态已变更后结束。

### 4.2 展开版

这个问题其实是在考“最终动作的幂等性”。

因为分块全部上传完成后，可能会有多个地方同时触发合并：

1. 前端最后一块上传后主动触发
2. 后台异步任务兜底触发
3. 重试任务再次触发

如果没有控制，就可能出现重复合并。

所以比较稳的做法通常有两层：

第一层是状态控制。文件状态要明确区分，比如：

- `UPLOADING`
- `READY_TO_MERGE`
- `MERGING`
- `SUCCESS`

第二层是互斥控制。对同一个文件唯一标识做锁控制，比如 Redis 锁、数据库状态更新 CAS、或者本机锁配合分布式状态判断。

这样就算多个线程同时来了，也只有一个线程能把状态从 `READY_TO_MERGE` 推进到 `MERGING` 并真正开始做，其他线程发现状态已经变了，就不会重复执行。这里的关键不只是“加锁”，而是“状态流转和加锁一起做”。

## 5. 如果用户一直上传大文件但从不完成，怎么防止临时空间被打爆？

### 5.1 短答版

这类问题本质上要靠资源治理，不能只靠用户自觉。常见做法是给上传任务加过期时间、限制单用户容量和并发、定时清理超时未完成分块，再配合限流和告警。否则临时分块目录很容易被长期堆满。

### 5.2 展开版

这个问题很典型，因为上传系统最容易被忽视的不是“上传失败”，而是“用户一直不结束上传”。

如果系统没有治理策略，就可能出现：

1. 大量半成品分块一直占着磁盘
2. 某个用户持续占用临时空间
3. 最后把整台机器磁盘打满

所以比较稳的设计通常会有几层：

1. 给上传任务设置过期时间
2. 对单用户或单租户做容量上限
3. 限制并发上传任务数
4. 定时清理超时未完成的临时分块
5. 对磁盘使用率做监控和告警

我会把这题回答成“上传功能不只是能传，还要能治理资源”。因为一旦系统到了线上，磁盘空间本身就是要被运营和防护的资源。

## 6. 文件内容类型怎么校验？为什么不能只信 `Content-Type`？

### 6.1 短答版

因为 `Content-Type` 本质上也是客户端传过来的字段，完全可以被伪造，所以不能把它当成绝对可信。更稳妥的做法通常是把它当参考，再结合文件后缀、文件头特征、业务白名单规则一起校验。

### 6.2 展开版

这个问题和“不能只看文件后缀”本质上是一类安全边界问题。

我会明确说：`Content-Type` 很有用，但不能单独信任。因为它通常也是请求头里带来的，客户端完全可以把一个恶意脚本伪装成 `image/jpeg` 或 `application/pdf` 传进来。

所以比较稳的做法通常会组合几层：

1. 校验文件后缀是否在允许范围内
2. 校验 `Content-Type` 是否和业务预期一致
3. 校验文件头特征或魔数，判断真实类型
4. 对高风险类型做更严格限制或后续扫描

也就是说，`Content-Type` 更像是一个提示信号，而不是最终判定依据。面试里如果能说到“客户端输入本身不可信”，这题的工程味就出来了。

## 7. 如果从本地文件系统迁到 MinIO / OSS / COS，最先应该改哪一层？

### 7.1 短答版

最先应该抽离和替换的是“存储适配层”，而不是直接改业务逻辑层。因为上传任务、分块状态、秒传判重、权限控制这些大多还是业务逻辑，真正变化最大的 usually 是文件怎么存、怎么合并、怎么访问。

### 7.2 展开版

这个问题我会用“分层改造”的思路回答。

如果前期设计比较清楚，系统里其实可以拆成两部分：

1. 业务层：上传任务、分块状态、文件记录、权限、秒传、断点续传这些逻辑
2. 存储层：文件内容怎么落地、路径怎么组织、怎么合并、怎么生成访问地址

从本地文件系统迁到 MinIO / OSS / COS，变化最大的其实是第二层。因为：

1. 本地路径会变成对象 Key
2. 分块写入和最终合并方式可能改变
3. 预览和下载可能会改成签名 URL
4. 删除和回收策略也会跟着变

所以最先该改的通常不是业务表结构，也不是权限逻辑，而是把存储实现抽象出来，做成可替换的存储适配层。这样业务层尽量不动，迁移成本才可控。

## 8. 一句话总结

低频题真正有区分度的地方，不是答案更复杂，而是它要求你把“实现细节、事实源、状态幂等、资源治理、安全校验、分层演进”这些工程视角一起讲出来。
