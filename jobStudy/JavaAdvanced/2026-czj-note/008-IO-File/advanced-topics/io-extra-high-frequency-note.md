# IO 补充高频点

## 1. 这一份补什么

这一份不再重复前面已经讲过的主干内容，而是专门补几类“不是第一梯队必问，但面试里偶尔会顺手追到”的 IO 点：

1. 节点流和处理流
2. `DataInputStream` / `DataOutputStream`
3. `RandomAccessFile`
4. `AIO`

这些内容更像加分项。

也就是说：

- 不一定每场都问
- 但如果你能说上一两句，会显得知识面更完整

## 2. 什么是节点流和处理流

这组词很多人学过，但面试时反而容易一下子讲不顺。

可以先这样理解：

- 节点流：直接和数据源、数据目标打交道的流
- 处理流：套在已有流外面，给它增加功能的流

### 2.1 节点流怎么理解

节点流可以理解成“真正连到文件、数组、内存对象上的那层流”。

常见例子：

- `FileInputStream`
- `FileOutputStream`
- `FileReader`
- `FileWriter`

这些类本身就能直接读写，不依赖额外再包一层才成立。

### 2.2 处理流怎么理解

处理流本身通常不直接连接具体文件，而是建立在别的流之上，为原来的流增强能力。

常见例子：

- `BufferedInputStream`
- `BufferedOutputStream`
- `BufferedReader`
- `BufferedWriter`
- `ObjectInputStream`
- `ObjectOutputStream`
- `DataInputStream`
- `DataOutputStream`

这些流常见的特点是：

1. 外面包一层
2. 增强原流能力
3. 让读写更方便、更高效，或者支持特殊格式

### 2.3 面试里怎么说更稳

可以直接答：

节点流是直接连接数据源或目标的基础流，比如 `FileInputStream`、`FileReader`；处理流则是在已有流外面再包一层，用来增强功能，比如缓冲、对象序列化、按基本类型读写等，像 `BufferedReader`、`ObjectOutputStream`、`DataInputStream` 都算处理流。

## 3. `DataInputStream` 和 `DataOutputStream` 是干什么的

这两个类本质上是在问：

能不能把 Java 的基本类型按特定格式写出去，再按对应格式读回来。

比如：

- `int`
- `double`
- `boolean`
- `UTF` 字符串

### 3.1 它和普通字节流有什么区别

普通字节流更多是在处理“原始字节”。

而 `DataOutputStream` / `DataInputStream` 是：

- 把基本类型按约定格式编码写出
- 再按同样顺序和规则读回

所以它更像“面向基本类型的二进制读写工具”。

### 3.2 使用时最容易踩什么坑

最大的问题通常不是 API 本身，而是“写的顺序”和“读的顺序”必须严格一致。

比如你按下面顺序写：

1. `writeInt(18)`
2. `writeUTF("tom")`
3. `writeBoolean(true)`

那读的时候就要按同样顺序：

1. `readInt()`
2. `readUTF()`
3. `readBoolean()`

如果顺序乱了，读出来就会错，甚至直接异常。

### 3.3 面试里怎么回答

可以说：

`DataInputStream` 和 `DataOutputStream` 适合把基本类型按二进制格式写入和读取，比如 `int`、`double`、`boolean` 这些。它的核心不是单纯处理字节，而是提供面向基本类型的读写方法，不过写入顺序和读取顺序必须保持一致。

## 4. `RandomAccessFile` 是什么

`RandomAccessFile` 最值得记住的一点不是“它也是个文件类”，而是：

它支持随机访问文件中的某个位置。

也就是说，它不是只能从头读到尾，也不是只能从头写到尾，而是可以：

- 跳到指定位置
- 从那里开始读
- 从那里开始写

### 4.1 为什么叫随机访问

这里的“随机”不是乱来，而是说你可以通过位置偏移量直接定位。

典型方法是：

- `seek(long pos)`

你可以先把文件指针移动到某个位置，再继续读写。

### 4.2 它适合什么场景

面试里常见的理解场景有：

1. 需要从文件中某个固定位置读取内容
2. 需要覆盖文件的某一段
3. 需要做断点续传、分段写入这类“不是只从头到尾顺序处理”的事情

### 4.3 它和普通流最大的区别

普通流更像顺序读写。

`RandomAccessFile` 则多了“文件指针位置控制”这件事。

所以如果面试官问你它和 `FileInputStream`、`FileOutputStream` 的区别，核心就答：

- 普通流偏顺序处理
- `RandomAccessFile` 支持定位后再读写

### 4.4 面试里怎么回答

可以说：

`RandomAccessFile` 适合需要随机定位文件位置再进行读写的场景，它可以通过 `seek()` 调整文件指针，不像普通流那样主要按顺序读写。所以像分段写入、断点续传、覆盖固定偏移内容时，它会更自然。

## 5. 什么是 `AIO`

如果 `BIO` 和 `NIO` 已经比较熟了，面试官有时会继续问一句：

那你知道 `AIO` 吗？

这里先不要把自己带得太深。

对当前阶段，更合适的理解是：

`AIO` 指异步非阻塞 IO，强调的是发起 IO 操作后，不需要一直同步等待结果，而是在操作完成后通过回调或通知机制处理结果。

### 5.1 它和 `BIO`、`NIO` 怎么粗分

可以先粗暴分三层：

1. `BIO`：传统阻塞式
2. `NIO`：更偏缓冲区、通道、非阻塞思路
3. `AIO`：更强调异步完成后的回调通知

### 5.2 当前阶段要掌握到什么程度

如果你不是专门在做网络框架或高性能通信，一般不需要把 `AIO` 讲得特别深。

能说清下面这几句，通常就够了：

1. `AIO` 是异步非阻塞 IO
2. 它更强调“操作完成后再通知你”
3. 它和 `BIO`、`NIO` 一样，属于 IO 模型视角下的进一步区分

### 5.3 面试里怎么回答

可以说：

`AIO` 可以理解成异步非阻塞 IO，发起读写后不会像传统阻塞 IO 那样一直等结果，而是更强调完成后的通知或回调。对基础面试来说，通常先把 `BIO`、`NIO` 的区别讲清楚，再补一句知道 `AIO` 是更进一步的异步模型，就已经比较够用了。

## 6. 这几个点该怎么排优先级

如果时间有限，建议这么分：

1. 先掌握节点流 / 处理流
2. 再掌握 `RandomAccessFile`
3. 再知道 `DataInputStream` / `DataOutputStream`
4. 最后知道 `AIO` 的一句话定位

原因很简单：

- 节点流 / 处理流和整个 IO 体系理解直接相关
- `RandomAccessFile` 偶尔真会被单独提
- `DataInputStream` / `DataOutputStream` 是典型补充问法
- `AIO` 更偏知道概念即可

## 7. 当前这份最该记住的几句话

1. 节点流直接连数据源，处理流是在外面包一层增强功能
2. `DataInputStream` / `DataOutputStream` 是按基本类型格式读写，不是随便读写字节
3. `RandomAccessFile` 的关键能力是 `seek()`，也就是随机定位后读写
4. `AIO` 是异步非阻塞 IO，基础阶段知道它和 `BIO`、`NIO` 的粗区别就够了

## 8. 一句话总结

这一份补的不是 IO 主干，而是面试里偶尔会追到、答出来能让体系更完整的几个边角高频点。
