# MQ 可靠性、唯一性与幂等性高频追问

## 1. 这份文档补什么

前面的 MQ 文档已经讲了异步削峰。

这份文档补面试更容易追问的问题：

- 怎么保证消息可达
- 怎么保证消息不丢
- 怎么处理重复消息
- 怎么保证消费幂等
- 怎么保证消息唯一性
- 怎么处理顺序消息
- 怎么处理消息积压
- 怎么处理消费失败

## 2. MQ 能不能保证消息绝对不丢

严格说，很难说“绝对不丢”。

工程上一般说：

> 通过生产端确认、Broker 持久化、消费端确认、失败重试和补偿机制，尽量保证消息可靠投递和最终可达。

不要面试里直接说：

> MQ 保证消息一定不丢。

更稳的说法是：

> MQ 本身提供可靠性机制，但业务还要配合确认、重试、幂等和补偿。

## 3. 怎么保证消息可达

通常从三段看：

1. 生产者到 MQ
2. MQ 自身存储
3. MQ 到消费者

生产者侧：

- 发送确认
- 失败重试
- 本地消息表
- 事务消息

Broker 侧：

- 持久化
- 主从复制
- 高可用部署

消费者侧：

- 消费成功后再 ack
- 消费失败重试
- 死信队列
- 补偿任务

## 4. 生产者发送失败怎么办

常见方案：

- 同步等待发送结果
- 发送失败重试
- 记录本地消息表
- 定时任务扫描未成功消息重新发送

面试回答：

> 生产者不能发完就不管，一般要确认消息是否发送成功。失败时可以重试，重要消息可以先落本地消息表，再异步发送，发送成功后更新状态，失败则定时补偿。

## 5. 消费者消费失败怎么办

常见方案：

- 不 ack，让 MQ 重投
- 重试有限次数
- 超过次数进入死信队列
- 人工或定时补偿

面试回答：

> 消费端要在业务处理成功后再确认消费。如果处理失败，就不要确认或主动返回失败，让 MQ 触发重试。重试次数超过阈值后可以进入死信队列，再通过补偿任务或人工排查处理。

## 6. 为什么会有重复消息

重复消息很常见。

原因包括：

- 生产者重试导致重复发送
- Broker 投递后消费者处理成功但 ack 失败
- 消费者超时导致 MQ 重新投递
- 网络抖动导致状态不确定

所以 MQ 系统一般默认要接受一个事实：

> 消息可能重复，消费者必须保证幂等。

## 7. 怎么保证消费幂等

常见做法：

- 业务唯一键
- 消息唯一 ID
- 去重表
- Redis setnx
- 数据库唯一索引
- 状态机校验

比如订单支付成功消息：

```text
pay_success:orderId
```

如果订单已经是支付成功状态，再收到重复消息就直接返回成功。

## 8. 怎么保证消息唯一性

消息唯一性通常不是指 MQ 永远只投递一次。

更准确地说：

> 给每条业务消息生成业务唯一标识，消费端根据这个标识做去重。

常见唯一标识：

- 订单号
- 支付流水号
- 业务请求号
- messageId

注意：

> 不要只依赖 MQ 自动生成的 messageId，最好使用业务唯一键。

## 9. 幂等和唯一性有什么区别

唯一性是识别同一条业务消息。

幂等性是重复处理同一条消息时，结果仍然正确。

比如：

- 唯一性：这条消息对应订单 `10001`
- 幂等性：订单 `10001` 重复收到支付成功消息，也只更新一次状态

一句话：

> 唯一性用于识别重复，幂等性用于处理重复。

## 10. 怎么保证顺序消息

如果业务要求顺序，比如同一个订单的状态变更：

```text
创建 -> 支付 -> 发货 -> 完成
```

可以：

- 同一个业务 key 发送到同一个队列或分区
- 单线程或有序消费
- 消费端用状态机兜底

注意：

> 全局有序成本很高，通常只保证同一业务 key 局部有序。

## 11. 消息积压怎么办

常见原因：

- 消费者处理慢
- 消费者实例太少
- 下游数据库慢
- 消费失败大量重试
- 突发流量过大

处理方向：

- 扩容消费者
- 提高消费并发
- 优化消费逻辑
- 批量消费
- 排查下游瓶颈
- 暂停非核心消费
- 临时扩队列或分区

面试回答：

> 消息积压先看是生产太快还是消费太慢，再看消费者实例、消费耗时、下游数据库和失败重试。处理时可以扩容消费者、提高并发、优化下游、批量处理，并监控积压量变化。

## 12. MQ 怎么做削峰

削峰的核心是：

> 前端高峰请求先写入 MQ，后端消费者按自己能力慢慢处理。

好处：

- 避免瞬时流量打垮后端
- 解耦生产者和消费者
- 提高系统抗峰值能力

但要注意：

- MQ 会引入延迟
- 消息可能积压
- 消费端要保证幂等
- 失败要有补偿

## 13. MQ 和分布式事务是什么关系

MQ 常用于最终一致。

比如：

1. 订单创建成功
2. 发送扣库存消息
3. 库存服务消费消息
4. 失败则重试或补偿

和 Seata 对比：

- Seata 更偏同步事务协调
- MQ 更偏异步最终一致

## 14. 面试怎么完整回答“MQ 怎么保证可靠性”

可以这样说：

MQ 的可靠性要从生产者、Broker、消费者三段看。生产者侧要确认发送结果，失败重试，重要消息可以先落本地消息表；Broker 侧依赖持久化、主从复制和高可用部署；消费者侧要业务处理成功后再 ack，失败则重试，超过次数进入死信队列，再通过补偿任务处理。同时由于消息可能重复，消费者必须做幂等，比如用业务唯一键、数据库唯一索引、Redis setnx 或状态机控制，保证重复消费不会造成重复扣减或重复更新。

## 15. 一句话总结

MQ 面试高频不是“用了 MQ”，而是可靠投递、重复消费、幂等、顺序、积压和补偿这一整套闭环。
