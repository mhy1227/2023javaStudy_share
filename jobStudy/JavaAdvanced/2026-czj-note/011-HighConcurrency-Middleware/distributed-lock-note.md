# 分布式锁补充

## 1. 为什么这一份要单独补

线程锁和分布式锁不是一个层级的问题。

线程锁解决的是：

- 单机进程内并发

分布式锁解决的是：

- 多机器、多实例之间的并发协调

## 2. 先一句话理解分布式锁

可以先记一个口径：

- 分布式锁是为了解决多个服务实例同时操作同一份共享资源时的互斥问题

## 3. 为什么单机锁不够

因为单机锁只在当前 JVM 进程里有效。

如果你的服务部署了多个实例：

- 实例 A 加了本地锁
- 实例 B 并不知道

所以本地锁不能解决分布式环境下的互斥问题。

## 4. 常见实现思路

最常见的是：

1. Redis 分布式锁
2. Zookeeper 分布式锁
3. 数据库悲观锁/唯一约束辅助方案

## 5. Redis 分布式锁为什么常见

因为它实现相对简单，而且性能高。

高频关键词包括：

- `SET key value NX PX`
- 唯一标识
- 过期时间
- 解锁时校验 owner

## 6. 分布式锁最核心的设计点是什么

至少要考虑这几个问题：

1. 怎么加锁
2. 怎么保证锁不会被别人误删
3. 业务执行太久怎么办
4. 锁过期了怎么办
5. Redis 异常时怎么办

也就是说，分布式锁不是“加上去就完了”，而是要考虑锁的完整生命周期。

## 7. 为什么要有唯一标识

因为加锁和解锁必须对应到同一个持有者。

如果没有唯一标识，最危险的问题就是：

- 线程 A 的锁过期了
- 线程 B 拿到了新锁
- 线程 A 又来执行解锁
- 把线程 B 的锁删掉了

所以解锁时要先校验：

- 这个锁是不是我自己加的

## 8. 为什么要设置过期时间

因为如果业务线程异常退出或者机器宕机，没有过期时间，锁可能永远不释放。

所以分布式锁一般都要有：

- 过期时间

但新的问题也来了：

- 如果业务执行时间超过了锁的过期时间怎么办？

这也是面试里很喜欢继续追的问题。

## 9. 业务执行太久怎么办

常见思路有两种：

1. 业务尽量控制执行时间
2. 用看门狗机制自动续期

这也是很多人提到 `Redisson` 的原因。

因为它常见的一个能力就是：

- 锁自动续期

## 10. Redis 分布式锁一定绝对安全吗

面试里如果问深一点，答案通常不能说绝对安全。

更稳的回答是：

- Redis 分布式锁在很多业务场景里是常用方案，但它依然要结合过期时间、网络抖动、主从切换、业务执行时长这些问题综合考虑

## 11. Zookeeper 分布式锁一般怎么理解

你当前阶段不一定要展开太深，但可以知道一个粗主线：

- 它更偏基于临时节点和监听机制来实现锁

很多人会觉得它相比 Redis：

- 一致性语义更强
- 实现和运维理解成本也更高

## 12. 面试里怎么回答“为什么需要分布式锁”

可以直接背一版：

本地锁只能控制单个 JVM 实例内的并发，如果服务是多实例部署，就可能出现多个实例同时操作同一份资源，所以这时需要分布式锁来做跨实例互斥控制。

## 13. 面试里怎么回答“Redis 分布式锁要注意什么”

可以直接背一版：

Redis 分布式锁最核心要注意三件事：第一是加锁要带唯一标识和过期时间；第二是解锁时不能直接删，要先校验这个锁是不是自己持有的；第三是要考虑业务执行时间超过锁过期时间的情况，必要时要有自动续期机制。

## 14. 当前阶段最值得记什么

1. 分布式锁解决的是多实例之间的互斥，不是单机锁
2. Redis 是最常见方案之一
3. 锁要有唯一标识和过期时间
4. 解锁要防误删
5. 业务执行太久时要考虑续期问题

## 15. 一句话总结

分布式锁解决的是多实例之间的并发互斥问题，不是单机线程锁的简单放大版，而 Redis 分布式锁最核心的点就是唯一标识、过期时间和安全解锁。
