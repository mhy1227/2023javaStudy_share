# 跨库 Join 与非分片键查询

## 1. 这篇文档解决什么问题

分库分表题里，最容易被继续追的两个点就是：

- 跨库 `join` 怎么办
- 非分片键查询怎么办

因为这两个问题非常容易暴露你是不是只会讲“拆”，不会讲“拆完以后怎么查”。

## 2. 为什么这两个点高频

因为很多人一讲分库分表，只会说：

- 数据量大了就拆
- 按 `user_id` 或 `order_id` 路由

但面试官真正想看的是：

- 如果查询条件不是分片键怎么办
- 如果两个表不在一个库里，还能不能像以前一样直接 `join`

也就是说：

> 真正难的不是“怎么拆”，而是“拆完以后查询路径怎么设计”。

## 3. 先说跨库 Join 为什么麻烦

单库单表时代，很多 SQL 很自然：

- `join`
- `order by`
- `group by`

都可以直接交给数据库。

但一旦分库分表：

- 两张表可能不在一个实例里
- 数据可能分散在多个分片

这时候数据库本身就很难再像单库那样帮你一次性做完。

## 4. 面试里怎么回答跨库 Join

一个比较稳的口径是：

> 分库分表后，跨库 join 一般不建议像单库时代那样直接依赖数据库完成，因为成本和复杂度都会明显上升。更常见的思路是尽量避免跨库 join，比如通过冗余字段、全局表、应用层聚合，或者设计时让强关联数据尽量落在同一分片。

## 5. 常见解决思路有哪些

比较高频的口径通常有：

- 应用层聚合
- 冗余字段
- 全局表
- ER 同分片
- 搜索引擎或离线数仓兜底

## 6. 应用层聚合怎么理解

就是把原来一条 SQL 干的事，拆成：

- 先查主数据
- 再查关联数据
- 最后在应用层拼装

它的优点是：

- 简单直接
- 不强依赖数据库跨库能力

它的问题是：

- 代码复杂度会上升
- 查询次数可能变多
- 对分页和排序不够友好

## 7. 冗余字段怎么理解

如果某些关联信息查询特别高频，就可能不会每次都临时跨库查，而是：

- 在主表里冗余一部分常用字段

比如：

- 订单表里冗余商品名、店铺名、用户昵称快照

它的好处是：

- 查询快
- 避免频繁跨库拼接

它的问题是：

- 数据一致性维护更复杂

## 8. 全局表怎么理解

全局表更适合：

- 数据量不大
- 但每个分片都会频繁用到

比如：

- 地区码表
- 配置字典表

这种表通常可以在多个库里各放一份。

这样很多关联查询就不必每次都跨库。

## 9. ER 同分片为什么常被提

它的核心思想是：

- 强关联数据尽量按同一个分片键路由到同一分片

比如：

- 订单和订单明细都按 `order_id` 或 `user_id` 路由

这样虽然系统整体还是分库分表了，但这类高频关联查询仍然更容易在单分片内完成。

这会比事后到处补跨库 join 更稳。

## 10. 再说非分片键查询为什么麻烦

如果你的分片键是：

- `user_id`

但面试官继续问：

- 如果我现在按手机号查用户怎么办

这就属于：

- 非分片键查询

问题在于：

- 路由系统不知道该打到哪个分片

最差情况下就会变成：

- 全路由
- 扫所有分片

## 11. 非分片键查询怎么答

比较稳的口径是：

> 非分片键查询是分库分表后的高频难点，因为系统没法像按分片键那样直接定位分片。常见思路包括建立映射关系、增加辅助索引表、搜索引擎兜底，或者尽量把高频查询条件提前纳入分片设计。最差情况才是全路由扫描，因为那样代价最高。

## 12. 常见解决思路

高频回答通常包括：

- 映射表
- 辅助索引表
- 搜索引擎
- 全路由兜底
- 业务上限制查询路径

## 13. 映射表 / 辅助索引表怎么理解

比如系统按 `user_id` 分片，但业务又经常按手机号查。

这时候可能会维护一张：

- `mobile -> user_id`

的映射关系。

查的时候先用手机号找到 `user_id`，再按 `user_id` 路由到目标分片。

这是一种很典型的工程思路。

## 14. 搜索引擎兜底怎么理解

如果查询条件更复杂，比如：

- 多字段组合搜索
- 模糊搜索
- 条件很多的后台检索

那很多系统就不会硬让分库分表后的数据库扛所有搜索需求，而会考虑：

- Elasticsearch 这类搜索引擎

所以这里的重点不是“数据库不行”，而是：

- 查询模型已经不适合只靠分片后的 OLTP 数据库来硬查

## 15. 全路由为什么通常是最后兜底

因为全路由意味着：

- 每个分片都打一遍

它的代价通常是：

- 延迟高
- 开销大
- 分片越多越痛苦

所以更稳的说法是：

- 全路由不是不能用，但通常只能作为低频、后台、兜底型查询

## 16. 面试里最容易犯的错误

1. 只会说分库分表提升性能，不会说查询会变难
2. 把跨库 join 讲成“中间件会自动解决”
3. 遇到非分片键查询就只会答“扫所有库”
4. 不知道 ER 同分片、冗余字段、映射表这些常见工程口径

## 17. 一个比较稳的口语版回答

> 分库分表之后，最容易被追问的就是跨库 join 和非分片键查询，因为真正难的不是拆，而是拆完以后怎么查。跨库 join 一般不建议像单库时代那样完全依赖数据库做，更常见的思路是应用层聚合、冗余字段、全局表，或者设计时让强关联数据尽量 ER 同分片。非分片键查询也类似，因为系统没法直接按路由键定位分片，所以通常会配合映射表、辅助索引表或者搜索引擎解决，最差情况才做全路由扫描。面试里如果能把这些点讲出来，会比只会说“数据量大了就分库分表”成熟很多。

## 18. 一句话收尾

> 分库分表题真正拉开差距的，往往不是“怎么拆”，而是“拆完以后跨库 join 和非分片键查询怎么收口”。
