# 分片键怎么选

## 1. 这篇文档解决什么问题

分库分表题里，分片键几乎一定会被问。

因为很多人会说：

- 我们按 `user_id` 分片
- 我们按 `order_id` 分片

但面试官真正想看的是：

- 你为什么这么选
- 选错会有什么后果
- 这个字段到底适不适合做分片键

所以这一篇的重点不是列字段，而是讲：

> 分片键为什么重要，应该按什么标准选，答题时怎么说才像真的做过设计。

## 2. 为什么分片键这么关键

分片键本质上决定了三件事：

- 数据怎么分布
- 查询怎么路由
- 后续扩容会不会痛苦

所以它不是一个普通字段选择问题，而是：

- 架构层决策

很多分库分表方案最后效果不好，不是“不会分片”，而是：

- 分片键选错了

## 3. 面试官为什么特别爱问

因为这个问题最能区分：

- 你是真的理解分库分表
- 还是只会背“按 ID 取模”

面试官问分片键，本质上是在看你有没有意识到：

- 数据分布
- 查询模型
- 热点风险
- 迁移扩容

这几个维度是绑在一起的。

## 4. 常见候选字段有哪些

高频候选通常包括：

- `user_id`
- `order_id`
- `merchant_id`
- `tenant_id`
- 时间维度字段

但注意：

- 常见不等于一定合适

## 5. 选分片键最核心的 4 个标准

一个比较稳的回答通常会围绕这几个点：

1. 数据分布尽量均匀
2. 查询尽量能命中单分片
3. 热点风险可控
4. 后续扩容不要太痛苦

## 6. 数据分布尽量均匀怎么理解

这是最基础的一条。

如果某个字段天然就会导致：

- 某些值数据特别大
- 某些值访问特别集中

那它就不一定是一个很稳的分片键。

比如：

- 按省份分片

就很可能出现：

- 大省份数据量远高于小省份

这就会天然带来倾斜。

## 7. 查询尽量命中单分片怎么理解

这是很多人最容易忽略的一条。

如果系统经常按某个字段查数据，那这个字段最好和分片策略关系比较强。

否则后面就会出现：

- 查询条件不是分片键
- 系统无法直接路由
- 只能全路由扫多个分片

所以更稳的说法是：

- 分片键不只是看“好不好分”，还要看“好不好查”

## 8. 热点风险可控怎么理解

有些字段虽然查起来方便，但热点太集中。

比如：

- 某个超级大商户
- 某个活跃租户
- 某个爆款商品

如果按这种字段直接分片，就可能形成热点分片。

所以分片键不能只看业务语义，还要看：

- 真实流量分布

## 9. 扩容不要太痛苦怎么理解

分片键选得再漂亮，如果后续一扩容就大迁移，那也不算好设计。

所以分片键和分片算法一起考虑时，必须想到：

- 未来节点变多怎么办
- 迁移代价高不高

这也是为什么很多面试官会把分片键和扩容问题连着问。

## 10. `user_id` 为什么常被拿来做分片键

因为它通常有几个优点：

- 数据量大
- 业务经常按用户维度查询
- 路由逻辑比较稳定

但它也不是绝对完美。

如果某些超级活跃用户请求量特别高，依然可能有热点问题。

## 11. `order_id` 为什么常被拿来做分片键

因为订单数据本身通常量大，而且：

- 全局唯一
- 路由容易
- 趋势上更容易和分布式 ID 结合

但它的弱点是：

- 如果业务高频按 `user_id` 查订单
- 那么只按 `order_id` 分片，查询不一定最顺

所以这时候就要看：

- 你的主查询路径到底是什么

## 12. 时间字段为什么容易出问题

按时间分片很直观，比如：

- 按月分表
- 按天分片

它的优点是：

- 好管理
- 好归档
- 冷热数据清晰

但风险也明显：

- 新数据会集中打到最新分片
- 最新分片特别热

所以时间维度分片很常见，但也非常容易被继续追热点问题。

## 13. 什么样的字段通常不适合直接做分片键

比较典型的包括：

- 分布天然不均的字段
- 高频查询却不稳定的字段
- 后续可能大规模变化的字段
- 业务上容易形成超级热点的字段

面试里你不一定要给特别多例子，但要体现出：

- 不是所有业务字段都值得直接拿来分片

## 14. 面试里怎么回答“分片键怎么选”

一个比较稳的口径是：

> 选分片键时，我会先看这个字段能不能让数据尽量均匀分布，再看业务高频查询是不是能尽量命中单分片，同时还要考虑热点风险和未来扩容代价。也就是说，分片键不是只看“好不好拆”，还要看“好不好查、会不会热、后面能不能扩”。像 `user_id`、`order_id` 这种字段经常会被选中，就是因为它们通常更稳定，也更容易和主查询路径结合。

## 15. 面试里最容易犯的错误

1. 一上来就答“按 ID 取模”结束
2. 只看数据量，不看查询路径
3. 不考虑热点风险
4. 不考虑扩容迁移代价
5. 把分片键讲成纯数据库字段选择问题

## 16. 一个比较稳的口语版回答

> 分片键是分库分表里最关键的设计点之一，因为它直接决定数据分布、查询路由和后续扩容成本。选分片键时，我一般会同时看四个点：第一是数据尽量均匀，不要天然倾斜；第二是业务高频查询最好能尽量命中单分片；第三是要考虑热点风险，比如某些用户、商户或时间段会不会特别集中；第四是未来扩容不要太痛苦。像 `user_id`、`order_id` 这类字段之所以经常被选中，就是因为它们通常更稳定，也更容易和主查询路径结合。这个问题真正考的不是你会不会说出一个字段名，而是你知不知道分片键背后其实是在做分布、查询和扩展性的平衡。

## 17. 一句话收尾

> 分片键题真正想考的，不是“你选哪个字段”，而是“你知不知道这个字段会把系统带向什么样的分布、查询路径和扩容代价”。
