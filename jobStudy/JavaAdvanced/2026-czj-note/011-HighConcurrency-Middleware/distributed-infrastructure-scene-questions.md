# 常见分布式基础设施场景题

## 1. 这篇文档解决什么问题

前面这几篇如果分开看，你会觉得每一篇都懂一点：

- MQ 削峰
- 分布式锁
- 分布式 ID
- 分库分表
- 分库分表后的一致性治理

但真实面试里，面试官不一定按目录顺序问你。

更常见的是直接丢一个场景：

- 你怎么设计秒杀系统
- 订单系统为什么要用分布式 ID
- 分库分表之后怎么保证一致性
- 库存超卖怎么处理
- 热点数据和热点分片怎么处理

所以这一篇的重点不是再讲单点知识，而是把这些基础设施能力串成：

> 可以直接拿来讲的场景题回答路径。

## 2. 为什么场景题最容易暴露问题

因为单点知识题很多人都能背：

- 雪花算法是什么
- Redis 锁怎么加
- MQ 为什么能削峰

但一旦放进场景里，马上就会暴露你是不是真的能串起来。

比如一个下单链路，面试官可能连续追：

1. 流量高怎么办
2. 库存怎么扣
3. 库存超卖怎么防
4. 下单和扣库存不在一个库怎么办
5. 订单号怎么生成
6. 分库分表后怎么查

所以场景题的本质是：

- 看你会不会把多个技术点拼成完整链路

## 3. 做场景题时先不要急着报技术名词

最容易犯的错误是：

- 上 Redis
- 上 MQ
- 上分库分表
- 上分布式锁

这些不一定错，但如果一上来就报方案，会显得你没有先判断问题。

更稳的回答方式应该是先看：

- 瓶颈在哪
- 正确性要求多高
- 哪些链路必须同步
- 哪些链路可以异步
- 哪些地方会有热点

然后再决定：

- 用什么基础设施

## 4. 秒杀系统为什么是经典中的经典

因为它几乎把高并发面试的核心问题都压在一起了：

- 瞬时流量很高
- 热点极其集中
- 库存不能超卖
- 数据库很容易被打爆
- 用户还要求响应快

所以你只要能把秒杀系统讲顺，很多高并发基础设施问题就都能顺带覆盖。

## 5. 秒杀系统的稳妥回答主线

可以先按这条线讲：

1. 前端和网关先做限流、防刷、验证码
2. 商品信息和库存状态尽量缓存化
3. 高峰请求先进缓冲层，必要时通过 MQ 削峰
4. 扣库存时要保证正确性，避免超卖
5. 订单、库存、支付等链路拆分后要考虑一致性
6. 非核心动作异步化
7. 最后补监控、补偿、降级和对账

这个回答的好处是：

- 不会一上来就只背一个技术点
- 会显得你在按链路做系统设计

## 6. 场景一：库存超卖怎么处理

这是最经典的继续追问。

面试官问这个，本质上是在看你知不知道：

- 高并发下正确性优先

常见思路包括：

- 数据库条件更新
- Redis 预扣减
- 分布式锁辅助控制
- 状态校验和幂等

回答时要注意：

- 不要只说“加锁”

因为“加锁”只是手段，不是完整答案。

更稳的说法是：

> 库存扣减首先要保证原子性和正确性，可以通过数据库条件更新或缓存预扣减做第一层控制。如果是多实例环境，还要考虑并发互斥和重复请求问题，必要时再结合分布式锁、幂等和补偿机制。

## 7. 场景二：为什么这里会想到 MQ 削峰

如果所有秒杀请求都直接打数据库，最容易出现：

- 数据库连接打满
- 行锁竞争严重
- 服务线程被占满

这时候 MQ 的价值就很清晰：

- 先把瞬时流量缓冲下来
- 后端按承受能力慢慢消费

但回答不能只停在这里，你最好主动补一句：

- MQ 只是削峰，不是让问题消失

因为面试官下一句很可能就会问：

- 那消息积压怎么办
- 消费失败怎么办

## 8. 场景三：为什么订单号通常不用数据库自增

这个问题非常常见，尤其是在讲分库分表时。

原因通常有这几个：

- 多库多表之后自增主键不好统一
- 需要全局唯一 ID
- 最好还能趋势递增
- 便于路由、关联和排障

所以订单号场景里很自然会引出：

- 雪花算法
- 号段模式
- 统一发号服务

你可以把它答成：

> 订单系统通常是高并发核心链路，而且后续很可能做分库分表，所以一般不会长期依赖单库自增主键，而是更偏向分布式 ID 方案，这样才能兼顾全局唯一、扩展性和路由能力。

## 9. 场景四：分库分表为什么常出现在订单系统

因为订单类业务天然具备几个特点：

- 数据量大
- 写入频繁
- 查询也多
- 生命周期长

这就很容易让单表变大、索引变重、单库压力上升。

所以订单系统常常是最容易被问：

- 什么时候考虑分库分表

回答时要强调：

- 不是业务一大就立刻拆
- 通常是先做 SQL/索引优化、缓存、读写分离，再考虑拆

## 10. 场景五：分库分表后订单和库存不在一个库怎么办

这是典型的一致性题。

它本质上在问：

- 本地事务边界被打破后，系统怎么收口

更稳的思路通常是：

1. 先区分核心和非核心动作
2. 核心链路考虑更严格的一致性策略
3. 更多互联网场景下采用最终一致
4. 通过 MQ、本地消息表、重试、补偿、幂等、对账兜底

你不要只回答：

- 用分布式事务

因为这种回答太空。

## 11. 场景六：为什么分布式锁不是万能答案

很多人一紧张就会说：

- 那就加分布式锁

但分布式锁能解决的是：

- 多实例并发互斥

它不能直接解决：

- 跨库事务
- 消息可靠送达
- 最终一致收口

所以如果面试官问：

- 库存超卖怎么处理

你可以提分布式锁，但不能把它说成总方案。

更准确的表达是：

- 分布式锁主要用于并发控制，不是完整一致性方案

## 12. 场景七：热点数据和热点分片怎么处理

这是分库分表后很容易继续追的问题。

比如：

- 某个爆款商品访问量特别高
- 某个大商户数据量特别大
- 某个用户维度的数据特别集中

这时候即使做了分片，也可能出现：

- 个别分片特别热

应对思路通常包括：

- 更合理的分片键
- 缓存前置
- 热点隔离
- 局部扩容
- 读写分离

回答时要体现一个意识：

- 分片不代表天然均匀

## 13. 场景八：扩容为什么是难点

很多人会说：

- 先按 `id % 4` 拆

但面试官继续追：

- 后面从 4 个分片扩到 8 个怎么办

这时候就能看出你是否真正理解分片代价。

因为扩容通常意味着：

- 路由规则变化
- 数据迁移
- 灰度切换
- 双写或停机窗口

所以更稳的回答是：

- 分库分表设计时就要考虑未来扩容，不然简单取模后期会非常痛苦

## 14. 场景九：下单成功但发券失败怎么办

这个场景非常适合考你是否分得清核心和非核心链路。

一般来说：

- 创建订单更核心
- 发券通常相对后置

所以更自然的治理方式是：

- 订单主流程先保证成功
- 发券走异步
- 发券失败后重试、补偿或人工介入

你如果把所有事情都强绑在一个强一致大事务里，通常不是最优解。

## 15. 场景十：消息重复消费怎么办

只要用了 MQ，这题几乎迟早会被问。

核心不是“能不能完全不重复”，而是：

- 重复了以后系统能不能稳住

常见思路包括：

- 幂等表
- 业务唯一单号
- 状态机控制
- 唯一索引去重

最关键的口径是：

- 重复消费不可怕，没有幂等才可怕

## 16. 场景十一：为什么很多场景都绕不开对账

因为你只要用了：

- 异步
- 重试
- 补偿
- 分库

就意味着系统里可能出现短时间中间态。

这时候单靠“理论上应该成功”是不够的。

所以很多核心业务都需要：

- 定时对账

它的价值是：

- 把那些链路上漏掉的异常单捞出来

## 17. 场景十二：如果你来讲一个完整下单链路

一个比较稳的回答路径可以是：

1. 用户请求先过限流和防刷
2. 商品详情和静态信息尽量走缓存
3. 高峰请求必要时通过 MQ 削峰
4. 下单主流程里先保证核心业务落库
5. 订单号采用分布式 ID
6. 数据量大后订单系统再考虑分库分表
7. 分库后通过最终一致方案处理订单、库存、优惠等协同
8. 用幂等、补偿、对账和监控把尾部风险收口

这条链路一旦讲顺，说明你不是只会背单点知识。

## 18. 场景题里最容易犯的错误

1. 一上来就报中间件名词，不先讲问题和瓶颈
2. 只说性能，不说正确性和一致性
3. 只说分布式锁，不说幂等和补偿
4. 只说 MQ，不说消息丢失、重复和积压
5. 只说分库分表，不说查询、扩容和事务代价
6. 不区分核心链路和非核心链路

## 19. 面试里怎么把场景题讲得更像做过项目

一个很重要的细节是：

- 先分层
- 再分主次
- 再讲治理手段

比如你不要一上来就说：

- 我们用了 Redis、MQ、分库分表、雪花算法

更好的表达是：

- 先说业务瓶颈是什么
- 再说哪个环节需要保护
- 再说为什么选这个中间件
- 最后补边界和代价

这样面试官会更容易觉得你是真的在做取舍，不是在背题库。

## 20. 一个比较稳的口语版回答

> 分布式基础设施场景题本质上不是在考单个技术点，而是在看你能不能把高并发治理、并发控制、全局 ID、数据分片和一致性治理串成完整链路。比如秒杀或下单场景里，先要看流量和瓶颈，再决定要不要做限流、缓存和 MQ 削峰；库存扣减要优先保证正确性，必要时结合原子更新、幂等和分布式锁；订单号通常要用分布式 ID，后面数据规模上来才会考虑分库分表；一旦分库之后，本地事务边界被打破，就要用 MQ、本地消息表、补偿、对账这些方式做最终一致。真正重要的不是报了多少中间件名字，而是每个手段解决了什么问题、又引入了什么复杂度。

## 21. 一句话收尾

> 场景题真正想看的，不是你记住了多少名词，而是你能不能把“流量、锁、ID、分片、事务、一致性”讲成一条有取舍的完整链路。
