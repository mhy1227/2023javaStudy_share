# 分库分表

## 1. 这篇文档解决什么问题

前面已经补了：

- 高并发系统设计
- 分布式 ID

那下一步非常自然就会追到：

- 数据量上来了怎么办
- 单库单表为什么会扛不住
- 分库分表到底是解决什么问题
- 是先分库还是先分表
- 分完以后查询、事务、扩容是不是会更麻烦

所以这一篇的重点不是背框架名词，而是讲清楚：

> 为什么会走到分库分表，怎么拆，拆完会带来什么收益和代价，面试里怎么讲得像真正做过架构取舍。

## 2. 为什么会需要分库分表

很多系统一开始都不会直接做分库分表。

因为在早期阶段：

- 单库单表最简单
- SQL 最直接
- 事务边界最清晰
- 运维成本也最低

但业务量一旦上来，单库单表就会慢慢出现这些问题：

- 单表数据量太大
- 索引体积越来越大
- 查询和写入变慢
- 数据库连接数和 CPU 压力增大
- 单库容量、IO、磁盘都越来越吃紧

这时候你会发现：

> 问题不一定是 SQL 不会写，而是数据规模已经逼近单机数据库的承载边界。

## 3. 分库分表本质上在解决什么

可以先记一句话：

> 分库分表本质上是在做数据水平拆分，把原来集中在一个库或一张表上的压力，分散到多个物理节点或多个表上。

它主要解决的是：

- 存储容量问题
- 单表性能问题
- 高并发写入压力
- 单库热点压力

注意这里要说准：

- 分库分表主要解决的是容量和吞吐
- 它不是银弹
- 也不会自动解决所有慢查询问题

## 4. 什么情况下才值得考虑分库分表

面试里不要回答成：

- 数据一大就分库分表

更稳的说法是先看边界。

通常要先经历这些优化：

- 索引优化
- SQL 优化
- 缓存
- 读写分离
- 归档冷数据
- 业务层限流和削峰

如果这些都做了，还是遇到：

- 单表太大
- 写入持续高压
- 扩容空间越来越小
- 数据库已经成为核心瓶颈

这时候才更像分库分表的信号。

也就是说：

> 分库分表通常是容量和性能治理的后手，而不是第一手。

## 5. 先区分几种常见拆法

面试里最容易混的是：

- 垂直拆分
- 水平拆分
- 分库
- 分表

这几个词最好分清。

### 5.1 面试里为什么经常顺带问横向和纵向

因为“分库分表”这个词本身比较宽。

面试官追问横向和纵向，通常是在看你有没有分清：

- 是按业务模块拆
- 还是按数据行拆
- 是先解决业务边界问题
- 还是先解决单表数据量和单库压力问题

所以这里不要只答：

- 一个横着拆，一个竖着拆

更稳的口径是：

> 纵向拆分更偏按业务领域或字段职责拆，比如用户库、订单库、商品库；横向拆分更偏同一类数据量太大，把同样结构的数据按某个分片键拆到多个库表里。真正面试里说分库分表，更多是在追水平拆分，也就是横向拆分，但垂直拆分通常会作为前置架构拆分一起问。

你可以简单记成：

- 纵向拆分：按业务拆，解决职责和业务边界
- 横向拆分：按数据拆，解决容量和吞吐压力

## 6. 垂直拆分是什么

垂直拆分更像是按业务维度拆。

比如原来一个库里什么都有：

- 用户
- 订单
- 商品
- 支付

后面按业务域拆成：

- 用户库
- 订单库
- 商品库
- 支付库

它的特点是：

- 按业务模块拆
- 表结构不一定变
- 更像系统边界和业务边界的拆分

垂直拆分的价值主要是：

- 降低业务耦合
- 让不同业务各自扩展
- 减少一个库承载所有业务的压力

## 7. 水平拆分是什么

水平拆分更像是：

- 同一张表数据太多
- 把同样结构的数据拆到多个表或多个库里

比如 `order` 表太大，拆成：

- `order_00`
- `order_01`
- `order_02`
- `order_03`

或者进一步拆到不同库：

- `db_order_0.order_00`
- `db_order_1.order_01`

它的本质是：

- 表结构基本一样
- 拆的是数据行

这通常才是大家口中的“分库分表”核心。

## 8. 分表和分库有什么区别

### 8.1 分表

分表通常还是在一个数据库实例里，把一张大表拆成多张表。

它主要解决：

- 单表数据量过大
- 单表索引膨胀

### 8.2 分库

分库是把数据进一步拆到不同数据库实例。

它主要解决：

- 单库连接数压力
- 单库写入瓶颈
- 单机资源上限

所以可以把它记成一句话：

- 分表偏向解决单表问题
- 分库偏向解决单机数据库整体压力问题

很多系统是先分表，再看是否需要进一步分库。

## 9. 常见的分片键怎么选

分库分表里最关键的设计点之一，就是：

- 按什么字段拆

这个字段通常叫：

- 分片键
- 路由键

常见选择包括：

- `user_id`
- `order_id`
- `merchant_id`
- `tenant_id`

选分片键时，通常要看几个点：

- 分布是否均匀
- 是否高频参与查询
- 是否能稳定落到单分片
- 会不会形成热点

## 10. 为什么分片键选错会很痛苦

如果分片键选得不好，后续常见问题会很多：

- 数据分布不均
- 某几个分片特别热
- 查询经常跨库跨表
- 路由规则很难维护

比如按省份拆，有可能出现：

- 大省份数据量极大
- 小省份数据量很小

这就会导致严重倾斜。

所以分片键本质上不是“随便挑一个业务字段”，而是：

> 要同时考虑数据分布、查询路径和未来扩容。

## 11. 常见路由算法有哪些

最常见的口径通常有：

- 取模
- 范围
- 哈希
- 一致性哈希

## 12. 取模分片怎么理解

最常见的例子就是：

- `user_id % 4`

优点是：

- 简单
- 路由快
- 分布通常比较直观

但它的问题也很经典：

- 后续扩容不方便

比如原来是 4 个分片，后面改成 8 个，很多历史数据路由规则都会变。

## 13. 范围分片怎么理解

范围分片常见于按时间或区间拆：

- 按月份分表
- 按订单号区间分库

它的优点是：

- 很直观
- 方便做冷热数据管理

但问题也很明显：

- 容易形成热点
- 新数据可能持续落在最后一个分片

所以按时间分片很常见，但也要小心“最新分片过热”。

## 14. 哈希分片怎么理解

哈希分片本质上是：

- 对分片键做哈希
- 再落到不同分片

它比简单取模更强调：

- 分布均匀性

但如果底层分片数量变化，依然会面临数据迁移问题。

## 15. 一致性哈希为什么会被提到

一致性哈希更多在：

- 节点动态扩缩容

这种场景里被提到。

它的价值是：

- 扩容时减少数据迁移量

但在传统数据库分库分表里，不一定是第一选择。

面试里知道它的核心价值就够了，不必强行说所有数据库分片都用它。

## 16. 分库分表之后会带来哪些代价

这是面试里非常关键的部分。

因为很多人只会说收益，不会说副作用。

最常见的代价包括：

- 跨库跨表查询更复杂
- `join` 变麻烦
- 分页更麻烦
- 排序和聚合更麻烦
- 全局事务更复杂
- 扩容和数据迁移成本更高
- 运维和排障难度上升

所以回答时一定要体现：

> 分库分表不是“优化一下就好了”，而是拿复杂度换容量和吞吐。

## 17. 为什么跨库查询会麻烦

因为原来你只查一张表，现在可能要查：

- 多张分表
- 多个分库

这就意味着：

- SQL 可能要改写
- 中间件可能要帮你路由和聚合
- 应用层可能要自己合并结果

尤其是：

- 跨库 `join`
- 全局排序
- 全局分页

这些能力在分库分表后都会变得更贵。

## 18. 分页和排序为什么容易出问题

比如你原来写：

- `order by create_time desc limit 20`

在单表里很自然。

但分成 8 张表后，问题就变成：

- 每张表先取多少
- 最后怎么合并
- 怎么保证全局有序

所以分库分表后很多分页查询不会再完全依赖传统 offset 分页，而会更偏向：

- 游标分页
- 按主键或时间范围翻页

## 19. 为什么分库分表会让事务更复杂

原来单库事务很直接：

- 一个本地事务就够了

但分库之后，一次业务操作可能涉及多个库：

- 扣库存一个库
- 写订单另一个库
- 写流水又一个库

这时候本地事务就不够了。

你就会面临：

- 分布式事务
- 最终一致性
- 补偿机制

所以分库分表经常和这些话题一起出现。

## 20. 分库分表和分布式 ID 为什么经常一起问

因为一旦拆库拆表：

- 每个库自己的自增主键就不够用了
- 你需要统一主键体系
- 路由、关联、排障都更依赖全局唯一 ID

所以分库分表之后，通常更需要：

- 雪花算法
- 号段模式
- 统一发号服务

这就是前一篇分布式 ID 文档和这一篇能自然接起来的原因。

## 21. 工程上一般怎么落地

现实里常见的做法通常不是“全手写 SQL 路由”，而是：

- 用分库分表中间件
- 在应用层定义路由规则
- 统一分片键和主键策略
- 做迁移、扩容、双写或灰度切换

也就是说，真正的重点通常不是：

- 会不会写 `% 8`

而是：

- 你能不能把拆分规则、查询路径、迁移方式和一致性问题一起考虑清楚

## 22. 扩容为什么是高风险点

很多人一开始只想到“先拆了再说”，但真正难的是后续扩容。

因为扩容通常意味着：

- 新增分片
- 迁移历史数据
- 更新路由规则
- 保障迁移期间业务可用

如果最初分片策略设计得太死，比如简单固定取模，后续扩容就会很痛。

所以面试里如果你主动提：

- 分片策略要考虑未来扩容

会比只会说“按 ID 取模”更稳。

## 23. 什么时候不建议急着上分库分表

这点也很重要。

如果系统现在只是：

- 索引没建好
- SQL 很烂
- 缓存没用起来
- 读写分离也没做

那这时候急着上分库分表，往往是：

- 用最重的方案解决最基础的问题

所以更合理的顺序通常是：

1. 先做 SQL 和索引优化
2. 再做缓存和读写分离
3. 再看冷热分层和归档
4. 最后才考虑分库分表

## 24. 面试里怎么把分库分表这题讲顺

可以按这条线回答：

1. 先说单库单表什么时候会遇到瓶颈
2. 再说分库分表本质上是在做数据水平拆分
3. 再区分垂直拆分、水平拆分、分库、分表
4. 再说分片键和路由规则怎么选
5. 最后主动补它带来的代价，比如跨库查询、分页、事务、扩容复杂度

这样会比只说“数据量大了就分库分表”成熟很多。

## 25. 一个比较稳的口语版回答

> 分库分表一般是在单库单表已经接近容量或性能瓶颈时才考虑，它本质上是把原来集中在一个库或一张表上的数据和压力拆散。垂直拆分更偏按业务模块拆，水平拆分更偏把同样结构的数据分到多个库表里。真正做分库分表时，最关键的是分片键设计，因为它会直接影响数据是否均匀、查询是否容易命中单分片，以及后续扩容是否痛苦。分库分表能提升容量和吞吐，但代价也很明显，比如跨库 join、分页排序、聚合、事务一致性和扩容迁移都会变复杂。所以它不是第一选择，通常是在索引优化、缓存、读写分离这些手段之后，才会上的重方案。另外分库分表之后通常也要配合分布式 ID 方案，不然全局主键、路由和数据关联都会变得很混乱。

## 26. 一句话收尾

> 分库分表真正考的不是“会不会拆”，而是你知不知道为什么拆、按什么拆、拆完以后会多出哪些复杂度。
