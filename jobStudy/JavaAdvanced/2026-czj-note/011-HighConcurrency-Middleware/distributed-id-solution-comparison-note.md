# 分布式 ID 方案对比

## 1. 这篇文档解决什么问题

前一篇已经单独讲过雪花算法。

但真实面试里，面试官很少只问：

- 雪花算法是什么

更常见的追问是：

- 为什么你们不用数据库自增
- UUID、Redis 发号、雪花算法怎么比较
- 号段模式是什么
- 分库分表之后为什么更需要统一 ID 方案
- 业务里到底该怎么选

所以这一篇重点不是讲某一个算法细节，而是讲：

> 分布式 ID 这件事，常见方案有哪些，各自优缺点是什么，面试里怎么做工程化取舍。

## 2. 为什么分布式系统一定会遇到 ID 问题

单体系统里，很多时候一张表一个自增主键就结束了。

但系统一旦往下面这些方向走：

- 服务多实例部署
- 分库分表
- 多机房
- 高并发写入
- 订单、支付、消息、流水等核心业务

ID 问题就会从“数据库顺手生成一下”变成一个基础设施问题。

这时候通常会同时关注几个目标：

- 全局唯一
- 最好趋势递增
- 生成性能高
- 不要强依赖单点
- 最好对数据库索引友好

## 3. 常见方案大概有哪些

面试里最常见的分布式 ID 方案可以先分成这几类：

- 数据库自增
- UUID
- Redis 发号
- 数据库号段模式
- 雪花算法

你不用把每种方案都讲成源码分析题，但至少要知道：

- 它解决什么问题
- 它为什么不一定适合所有场景
- 为什么很多系统最后会走到雪花算法或号段模式

## 4. 数据库自增

### 4.1 它的优点

数据库自增最大的优点是简单。

它通常具备这些特点：

- 实现成本低
- 天然递增
- 业务方几乎不用操心生成逻辑

如果系统还处在：

- 单体
- 单库单表
- 写入量不大

那数据库自增完全够用。

### 4.2 它的问题

但它的问题也很明显：

- 发号能力依赖数据库
- 容易把数据库变成单点
- 分库分表后全局唯一不好统一
- 业务和存储耦合比较重

所以数据库自增不是“错”，而是它的适用边界比较靠前。

## 5. UUID

### 5.1 它的优点

UUID 的优点很直接：

- 本地就能生成
- 不依赖中心节点
- 唯一性通常够用

### 5.2 它的问题

UUID 最大的问题不在唯一性，而在工程体验：

- 太长
- 可读性差
- 作为数据库主键不够友好
- 一般不具备明显递增趋势

如果把 UUID 当主键，常见风险是：

- 索引更分散
- 页分裂压力更大
- 排序和排查不直观

所以 UUID 更像是：

- 唯一性很好
- 但不一定适合作为高并发核心业务表主键

## 6. Redis 发号

Redis 发号最常见的做法就是：

- 用 `INCR`
- 或者按业务维度做 `INCRBY`

### 6.1 它的优点

它的优点是：

- 简单
- 全局递增口径清晰
- 性能通常比数据库发号更高

### 6.2 它的问题

但它本质上还是中心化发号：

- 发号能力依赖 Redis
- Redis 出问题会影响发号
- 高可用、持久化、主从切换都要考虑

所以 Redis 发号更适合：

- 需要统一单调递增序列
- 可接受中心化发号
- 对运维稳定性有把控

它不太像“完全去中心化方案”。

## 7. 数据库号段模式

这一类方案面试里也很常见，尤其适合从工程视角拉开层次。

它的大致思路是：

- 不是每来一条数据都去数据库拿一个 ID
- 而是一次从数据库申请一段号段
- 应用在本地内存里把这段号慢慢发完

比如一次申请：

- `10001 ~ 20000`

那应用后续在本地就可以直接发这一段里的 ID。

### 7.1 它的优点

- 比逐条数据库自增性能高很多
- 仍然比较容易理解和管理
- 中心发号压力大幅下降

### 7.2 它的问题

- 本质上仍然依赖中心号段分配
- 要处理号段预加载
- 要处理实例重启后号段浪费
- 不一定天然带机器维度语义

面试里提到号段模式，会显得你知道：

- 不是所有系统都直接用雪花算法
- 很多公司会在“中心化发号”和“本地高性能”之间做折中

## 8. 雪花算法

雪花算法前一篇已经单独讲过，这里只做对比口径总结。

### 8.1 它的优点

- 本地生成
- 性能高
- 分布式多节点友好
- 通常是 `long` 型
- 趋势递增

### 8.2 它的问题

- 依赖机器号配置
- 依赖时间戳
- 需要处理时钟回拨
- 单毫秒内序列号存在上限

所以雪花算法不是“完美方案”，而是：

> 在高并发分布式系统里，一个综合性很强、很常见、性价比很高的方案。

## 9. 从几个维度横向比较

如果面试官让你比较，可以从下面几个维度讲。

### 9.1 唯一性

- 数据库自增：单库内好保证，全局不一定
- UUID：唯一性强
- Redis 发号：统一中心下好保证
- 号段模式：依赖号段分配逻辑
- 雪花算法：依赖时间、机器号、序列组合

### 9.2 递增性

- 数据库自增：最好
- Redis 发号：也比较直接
- 号段模式：通常也能保持递增趋势
- 雪花算法：大致递增
- UUID：通常最差

### 9.3 性能

- 数据库自增：相对一般
- UUID：本地生成快
- Redis 发号：高，但依赖远程调用
- 号段模式：高
- 雪花算法：通常最高的一档

### 9.4 架构依赖

- 数据库自增：依赖数据库
- Redis 发号：依赖 Redis
- 号段模式：依赖号段中心
- 雪花算法：依赖机器号和时钟
- UUID：依赖最少

### 9.5 数据库索引友好性

- 数据库自增：好
- Redis 发号：好
- 号段模式：好
- 雪花算法：通常较好
- UUID：通常最不友好

## 10. 分库分表之后为什么更需要统一 ID

这其实是面试里一个非常顺的追问点。

一旦分库分表：

- 每个库自己自增会重复
- 订单、支付、履约等链路需要全局唯一标识
- 数据汇总、路由、排查都需要统一主键体系

这时候如果没有统一 ID：

- 跨库数据不好关联
- 很多表主键策略会混乱
- 排查问题时也不方便

所以分库分表和分布式 ID 经常是成对出现的话题。

## 11. 分库分表和 ID 方案之间是什么关系

可以把它理解成一句话：

> 分库分表解决的是数据怎么拆，分布式 ID 解决的是拆完以后主键怎么还能全局统一。

两者经常一起出现，原因是：

- 数据量大才会考虑拆库拆表
- 数据量大时通常写入也多
- 写入一多，主键生成方案就不能太脆弱

所以如果你后面继续补“分库分表”文档，这里正好是前置知识。

## 12. 工程上到底怎么选

面试里不要回答成：

- 只有雪花算法能用

更稳的说法是按场景选。

### 12.1 系统简单、规模不大

可以用：

- 数据库自增

因为实现成本最低。

### 12.2 只要求唯一，不太关心递增和索引体验

可以考虑：

- UUID

### 12.3 想要统一中心发号，而且逻辑简单直观

可以考虑：

- Redis 发号

### 12.4 想减少中心压力，又希望统一管理

可以考虑：

- 号段模式

### 12.5 多节点高并发，强调性能和分布式扩展

通常更适合：

- 雪花算法

## 13. 面试里怎么把“方案对比题”讲顺

一个比较稳的回答路径是：

1. 先说为什么分布式系统需要统一 ID
2. 再说数据库自增在单体里够用，但扩展性有限
3. 再说 UUID 唯一性强，但不够适合主键索引场景
4. 再说 Redis 发号和号段模式属于中心化或半中心化思路
5. 最后再说雪花算法更适合高并发分布式场景，但也要处理时钟回拨和机器号配置

这会比只背结论更像真正做过架构取舍。

## 14. 一个比较稳的口语版回答

> 分布式 ID 方案常见有数据库自增、UUID、Redis 发号、号段模式和雪花算法。数据库自增实现最简单，但更适合单库单表；UUID 唯一性强，但太长而且对索引不够友好；Redis 发号能做到全局递增，但它本质上还是中心化发号；号段模式是一种折中方案，先从数据库批量申请号段，再在本地发号，性能比逐次申请好很多。雪花算法则更适合多节点高并发场景，它把时间戳、机器号和序列号拼成一个 long 型 ID，能做到本地高性能生成和大致递增，但要注意时钟回拨和机器号配置问题。实际怎么选，要看系统规模、是否分库分表、是否追求递增以及能不能接受中心化依赖。

## 15. 一句话收尾

> 分布式 ID 题真正想考的，不是你会背几种方案名字，而是你能不能说清楚每种方案的适用场景、代价和工程取舍。
