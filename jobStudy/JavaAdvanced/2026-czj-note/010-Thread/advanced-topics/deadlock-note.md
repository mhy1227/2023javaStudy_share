# 死锁与排查说明

## 1. 为什么死锁值得单独写

因为线程这一章讲到锁之后，面试里非常容易继续追问：

- 什么是死锁？
- 怎么避免？
- 真出了问题怎么排查？

也就是说，死锁并不是单纯的概念题，它通常会从：

- 定义
- 原因
- 避免
- 排查

一路往下问。

## 2. 什么是死锁

可以先给一个容易口述的定义：

- 多个线程互相等待对方释放资源，结果谁都走不下去，这种状态就叫死锁

更直白一点说：

- A 等 B
- B 等 A
- 最后都卡住了

## 3. 一个最经典的死锁例子

例如：

- 线程 1 先拿锁 A，再等锁 B
- 线程 2 先拿锁 B，再等锁 A

这时如果两边都不释放，就会卡死。

这也是为什么面试里经常会问：

- 多把锁为什么危险

### 3.1 一个简单代码示意

```java
Object lockA = new Object();
Object lockB = new Object();

Thread t1 = new Thread(() -> {
    synchronized (lockA) {
        synchronized (lockB) {
            System.out.println("t1");
        }
    }
});

Thread t2 = new Thread(() -> {
    synchronized (lockB) {
        synchronized (lockA) {
            System.out.println("t2");
        }
    }
});
```

这个例子里，如果 `t1` 先拿到 `lockA`，`t2` 先拿到 `lockB`，后面就可能互相等待。

## 4. 死锁的四个必要条件

这是高频必问。

通常会讲这四个：

1. 互斥条件
2. 请求与保持条件
3. 不可剥夺条件
4. 循环等待条件

### 4.1 互斥条件

资源同一时刻只能被一个线程占用。

### 4.2 请求与保持条件

线程在已经持有部分资源的同时，又去请求新的资源。

### 4.3 不可剥夺条件

线程拿到的资源，在没有用完前不能被强行夺走。

### 4.4 循环等待条件

多个线程形成首尾相接的等待环。

## 5. 面试里这四个条件怎么用

不一定要一字不差地背定义，但你至少要知道：

- 死锁不是随机发生的
- 它通常和多把锁、持锁等待、顺序不一致有关

## 6. 怎么避免死锁

这是比“什么是死锁”更常见的第二问。

最常见的回答思路：

1. 固定加锁顺序
2. 尽量减少同时持有多把锁
3. 缩小锁粒度
4. 在复杂场景下考虑 `tryLock()`
5. 尽量避免嵌套锁设计过深

## 7. 为什么固定加锁顺序很重要

因为很多死锁就是这么来的：

- 线程 A 先拿锁 1，再拿锁 2
- 线程 B 先拿锁 2，再拿锁 1

如果统一成：

- 所有线程都先拿锁 1，再拿锁 2

死锁风险就会明显下降。

## 8. `tryLock()` 为什么能帮助降低死锁风险

因为它不是一直死等。

如果拿不到锁，可以：

- 放弃
- 重试
- 走降级逻辑

所以在复杂场景下，它比一味阻塞等待更灵活。

## 9. 死锁和阻塞是一回事吗

不是。

阻塞更宽泛，可能只是：

- 当前拿不到锁，先等一下

而死锁是：

- 多方互相等待，形成闭环，谁都走不下去

所以：

- 阻塞不一定是死锁
- 死锁一定是一种严重阻塞

## 10. 死锁、活锁、饥饿要不要区分

如果是基础面试，不一定每次都会问到，但知道这几个词的区别会加分。

### 10.1 死锁

- 大家都卡住，谁也走不动

### 10.2 活锁

- 线程没有卡死，但一直互相让来让去，结果谁都做不成事

### 10.3 饥饿

- 某些线程一直拿不到执行机会或资源

当前阶段你不需要展开很深，但至少别把它们混成一个词。

## 11. 怎么排查死锁

这也是面试里很值得补的一点。

当前阶段不需要讲太复杂，但至少可以知道：

1. 看程序现象：线程长期卡住不动
2. 看线程栈信息
3. 看锁等待关系
4. 借助工具排查

## 12. Java 里常见的排查思路

比较常见的思路有：

1. `jstack`
2. 线程 dump
3. 结合日志和线程名分析

### 12.1 `jstack` 这件事要知道到什么程度

当前阶段不要求你把命令参数背得很细，但至少要知道：

- `jstack` 可以看线程栈
- 线程栈里可以看到线程卡在什么锁上
- 如果是典型死锁，工具通常会给出比较明确的提示

面试里说到这一步，通常已经够用了。

面试里如果被问“线上怎么查”，可以先稳妥回答：

- 先看线程状态和线程栈，确认线程是不是互相等待对方持有的锁，再结合日志和线程名定位业务代码位置

## 13. 排查时应该重点看什么

可以先按这条线记：

1. 哪些线程长时间不动
2. 它们分别在等哪把锁
3. 锁现在又被哪个线程持有
4. 是否形成了闭环等待
5. 对应到哪段业务代码

这比单纯说“我会看日志”更像真正排查过问题的人。

## 14. 面试里最容易漏的点

### 12.1 只会说“两个线程互相卡住”

这太简略了。

至少要补一句：

- 本质是循环等待资源

### 12.2 只会背四个必要条件

但说不清怎么避免。

一定要补：

- 固定加锁顺序

### 12.3 只会说“可以查日志”

还要补：

- 看线程栈
- 看锁等待关系

### 12.4 只会说死锁，不会说和普通阻塞的区别

至少要补一句：

- 阻塞不一定是死锁，死锁强调循环等待

## 15. 面试里怎么回答“什么是死锁”

可以这样答：

死锁指的是多个线程互相等待对方释放资源，导致所有线程都无法继续执行的状态。它通常出现在多个线程以不一致顺序获取多把锁的场景里，本质上是形成了循环等待。

## 16. 面试里怎么回答“怎么避免死锁”

可以这样答：

最常见的避免思路是固定加锁顺序，尽量减少同时持有多把锁，缩小锁粒度，在复杂场景下可以考虑用 `tryLock()` 这种更灵活的方式，避免线程长时间互相等待。

## 17. 面试里怎么回答“怎么排查死锁”

可以这样答：

排查死锁时，一般先看程序现象是不是线程长时间卡住，然后抓线程栈或线程 dump，重点看线程之间是不是在互相等待对方持有的锁，再结合日志、线程名和具体业务代码去定位问题。

## 18. 当前阶段最该记住的结论

1. 死锁本质上是循环等待资源
2. 死锁和普通阻塞不是一回事
3. 固定加锁顺序是最常见的避免手段
4. `tryLock()` 在复杂场景下更灵活
5. 排查时重点看线程栈和锁等待关系

## 19. 一句话总结

死锁这一块最重要的不是死背四个条件，而是先讲清楚：它为什么发生、怎么避免，以及真出了问题大概怎么排查。
