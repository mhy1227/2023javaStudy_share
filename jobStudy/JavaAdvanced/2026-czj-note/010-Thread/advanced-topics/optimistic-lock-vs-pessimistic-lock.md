# 乐观锁与悲观锁说明

## 1. 为什么这一份要单独补

面试里问锁分类，最常被单独追着问的就是：

- 什么是乐观锁？
- 什么是悲观锁？
- 它们有什么区别？
- 分别适合什么场景？

这组问题出现频率很高，而且会直接从这里顺着追到：

- CAS
- 数据库版本号
- `synchronized`
- `ReentrantLock`

所以这一份值得单独整理。

## 2. 先一句话理解乐观锁和悲观锁

可以先记一个很稳的口径：

- 乐观锁：先假设冲突不多，先执行，提交时再检查
- 悲观锁：先假设冲突容易发生，所以先锁住再操作

更口语一点说：

- 乐观锁更像“先做，最后校验”
- 悲观锁更像“先管住，再处理”

## 3. 为什么会有这两种思路

因为并发场景下，大家面对的问题不一样。

有的场景：

- 冲突不频繁
- 大多数时候大家不会撞在一起

这时如果每次都先重锁住，可能不划算。

但有的场景：

- 共享资源非常敏感
- 冲突概率高
- 一旦冲突就不能接受错误结果

这时就更倾向先做保护。

所以乐观锁和悲观锁，本质上是在回答：

- 你到底对并发冲突有多“乐观”或者多“保守”

## 4. 什么是乐观锁

乐观锁的核心思路是：

- 不先把线程锁住
- 先尝试更新
- 提交时再检查中间有没有别人改过

如果没人改过，就成功。

如果有人改过，就失败、重试或者回滚。

所以它更强调：

- 检查更新时机

而不是：

- 先阻塞其他线程

## 5. 什么是悲观锁

悲观锁的核心思路是：

- 先假设别人很可能会来抢
- 所以先加锁保护
- 再做后续操作

它更强调：

- 先保证互斥访问

而不是到最后再检查。

所以像下面这些你都可以先粗归到悲观锁思路：

- `synchronized`
- `ReentrantLock`

## 6. 乐观锁和悲观锁最直观的区别

可以先记成三句话：

1. 乐观锁偏“先做再校验”
2. 悲观锁偏“先锁再处理”
3. 乐观锁不代表完全没成本，悲观锁也不代表一定更慢

这里第三句特别重要。

因为很多人会把它们背成：

- 乐观锁一定更高级
- 悲观锁一定更笨重

这都不准确。

## 7. 乐观锁常见例子

当前阶段最常见的例子有两个：

1. CAS
2. 数据库版本号

CAS 的思路是：

- 比较当前值和预期值

数据库版本号的思路是：

- 更新时带上版本条件

例如可以粗理解成：

- `where id = ? and version = ?`

如果版本不一致，说明别人改过，本次失败。

## 8. 悲观锁常见例子

当前阶段最常见的例子通常有：

1. `synchronized`
2. `ReentrantLock`
3. 数据库里的排他锁思路

它们的共同点都是：

- 更强调先控制访问，再执行业务

## 9. 乐观锁适合什么场景

比较适合的场景通常有：

1. 冲突概率不高
2. 读多写少
3. 单次更新比较轻量
4. 可以接受失败后重试

例如：

- 简单状态更新
- 版本号校验更新
- 一些轻量级原子操作

## 10. 悲观锁适合什么场景

比较适合的场景通常有：

1. 共享资源竞争明显
2. 冲突概率高
3. 业务逻辑复杂
4. 更强调过程中的互斥保护

例如：

- 库存扣减
- 余额修改
- 多步骤复合更新
- 共享集合复合操作

## 11. 乐观锁有什么问题

乐观锁不是没有代价。

它常见的问题包括：

1. 冲突激烈时失败重试会很多
2. 可能带来自旋开销
3. 有 ABA 这类问题需要额外处理
4. 不适合很复杂的复合业务流程

所以它更适合：

- 简单、轻量、可重试

## 12. 悲观锁有什么问题

悲观锁也不是没有代价。

它常见的问题包括：

1. 线程阻塞等待
2. 上下文切换成本
3. 并发度下降
4. 设计不好会有死锁风险

所以它虽然稳，但也会更重。

## 13. 面试里怎么回答“乐观锁和悲观锁有什么区别”

可以直接背一版：

乐观锁和悲观锁的核心区别在于对并发冲突的假设不同。乐观锁先假设冲突不多，所以通常先执行，提交时再检查有没有别人改过，CAS 和数据库版本号就是典型例子；悲观锁先假设冲突容易发生，所以会先加锁保护，再执行业务，`synchronized` 和 `ReentrantLock` 更偏这种思路。乐观锁适合轻量、冲突不高、可重试的场景，悲观锁更适合竞争明显、逻辑复杂、过程必须互斥保护的场景。

## 14. 面试里怎么回答“什么时候用乐观锁，什么时候用悲观锁”

可以背一版短的：

如果场景冲突不高、更新比较轻量、失败后可以重试，我会优先考虑乐观锁思路；如果共享资源竞争明显、业务逻辑复杂、过程中的互斥保护更重要，我会更倾向悲观锁。核心不是背概念，而是先看冲突概率和业务复杂度。

## 15. 当前阶段记忆重点

这一份最少要记住：

1. 乐观锁是先做再校验
2. 悲观锁是先锁再处理
3. CAS 和版本号常被归到乐观锁思路
4. `synchronized`、`ReentrantLock` 更偏悲观锁
5. 乐观锁适合轻量、可重试、冲突不高
6. 悲观锁适合竞争明显、过程必须互斥保护
7. 二者都不是“绝对更高级”的方案
