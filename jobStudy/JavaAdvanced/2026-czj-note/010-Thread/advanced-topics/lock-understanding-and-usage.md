# 锁的理解与使用说明

## 1. 为什么这一份要单独写

因为面试里问锁，通常不会只问你：

- 什么是 `synchronized`
- 什么是 `ReentrantLock`

更常见的问法是：

- 你怎么理解锁？
- 你什么时候会用锁？
- 常见的锁的使用方式有哪些？
- 为什么这里要加锁，不加会怎样？

也就是说，面试官想听的不是某个类名的定义，而是你有没有“锁的整体理解”。

## 2. 先用一句话理解什么是锁

可以先给一个很稳的口径：

- 锁本质上是一种并发控制手段，用来约束多个线程访问共享资源的顺序，避免临界区代码在并发下出现错误结果

更直白一点说：

- 锁就是让“本来可能乱掉的共享操作”，在必要的时候变得有秩序

## 3. 为什么需要锁

前面线程安全那份已经讲过，线程安全问题通常满足两个条件：

1. 有共享资源
2. 有多个线程并发访问或修改

如果这两个条件同时成立，就容易出问题。

例如：

- `i++`
- 卖票
- 库存扣减
- 余额修改
- 共享集合更新

这些都属于：

- 访问临界区代码

所以锁的出现，本质上是为了保护临界区。

## 4. 我怎么理解锁

如果面试官直接问：

- 你怎么理解锁？

比较合适的回答主线是：

1. 锁是解决线程安全问题的一种手段
2. 它的核心是控制多个线程进入临界区的顺序
3. 本质上是用互斥访问来减少共享资源竞争
4. 它不是目的，线程安全才是目的

这个最后一点很重要：

- 锁是手段，不是目的

## 5. 锁解决的主要是什么问题

从线程安全三要素去看，锁最核心地帮助解决的是：

1. 原子性
2. 一定的可见性
3. 一定的有序性

但如果是基础面试，优先先说：

- 锁最直观解决的是“多个线程同时进入临界区”带来的原子性问题

## 6. 什么时候需要用锁

这个问题非常常见。

比较稳的回答是：

- 当多个线程会并发访问同一份共享资源，而且这段逻辑不是单一步骤原子动作时，就要考虑加锁

典型场景：

1. 共享变量复合更新
2. 先判断再执行
3. 多步骤状态修改
4. 多线程操作同一集合
5. 库存、金额、计数器等共享业务状态更新

## 7. 什么时候不一定需要锁

这也是面试里一个加分点。

不是所有并发问题都一定先上锁。

下面这些场景不一定要用传统锁：

1. 不共享数据
2. 使用不可变对象
3. 用 `ThreadLocal` 做线程隔离
4. 简单计数用原子类
5. 只是状态标记，用 `volatile`

所以你可以顺手补一句：

- 加锁不是唯一思路，先看问题本质是什么

## 8. 常见的锁使用方式有哪些

当前阶段最值得先掌握的主要有三类：

1. `synchronized`
2. `ReentrantLock`
3. 读写锁的概念级认知

## 9. 第一类：`synchronized`

这是最基础、最经典的锁使用方式。

最常见的写法有三种：

1. 同步实例方法
2. 同步静态方法
3. 同步代码块

### 9.1 同步实例方法

```java
public synchronized void add() {
    count++;
}
```

锁的是：

- 当前对象实例 `this`

### 9.2 同步静态方法

```java
public static synchronized void add() {
    count++;
}
```

锁的是：

- 当前类对应的 `Class` 对象

### 9.3 同步代码块

```java
synchronized (lock) {
    count++;
}
```

锁的是：

- 括号里指定的对象

## 10. 第二类：`ReentrantLock`

`ReentrantLock` 是最常见的显式锁。

基础写法：

```java
private final Lock lock = new ReentrantLock();

public void add() {
    lock.lock();
    try {
        count++;
    } finally {
        lock.unlock();
    }
}
```

它和 `synchronized` 的目标类似，但控制更灵活。

最常见的面试补充点：

1. `tryLock()`
2. `lockInterruptibly()`
3. 公平锁 / 非公平锁

## 11. 第三类：读写锁

如果只是基础阶段，不需要把读写锁讲得太深，但你可以知道：

- 它把“读”和“写”分开控制

常见口径：

- 读多写少场景下，读写锁有时更合适

当前阶段知道这层意思就够，不需要展开到底层实现。

## 12. 我怎么使用锁比较合适

这是面试里很像真实开发者的问题。

一个更像实际开发的回答方向是：

1. 先判断是不是共享资源问题
2. 再判断是不是复合操作
3. 简单场景优先考虑简单方案
4. 锁范围尽量小，避免无意义扩大临界区
5. 如果用显式锁，记得 `unlock()` 放 `finally`

## 13. 常见的锁使用习惯

### 13.1 锁尽量只包住临界区

不要把无关逻辑全包进去。

因为锁范围越大：

- 并发度越低
- 线程等待越多

### 13.2 保证多个线程竞争的是同一把锁

这是高频易错点。

例如：

```java
synchronized (new Object()) {
    count++;
}
```

这种写法通常没意义，因为每次都是新对象。

### 13.3 显式锁记得释放

这就是为什么：

- `unlock()` 一般必须放 `finally`

### 13.4 尽量固定加锁顺序

这和死锁直接相关。

## 14. 常见的锁使用场景

你可以把常见业务场景和锁绑定起来：

1. 卖票 / 扣库存
   常见用 `synchronized` 或 `ReentrantLock`
2. 简单计数
   优先考虑 `AtomicInteger`
3. 停止标记
   优先考虑 `volatile`
4. 线程隔离场景
   优先考虑 `ThreadLocal`
5. 读多写少的复杂共享结构
   可以知道读写锁思路

## 15. 锁的代价也要会说

面试里如果你只说锁的好处，会显得比较浅。

锁的代价常见有：

1. 降低并发度
2. 增加等待时间
3. 容易引入死锁风险
4. 代码复杂度上升

所以更像真实开发者的说法是：

- 锁很有用，但也有成本，不能无脑加

## 16. 面试里最容易漏的点

### 16.1 只会背锁的定义，不会说为什么需要锁

要补一句：

- 因为共享资源在并发访问下会产生竞争

### 16.2 只会说“加锁保证线程安全”

还要补：

- 锁本质上是控制临界区访问顺序

### 16.3 只会说 `synchronized` 和 `ReentrantLock`

但说不清什么时候用谁

更稳的口径是：

- 简单场景 `synchronized` 往往就够
- 更复杂控制需求再考虑 `ReentrantLock`

### 16.4 以为所有并发问题都要加锁

这也不对。

要知道：

- 有些问题适合 `volatile`
- 有些适合原子类
- 有些适合线程隔离

## 17. 面试里怎么回答“你怎么理解锁”

可以这样答：

我理解锁本质上是一种并发控制手段，用来约束多个线程访问共享资源的顺序，避免临界区代码在并发环境下出现错误结果。它最核心解决的是共享资源竞争问题，本质上是通过互斥访问来帮助保证原子性，同时也能带来一定的可见性和有序性保证。但锁只是手段，线程安全才是目标。

## 18. 面试里怎么回答“你怎么使用锁”

可以这样答：

我一般会先判断是不是共享资源并发访问的问题，如果只是简单状态标记，可能会先考虑 `volatile`；如果是简单原子计数，会先考虑原子类；如果是典型临界区更新，比如库存扣减、余额修改、共享集合复合操作，我会考虑用锁。简单场景通常 `synchronized` 就够了，如果需要更灵活的控制，比如 `tryLock()`、可中断获取锁、公平锁这些，再考虑 `ReentrantLock`。

## 19. 面试里怎么回答“常见的锁使用方式有哪些”

可以这样答：

最常见的锁使用方式，基础层面主要是 `synchronized` 和 `ReentrantLock`。`synchronized` 常见有同步实例方法、同步静态方法和同步代码块三种写法；`ReentrantLock` 则是显式加锁和释放锁，通常配合 `try-finally` 使用。另外如果面试继续往下追，也可以补充读写锁这种读写分离控制思路。

## 20. 当前阶段最该记住的结论

1. 锁本质上是并发控制手段
2. 锁的核心是保护临界区
3. 不是所有并发问题都一定先上锁
4. 简单场景优先简单方案
5. `synchronized` 和 `ReentrantLock` 是最常见的锁使用方式
6. 加锁有收益，也有代价

## 21. 一句话总结

面试里聊锁，最重要的不是背多少“锁的分类”，而是先讲清楚：为什么需要锁、什么时候该用锁、常见怎么用锁、以及锁会带来什么代价。
