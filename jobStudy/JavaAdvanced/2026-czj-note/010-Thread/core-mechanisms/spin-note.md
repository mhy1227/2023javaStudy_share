# 自旋说明

## 1. 为什么这一份要单独补

当你已经学到：

- 锁
- CAS
- `synchronized`
- `ReentrantLock`

面试官很容易继续往下追：

- 什么是自旋？
- 自旋和阻塞有什么区别？
- 为什么有的锁不立刻挂起线程？
- 自旋为什么有时候快，有时候反而浪费 CPU？

所以这一份主要补：

- 自旋的基本理解
- 自旋和阻塞的区别
- 自旋为什么会和锁、CAS 连在一起

## 2. 先一句话理解什么是自旋

可以先记一个很直白的说法：

- 自旋就是线程没拿到资源时，先不立刻挂起，而是短时间内反复尝试

更口语一点说：

- 它不是“睡着等”，而是“先站在门口再试几次”

## 3. 为什么会有自旋

线程如果一拿不到锁就立刻阻塞，再等系统把它唤醒，中间会涉及：

- 用户态和内核态切换
- 线程挂起和恢复
- 调度开销

这些成本并不低。

如果锁持有时间其实很短，比如：

- 另一个线程马上就释放锁了

那这时候如果直接阻塞，反而不划算。

所以就有了一个思路：

- 先自旋几次看看，万一很快就拿到了

## 4. 自旋和阻塞的区别

可以先用最容易答的方式记：

- 自旋：线程不挂起，继续占着 CPU 尝试
- 阻塞：线程挂起，让出 CPU，等后面再被唤醒

它们的本质区别在于：

- 一个更像拿 CPU 时间换等待结果
- 一个更像先休息，等系统后面再叫醒

## 5. 自旋的优点

自旋的优点主要在于：

1. 避免频繁线程切换
2. 如果等待时间很短，可能比阻塞更快
3. 在轻量竞争下响应更直接

所以它更适合：

- 锁很快会释放
- 冲突时间很短
- 临界区很小

## 6. 自旋的缺点

自旋的代价也很明显：

1. 会持续消耗 CPU
2. 如果锁迟迟不释放，就会空转
3. 竞争激烈时，整体效率可能更差

所以自旋不是越久越好。

你可以简单理解成：

- 自旋适合“短等”
- 不适合“长等”

## 7. 自旋为什么会和 CAS 放在一起讲

因为 CAS 失败后，很多时候不是马上放弃，而是会：

- 再读一次
- 再比较一次
- 再尝试更新一次

这种不断重试的过程，本质上就带有自旋味道。

所以很多人会把：

- CAS
- 自旋
- 无锁并发

这几个概念连起来讲。

## 8. 自旋为什么会和锁放在一起讲

因为在锁竞争中，也不一定是一上来就挂起线程。

有些实现会先尝试：

- 短暂自旋

看看能不能很快拿到锁。

如果还是不行，再考虑：

- 阻塞

这背后的考虑就是：

- 线程切换有成本

## 9. 自旋适合什么场景

比较适合的场景通常有：

1. 临界区代码很短
2. 锁持有时间很短
3. 竞争不是特别夸张
4. 对响应速度有要求

例如：

- 简单状态位更新
- 很短的小段同步逻辑
- 轻量并发控制

## 10. 自旋不适合什么场景

不适合的场景通常有：

1. 临界区很大
2. 锁持有时间长
3. 竞争很激烈
4. CPU 已经很紧张

如果这种场景还一直自旋，结果往往是：

- 锁也没抢到
- CPU 还被白白烧掉

## 11. `park/unpark` 和自旋怎么联系

这一块你当前阶段不一定要讲太底层，但最好知道个大概：

- `LockSupport.park()` 更偏让线程进入等待
- `LockSupport.unpark()` 更偏唤醒线程继续执行

所以如果把思路放宽一点：

- 自旋是“先不睡，继续试”
- `park` 是“先停下来，等你叫我”

这两个可以理解成并发控制中的两种不同等待策略。

## 12. 面试里怎么回答“什么是自旋”

可以直接背一版：

自旋可以理解成线程在没拿到锁或者没更新成功时，不立刻挂起，而是短时间内反复尝试。它的好处是可以避免线程阻塞和唤醒带来的切换成本，适合等待时间很短的场景；但如果竞争很激烈或者持锁时间很长，自旋就会白白消耗 CPU，所以它不是越多越好。

## 13. 面试里怎么回答“自旋和阻塞有什么区别”

可以背一版短的：

自旋是不挂起线程，继续占用 CPU 去重试；阻塞是线程先挂起，让出 CPU，等后续再被唤醒。前者适合短时间等待，后者适合等待时间不确定或者比较长的场景。

## 14. 当前阶段记忆重点

这一份最少要记住：

1. 自旋是短时间反复尝试
2. 自旋是拿 CPU 换线程切换成本
3. 自旋适合短临界区、短等待
4. 自旋时间太长会浪费 CPU
5. CAS 和自旋常会一起出现
6. 自旋和阻塞是两种不同等待策略
