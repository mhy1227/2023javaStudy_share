# CAS 说明

## 1. 为什么这一份要单独补

线程这一章前面已经讲了：

- `synchronized`
- `volatile`
- `Lock`
- 原子类和并发工具类

但面试继续往下问时，经常会追到：

- 什么是 CAS？
- `AtomicInteger` 为什么能保证原子性？
- 乐观锁和悲观锁怎么联系起来？
- CAS 有什么问题？

所以这一份主要是把 `CAS` 这个并发里的高频底层机制单独拆开。

## 2. 先一句话理解什么是 CAS

可以先记一个口径：

- CAS 是 Compare And Swap，也常叫 Compare And Set

它的核心思想是：

- 先比较共享变量当前值是不是预期值，如果是，就更新成新值；如果不是，就说明有别的线程改过，这次更新失败

更直白一点说：

- CAS 不是“先锁住再改”，而是“先判断有没有人抢先改过”

## 3. 为什么会需要 CAS

前面已经讲过：

- `i++` 不是原子操作

如果想让一个简单计数在并发下也正确，当然可以用锁。

但很多场景下，大家也会想：

- 能不能不阻塞线程
- 能不能不每次都加重锁

这时候，CAS 就是一种很典型的思路。

它更偏：

- 乐观并发控制

## 4. CAS 的基本执行过程

可以把它理解成下面这几步：

1. 读取当前值
2. 拿当前值和预期值比较
3. 如果一致，就更新
4. 如果不一致，说明被别人改过，本次失败

例如：

- 当前值是 `10`
- 线程 A 预期它还是 `10`
- A 想把它改成 `11`

如果比较时它确实还是 `10`，那就更新成功。

如果比较时它已经被别的线程改成了 `12`，那 A 这次就失败。

## 5. `AtomicInteger` 和 CAS 是什么关系

面试里很常见的一组追问是：

- `AtomicInteger` 为什么线程安全？

一个比较合适的回答是：

- 因为它底层很多更新操作不是直接裸写，而是通过 CAS 这种比较并更新的方式完成，从而保证单次更新的原子性

也就是说：

- `AtomicInteger` 是工具类
- CAS 是它背后的重要思想之一

## 6. CAS 更像乐观锁

如果把它和前面锁的知识串起来，可以这样理解：

- 悲观锁：先假设竞争很激烈，所以先锁住再说
- 乐观锁：先假设冲突不多，先试着改，失败了再重试

CAS 很明显更偏第二种。

所以它常被拿来和悲观锁对比。

## 7. CAS 的优点

比较值得记住的优点有：

1. 不一定需要阻塞线程
2. 在简单并发更新场景下性能往往不错
3. 适合计数器、状态位更新这类轻量原子操作

你可以简单记成：

- CAS 更适合简单、小粒度、冲突没那么夸张的更新场景

## 8. CAS 的问题和局限

CAS 不是万能的，面试里最好主动补这一点。

### 8.1 自旋重试会消耗 CPU

CAS 失败后，很多实现会继续重试。

这种不断尝试的过程，本质上就是一种自旋。

如果竞争非常激烈，就可能出现：

- 一直失败
- 一直重试
- CPU 白白消耗

所以 CAS 不是“零成本”。

### 8.2 只能比较一个值的当前状态

CAS 很适合：

- 单个变量的原子更新

但如果你的业务逻辑是：

- 先判断 A
- 再更新 B
- 再联动更新 C

这种复合逻辑通常就不能只靠一个简单的 CAS 解决。

这时更常见的做法还是：

- 加锁
- 或者重新设计数据结构

### 8.3 ABA 问题

这是 CAS 面试里特别高频的点。

所谓 ABA，可以粗理解成：

1. 线程 1 读到值是 `A`
2. 线程 2 把它改成了 `B`
3. 线程 2 又把它改回了 `A`
4. 线程 1 再去 CAS 时，会发现值“还是 A”

从比较结果看，好像没变。

但实际上：

- 它中间已经被改过了

这就是 ABA 问题。

### 8.4 ABA 怎么处理

比较常见的口径是：

- 不只比较值，还要比较版本

比如：

- `AtomicStampedReference`

它的思路就是：

- 值一样还不够，版本也得对得上

## 9. CAS 适合什么场景

比较适合的场景通常有：

1. 简单计数
2. 状态标记切换
3. 轻量级无锁更新
4. 冲突不算特别重的并发修改

例如：

- 访问次数计数
- 某个任务状态从 `INIT` 改成 `RUNNING`
- 简单的开关位更新

## 10. CAS 不适合什么场景

不太适合的场景通常有：

1. 复合业务操作很多
2. 竞争特别激烈
3. 需要强事务语义
4. 多个变量必须整体一致

这时候通常更应该优先考虑：

- `synchronized`
- `ReentrantLock`
- 或者业务层面的事务与一致性方案

## 11. `AtomicInteger` 和 `LongAdder` 怎么讲

这也是比较常见的追问。

可以先记一个偏面试化的结论：

- 低到中等竞争下，`AtomicInteger` 很常见
- 高并发计数场景下，`LongAdder` 往往更适合

原因可以粗理解成：

- `AtomicInteger` 更像大家抢一个点
- `LongAdder` 更像把压力分散到多个槽位，再汇总结果

所以在高并发累加场景下，`LongAdder` 通常更抗竞争。

## 12. 面试里怎么回答“什么是 CAS”

可以直接背一版：

CAS 可以理解成一种无锁的原子更新思想，它会先比较当前值和预期值是否一致，一致才更新，不一致就失败或重试。它常用于 `AtomicInteger` 这类原子类中，优点是轻量、不一定阻塞线程，但也有 ABA 和自旋开销这些问题，所以它更适合简单的原子更新，不是所有并发问题都靠 CAS 解决。

## 13. 当前阶段记忆重点

这一份最少要记住：

1. CAS 是比较并更新
2. 它更偏乐观锁思想
3. `AtomicInteger` 背后常用到 CAS
4. CAS 会有自旋开销
5. CAS 有 ABA 问题
6. 高并发计数常会提到 `LongAdder`
