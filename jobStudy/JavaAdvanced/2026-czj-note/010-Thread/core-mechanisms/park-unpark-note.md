# park / unpark 说明

## 1. 为什么这一份要单独补

前面已经补了：

- CAS
- 自旋
- AQS

再往下时，面试里很容易继续追：

- `park` 和 `unpark` 是什么？
- 它和 `wait` / `notify` 有什么关系？
- 为什么并发组件里总会提到 `LockSupport`？

所以这一份主要补：

- `park/unpark` 的基本理解
- 它和阻塞唤醒的关系
- 它和 `wait/notify` 的区别

## 2. 先一句话理解 `park` / `unpark`

可以先记一个最直白的说法：

- `park()` 让线程停下来等
- `unpark()` 让线程继续往下走

更口语一点说：

- `park` 像“先停车”
- `unpark` 像“给你放行”

## 3. 它们属于谁

这两个方法常和下面这个类一起出现：

- `LockSupport`

所以平时你更常见的是：

```java
LockSupport.park();
LockSupport.unpark(thread);
```

## 4. 为什么会需要它

并发里经常会遇到一种情况：

- 线程现在还不能继续执行
- 但以后满足条件了又要把它唤醒

如果每个工具类都自己去实现一套阻塞和唤醒逻辑，会很重复。

所以你可以粗理解成：

- `park/unpark` 提供了一种更底层的“线程挂起和恢复”手段

## 5. `park()` 做了什么

你可以先粗理解成：

- 让当前线程进入等待状态

调用以后，当前线程通常就先不继续往下执行了，要等后面有合适时机再恢复。

## 6. `unpark()` 做了什么

可以粗理解成：

- 给目标线程一个“可以继续执行”的许可

如果目标线程已经在 `park` 里等着，那它后面就有机会继续往下执行。

如果目标线程还没 `park`，也可以先给它这个许可。

这一点是它和 `wait/notify` 很值得区分的地方。

## 7. `park/unpark` 和 `wait/notify` 的区别

这是面试特别爱问的一组。

可以先记最核心的几条：

1. `wait/notify` 是 `Object` 的方法
2. `park/unpark` 常通过 `LockSupport` 使用
3. `wait` 必须配合同步块使用
4. `park` 不要求你必须先拿某个对象锁
5. `unpark` 可以先于 `park`

其中最后一点很重要。

因为：

- `notify` 如果先发生，后面再 `wait`，可能就错过了
- `unpark` 先给许可，后面线程再 `park`，通常还能消费这个许可

## 8. 为什么说 `park/unpark` 更灵活

因为它不像 `wait/notify` 那样强依赖：

- 某个对象监视器
- 必须在同步块里配合使用

所以很多显式锁和并发工具类更容易基于它去组织线程等待和唤醒。

可以简单理解成：

- `wait/notify` 更像传统内置同步配套
- `park/unpark` 更像 JUC 这边更底层的阻塞唤醒工具

## 9. 它和 AQS 的关系怎么讲

这一块你当前阶段不用抠源码，但最好知道主线：

- AQS 管理状态和等待队列
- 线程真正阻塞和恢复时，常会和 `park/unpark` 这类能力连起来

所以如果面试官问：

- 为什么 `ReentrantLock`、`CountDownLatch` 这些同步器能做线程挂起和唤醒？

你至少要有这个认知：

- 它们底层会和 AQS、`LockSupport` 这类机制联系起来

## 10. `park/unpark` 和自旋怎么联系

你可以把它们理解成两种不同的等待思路：

- 自旋：先不睡，继续试
- `park`：先停下来，等条件到了再继续

所以如果把主线串起来，就是：

- 轻量等待可能先自旋
- 继续等就可能进入阻塞
- 阻塞恢复时会涉及 `park/unpark`

## 11. 面试里怎么回答“什么是 `park/unpark`”

可以直接背一版：

`park/unpark` 可以理解成 Java 并发里一组更底层的线程阻塞和唤醒机制，通常通过 `LockSupport` 来使用。`park` 会让线程先停下来等待，`unpark` 相当于给线程继续执行的许可。和 `wait/notify` 相比，它不强依赖对象监视器，而且 `unpark` 可以先于 `park`，所以在很多并发组件里会更灵活。

## 12. 面试里怎么回答“它和 `wait/notify` 有什么区别”

可以背一版短的：

`wait/notify` 是基于对象监视器的，通常要放在同步代码块里用；`park/unpark` 更底层，通常通过 `LockSupport` 使用，不强依赖某个对象锁，而且 `unpark` 可以先发许可，再由后续的 `park` 消费，所以使用上更灵活。

## 13. 当前阶段记忆重点

这一份最少要记住：

1. `park` 是停下来等
2. `unpark` 是给继续执行的许可
3. 它们常通过 `LockSupport` 使用
4. 它们和 `wait/notify` 不完全是一条路
5. `unpark` 可以先于 `park`
6. AQS 和很多并发组件会和它们联系起来
