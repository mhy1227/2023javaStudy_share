# 原子类说明

## 1. 为什么这一份要单独补

线程这一章继续往下问时，原子类是非常高频的一组内容。

常见问法包括：

- 什么是原子类？
- 为什么有了锁还需要原子类？
- `AtomicInteger` 为什么线程安全？
- `AtomicInteger` 和 `LongAdder` 怎么选？
- `AtomicReference` 是干什么的？
- ABA 问题怎么处理？

所以这一份主要是把：

- 原子类的整体定位
- 常见成员
- 使用边界
- 面试高频主线

系统地收起来。

## 2. 先一句话理解什么是原子类

可以先记一个口径：

- 原子类是 Java 并发包里一组用来做原子更新的工具类

更直白一点说：

- 它适合把一些简单共享变量的更新，做得更轻量

## 3. 为什么会需要原子类

前面已经讲过：

- `i++` 不是原子操作
- 多线程并发更新共享变量时，结果可能会错

解决这类问题当然可以上锁。

但如果场景只是：

- 简单计数
- 状态位切换
- 某个引用替换

这时候直接用锁，有时会显得太重。

所以原子类可以理解成：

- 在一部分简单并发更新场景下，对锁的一种更轻量补充

## 4. 原子类和锁是什么关系

这个点面试里特别容易问。

比较稳的理解方式是：

- 锁和原子类都能解决一部分线程安全问题
- 但它们的适用边界不一样

你可以简单记成：

- 锁更适合临界区、复合业务逻辑、多个步骤一起保护
- 原子类更适合单变量、简单原子更新

所以原子类不是“取代锁”，而是：

- 补充锁

## 5. 原子类底层为什么能线程安全

当前阶段可以先记主线，不用一下子深抠源码：

- 很多原子类底层会用到 CAS

也就是说，它常见的思路不是先把线程全锁住，而是：

- 先比较当前值
- 再尝试更新
- 失败就重试

所以你可以把这条主线记牢：

- 原子类和 CAS 关系很紧

## 6. 常见原子类有哪些

当前阶段最值得先掌握的几类有：

1. `AtomicInteger`
2. `AtomicLong`
3. `AtomicBoolean`
4. `AtomicReference`
5. `AtomicStampedReference`
6. `LongAdder`

其中前五个更像：

- 原子变量 / 原子引用工具

而 `LongAdder` 更像：

- 高并发计数优化方案

## 7. `AtomicInteger` / `AtomicLong` 适合什么

这两个最常见，也最容易被问。

它们通常适合：

1. 简单计数
2. 自增自减
3. 轻量级状态值更新

例如：

- 访问次数统计
- 生成流水号中的本地计数部分
- 简单并发计数器

## 8. `AtomicBoolean` 适合什么

这个很适合做：

- 开关状态
- 标记位
- 是否已经初始化

例如：

- 某个任务是否已启动
- 某个资源是否已经关闭

这种题虽然简单，但面试里很爱用来看看你会不会只背 `AtomicInteger`。

## 9. `AtomicReference` 是干什么的

可以先记一个很直白的说法：

- 它用来原子地更新一个对象引用

也就是说，它不只是能改数字，还可以：

- 原子地替换某个对象

例如：

- 用一个新配置对象替换旧配置对象
- 用一个新状态对象替换旧状态对象

## 10. `AtomicStampedReference` 是干什么的

这通常和 ABA 问题一起问。

它可以粗理解成：

- 不只看值，还顺带看版本

所以它适合用来应对：

- 值虽然看起来一样，但中间其实被改过的情况

也就是：

- ABA 问题

## 11. `LongAdder` 为什么也常放到原子类这块讲

因为面试官在问原子类时，经常会顺带追：

- 高并发计数为什么不用 `AtomicLong`？

所以你要知道：

- `LongAdder` 虽然和传统单变量原子类不完全一样，但它常被放在原子更新这一层一起理解

它的重点是：

- 在高并发累加场景下，减少单点竞争

## 12. 原子类适合什么场景

比较适合的场景通常有：

1. 单变量原子更新
2. 简单计数
3. 标志位切换
4. 引用替换
5. 冲突不算特别夸张的轻量并发更新

你可以记成一句话：

- 简单共享变量更新，优先考虑原子类；复杂临界区逻辑，再考虑锁

## 13. 原子类不适合什么场景

这一点特别重要，很多人只会背优点，不会说边界。

原子类不太适合：

1. 多步骤复合业务逻辑
2. 多个变量必须整体一致
3. 复杂事务语义
4. 竞争特别激烈且只是单点更新的极端场景

例如：

- 先判断库存
- 再扣库存
- 再写订单

这种就不是一个原子类名义上“线程安全”能兜住的。

## 14. 单个原子类线程安全，不等于整个业务线程安全

这是很值得单独记的一句。

例如：

- `count.incrementAndGet()` 是线程安全的

但如果你的逻辑变成：

1. 先读 `count`
2. 判断某个条件
3. 再更新另一个共享变量

那整个过程未必还是线程安全。

所以要记住：

- 单操作原子，不等于复合流程原子

## 15. `AtomicInteger` 和 `LongAdder` 怎么选

这个点因为太高频，最好在这一份里也单独收一下。

可以先记结论：

- 一般原子计数，`AtomicInteger` / `AtomicLong` 可以
- 高并发累加统计，`LongAdder` 更常见

原因可以粗理解成：

- 原子类更像大家抢同一个点
- `LongAdder` 更像把压力分散掉

## 16. 面试里怎么回答“为什么有了锁还需要原子类”

可以直接背一版：

因为不是所有线程安全问题都需要上锁。如果场景只是简单共享变量更新，比如计数、自增、标志位切换、引用替换，直接上锁有时会偏重。原子类通常基于 CAS 这类思想实现，使用上更轻量，也更适合单变量原子更新场景。但如果是多步骤复合业务逻辑，原子类就不一定够，这时候还是要回到锁或者更高层的一致性方案。

## 17. 面试里怎么回答“原子类有哪些”

可以背一版短的：

当前阶段我最先掌握的是 `AtomicInteger`、`AtomicLong`、`AtomicBoolean`、`AtomicReference` 和 `AtomicStampedReference`。如果问高并发计数，我还会补 `LongAdder`。前几个更偏单变量原子更新，`AtomicReference` 适合原子替换对象引用，`AtomicStampedReference` 常用来处理 ABA，`LongAdder` 则更适合高并发累加统计。

## 18. 当前阶段记忆重点

这一份最少要记住：

1. 原子类适合简单共享变量更新
2. 原子类和 CAS 关系很紧
3. `AtomicInteger` 适合简单计数
4. `AtomicBoolean` 适合标志位
5. `AtomicReference` 适合原子替换对象引用
6. `AtomicStampedReference` 常和 ABA 一起讲
7. `LongAdder` 更适合高并发累加
8. 单操作原子，不等于整个业务线程安全
