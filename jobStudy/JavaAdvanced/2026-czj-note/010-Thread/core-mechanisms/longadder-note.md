# LongAdder 说明

## 1. 为什么这一份要单独补

前面讲 CAS 时已经提到：

- `AtomicInteger`
- `LongAdder`

但面试里很常继续追：

- 为什么高并发计数经常不直接用 `AtomicInteger`？
- `LongAdder` 是解决什么问题的？
- `LongAdder` 和 `AtomicLong` 怎么选？

所以这一份主要补：

- `LongAdder` 的定位
- 它和原子类的区别
- 为什么它更适合高并发计数

## 2. 先一句话理解 `LongAdder`

可以先记一个偏面试化的口径：

- `LongAdder` 是一个更适合高并发累加场景的计数器

更直白一点说：

- 如果很多线程同时加同一个数，`LongAdder` 往往比大家一起抢一个点更自然

## 3. 为什么会需要它

前面已经讲过：

- `AtomicInteger` / `AtomicLong` 可以用 CAS 做原子更新

但如果并发特别高，很多线程都去更新同一个值，就会出现：

- 大家一直 CAS 失败
- 一直重试
- 冲突越来越明显

所以就会有一个自然思路：

- 能不能别让所有线程都去抢同一个位置

`LongAdder` 基本就是顺着这个方向来的。

## 4. 它的大致思路是什么

当前阶段不用死抠源码，可以先粗理解成：

- 它不是只维护一个点
- 而是把累加压力分散到多个槽位上
- 最后再把这些槽位汇总起来

你可以把它想成：

- `AtomicLong` 更像所有人排队改一个窗口
- `LongAdder` 更像开了多个窗口分流

## 5. 为什么它在高并发下更有优势

因为高并发场景下，最大的痛点往往不是“能不能原子更新”，而是：

- 同一个位置竞争太激烈

而 `LongAdder` 的思路是：

- 把竞争拆开

这样大家不必都盯着同一个 CAS 更新点，冲突就更容易被摊开。

## 6. `LongAdder` 和 `AtomicLong` 怎么选

这是最常见的一题。

可以先记一个简化结论：

- 普通原子计数可以用 `AtomicLong`
- 高并发累加更倾向 `LongAdder`

如果想再说完整一点：

- 如果你更关心简单直接、语义清晰，`AtomicLong` 很常见
- 如果你更关心高并发下的累加吞吐，`LongAdder` 往往更适合

## 7. `LongAdder` 适合什么场景

比较适合的场景通常有：

1. 访问量统计
2. 请求次数统计
3. 高并发计数
4. 监控指标累加

例如：

- 接口 QPS 计数
- 用户行为次数统计
- 某个模块的调用次数累加

## 8. `LongAdder` 不太适合什么场景

这里也最好主动补一句边界：

- 它更适合“累加统计”

如果你需要的是：

- 每次都非常强调立刻拿到一个绝对精确、瞬时一致的单点值

那你就要更谨慎。

因为它的核心优势是：

- 分散竞争

不是所有业务都只看吞吐。

## 9. 常见方法先记哪些

当前阶段最常见的先记这几个就够了：

```java
LongAdder adder = new LongAdder();
adder.increment();
adder.add(5);
long result = adder.sum();
```

## 10. 它和 CAS 的关系怎么讲

这一点也经常会被追。

可以先记这个主线：

- `AtomicLong` 更像围绕单点 CAS 做更新
- `LongAdder` 更像在高竞争下把压力分散掉

所以它不是否定 CAS，而是：

- 在高并发累加这个特定问题上，做了更适合吞吐的设计

## 11. 面试里怎么回答“为什么高并发计数更常提 `LongAdder`”

可以直接背一版：

因为在高并发场景下，如果很多线程都去更新同一个原子变量，CAS 冲突会比较明显，失败重试也会增加。`LongAdder` 的思路是把更新压力分散到多个槽位，最后再汇总结果，所以它在高并发累加场景下往往吞吐更好，因此像访问量、请求数、监控指标这类统计问题，经常会优先想到它。

## 12. 面试里怎么回答“`LongAdder` 和 `AtomicLong` 区别”

可以背一版短的：

`AtomicLong` 更适合一般原子更新，思路更直接；`LongAdder` 更适合高并发累加，它通过分散竞争来减少单点冲突，所以在纯计数统计场景下往往表现更好。

## 13. 当前阶段记忆重点

这一份最少要记住：

1. `LongAdder` 适合高并发累加
2. 它主要是为了解决单点竞争太重的问题
3. 它比 `AtomicLong` 更适合高并发统计
4. 常见业务是访问量、次数、指标计数
5. 它不是所有原子更新场景都要替代原子类
