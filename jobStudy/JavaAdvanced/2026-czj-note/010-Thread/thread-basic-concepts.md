# 线程基础认知

## 1. 为什么线程这一章要先讲这些基础概念

如果一上来就直接看：

- `Thread`
- `Runnable`
- `Callable`
- `synchronized`
- `volatile`
- 线程池

很容易出现的问题是：

- 代码会写一点
- 题也背了一些
- 但一旦面试官往前追问“为什么要有线程”“CPU 是怎么参与调度的”“协程和线程什么关系”，就容易断掉

所以线程这一章，前面的地基必须先补：

- CPU
- 进程
- 线程
- 并发
- 并行
- 上下文切换
- 协程

## 2. 为什么先讲 CPU

因为线程的执行，本质上离不开 CPU 调度。

可以先用一句不那么死板的话理解：

- 程序想运行，最终都要拿到 CPU 时间片

也就是说，不是你写了一个线程，它就会一直霸占 CPU；
而是操作系统会根据调度策略，让不同线程轮流获得执行机会。

所以后面很多现象都和 CPU 调度有关，比如：

- 为什么多个线程不是严格按你写的顺序跑
- 为什么会发生上下文切换
- 为什么线程开太多反而可能更慢

## 3. 什么是进程

进程可以理解成一个“正在运行中的程序实例”。

例如：

- 启动一个 Java 程序
- 打开一个浏览器
- 打开一个微信

这些都可以看成进程。

进程通常有自己独立的资源空间，比如：

- 地址空间
- 打开的文件资源
- 一些系统资源

面试里可以先记一句：

- 进程是资源分配的基本单位

## 4. 什么是线程

线程是进程中的执行单元。

一个进程里可以有一个线程，也可以有多个线程。

多个线程共享同一个进程的大部分资源，所以它们之间切换和协作的成本，通常比进程间切换更低。

面试里也经常会说：

- 线程是 CPU 调度的基本单位

你可以先这么理解：

- 进程更像一个“运行环境”
- 线程更像这个环境里的“执行路径”

## 5. 进程和线程的区别

最常见的回答主线：

1. 进程是资源分配的基本单位，线程是 CPU 调度的基本单位
2. 进程之间资源相对独立，线程共享同一进程内的大部分资源
3. 进程切换成本通常更高，线程切换成本通常相对更低
4. 线程通信相对更容易，但也更容易带来线程安全问题

## 6. 为什么要有线程

线程存在，不是为了让程序“看起来更高级”，而是为了解决几个实际问题：

### 6.1 提高程序响应能力

例如：

- 一个线程负责界面响应
- 一个线程负责后台加载数据

这样用户不会觉得整个程序卡死。

### 6.2 提高资源利用率

当一个任务在等待 IO 时，CPU 不一定要一直空着，可以去执行别的线程。

### 6.3 更适合拆分任务

有些任务天然就适合拆开做，例如：

- 多个文件同时处理
- 网络请求并发发送
- 后台异步日志处理

## 7. 什么是并发

并发强调的是：

- 在同一段时间内，多个任务都在推进

它不一定要求某个时刻真的完全同时执行。

例如单核 CPU 下，多个线程轮流获得时间片，也可以形成并发效果。

## 8. 什么是并行

并行强调的是：

- 同一时刻，多个任务真的在同时执行

这通常更依赖多核 CPU。

所以面试里很常见的一句区分是：

- 并发看的是“时间段内一起推进”
- 并行看的是“同一时刻同时执行”

## 9. 为什么会有并发

因为现实中的程序很少只有一条单一执行路径。

常见原因包括：

- 一个任务在等 IO，不想把 CPU 浪费掉
- 多个任务彼此独立，希望一起推进
- 希望提升吞吐量或用户体验

换句话说，并发本质上是为了更高效地利用系统资源，以及更自然地描述复杂任务。

## 10. 为什么并发会带来问题

因为多个线程一旦共享资源，执行顺序就不再稳定。

这时会出现很多经典问题：

- 共享变量被多个线程同时修改
- 一个线程改了值，另一个线程暂时看不到
- 指令执行顺序和你想象的不完全一致

所以后面才会引出并发三大核心问题：

1. 原子性
2. 可见性
3. 有序性

## 11. 什么是上下文切换

上下文切换可以简单理解为：

- CPU 从当前线程切走
- 保存它当前执行到哪
- 再切到另一个线程继续执行

这个过程不是免费的，是有开销的。

所以线程也不是越多越好。

如果线程数量太多，CPU 可能花很多时间在切换上，而不是在真正执行任务上。

这也是为什么后面会强调：

- 合理使用线程池
- 不要无脑创建大量线程

## 12. 协程是什么

协程可以先粗理解成：

- 比线程更轻量的一种协作式执行单元

它和线程最大的直观区别在于：

- 线程通常更多依赖操作系统调度
- 协程通常更多在用户态、框架或运行时层面调度

也就是说，线程更像“系统级调度单元”，协程更像“程序自己管理的轻量任务”。

## 13. 协程和线程的区别

面试里如果被问到，可以先用保守回答：

1. 线程通常由操作系统参与调度，协程通常由用户态运行时或框架调度
2. 协程更轻量，切换成本通常更低
3. 一个线程里通常可以承载多个协程
4. 协程适合大量高并发、等待时间较多的场景

但这里也要注意：

- Java 基础面试里，协程通常不是第一优先级
- 如果不是问到虚拟线程、Reactor、Netty、Kotlin 协程这类延伸内容，不需要把协程讲得过深

## 14. 线程和协程是什么关系

可以先这样理解：

- 线程是更底层、更传统的执行单位
- 协程是构建在更高层调度思想上的轻量任务模型

很多协程最终还是要落到线程上执行，只是中间多了一层更轻的调度。

## 15. 面试里这部分怎么回答比较稳

如果面试官问：

- 为什么要有线程？
- 并发和并行区别是什么？
- 协程和线程区别是什么？

你可以按这条线去答：

1. 先讲 CPU 调度和程序执行需要时间片
2. 再讲进程和线程的定义与区别
3. 再讲为什么要并发
4. 再讲并发带来的线程安全问题
5. 最后再补一句协程是更轻量的任务模型

## 16. 当前阶段最该记住的结论

1. 线程不是孤立概念，前面一定连着 CPU、进程、调度
2. 并发不等于并行
3. 线程带来效率，也带来共享资源竞争问题
4. 上下文切换有成本，所以线程不是越多越好
5. 协程可以知道，但在线程基础阶段先不必讲太深

## 17. 一句话总结

线程这一章真正的前置地基，不是 `Thread` 类本身，而是先分清：CPU 怎么调度、进程和线程分别是什么、为什么会有并发、并发为什么会出问题，以及协程和线程大概是什么关系。
