# 生产者消费者练习题

## 1. 这一份练什么

这一题主要练：

- 条件等待
- 条件唤醒
- 缓冲区满和空的判断
- 多线程协作模型

它是锁题里最经典的一类，也是非常像面试手撕题的一类。

## 2. 题目核心目标

通常题目会给出类似目标：

- 一个或多个线程生产数据
- 一个或多个线程消费数据
- 缓冲区满时不能继续生产
- 缓冲区空时不能继续消费

这类题的重点不在“队列怎么实现得多高级”，而在于：

- 线程怎么正确等待和唤醒

## 3. 这类题核心考察什么

最核心的通常有四点：

1. 条件判断
2. 等待机制
3. 唤醒机制
4. 状态变更顺序

你可以把它理解成：

- 生产者等“有空位”
- 消费者等“有数据”

## 4. 最常见写法

当前阶段最值得先练两版：

1. `synchronized + wait/notifyAll`
2. `ReentrantLock + Condition`

第一版更适合理清：

- 基础等待唤醒主线

第二版更适合理清：

- 多条件队列的好处

## 5. 为什么这题很适合练 `Condition`

因为它天然有两类等待条件：

1. 队列非满
2. 队列非空

如果用 `Condition`，你可以把它们拆成不同等待队列。

这比所有线程都混在一个等待队列里更清晰。

## 6. 这类题最容易犯的错

- 用 `if` 判断，而不是 `while`
- 队列状态更新后没有正确唤醒对方线程
- 只 `notify` / `signal` 了错误的一侧
- 锁内逻辑和状态修改顺序混乱
- 多生产者多消费者场景下思路没扩展开

## 7. 为什么条件判断通常写在 `while`

这是生产者消费者题的高频点。

当前阶段你先记这个结论就够了：

- 线程被唤醒后，不代表条件一定还满足，所以要重新判断

所以这类题你一旦写成：

- `if`

通常就容易被追问。

## 8. 做题时建议先想什么

不要一上来就写代码。

建议先想这几个问题：

1. 缓冲区状态怎么表示
2. 什么情况下生产者要等
3. 什么情况下消费者要等
4. 生产完之后该唤醒谁
5. 消费完之后该唤醒谁

## 9. 面试里怎么描述这类题

如果面试官问你做过什么线程协作题，可以这样说：

我练过生产者消费者题，重点是处理“队列满时生产者等待、队列空时消费者等待”这两类条件。这个题我一般会先用 `synchronized + wait/notifyAll` 写一版，再用 `ReentrantLock + Condition` 写一版。它很适合考察线程协作、条件等待、唤醒时机，以及为什么判断条件要放在 `while` 中。

## 10. 复盘时要问自己的问题

- 我有没有把“满”和“空”这两类条件区分清楚？
- 我有没有想清楚生产完和消费完分别该唤醒谁？
- 我有没有把条件判断放在 `while` 中？
- 如果变成多生产者、多消费者，我的思路还能不能成立？
- 如果改成 `Condition`，我能不能说出它比 `wait/notify` 更清楚在哪里？
