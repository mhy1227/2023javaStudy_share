# 库存扣减题答案版

## 1. 这一份怎么用

这一份不是完整业务代码，而是库存扣减题的推荐解法主线。

适合用法：

- 先自己分析场景
- 再回来对照推荐思路
- 最后用来练“怎么把技术方案和业务边界一起讲清楚”

## 2. 推荐答案主线

库存扣减题最核心的主线通常是：

- 库存是共享资源
- 扣减不是单一步骤
- “判断是否足够 + 扣减库存”要作为一个整体保护

所以它和简单计数题最大的区别就在于：

- 它是复合业务逻辑

## 3. 当前阶段最自然的本地解法

如果题目语境是：

- 单机内存练习

那当前阶段最自然的本地方案通常是：

- `synchronized`
- 或 `ReentrantLock`

因为你要保护的不是一个单独的加减动作，而是：

- 先判断
- 再扣减

这一整段临界区。

## 4. 为什么不能直接答成原子类

这是库存扣减题最容易被问死的点。

一个稳的回答是：

- 如果只是单个变量自减，原子类可以解决单次更新原子性
- 但库存题很多时候是“判断 + 扣减”这种复合逻辑
- 所以单个原子操作线程安全，不等于整个扣减流程线程安全

也就是说，这题更应该先从：

- 临界区保护

的角度去想，而不是只盯着：

- `AtomicInteger`

## 5. 第一版推荐解法

推荐先写：

- `synchronized`

原因很简单：

- 它更直接
- 更容易把“判断 + 扣减”这一整段包成一个整体

### 标准思路

1. 拿锁
2. 判断库存是否足够
3. 如果足够，执行扣减
4. 返回扣减结果

## 6. 第二版推荐解法

如果面试官继续追，或者你想体现对显式锁更熟，可以再补：

- `ReentrantLock`

这版更适合顺手带出：

- 可中断
- `tryLock()`
- 更灵活的锁控制

但在库存扣减这类基础题里，核心仍然不是炫 API，而是：

- 你有没有先把临界区保护思路讲清楚

## 7. 这题真正的延伸点是什么

库存题特别适合继续往后追：

- 单机可以这样做，多机怎么办？
- 数据库层怎么做？
- 版本号和条件更新怎么理解？
- 分布式锁什么时候会引出来？

所以你做这题时，最好主动有一个认知：

- 本地锁只能保护当前进程内线程

这句话一说出来，层次就会明显高一点。

## 8. 最容易丢分的点

- 把库存题误答成简单计数题
- 完全不提“判断 + 扣减”是复合逻辑
- 只会说加锁，不会说为什么这里必须保护整段逻辑
- 不去区分单机并发和多机并发
- 一点业务边界都不提

## 9. 面试口径

如果让我讲库存扣减题，我会先强调它和计数器题不是一个层级。库存扣减通常不是单个自减动作，而是“先判断库存是否足够，再执行扣减”的复合业务逻辑，所以我会优先考虑用锁把这段临界区整体保护起来。当前阶段如果是单机练习，我会先用 `synchronized`，需要更灵活控制时再用 `ReentrantLock`。同时我会主动补一句，本地锁只能解决单进程内并发，真实业务里还要考虑数据库条件更新或分布式场景。

## 10. 最后复盘

做完这一题，至少要能讲清楚：

- 为什么库存题不是简单计数题
- 为什么“判断 + 扣减”必须整体保护
- 为什么本地锁只是当前阶段答案
- 为什么这题天然能顺势追到数据库版本号和分布式锁
