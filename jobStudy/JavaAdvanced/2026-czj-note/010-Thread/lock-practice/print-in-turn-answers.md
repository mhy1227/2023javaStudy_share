# 顺序打印题答案版

## 1. 这一份怎么用

这一份不是完整代码仓，而是顺序打印题的标准解题思路整理。

适合用法：

- 先自己写
- 写不出来时回来对照主线
- 写完后拿它做复盘

## 2. 题 1：两线程交替打印

### 推荐答案主线

最常见、最稳的写法是：

- 一个共享状态
- 一把共享锁
- 条件不满足时等待
- 打印后切换状态
- 唤醒对方线程

### 推荐实现路线

第一版：

- `synchronized + wait/notifyAll`

第二版：

- `ReentrantLock + Condition`

### 关键点

- 锁对象必须一致
- 条件判断通常放在 `while`
- 打印后要先改状态，再做唤醒

### 为什么这样写

因为这类题的核心不是打印，而是：

- 谁轮到执行
- 谁该等待
- 谁负责叫醒对方

### 面试口径

如果让我写两线程交替打印，我会先定义一个共享状态来表示当前轮到谁执行，然后用一把共享锁把判断、等待、打印和状态切换包在一起。条件不满足时线程进入等待，打印完成后切换状态并唤醒对方。基础版我会先用 `synchronized + wait/notifyAll`，进阶版再用 `ReentrantLock + Condition`。

## 3. 题 2：三线程循环打印 `ABCABCABC`

### 推荐答案主线

最常见主线是：

- 一个共享状态变量控制当前轮次
- 三个线程各自等待自己的条件
- 打印后把状态切到下一个线程

### 推荐实现路线

第一版：

- 状态变量 + `synchronized`

第二版：

- `ReentrantLock + 3 个 Condition`

### 关键点

- 状态设计要清楚，比如 `0/1/2` 分别表示轮到谁
- 三个线程都看同一份共享状态
- 条件不满足就继续等待

### 为什么第二版更清楚

因为多个 `Condition` 更容易表达：

- A 等 A 的条件
- B 等 B 的条件
- C 等 C 的条件

比所有线程混在一个等待队列里更容易讲。

### 面试口径

三线程顺序打印的核心是轮转控制。我一般会先用一个共享状态来表示当前轮到 A、B 还是 C 执行。每个线程都在同一把锁下检查状态，不是自己的轮次就等待，轮到自己时打印并切换到下一个状态。基础版我会先写 `synchronized`，如果要体现对显式锁更熟，我会再写一版 `ReentrantLock + 多个 Condition`。

## 4. 题 3：固定顺序执行 `first` / `second` / `third`

### 推荐答案主线

它和交替打印不完全一样，核心是：

- 先后依赖

不是：

- 多线程轮流反复执行

### 推荐实现路线

第一版：

- 状态变量 + `synchronized`

第二版：

- `CountDownLatch`

### 关键点

- `second` 要等 `first`
- `third` 要等 `second`

### 为什么 `CountDownLatch` 很适合

因为它天生就适合表达：

- 前一个完成后，后一个再开始

### 面试口径

固定顺序执行这类题和交替打印不同，它更像是执行依赖约束。我会先用状态控制加 `synchronized` 写一版，更直接的扩展写法是用 `CountDownLatch`，让 `second` 等 `first` 完成、`third` 等 `second` 完成。

## 5. 顺序打印题最容易丢分的点

- 用 `if` 而不是 `while`
- 多线程看见的不是同一把锁
- 状态切换和唤醒顺序写乱
- 三线程题还按两线程思路硬套
- 只能写代码，讲不清线程协作主线

## 6. 最后复盘

做完这一类题，至少要能讲清楚：

- 共享状态是什么
- 等待条件是什么
- 谁来唤醒谁
- 为什么判断要写在 `while`
- 为什么 `Condition` 版更适合多条件协作
