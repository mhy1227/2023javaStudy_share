# 读多写少缓存访问练习题

## 1. 这一份练什么

这一题主要练：

- 锁选型
- 读写场景分析
- 为什么普通独占锁不一定最优
- 怎么把“读多写少”讲成一个合理的并发设计题

它很适合把锁题从“只会加锁”往“会做场景判断”推进一步。

## 2. 题目核心目标

通常题目会给出类似目标：

- 大量线程读取缓存
- 少量线程更新缓存
- 并发下保证结果正确
- 同时尽量不要把读操作全部串行化

这类题的重点不在于缓存实现得多复杂，而在于：

- 你能不能识别这是典型的读多写少场景

## 3. 这类题真正考什么

最核心的通常有四点：

1. 场景识别
2. 锁选型
3. 读写分离思维
4. 性能和正确性的平衡

也就是说，面试官不只是想听：

- 这里加个锁

而是想听：

- 为什么这里不能简单一把独占锁全包住

## 4. 为什么普通独占锁不一定最优

因为在读多写少的场景下，很多读操作之间其实并不冲突。

如果你用普通独占锁把所有读写都互斥掉，就会导致：

- 大量本来可以并发的读，被无谓地串行化

所以这类题最想引出的一个点就是：

- 读操作不一定都该互斥

## 5. 当前阶段最常见解法怎么想

当前阶段最自然的思路通常是：

- `ReentrantReadWriteLock`

因为它能比较清楚地表达：

- 读锁
- 写锁

这种分离设计。

如果你只停留在：

- `synchronized`

也不是完全错，但会显得：

- 方案太粗

## 6. 为什么这题很适合讲读写锁

因为它天然就是：

- 读很多
- 写很少

所以用读写锁时很容易讲清楚：

- 读尽量并发
- 写继续严格互斥

这比单纯背“读写锁适合读多写少”更像真正理解了场景。

## 7. 这类题还能继续往哪追

面试里很容易继续追到：

- 为什么不是普通锁？
- 读写锁在什么情况下也不一定有优势？
- `StampedLock` 是什么？

所以你做这题时最好有这个认知：

- 读写锁也不是所有场景都更好

如果写很多，或者读写冲突仍然很重，收益未必明显。

## 8. 这类题最容易犯的错

- 只会说“加锁”，不会说为什么选读写锁
- 把读多写少和“永远适合读写锁”画等号
- 不去区分读操作之间是否真的冲突
- 完全不提 `StampedLock` 这个延伸认知点
- 只会讲性能，不会讲正确性边界

## 9. 做题时建议先想什么

建议先想这几个问题：

1. 哪些操作是读
2. 哪些操作是写
3. 读和读之间是否必须互斥
4. 写和写之间是否必须互斥
5. 读和写之间怎么保证一致性

## 10. 面试里怎么描述这类题

如果面试官问你练过什么偏锁选型的题，可以这样说：

我练过读多写少缓存访问这类题，它不是简单问你会不会加锁，而是想看你能不能识别场景。因为大量读操作之间通常不冲突，如果用普通独占锁把所有读写都串起来，会比较浪费，所以这类题更自然会想到读写锁。当前阶段我会优先用 `ReentrantReadWriteLock` 去讲，顺手再补一句 `StampedLock` 是进一步的加分认知。

## 11. 复盘时要问自己的问题

- 我有没有先识别出这是读多写少场景？
- 我有没有说清为什么普通独占锁不一定最优？
- 我能不能解释为什么这里更容易想到读写锁？
- 我有没有意识到读写锁也不是任何场景都占优？
- 如果面试官继续追，我能不能顺势提到 `StampedLock`？
