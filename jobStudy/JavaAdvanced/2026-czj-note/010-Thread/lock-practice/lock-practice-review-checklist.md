# 锁练习题考前复盘 Checklist

## 1. 这一份怎么用

这一份不是题解版，而是锁练习题的考前压缩复盘清单。

适合用法：

- 面试前最后 10 到 20 分钟快速过一遍
- 检查自己哪几类题还答不顺
- 复盘“题目主线、易错点、面试怎么说”

复习时建议每题只问自己四个问题：

- 这题核心考什么？
- 推荐方案是什么？
- 最容易错在哪？
- 面试里我会怎么讲？

## 2. 顺序打印题

主线：

- 线程协作，不是打印本身

推荐方案：

- `synchronized + wait/notifyAll`
- `ReentrantLock + Condition`

必须记住：

- 一把共享锁
- 一个共享状态
- 条件判断通常放 `while`
- 打印后切状态，再唤醒

最容易错：

- `if` 代替 `while`
- 锁对象不一致
- 状态切换和唤醒顺序写乱

面试怎么说：

- 这类题核心是“谁该执行、谁该等待、谁来唤醒下一个线程”

## 3. 生产者消费者题

主线：

- 满时生产者等，空时消费者等

推荐方案：

- `synchronized + wait/notifyAll`
- `ReentrantLock + Condition`

必须记住：

- 队列非满
- 队列非空
- 条件判断放 `while`
- 状态变化后唤醒对方

最容易错：

- 唤醒错误线程
- 条件判断写成 `if`
- 多生产者多消费者场景一扩展就乱

面试怎么说：

- 这题考的是条件等待与唤醒，不是队列写得多复杂

## 4. 有界阻塞队列题

主线：

- 把生产者消费者模型组件化

推荐方案：

- `synchronized + wait/notifyAll`
- `ReentrantLock + Condition`

进阶关键词：

- `put`
- `take`
- `notFull`
- `notEmpty`

最容易错：

- 容量控制不严谨
- 组件状态和线程协作逻辑混在一起
- 讲不清为什么它比生产者消费者更像组件题

面试怎么说：

- 这题不只是线程协作，还在考你能不能把协作逻辑封装成组件

## 5. 库存扣减题

主线：

- 这是复合业务逻辑，不是简单计数

推荐方案：

- `synchronized`
- `ReentrantLock`

必须记住：

- “判断库存是否足够 + 扣减库存”要整体保护
- 本地锁只能解决单机进程内问题

最容易错：

- 误答成 `AtomicInteger.decrementAndGet()`
- 完全不提单机和多机的边界

面试怎么说：

- 这题重点不是会不会加锁，而是能不能识别“判断 + 修改”是一个临界区

## 6. 读多写少缓存访问题

主线：

- 锁选型题，不是简单加锁题

推荐方案：

- `ReentrantReadWriteLock`

延伸认知：

- `StampedLock`

必须记住：

- 读尽量并发
- 写继续互斥
- 普通独占锁不一定最优

最容易错：

- 把读写锁说成任何场景都优于普通锁
- 只谈性能，不谈正确性边界

面试怎么说：

- 这题的关键是先识别“读多写少”，再自然引出读写锁

## 7. 计数器选型题

主线：

- 不是只有一个答案，要看场景

推荐方案：

- 简单同步计数：`synchronized`
- 一般原子计数：`AtomicInteger` / `AtomicLong`
- 高并发统计：`LongAdder`

必须记住：

- 单变量更新和复合临界区不是一回事
- 高并发统计为什么会追到 `LongAdder`
- 原子类和 CAS 关系很紧

最容易错：

- 所有计数题都答同一个方案
- 把 `LongAdder` 说成全面优于原子类

面试怎么说：

- 这题核心是场景分层：简单计数、一般并发原子更新、高并发累加统计，各自答案不一样

## 8. 考前最后五个总复盘点

- 我能不能把“线程协作题”和“业务锁题”区分开？
- 我是不是已经形成了“先看场景，再选方案”的习惯？
- 我能不能主动说出 `while`、共享状态、等待条件、唤醒时机这些关键词？
- 我能不能把本地锁答案和数据库 / 分布式延伸区分开？
- 我能不能把锁、CAS、原子类、LongAdder 串成一条完整主线？

## 9. 最后一轮快问快答

- 顺序打印题考什么？
- 为什么生产者消费者里条件判断通常放 `while`？
- 有界阻塞队列和生产者消费者是什么关系？
- 为什么库存题不是简单计数题？
- 为什么读多写少会想到读写锁？
- 为什么计数器选型题不能只背一个答案？
