# 有界阻塞队列题答案版

## 1. 这一份怎么用

这一份不是完整代码仓，而是有界阻塞队列题的标准解题主线整理。

适合用法：

- 先自己写
- 卡住时回来对照思路
- 写完后再用它复盘组件化设计有没有讲清楚

## 2. 推荐答案主线

这类题最常见、最稳的主线通常是：

- 一个共享队列
- 一个容量上限
- 一把共享锁
- `put` 时处理“满”
- `take` 时处理“空”
- 状态变化后唤醒对应等待线程

它本质上还是生产者消费者模型，只是更强调：

- 组件化封装

## 3. 第一版推荐解法

推荐先写：

- `synchronized + wait/notifyAll`

因为这版更适合理清下面这些主线：

- 队列状态怎么表示
- `put` 在什么条件下等待
- `take` 在什么条件下等待
- 操作完成后该唤醒谁

### 标准思路

`put` 逻辑：

1. 拿锁
2. 如果队列已满，等待
3. 放入元素
4. 唤醒可能在等数据的线程

`take` 逻辑：

1. 拿锁
2. 如果队列为空，等待
3. 取出元素
4. 唤醒可能在等空位的线程

## 4. 第二版推荐解法

更清晰的进阶版通常是：

- `ReentrantLock + Condition`

而且最好拆出两个条件队列：

- `notFull`
- `notEmpty`

这样更容易表达：

- 放入方等“非满”
- 取出方等“非空”

## 5. 为什么这一题比生产者消费者更像面试

因为生产者消费者更像：

- 场景协作题

而有界阻塞队列更像：

- 组件实现题

面试官想看的通常不只是：

- 你会不会 `wait`

而是：

- 你能不能把共享数据结构、锁、条件控制、唤醒逻辑一起组织成一个可复用组件

## 6. 最容易丢分的点

- 容量判断不严谨
- `put` / `take` 等待条件写错
- 用 `if` 代替 `while`
- 先唤醒再修改队列状态
- 只会说主线，讲不清为什么要分 `notFull` / `notEmpty`

## 7. 为什么条件判断还要放在 `while`

这和生产者消费者题是同一条主线：

- 线程被唤醒后，不代表条件此时一定仍然满足

所以条件判断仍然应该写在：

- `while`

而不是：

- `if`

## 8. 面试口径

如果让我讲有界阻塞队列题，我会先把它定义成“把生产者消费者模型组件化”。核心是一个共享队列和两个条件：队列满时 `put` 等待，队列空时 `take` 等待。基础版我会先用 `synchronized + wait/notifyAll` 写，把等待和唤醒主线理清；如果要写得更清楚，我会用 `ReentrantLock + Condition`，并拆成 `notFull` 和 `notEmpty` 两个条件队列。

## 9. 最后复盘

做完这一题，至少要能讲清楚：

- 为什么它和生产者消费者是同一主线
- 为什么它又比生产者消费者更像组件题
- `put` 和 `take` 的等待条件分别是什么
- 为什么 `Condition` 版更适合把逻辑讲清楚
- 为什么判断条件仍然要写在 `while`
