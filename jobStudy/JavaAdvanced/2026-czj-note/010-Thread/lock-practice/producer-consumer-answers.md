# 生产者消费者题答案版

## 1. 这一份怎么用

这一份不是完整代码答案库，而是生产者消费者题的推荐解法主线。

适合用法：

- 自己先写一版
- 卡住时对照核心思路
- 写完后拿它做复盘

## 2. 推荐答案主线

最核心的主线通常是：

- 一个共享缓冲区
- 一把共享锁
- 满时生产者等待
- 空时消费者等待
- 状态变化后唤醒对方

这类题最重要的不是队列多花哨，而是：

- 满和空两类条件是不是控制清楚了

## 3. 第一版推荐解法

推荐先写：

- `synchronized + wait/notifyAll`

因为这版最能帮助你理清：

- 等待条件
- 唤醒时机
- 状态更新顺序

### 标准思路

生产者逻辑：

1. 拿锁
2. 如果满了，等待
3. 放入数据
4. 唤醒可能在等数据的消费者

消费者逻辑：

1. 拿锁
2. 如果空了，等待
3. 取出数据
4. 唤醒可能在等空位的生产者

## 4. 为什么条件判断要写在 `while`

这是生产者消费者题的高频追问。

一个很稳的回答是：

- 线程被唤醒后，不代表条件此时一定还满足，所以必须重新检查

所以这类题一旦写成：

- `if`

就很容易被追问。

## 5. 第二版推荐解法

更清晰的进阶版通常是：

- `ReentrantLock + Condition`

而且最好拆成两个条件：

- `notFull`
- `notEmpty`

这样更容易表达：

- 生产者等“非满”
- 消费者等“非空”

## 6. 为什么 `Condition` 版更适合讲

因为它天然能把不同等待条件分开。

相比 `wait/notifyAll`，它更容易把逻辑讲清楚：

- 哪些线程在等空位
- 哪些线程在等数据

所以如果面试官想看你对显式锁更熟，这一版很加分。

## 7. 生产者消费者题最容易丢分的点

- `if` 写成条件判断而不是 `while`
- 队列状态没更新好就先唤醒
- 唤醒了错误类型的线程
- 只会写单生产者单消费者，扩展到多个就乱
- 代码能跑，但讲不清等待与唤醒逻辑

## 8. 面试里怎么描述这类题

如果让我讲生产者消费者题，我会先把模型说清楚：共享缓冲区有两个核心条件，一个是“非满”，一个是“非空”。生产者在满的时候要等待，消费者在空的时候要等待。基础版我会先用 `synchronized + wait/notifyAll` 写，先把等待和唤醒的主线理顺；如果要体现对显式锁的掌握，我会再用 `ReentrantLock + Condition` 写一版，并把 `notFull` 和 `notEmpty` 分成两个条件队列。

## 9. 和有界阻塞队列是什么关系

可以直接记成：

- 生产者消费者更像场景题
- 有界阻塞队列更像组件题

前者强调：

- 线程角色之间怎么协作

后者强调：

- 怎么把这种协作封装进一个可复用组件

## 10. 最后复盘

做完这一题，至少要能讲清楚：

- 生产者为什么要等
- 消费者为什么要等
- 满和空这两类条件怎么控制
- 为什么判断写在 `while`
- 为什么 `Condition` 版更适合多条件协作
