# 库存扣减练习题

## 1. 这一份练什么

这一题主要练：

- 业务场景下的锁使用
- 复合逻辑保护
- 为什么“单个原子变量”不一定够
- 怎么把技术方案和业务约束一起讲清楚

它比顺序打印、生产者消费者更像真实项目里的并发题。

## 2. 题目核心目标

通常题目会给出类似目标：

- 多个线程同时扣减同一商品库存
- 最终库存不能出现负数
- 并发下结果仍然正确

如果是面试题，很多时候面试官更想听的是：

- 你怎么分析问题
- 你为什么选这个方案

不只是看你会不会写一把锁。

## 3. 这类题真正考什么

这类题最核心的通常有四点：

1. 共享资源识别
2. 复合逻辑保护
3. 技术方案选型
4. 业务边界认知

也就是说，你不仅要知道：

- 库存是共享资源

还要知道：

- “先判断再扣减”本身就是复合操作

## 4. 为什么它通常不能只靠一个简单原子变量就完事

这是这类题最值得先想清楚的一点。

因为库存扣减很多时候不是单个“减 1”这么简单。

更常见的逻辑是：

1. 先判断库存是否足够
2. 再执行扣减
3. 可能还要记录订单或流水

这已经不是一个简单的单变量原子更新问题了。

所以你至少要知道：

- 单个原子操作线程安全，不等于整个扣减流程线程安全

## 5. 当前阶段常见解法怎么想

如果只是在单机内存场景下做练习，当前阶段常见可以先想这几种：

1. `synchronized`
2. `ReentrantLock`
3. 数据库层版本号或条件更新思路

当前目录主要练的是：

- 本地锁思路

所以最自然的是：

- 先用 `synchronized` 写一版
- 再用 `ReentrantLock` 写一版

## 6. 这类题为什么比计数器题更复杂

因为计数器题很多时候只是：

- 一个值加一

而库存题更像：

- 先判断
- 再修改
- 还要保证结果不越界

所以它更能体现：

- 你会不会区分“简单原子更新”和“复合业务临界区”

## 7. 面试里如果继续追，会问什么

这类题很容易继续追到：

- 如果是单机可以怎么做？
- 如果是多机怎么办？
- 为什么要考虑数据库约束？
- 为什么不能只靠本地锁？

所以你做这类题时，最好就有这个认知：

- 本地锁只能保护本进程内的线程

这为后面分布式锁和数据库层并发控制做铺垫。

## 8. 这类题最容易犯的错

- 只盯着“减库存”这一步，忽略前面的判断逻辑
- 把库存题误答成简单 `AtomicInteger.decrementAndGet()`
- 不去区分单机锁和多机并发
- 只会说“加锁就行”，不会说为什么这里需要锁
- 完全不提业务层约束和数据库一致性

## 9. 做题时建议先想什么

不要一上来就写代码。

建议先想这几个问题：

1. 共享资源是什么
2. 哪几步必须作为一个整体保护
3. 这是单机练习题，还是要考虑多实例
4. 结果正确的标准是什么
5. 如果继续扩展到真实项目，还要补什么约束

## 10. 面试里怎么描述这类题

如果面试官问你练过什么业务型并发题，可以这样说：

我练过库存扣减题，这类题和简单计数不一样，因为它通常不是单个原子更新，而是“先判断库存是否足够，再执行扣减”这种复合逻辑，所以更适合用锁把整个临界区保护起来。当前阶段我会先从单机本地锁思路去写，比如 `synchronized` 或 `ReentrantLock`，同时我也会主动补一句，本地锁只能解决单进程内的问题，真实业务里还要考虑数据库约束或分布式场景。

## 11. 复盘时要问自己的问题

- 我有没有清楚识别这是复合业务逻辑，不是简单计数？
- 我有没有把“判断 + 扣减”当成一个整体？
- 我有没有区分单机并发和多机并发？
- 我有没有意识到库存题和计数器题不是一个层级？
- 如果面试官继续追，我能不能顺势讲到数据库版本号或分布式锁？
