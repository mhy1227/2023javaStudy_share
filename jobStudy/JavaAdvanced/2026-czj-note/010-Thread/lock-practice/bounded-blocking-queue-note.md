# 有界阻塞队列练习题

## 1. 这一份练什么

这一题可以看作是生产者消费者题的升级版。

它主要练：

- 共享队列保护
- 满和空的条件控制
- 阻塞等待与唤醒
- 用锁把一个简单数据结构包成线程安全组件

## 2. 题目核心目标

通常目标会是：

- 自己实现一个简化版有界阻塞队列
- 支持放入和取出
- 队列满时，放入方等待
- 队列空时，取出方等待

本质上它还是生产者消费者模型，只不过更接近：

- 一个可复用的组件题

## 3. 这类题和生产者消费者有什么关系

可以直接这样理解：

- 生产者消费者更像场景题
- 有界阻塞队列更像组件题

前者关注：

- 线程角色之间怎么协作

后者关注：

- 怎么把这种协作封装进一个队列实现里

## 4. 这类题最常考什么

最常考的通常有：

1. 队列容量控制
2. 满和空时的等待条件
3. 放入后唤醒谁
4. 取出后唤醒谁
5. `synchronized` 和 `ReentrantLock` 两种写法对比

## 5. 建议练法

当前阶段建议先做两版：

1. `synchronized + wait/notifyAll`
2. `ReentrantLock + Condition`

如果你想拉开层次，可以把第二版再拆成：

- `notFull`
- `notEmpty`

两个条件队列。

## 6. 这类题容易出什么错

- 容量判断不严谨
- 先修改状态还是先唤醒的顺序写乱
- 条件判断没有放在 `while`
- 只会写单生产者单消费者，题一变就乱
- 把队列操作和线程控制逻辑混得太杂

## 7. 为什么它比普通打印题更像面试

因为它更贴近真实组件设计。

面试官想看的通常不只是：

- 你会不会 `wait`

而是：

- 你能不能把锁、条件、共享数据结构、线程协作一起组织清楚

所以这类题比顺序打印题更能体现：

- 设计能力

## 8. 做题时建议先拆哪些点

建议先拆成四部分想：

1. 队列本身怎么存数据
2. `put` 在什么条件下要等
3. `take` 在什么条件下要等
4. 每次成功操作后该唤醒哪一类线程

## 9. 面试里怎么描述这类题

如果面试官问你做过什么线程协作或锁题，可以这样说：

我练过简化版有界阻塞队列，这个题本质上是在生产者消费者模型上再往组件化走一步。它会同时考察共享队列保护、满和空两类等待条件、正确唤醒以及 `synchronized` 和 `ReentrantLock + Condition` 两种写法的区别。这类题比简单打印题更接近真实组件实现。

## 10. 复盘时要问自己的问题

- 我有没有把队列状态和线程协作逻辑拆清楚？
- 我能不能明确说出 `put` 和 `take` 各自的等待条件？
- 我有没有正确使用 `while` 重新判断条件？
- 我能不能用 `Condition` 把 `notFull` 和 `notEmpty` 分开？
- 如果要从“题目能做”升级到“组件能说清”，我还差哪一步？
