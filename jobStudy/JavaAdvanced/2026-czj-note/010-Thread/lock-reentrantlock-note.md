# Lock 与 ReentrantLock 说明

## 1. 为什么在 `synchronized`、`volatile` 之后讲 `Lock`

因为到这里，线程同步已经出现了两条很典型的思路：

1. `synchronized`
2. `volatile`

但面试和实际开发里还会继续问：

- 为什么 Java 还要提供 `Lock`
- `ReentrantLock` 到底比 `synchronized` 多了什么

这就说明：

- `synchronized` 虽然基础、常用，但不是唯一方案
- `Lock` 是另一套更显式、更灵活的同步方式

## 2. 什么是 `Lock`

可以先给一个容易口述的定义：

- `Lock` 是 Java 并发包里提供的锁接口，用来对共享资源进行显式加锁和解锁控制

这里最关键的词是：

- 显式

也就是说，它不像 `synchronized` 那样把加锁写在语法结构里，而是需要你手动：

1. `lock()`
2. 执行业务代码
3. `unlock()`

## 3. 什么是 `ReentrantLock`

`ReentrantLock` 是 `Lock` 接口最常见的实现类之一。

它可以先粗理解成：

- 功能更丰富的显式可重入锁

面试里如果问到 `Lock`，通常真正展开讲的，基本就是 `ReentrantLock`。

## 4. 为什么还需要 `Lock`

这是高频问题。

因为 `synchronized` 虽然简单，但控制能力比较固定。

而 `Lock` 提供了更灵活的控制方式，例如：

1. 手动加锁和解锁
2. 可中断地获取锁
3. 尝试获取锁
4. 可设置公平锁 / 非公平锁
5. 更适合扩展更复杂的并发控制场景

所以不是说 `synchronized` 不好，而是：

- `Lock` 在一些复杂场景下更灵活

## 5. 最基础的写法

```java
private final Lock lock = new ReentrantLock();

public void add() {
    lock.lock();
    try {
        count++;
    } finally {
        lock.unlock();
    }
}
```

这一段里最重要的不是语法长什么样，而是你要记住这条模式：

1. `lock.lock()`
2. `try`
3. 业务代码
4. `finally`
5. `lock.unlock()`

## 6. 为什么 `unlock()` 一般必须放在 `finally`

这是非常重要的实践点。

因为如果你的业务代码中间抛了异常，而你没有在 `finally` 里解锁，就可能导致：

- 锁一直不被释放

后果就是：

- 其他线程一直拿不到锁
- 程序卡住

所以面试里可以直接说：

- `Lock` 一般要和 `try-finally` 配套使用，确保锁一定被释放

## 7. 什么叫可重入

`ReentrantLock` 里的 “Reentrant” 指的是：

- 同一个线程在已经拿到这把锁的前提下，还可以再次进入并继续获取这把锁

例如一个同步方法里又调用另一个同样需要这把锁的方法，如果锁不可重入，就会把自己卡死。

所以“可重入”这个能力非常重要。

## 8. `ReentrantLock` 为什么叫可重入锁

就是因为：

- 同一线程可以重复获得同一把锁

你可以把它理解成：

- 这把锁认识“是谁已经拿到了它”

所以同一个线程再次进入时，不会把自己挡在外面。

## 9. `lock()` 是什么

`lock()` 的作用是：

- 获取锁

如果锁当前没人持有，就拿到锁继续往下执行；
如果锁被别人占着，通常就得等待。

## 10. `unlock()` 是什么

`unlock()` 的作用是：

- 释放锁

这一步不是可有可无，而是显式锁模式里必须认真处理的一步。

## 11. `tryLock()` 是什么

`tryLock()` 可以理解成：

- 尝试获取锁

和 `lock()` 的区别在于：

- `lock()` 拿不到通常就继续等
- `tryLock()` 拿不到可以直接返回，不一定一直阻塞

这在一些不想长时间等待锁的场景里很有用。

## 12. `lockInterruptibly()` 是什么

`lockInterruptibly()` 可以理解成：

- 线程在等待锁的时候，可以响应中断

这也是 `Lock` 相比 `synchronized` 更灵活的地方之一。

如果面试里只是基础阶段，知道这个能力即可，不需要展开太深。

## 13. 公平锁和非公平锁

`ReentrantLock` 还可以配置成：

1. 公平锁
2. 非公平锁

### 13.1 公平锁

更偏向：

- 按等待顺序获取锁

### 13.2 非公平锁

更偏向：

- 不严格按先来后到
- 谁机会好谁先拿到

一般来说：

- 非公平锁吞吐量通常更高一些
- 公平锁更强调等待顺序上的公平

## 14. 一个简单例子：用 `ReentrantLock` 改写计数器

```java
class Counter {
    private int count = 0;
    private final Lock lock = new ReentrantLock();

    public void add() {
        lock.lock();
        try {
            count++;
        } finally {
            lock.unlock();
        }
    }
}
```

这个例子和前面的 `synchronized` 版本作用类似，都是为了保护临界区。

## 15. `Lock` 主要解决什么问题

从本质上说，它和 `synchronized` 一样，核心还是：

- 保护共享资源
- 控制临界区互斥访问

所以它同样能帮助解决：

1. 原子性问题
2. 一定的可见性问题
3. 一定的有序性问题

不同点更多在于：

- 控制方式更灵活

## 16. `Lock` 的优点

面试里可以先记这几条：

1. 使用更灵活
2. 可以手动控制加锁和释放锁
3. 支持 `tryLock()`
4. 支持 `lockInterruptibly()`
5. 支持公平锁 / 非公平锁

## 17. `Lock` 的代价

它也不是天然完胜。

相对 `synchronized` 来说，它的代价在于：

1. 写法更繁琐
2. 需要手动释放锁
3. 如果忘记 `unlock()`，风险更大

所以它更灵活，但也更考验编码规范。

## 18. `Lock` 和 `synchronized` 的关系

当前阶段不要上来就背太长对比表，可以先用一句核心话去记：

- 它们本质上都是为了解决共享资源并发访问时的同步问题

区别在于：

- `synchronized` 更偏语法层面的内置锁，简单直观
- `Lock` 更偏 API 层面的显式锁，控制更灵活

## 19. 什么时候更适合考虑 `ReentrantLock`

如果只是很基础的同步保护，`synchronized` 通常已经够用。

更适合考虑 `ReentrantLock` 的场景一般是：

1. 需要尝试获取锁
2. 需要可中断获取锁
3. 需要公平锁
4. 需要更复杂的并发控制

## 20. 面试里最容易漏的点

### 20.1 只会说 `Lock` 更灵活，但说不出灵活在哪

至少要补：

- `tryLock()`
- `lockInterruptibly()`
- 公平锁 / 非公平锁

### 20.2 忘记说 `unlock()` 要放 `finally`

这是非常高频的实践点。

### 20.3 把“可重入”解释得太模糊

要讲清楚：

- 同一个线程已经拿到锁后，还能再次拿同一把锁

### 20.4 以为 `Lock` 和 `synchronized` 是完全不同目标

其实目标类似，都是为了解决同步问题，只是控制方式不同。

## 21. 面试里怎么回答“为什么还要有 `ReentrantLock`”

可以这样答：

因为 `synchronized` 虽然简单常用，但控制能力相对固定，而 `ReentrantLock` 提供了更灵活的锁控制方式，比如可以手动加锁和释放锁、支持 `tryLock()`、支持可中断获取锁、还能配置公平锁和非公平锁。所以在一些更复杂的并发控制场景下，`ReentrantLock` 会更合适。

## 22. 面试里怎么回答“什么是可重入”

可以这样答：

可重入指的是同一个线程在已经持有某把锁的前提下，还可以再次获得这把锁，而不会把自己阻塞住。`ReentrantLock` 是可重入锁，`synchronized` 也是可重入的。

## 23. 当前阶段最该记住的结论

1. `Lock` 是显式锁接口，`ReentrantLock` 是常见实现
2. 它和 `synchronized` 的目标类似，都是保护临界区
3. `ReentrantLock` 的关键优势是更灵活
4. `unlock()` 一般必须放在 `finally`
5. 可重入指的是同一线程可以重复获取同一把锁

## 24. 一句话总结

`Lock / ReentrantLock` 这一块最重要的不是记很多方法名，而是先讲清楚：为什么在 `synchronized` 之外还需要显式锁、它灵活在哪、为什么 `unlock()` 要放进 `finally`，以及“可重入”到底是什么意思。
