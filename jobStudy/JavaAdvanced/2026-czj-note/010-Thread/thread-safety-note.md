# 线程安全说明

## 1. 为什么先讲线程安全

线程这一章真正的分水岭，其实不是“会不会创建线程”，而是：

- 你知不知道多线程为什么会出问题
- 你能不能讲清楚什么叫线程安全

因为后面的：

- `synchronized`
- `volatile`
- `Lock`
- 原子类
- 并发容器

本质上都是在解决线程安全问题。

所以如果这一份没讲清楚，后面所有同步手段都会变成零散知识点。

## 2. 什么是线程安全

先给一个偏面试化、但容易理解的说法：

- 多个线程同时访问同一份共享数据或共享资源时，程序的结果仍然正确、可预期，这就可以叫线程安全

换句话说，线程安全关注的是：

- 在并发环境下，程序会不会出现错误结果
- 程序行为是不是稳定可控

## 3. 什么情况下容易出现线程安全问题

线程安全问题通常不是凭空出现的，一般要同时满足这两个条件：

1. 存在共享资源
2. 存在多个线程并发访问或修改

如果没有共享资源，各线程各干各的，问题通常不大。

最危险的情况往往是：

- 多个线程同时读写同一个变量
- 多个线程同时修改同一份集合
- 多个线程同时操作同一个对象状态

## 4. 什么是共享资源

共享资源可以简单理解成：

- 多个线程都能访问到的同一份数据或对象

常见例子：

- 共享计数器
- 共享余额
- 共享库存
- 共享集合

例如：

```java
class Counter {
    int count = 0;
}
```

如果多个线程同时改这个 `count`，就可能出问题。

## 5. 为什么多线程会不安全

因为多线程执行顺序是不稳定的。

你写代码时，脑子里常常是线性顺序：

1. 先读
2. 再改
3. 再写回去

但在并发环境里，多个线程可能交错执行。

于是就会出现：

- 线程 A 读到旧值
- 线程 B 也读到旧值
- 两边各自修改
- 最后结果被覆盖

这类问题本质上就是：

- 共享资源竞争

## 6. 一个最经典的例子：`i++`

很多人第一次接触线程安全时，都会以为：

- `i++` 很简单，应该没事

但它其实不是原子操作。

它通常包含三个步骤：

1. 读取 `i`
2. `i + 1`
3. 把结果写回去

如果两个线程同时执行这三步，就可能产生丢失更新。

示意：

```java
class Counter {
    int count = 0;

    public void add() {
        count++;
    }
}
```

多个线程同时执行 `add()`，最终结果可能比预期小。

## 7. 再看一个生活化例子：卖票

例如还有 1 张票：

- 线程 A 读取到票数是 1
- 线程 B 也读取到票数是 1
- A 卖掉一张
- B 也卖掉一张

结果就可能出现：

- 超卖

这就是非常典型的线程安全问题。

## 8. 线程安全的核心问题有哪些

Java 并发里，线程安全问题通常会继续拆成三大块：

1. 原子性
2. 可见性
3. 有序性

这三个词后面会反复出现，必须先理解。

## 9. 什么是原子性

原子性可以简单理解成：

- 一个操作要么全部做完，要么全部不做
- 中间不能被别的线程打断

例如：

- 单纯赋值 `x = 1`，可以近似理解为一个完整动作
- 但 `i++` 往往不是一个不可分割动作

所以 `i++` 的线程安全问题，本质上和原子性有关。

## 10. 什么是可见性

可见性关注的是：

- 一个线程修改了共享变量
- 另一个线程能不能及时看到这个新值

如果看不到，就可能一直拿着旧值工作。

例如一个线程把停止标记改成 `true`，另一个线程却还在循环，这背后就和可见性有关。

## 11. 什么是有序性

有序性关注的是：

- 程序执行顺序是不是完全按你写的代码顺序来

在单线程语境下，通常你感觉顺序是稳定的；
但在并发和编译优化、CPU 优化场景下，实际执行顺序不一定和源码顺序完全一致。

这就会引出：

- 指令重排序

所以有序性问题更多是“顺序是否符合预期”的问题。

## 12. 什么是竞态条件

竞态条件可以简单理解成：

- 程序最终结果取决于多个线程谁先执行、谁后执行

也就是说，结果不再稳定，而是和时机有关。

这通常就是并发 Bug 最难排查的原因：

- 它不是必现的
- 有时出问题，有时又没问题

## 13. 哪些代码天然更危险

线程安全里，下面这些操作要特别警惕：

1. 对共享变量做复合操作
2. 对共享集合做增删改
3. 检查再执行的逻辑
4. 多步骤更新同一状态

例如这种逻辑就很危险：

```java
if (count > 0) {
    count--;
}
```

看起来没问题，但在多线程下：

- 读和减不是一个原子整体

## 14. 什么是线程安全的类

线程安全的类，不是说“它永远不会出问题”，而是说：

- 在设计者规定的使用方式下
- 它可以在多线程环境中安全使用

例如：

- `String` 常常被认为是线程安全的
- `Vector` 的单个方法通常是同步的
- `ConcurrentHashMap` 是并发容器

但也要注意：

- 类本身某些方法线程安全，不等于你把多个操作拼起来后仍然绝对安全

## 15. 什么是线程不安全的类

线程不安全的类，在多线程环境下直接共享使用时，容易出问题。

典型例子：

- `ArrayList`
- `HashMap`
- 普通的可变成员变量对象

这不代表它们“不能用”，而是代表：

- 并发使用时要额外控制

## 16. 线程安全是不是等于加锁

不完全等于。

加锁当然是很重要的一种手段，但不是唯一手段。

常见解决思路包括：

1. 加锁同步
2. 使用 `volatile`
3. 使用原子类
4. 使用线程安全容器
5. 尽量避免共享
6. 使用不可变对象

所以面试里不要把线程安全简单理解成：

- 有锁 = 安全

更准确的说法应该是：

- 线程安全是目标，锁只是实现手段之一

## 17. 如何尽量避免线程安全问题

最基础的思路有几类：

### 17.1 少共享

能不共享就不共享。

### 17.2 共享就控制访问

如果必须共享，就要考虑：

- 同步
- 锁
- 原子类

### 17.3 尽量使用不可变对象

例如：

- `String`

不可变对象天然更容易在线程间安全共享。

### 17.4 使用并发容器

例如：

- `ConcurrentHashMap`
- `CopyOnWriteArrayList`

## 18. 面试里这一块最容易漏什么

### 18.1 只会背“线程安全就是加锁”

这太窄了。

要至少补充：

- 原子性
- 可见性
- 有序性

### 18.2 只会说 `i++` 不安全，但说不出为什么

一定要能补一句：

- 因为 `i++` 不是原子操作

### 18.3 只会说有共享资源会出问题，但说不出还需要什么条件

要补完整：

- 共享资源
- 多线程并发访问或修改

### 18.4 只会说线程不安全，却不会引出解决手段

至少要顺着说到：

- `synchronized`
- `volatile`
- `Lock`

## 19. 面试里怎么回答“什么是线程安全”

可以这样答：

线程安全指的是在多线程并发访问共享资源时，程序仍然能够得到正确、可预期的结果。线程安全问题通常出现在多个线程同时访问或修改同一份共享数据时，核心往往和原子性、可见性、有序性有关。后面一般会通过 `synchronized`、`volatile`、`Lock`、原子类或并发容器来解决。

## 21. 当前阶段最该记住的结论

1. 线程安全的前提通常是“共享资源 + 并发访问”
2. 多线程不安全的根源是执行顺序不可控和资源竞争
3. 线程安全的核心问题通常离不开原子性、可见性、有序性
4. `i++` 是经典的非线程安全例子
5. 线程安全不是只有加锁这一种思路

## 22. 一句话总结

线程安全这一块真正要搞清楚的，不是“哪些类安全、哪些类不安全”这么零散，而是先理解：为什么并发访问共享资源会出问题，这些问题具体体现在哪，以及 Java 后面为什么要提供那么多同步手段。
