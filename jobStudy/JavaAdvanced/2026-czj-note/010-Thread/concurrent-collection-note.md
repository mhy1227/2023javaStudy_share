# 并发容器说明

## 1. 为什么线程这一章还要讲并发容器

因为线程这一章走到这里，前面已经讲了：

- 线程安全
- `synchronized`
- `volatile`
- `Lock / ReentrantLock`
- 线程池

接下来最自然的问题就是：

- 如果多个线程要同时操作集合，怎么办

也就是说，线程安全不只是变量和锁的问题，集合也一样会在并发环境里出问题。

所以并发容器这一份要解决的核心就是：

- 为什么普通集合在并发下容易出问题
- Java 提供了哪些并发容器
- 它们各自适合什么场景

## 2. 为什么普通集合在并发环境下容易出问题

因为前面已经讲过，线程安全问题通常满足两个条件：

1. 存在共享资源
2. 多个线程并发访问或修改

集合本身就是非常典型的共享资源。

如果多个线程同时对同一个普通集合做：

- 添加
- 删除
- 修改
- 遍历

就可能出现：

- 数据不一致
- 覆盖更新
- 结构异常
- 遍历报错

## 3. 哪些普通集合是线程不安全的

最常见的几个：

1. `ArrayList`
2. `LinkedList`
3. `HashMap`
4. `HashSet`

它们本身在单线程场景下很好用，但在多线程并发读写场景下，就要非常小心。

## 4. 什么是并发容器

可以先给一个容易口述的定义：

- 并发容器是 Java 为多线程环境提供的一类集合，它们在设计上考虑了并发访问问题，能在一定规则下更安全地被多个线程同时使用

注意这里不要讲得太绝对。

更稳妥的理解是：

- 它们是为并发场景设计的集合
- 但也不是说任何复合操作都天然 100% 无脑安全

## 5. 这一块最常见的并发容器有哪些

当前阶段最常见、最容易被问到的通常有：

1. `ConcurrentHashMap`
2. `CopyOnWriteArrayList`
3. `BlockingQueue`

## 6. `ConcurrentHashMap` 是什么

`ConcurrentHashMap` 可以先粗理解成：

- 面向并发场景设计的 `Map`

它通常被拿来替代：

- `Hashtable`
- 或者多线程场景下直接用 `HashMap`

## 7. 为什么 `ConcurrentHashMap` 比 `Hashtable` 更常用

这是高频问题。

当前阶段你可以先用稳妥回答：

1. `Hashtable` 方式更老、控制更粗
2. `ConcurrentHashMap` 在并发场景下性能通常更好
3. `ConcurrentHashMap` 更符合现代并发容器设计思路

面试里最重要的不是把底层实现细节背得过细，而是先讲清：

- 它们都是线程安全 `Map`
- 但 `ConcurrentHashMap` 更常用、更适合高并发场景

## 8. `ConcurrentHashMap` 适合什么场景

它更适合：

1. 多线程读写共享 `Map`
2. 读多写多的缓存类场景
3. 需要线程安全字典结构的场景

例如：

- 共享配置缓存
- 统计结果存储
- 在线用户信息表

## 9. `CopyOnWriteArrayList` 是什么

`CopyOnWriteArrayList` 可以先粗理解成：

- 适合并发读多写少场景的线程安全列表

这里最关键的词是：

- 读多写少

## 10. 为什么叫 `CopyOnWrite`

名字本身就很好记：

- 写时复制

可以先粗理解成：

- 读的时候尽量不折腾
- 写的时候复制一份再改

所以它的特点就很明显：

1. 读操作体验较好
2. 写操作成本更高

## 11. `CopyOnWriteArrayList` 适合什么场景

它更适合：

1. 读远多于写
2. 数据量不是特别夸张
3. 遍历很多、修改较少

典型例子：

- 监听器列表
- 订阅者列表
- 配置快照列表

## 12. `CopyOnWriteArrayList` 不适合什么场景

如果写操作很多，它就不太合适，因为：

- 每次写都可能伴随复制成本

所以：

- 频繁增删改场景不适合它

## 13. `BlockingQueue` 是什么

`BlockingQueue` 可以先理解成：

- 带阻塞能力的队列接口

它最大的特点是：

- 队列空时，取元素可以阻塞等待
- 队列满时，放元素也可以阻塞等待

## 14. `BlockingQueue` 为什么常和线程池、生产者消费者一起出现

因为它非常适合做：

- 生产者消费者模型

例如：

- 生产者线程往队列里放任务
- 消费者线程从队列里取任务

如果队列没数据，消费者可以等；
如果队列满了，生产者也可以等。

这比自己手动写很多 `wait / notify` 要自然得多。

## 15. 常见的 `BlockingQueue` 使用场景

1. 任务缓冲
2. 生产者消费者模型
3. 线程池内部任务排队

所以前面讲线程池时，`workQueue` 本质上就和这类队列思路有关。

## 16. 并发容器是不是就等于“完全不用考虑线程安全”

不能这么理解。

这是一个很重要的边界。

并发容器通常能保证：

- 单次基本操作在并发场景下更安全

但如果你把多个步骤拼起来，例如：

1. 先判断
2. 再更新

这种复合逻辑，依然可能要额外考虑并发问题。

所以并发容器不是万能护身符。

## 17. 当前阶段最常见的选择思路

这一段很适合面试时快速回答。

### 17.1 如果要线程安全的 `Map`

优先先想到：

- `ConcurrentHashMap`

### 17.2 如果要线程安全的 `List`，而且读多写少

优先想到：

- `CopyOnWriteArrayList`

### 17.3 如果是任务传递、生产者消费者

优先想到：

- `BlockingQueue`

## 18. 为什么不直接全都加锁用普通集合

这也是一个常见延伸题。

当然可以通过外部加锁保护普通集合，但问题是：

1. 自己控制锁更容易写错
2. 粒度不好控制
3. 代码更繁琐
4. 并发容器通常已经针对这类场景做了更合适的设计

所以在合适场景下，直接使用并发容器通常更自然。

## 19. 面试里最容易漏的点

### 19.1 只会说 `ConcurrentHashMap` 是线程安全的

这还不够。

至少要补一句：

- 它比 `Hashtable` 更常用，更适合高并发场景

### 19.2 只会背 `CopyOnWriteArrayList` 名字

但说不清它最适合什么场景。

一定要记住：

- 读多写少

### 19.3 只会说 `BlockingQueue` 是队列

但说不出它为什么重要。

至少要补一句：

- 它非常适合生产者消费者模型和任务缓冲

### 19.4 以为并发容器能自动解决所有复合并发问题

这不准确。

复合逻辑依然要谨慎。

## 20. 面试里怎么回答“为什么 `ConcurrentHashMap` 比 `Hashtable` 更常用”

可以这样答：

它们都可以用于线程安全的 `Map` 场景，但 `Hashtable` 的设计更老、控制更粗，而 `ConcurrentHashMap` 更适合现代并发场景，通常性能和伸缩性更好，所以实际开发和面试里更常提到 `ConcurrentHashMap`。

## 21. 面试里怎么回答“`CopyOnWriteArrayList` 适合什么场景”

可以这样答：

`CopyOnWriteArrayList` 更适合读多写少的并发场景，因为它采用写时复制思路，读操作体验较好，但写操作成本更高，所以不适合频繁修改的数据结构。

## 22. 面试里怎么回答“`BlockingQueue` 有什么用”

可以这样答：

`BlockingQueue` 是带阻塞能力的队列接口，常用于生产者消费者模型和任务缓冲。队列空时消费者可以等待，队列满时生产者也可以等待，所以它在并发任务协作里非常常见，线程池内部也会涉及类似的任务排队思路。

## 23. 当前阶段最该记住的结论

1. 普通集合在并发读写场景下容易出问题
2. 并发容器是为多线程场景设计的集合
3. `ConcurrentHashMap` 是最常见的线程安全 `Map`
4. `CopyOnWriteArrayList` 最适合读多写少
5. `BlockingQueue` 很适合生产者消费者和任务缓冲

## 24. 一句话总结

并发容器这一块最重要的不是死背很多类名，而是先讲清楚：为什么普通集合在并发下会出问题，以及 `ConcurrentHashMap`、`CopyOnWriteArrayList`、`BlockingQueue` 这几类工具各自解决的是什么场景。
