# 线程池实战补充

## 1. 为什么这一份要单独补

前面的 [thread-pool-note.md](D:/CourseShop_2/2023javaStudy_share/jobStudy/JavaAdvanced/2026-czj-note/010-Thread/thread-pool-note.md) 已经把：

- 为什么需要线程池
- 核心参数
- `execute` / `submit`

这些基础主线补出来了。

但面试和项目里更容易继续追：

- 线程数怎么估
- 拒绝策略怎么选
- 任务堆积了怎么办
- CPU 密集和 IO 密集怎么区分

所以这一份主要补：

- 线程池怎么落到实战

## 2. 先一句话理解线程池实战在补什么

可以先记一个口径：

- 线程池基础是在讲“是什么”，线程池实战是在讲“怎么配置、怎么用、出事了怎么调”

## 3. 线程池参数为什么不能死背

因为线程池参数不是为了面试凑答案，而是要和业务场景对上。

例如：

- 任务是 CPU 密集还是 IO 密集
- 任务执行快还是慢
- 请求高峰是不是会堆积
- 能不能接受任务被丢弃

这些都会影响：

- 核心线程数
- 最大线程数
- 队列长度
- 拒绝策略

## 4. CPU 密集型任务怎么想

可以先记一个很常见的口径：

- CPU 密集型任务，线程数通常不宜开太多

因为这类任务大部分时间都在：

- 抢 CPU

所以线程太多反而会带来：

- 上下文切换开销

基础面试里可以先答：

- 通常会更接近 CPU 核数

## 5. IO 密集型任务怎么想

如果任务大部分时间在：

- 等待数据库
- 等待网络
- 等待磁盘

那这类更偏：

- IO 密集型

这时线程数通常可以比 CPU 核数更多一点，因为线程很多时间并不真的在跑 CPU。

所以面试里一个很稳的说法是：

- CPU 密集看核数，IO 密集更看等待比例

## 6. 队列怎么想

线程池不是只有线程数，队列也很关键。

你至少要先知道：

- 队列太大，任务容易大量堆积
- 队列太小，线程池更容易快速扩容或触发拒绝

所以队列不是越大越好。

很多时候面试官想听的不是精确数字，而是：

- 你知不知道“任务堆积”是个风险

## 7. 拒绝策略怎么想

如果线程满了、队列也满了，就会走到：

- 拒绝策略

这时你要先有一个实战认知：

- 拒绝策略不是冷知识，而是在保护系统

基础面试里你不一定要把每个策略全背成源码级，但至少知道：

1. 有的会直接抛异常
2. 有的会丢任务
3. 有的会丢最旧任务
4. 有的会让提交线程自己执行

最重要的是要知道：

- 不同业务能接受的代价不一样

## 8. 为什么不建议无脑用 `Executors`

这个点前面主文档已经提过，但实战里最好再记深一点。

一个稳的回答是：

- `Executors` 提供的默认工厂很方便，但很多参数是默认封装好的，容易让你忽略队列大小、线程数量和任务堆积风险，所以实际项目里通常更建议直接用 `ThreadPoolExecutor` 显式配置

## 9. 实战里最容易出现什么问题

最常见的通常有：

1. 线程池过小，任务处理不过来
2. 队列太大，任务堆积严重
3. 线程池过大，系统资源被打爆
4. 没有监控，出了问题不知道是线程满了还是任务慢了
5. 业务线程池混用，互相影响

## 10. 面试里怎么回答“线程池怎么配”

可以直接背一版：

线程池怎么配不能死背固定数字，我一般会先看任务类型。如果是 CPU 密集型任务，线程数通常不会开太大，更接近 CPU 核数；如果是 IO 密集型任务，线程数可以适当更多一些。然后再结合任务执行时长、请求峰值、是否允许堆积来调整队列长度和拒绝策略。实际项目里我会更倾向直接用 `ThreadPoolExecutor` 显式配置，而不是无脑用 `Executors` 默认工厂。

## 11. 面试里怎么回答“任务堆积了怎么办”

可以背一版短的：

如果任务堆积，我会先区分是线程池太小、任务执行太慢，还是下游依赖卡住了。然后再看队列长度、活跃线程数、任务耗时和拒绝次数这些指标，必要时调整线程池参数、拆分业务线程池、优化慢任务，或者从业务侧限流，不能只盯着把线程池开大。

## 12. 当前阶段记忆重点

这一份最少要记住：

1. 线程池实战核心是任务类型和业务场景
2. CPU 密集和 IO 密集的配法思路不同
3. 队列不是越大越好
4. 拒绝策略本质是在保护系统
5. 实际项目里更建议显式配置 `ThreadPoolExecutor`
6. 任务堆积不能只靠“加大线程数”解决
