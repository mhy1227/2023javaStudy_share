# 线程池监控与动态调优补充

## 1. 为什么这一份要单独补

前面的线程池文档已经讲了：

- 参数是什么
- 怎么建线程池
- 任务堆积了怎么排查

但项目面试里还会继续追问：

- 线程池上线后你怎么看它是否健康？
- 任务堆积你怎么发现？
- 线程池参数是不是能动态调？
- 你怎么判断该加线程还是该优化任务？

所以这一份主要补：

- 线程池运行时看什么
- 怎么判断线程池是否异常
- 参数调优应该怎么想

## 2. 先一句话理解线程池监控

可以先记一个口径：

- 线程池监控不是只看线程数，而是看线程、队列、任务耗时和拒绝情况是否持续异常

也就是说，线程池健康不健康，不能只问：

- 开了多少线程

而要看：

- 线程是不是长期打满
- 队列是不是长期堆积
- 任务是不是越跑越慢
- 有没有触发拒绝策略

## 3. 最常看的运行指标

当前阶段建议先记这些：

1. `corePoolSize`
2. `maximumPoolSize`
3. `poolSize`
4. `activeCount`
5. `queue.size()`
6. `taskCount`
7. `completedTaskCount`
8. `largestPoolSize`
9. 拒绝次数
10. 任务执行耗时

这些指标可以分成三类：

- 线程情况
- 队列情况
- 任务情况

## 4. 线程情况怎么看

### 4.1 `poolSize`

- 当前线程池实际线程数量

如果 `poolSize` 长期接近 `maximumPoolSize`，说明线程池经常被压到上限。

### 4.2 `activeCount`

- 当前正在执行任务的线程数

如果 `activeCount` 长期接近 `maximumPoolSize`，通常说明：

- 活跃线程长期打满
- 任务处理速度跟不上提交速度

但这还不能直接说明一定要加线程。

还要继续看：

- 任务是在算 CPU
- 还是卡在 IO / DB / RPC

## 5. 队列情况怎么看

最直接看的就是：

- `queue.size()`

如果队列长度短时间升高，可能只是流量峰值。

如果队列长度长期不下降，说明：

- 任务进入速度大于处理速度
- 线程池已经有积压风险

这时不能只想着：

- 把队列调大

因为队列越大，可能只是把问题藏起来，让请求延迟越来越长。

## 6. 任务情况怎么看

常看：

- 任务总数
- 已完成任务数
- 任务平均耗时
- 慢任务比例
- 异常任务数量

如果任务执行耗时突然变长，要继续判断：

1. 是业务代码变慢
2. 是下游数据库变慢
3. 是 RPC 变慢
4. 是锁竞争变严重
5. 是机器资源不足

线程池打满很多时候只是结果，不一定是根因。

## 7. 拒绝次数为什么很重要

拒绝次数说明：

- 线程池和队列都已经扛不住新任务了

如果拒绝次数开始增加，就要立刻关注：

- 是否有流量突增
- 是否有慢任务
- 是否下游异常
- 拒绝策略是否符合业务预期

拒绝不是单纯坏事。

它本质上是在告诉你：

- 系统已经开始自我保护

## 8. 怎么判断线程池是不是“打满”

可以看三个信号：

1. `activeCount` 长期接近 `maximumPoolSize`
2. 队列长度持续上升或长期不下降
3. 拒绝次数开始出现或持续增加

如果三者同时出现，基本可以认为：

- 线程池已经明显承压

## 9. 打满以后第一反应不是加线程

这是很关键的面试点。

线程池打满以后，不应该第一句话就说：

- 我把线程数调大

更稳的思路是：

1. 先判断任务为什么退不下来
2. 再判断是线程不够，还是任务太慢
3. 再判断下游是否已经成为瓶颈
4. 最后才考虑调线程数或队列

如果下游数据库已经慢了，盲目加线程可能会：

- 把下游打得更慢
- 让整体故障扩大

## 10. 什么时候可以考虑增加线程数

一般更适合在这些情况下考虑：

- 任务偏 IO 密集
- CPU 还有明显余量
- 下游还有承载余量
- 队列有堆积但任务本身不是异常慢

如果任务是 CPU 密集型，并且 CPU 已经很高，继续加线程通常意义不大。

## 11. 什么时候应该优化任务而不是调线程池

如果发现：

- 单个任务耗时明显变长
- 慢任务比例上升
- 下游响应变慢
- 锁等待明显增加

这时更应该优先看：

- 业务代码
- SQL / 索引
- RPC 超时
- 缓存命中率
- 锁粒度

不是所有问题都靠线程池参数解决。

## 12. 动态调参可以调哪些

`ThreadPoolExecutor` 里部分参数是可以运行时调整的，例如：

- `setCorePoolSize()`
- `setMaximumPoolSize()`
- `setKeepAliveTime()`

但要注意：

- 能动态调，不代表应该随便调

动态调参也要有依据：

- 当前指标
- 历史基线
- 流量峰值
- 下游承载能力

## 13. 动态调参的风险

动态调参如果做不好，会带来：

1. 线程数被调得过大，资源被打满
2. 队列堆积被掩盖，请求延迟持续恶化
3. 下游被更多并发压垮
4. 不同业务线程池互相影响

所以面试里可以说：

- 动态调参要结合监控、告警和业务边界，不能只做一个能改参数的页面

## 14. 为什么线程池要按业务隔离

监控调优里，这个点很重要。

如果所有业务共用一个线程池：

- 慢业务可能拖垮快业务
- 非核心业务可能挤占核心业务资源
- 出问题时不容易定位是哪类任务造成的

所以项目里更推荐：

- 核心业务和非核心业务隔离
- IO 密集和 CPU 密集任务隔离
- 不同下游依赖尽量隔离

## 15. 告警一般关注什么

可以先记这几个方向：

1. 活跃线程数长期接近最大线程数
2. 队列长度超过阈值
3. 拒绝次数大于 0 或持续增长
4. 任务平均耗时明显升高
5. 任务执行异常增多

如果项目里有监控系统，这些指标就很适合做告警。

## 16. 面试里怎么回答“你怎么监控线程池”

可以直接背一版：

我会重点看活跃线程数、当前线程数、队列长度、任务总数、完成任务数、拒绝次数和任务耗时。如果活跃线程长期接近最大线程数，同时队列持续堆积，甚至开始触发拒绝策略，就说明线程池已经明显承压。之后我会继续判断是任务执行慢、线程数不足，还是下游依赖卡住，而不是第一时间盲目加线程。

## 17. 面试里怎么回答“线程池打满了怎么调”

可以直接背一版：

线程池打满后我会先看任务为什么退不下来。如果是 IO 密集任务、CPU 还有余量、下游也能扛住，可以考虑适当增加线程数；如果是任务本身变慢、数据库或 RPC 卡住，就应该优先优化慢任务或下游，而不是继续加线程。队列和拒绝策略也要结合业务是否允许排队、是否允许降级来调整。

## 18. 当前阶段最值得记什么

这一份最少要记住：

1. 线程池监控看线程、队列、任务、拒绝
2. 队列长期堆积说明处理速度跟不上提交速度
3. 打满后先找根因，不要第一反应加线程
4. 动态调参要结合监控和业务边界
5. 线程池按业务隔离更利于风险控制和排障

## 19. 一句话总结

线程池调优的核心不是把参数调大，而是先通过监控判断：

- 压力在哪里
- 任务为什么慢
- 系统还能不能承受更多并发
