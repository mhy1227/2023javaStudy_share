- `CompletableFuture` 的进阶追问和项目口径

## 2. 先一句话理解进阶追问在问什么

可以先记一个口径：

- `CompletableFuture` 基础是在讲“会不会用”，进阶追问是在讲“你知不知道它在线上怎么安全使用”

## 3. 默认线程池是什么

这是高频问题。

如果你没有显式传线程池，很多 `CompletableFuture` 异步方法默认可能会使用：

- `ForkJoinPool.commonPool()`

这就是为什么它经常和 `ForkJoinPool` 一起被问到。

## 4. 为什么很多时候更推荐自定义线程池

因为默认公共线程池是共享资源。

如果很多业务都直接用默认线程池，可能出现：

1. 不同业务互相影响
2. 阻塞任务拖慢整个公共池
3. 排障时任务来源不清晰
4. 参数不够业务可控

所以实际项目里，一个更稳的口径是：

- 重要异步任务更推荐显式传入自定义线程池

## 5. `allOf()` 和 `anyOf()` 怎么理解

### 5.1 `allOf()`

- 等所有任务都完成

适合：

- 多个并行查询后统一汇总结果

### 5.2 `anyOf()`

- 任意一个任务先完成就可以继续

适合：

- 只要先拿到任意一个结果就能往下走的场景

## 6. 多个任务并行后怎么汇总结果

这是项目面试高频场景。

常见主线：

1. 用多个 `supplyAsync()` 并行提交任务
2. 用 `allOf()` 等待全部完成
3. 再调用各自的 `join()` 取结果
4. 最后做统一汇总

这类场景常见于：

- 查用户信息
- 查订单信息
- 查库存
- 查优惠券

## 7. `get()` 和 `join()` 有什么区别

这是高频追问。

可以先粗分成：

- `get()` 会抛受检异常
- `join()` 更偏运行时异常包装

面试里一个常见口径是：

- `get()` 更传统，异常处理更显式
- `join()` 在函数式链路里更顺手

## 8. 异常处理最值得记的三个方法

当前阶段建议先记：

1. `exceptionally()`
2. `handle()`
3. `whenComplete()`

### 8.1 `exceptionally()`

- 出异常时给一个兜底结果

### 8.2 `handle()`

- 不管成功还是失败，都能统一处理，并且能返回新结果

### 8.3 `whenComplete()`

- 更像回调通知
- 可以看成功或失败结果
- 但不负责真正替换最终结果

## 9. `exceptionally()`、`handle()`、`whenComplete()` 怎么区分

可以先记一个实用口径：

- `exceptionally()`：只管异常兜底
- `handle()`：成功失败都处理，还能改返回值
- `whenComplete()`：更像做收尾记录，不改变主结果链路

## 10. 超时怎么想

这也是很像项目经验的追问。

因为多个异步任务并行时，最怕的不是：

- 跑不起来

而是：

- 某一个任务拖太久，把整体接口也拖慢

所以项目里通常要考虑：

- 单任务超时
- 整体等待超时
- 超时后的降级和兜底

## 11. 为什么异步不一定更快

这是很值得讲的一点。

异步不一定更快，原因包括：

1. 任务本身不独立
2. 下游本来就是瓶颈
3. 线程池参数不合理
4. 任务拆得太碎，调度成本变高
5. 异常和超时处理不好

所以面试里可以说：

- 异步只是提高并行度的一种方式，不是所有场景都一定能提升整体性能

## 12. 为什么不能把所有查询都无脑并行化

因为并行也有代价。

比如：

- 会增加线程池压力
- 会增加下游并发压力
- 会增加超时和异常处理复杂度

所以你更稳的口径应该是：

- 只有在任务彼此独立、并且下游能承受并发时，才值得并行化

## 13. 一个常见项目回答模板

如果面试官问：

- 你项目里怎么用 `CompletableFuture`？

可以直接按这条线答：

1. 哪些任务彼此独立
2. 为什么考虑并行
3. 用了哪个自定义线程池
4. 怎么做超时控制
5. 怎么做异常兜底
6. 最后怎么汇总结果

这样回答会更像真实项目经验。

## 14. 面试里怎么回答“为什么自定义线程池”

可以直接背一版：

如果不显式传线程池，很多异步任务默认可能会走 `ForkJoinPool.commonPool()`。但公共线程池是共享资源，不同业务可能互相影响，所以实际项目里我更倾向给重要异步任务传入自定义线程池，这样线程数、队列和任务来源都更可控，也更方便隔离和排障。

## 15. 面试里怎么回答“异常怎么处理”

可以直接背一版：

`CompletableFuture` 里我通常会根据场景选择 `exceptionally()`、`handle()` 或 `whenComplete()`。如果只是给异常兜底结果，会先想到 `exceptionally()`；如果成功失败都要统一处理并返回新结果，会考虑 `handle()`；如果只是做日志、埋点、收尾动作，`whenComplete()` 会更自然。

## 16. 面试里怎么回答“多个任务怎么汇总”

可以直接背一版：

如果多个任务彼此独立，我会用多个 `supplyAsync()` 并行提交，再用 `CompletableFuture.allOf()` 等全部完成，最后统一 `join()` 拿结果并做汇总。同时我会配合自定义线程池、超时控制和异常兜底，避免某一个慢任务把整体链路拖垮。

## 17. 当前阶段最值得记什么

这一份最少要记住：

1. 默认线程池可能是 `ForkJoinPool.commonPool()`
2. 重要异步任务更推荐传自定义线程池
3. `allOf()` 和 `anyOf()` 的区别
4. `get()` 和 `join()` 的区别
5. `exceptionally()`、`handle()`、`whenComplete()` 的区别
6. 异步任务要考虑超时、异常、下游承载

## 18. 一句话总结

`CompletableFuture` 进阶追问的核心不是方法名背得多，而是要能回答：

- 线程池怎么选
- 异常怎么兜底
- 多任务怎么汇总
- 为什么异步不一定就更快
