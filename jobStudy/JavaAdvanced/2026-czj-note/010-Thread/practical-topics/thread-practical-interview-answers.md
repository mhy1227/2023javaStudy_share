# 线程实战专题示例回复

## 1. 这一份怎么用

这一份不是标准唯一答案，而是偏“面试里能讲顺”的口语化示例回复。

使用建议：

- 先自己答
- 再对照这一份查漏补缺
- 最后把表述改成更像你自己的说法

## 2. 为什么不推荐在业务代码里频繁 `new Thread()`？

示例回复：

因为频繁 `new Thread()` 的成本比较高，而且线程数量容易失控。线程一多，调度和上下文切换的开销也会上来，还不方便统一做监控、限流和异常处理。所以实际开发里更常见的做法是把任务交给线程池统一管理。

## 3. 线程池核心参数有哪些？

示例回复：

线程池最核心的几个参数一般是：核心线程数、最大线程数、空闲线程存活时间、阻塞队列、线程工厂和拒绝策略。面试里如果继续追，我会补一句，这些参数不是死背，而是要结合任务类型、任务耗时和流量峰值一起定。

## 4. CPU 密集型和 IO 密集型任务怎么配置线程池？

示例回复：

CPU 密集型任务一般线程数不会开太大，通常更接近 CPU 核数，因为这类任务主要在抢 CPU；IO 密集型任务因为很多时间都在等待网络、数据库或者磁盘，所以线程数通常可以适当放大一些。核心不是记公式，而是先判断任务到底卡 CPU 还是卡等待。

## 5. 为什么不建议无脑使用 `Executors` 默认线程池工厂？

示例回复：

因为有些默认工厂把线程数或者队列长度隐藏掉了，项目里不太利于风险控制。比如队列可能过大，容易导致任务积压得很深；线程数如果不受控，也可能把系统资源打满。所以更稳妥的做法一般是显式使用 `ThreadPoolExecutor`，把参数按业务场景配置清楚。

## 6. `submit()` 和 `execute()` 有什么区别？

示例回复：

`execute()` 更像是把任务直接扔进去执行，没有返回值；`submit()` 会返回 `Future`，后面可以拿结果，也能感知任务执行异常。实际开发里如果任务需要返回值，或者后续还要组合结果，通常会更常用 `submit()` 或者进一步用 `CompletableFuture`。

## 7. 线程池打满了你一般先看什么？

示例回复：

我一般会先看活跃线程数是不是已经打满，再看队列是不是在持续堆积，然后再看任务平均耗时有没有明显变长。接着我会判断是任务本身慢、锁竞争严重，还是下游数据库、RPC 之类的依赖卡住了。线程池打满通常只是现象，关键还是找到为什么任务退不下来。

## 8. `CompletableFuture` 是干什么的？

示例回复：

`CompletableFuture` 可以理解成 Java 里很常用的异步编排工具。它不只是把任务异步执行出去，更重要的是能把多个异步步骤串起来，比如前一个结果出来后继续加工、多个任务并行后统一汇总，或者把异常处理也接在链路上。

## 9. `Future` 和 `CompletableFuture` 有什么区别？

示例回复：

`Future` 更像一个异步结果占位符，提交任务后，后面去拿结果；`CompletableFuture` 则更进一步，能做链式编排、任务组合和异常处理。简单说，`Future` 更偏“拿结果”，`CompletableFuture` 更偏“组织整个异步流程”。

## 10. `runAsync()` 和 `supplyAsync()` 有什么区别？

示例回复：

`runAsync()` 适合没有返回值的异步任务，`supplyAsync()` 适合有返回值的异步任务。这两个是最基础的一组，很多后续链式操作其实都是从这里接出去的。

## 11. `thenApply()`、`thenAccept()`、`thenRun()` 有什么区别？

示例回复：

我一般这么记：`thenApply()` 是拿上一步结果继续加工，而且自己也有返回值；`thenAccept()` 是消费上一步结果，但自己不再返回值；`thenRun()` 是不关心上一步具体结果，只是任务结束后继续做下一个动作。这三个区分清楚以后，`CompletableFuture` 的主线就会顺很多。

## 12. 为什么 `CompletableFuture` 最好配合自定义线程池？

示例回复：

因为如果不显式指定，很多异步任务会走默认线程池。在简单示例里没问题，但一到真实项目里，不同业务的耗时和风险完全不同，如果全混在一起，很容易互相影响。所以更稳妥的做法通常是按业务隔离线程池，再把 `CompletableFuture` 任务明确丢到对应线程池里。

## 13. 多个异步任务并行查数据再汇总，你会怎么做？

示例回复：

我一般会用多个 `CompletableFuture.supplyAsync()` 并行去查，然后用 `allOf()` 等它们都完成，最后再统一做结果汇总。如果其中某个任务失败，还会配合异常处理逻辑决定是兜底返回、降级，还是直接整体失败。

## 14. CPU 飙高时你一般怎么查？

示例回复：

我会先确认是不是线程层面的热点问题，再定位占 CPU 高的线程，然后结合线程栈看它到底在执行什么代码。通常要判断是死循环、频繁自旋、热点代码计算过重，还是锁竞争导致线程反复抢占。也就是说，最终一定要从“CPU 高”落到“具体哪个线程在跑哪段代码”。

## 15. 怀疑死锁时你会怎么定位？

示例回复：

如果怀疑死锁，我会先拿线程 dump，看线程之间是不是互相持有对方需要的锁。这个时候 `jstack` 很常用，但更关键的是能从 dump 里看出线程在等谁、谁又被谁卡住，最后把锁关系还原出来。

## 16. 大量线程都在等待时，你怎么区分是锁问题还是下游阻塞？

示例回复：

我会先看线程状态和调用栈。如果很多线程都卡在同步块、锁获取或者 `park` 相关位置，那更像锁竞争；如果大量线程都卡在网络读写、数据库调用、RPC 或 IO 相关位置，那更像下游阻塞。判断这类问题，线程状态只是入口，最后还是要看栈里卡在哪个方法。

## 17. 如果面试官问“你在线上怎么用过线程”，你怎么组织回答？

示例回复：

我一般会按三层来讲。第一层讲场景，比如接口聚合、异步通知、批量任务处理；第二层讲方案，比如用了线程池还是 `CompletableFuture`，线程池怎么隔离；第三层讲风险控制，比如怎么监控任务堆积、怎么处理超时和异常、线程池打满时怎么兜底。这样回答会更像真实项目经验。

## 18. 一句话收尾

线程实战这一层，面试官真正想听的通常不是你背了多少类名，而是你能不能把“场景、方案、风险、排障”讲成一条完整链路。
