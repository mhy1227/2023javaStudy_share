# 线程场景题整理

## 1. 为什么这一份要单独补

线程这一章如果只背概念，面试里很容易卡在场景题上。

场景题常见特点是：

- 不直接问某个 API 是什么
- 而是给你一个问题，让你选择线程工具和排查思路

所以这一份主要整理：

- 线程池场景题
- 异步编排场景题
- `ThreadLocal` 场景题
- 线程排障场景题
- 多线程协作场景题

## 2. 场景题回答的通用套路

遇到线程场景题，建议先按这条线想：

1. 这是计算问题、IO 等待问题，还是线程协作问题？
2. 有没有共享资源？
3. 是要提高吞吐，还是要控制顺序？
4. 能不能接受失败、丢弃或降级？
5. 出问题时要看哪些指标？

不要一上来就报工具名。

先把问题性质判断清楚，再选择：

- 线程池
- 锁
- `CompletableFuture`
- `BlockingQueue`
- `CountDownLatch`
- `Semaphore`
- `ThreadLocal`

## 3. 场景一：线程池打满了怎么办

题目可能这样问：

- 线上线程池打满了，你怎么排查？
- 任务一直堆积，你怎么处理？

回答主线：

1. 看 `activeCount` 是否长期接近最大线程数
2. 看队列长度是否持续堆积
3. 看拒绝次数是否上升
4. 看任务耗时是否变长
5. 判断是任务慢、线程少，还是下游卡住

不要第一句就说：

- 加大线程数

更稳的口径是：

- 先找任务退不下来的原因，再决定是调线程池、优化任务、限流降级，还是处理下游瓶颈

## 4. 场景二：多个 RPC 并行调用后汇总

题目可能这样问：

- 一个接口里要查用户、订单、优惠券、库存，你怎么提高响应速度？

回答主线：

- 如果多个查询彼此独立，可以考虑并行调用
- Java 里可以用 `CompletableFuture`
- 多个任务并行后，用 `allOf()` 等待完成，再汇总结果

必须补的风险点：

- 要设置超时
- 要处理异常
- 要使用自定义线程池
- 要考虑下游承载能力

面试口径：

- 我不会简单把所有任务都异步化，而是先判断它们是否互相独立，再用 `CompletableFuture` 并行编排，并配合超时、异常兜底和自定义线程池隔离。

## 5. 场景三：100 个任务分 10 批执行，每批并发，批次有顺序

题目可能这样问：

- 有 100 个任务，每 10 个一批并发执行，但下一批必须等上一批结束后才能开始，怎么做？

回答主线：

1. 外层按批次循环
2. 每批提交 10 个任务到线程池
3. 当前批次用 `CountDownLatch` 等待全部完成
4. 当前批次结束后再进入下一批

可选工具：

- `CountDownLatch`
- `CompletableFuture.allOf()`
- 线程池

面试口径：

- 这类题核心不是创建多少线程，而是控制“批内并发、批间有序”。

## 6. 场景四：核心业务和非核心业务为什么要线程池隔离

题目可能这样问：

- 为什么不能所有异步任务共用一个线程池？

回答主线：

- 不同业务的耗时不同
- 优先级不同
- 失败代价不同

如果共用一个线程池，可能出现：

1. 非核心任务挤占核心任务资源
2. 慢任务拖垮快任务
3. 出问题时不好定位是哪类任务导致

面试口径：

- 线程池隔离的目的，是把不同业务的资源和风险隔开，避免一个慢任务或非核心任务拖垮核心链路。

## 7. 场景五：线程池里使用 `ThreadLocal` 为什么会有污染风险

题目可能这样问：

- 为什么在线程池里用 `ThreadLocal` 要 `remove()`？
- 请求 A 的数据为什么可能影响请求 B？

回答主线：

- 线程池线程会复用
- `ThreadLocal` 的值是跟线程绑定的
- 如果任务结束后不清理，后续任务复用同一个线程时可能读到旧值

推荐写法：

```java
try {
    CONTEXT.set(value);
    // 业务逻辑
} finally {
    CONTEXT.remove();
}
```

面试口径：

- 线程池里的线程生命周期很长，如果 `ThreadLocal` 不清理，就可能出现旧值残留和数据污染，所以一般放在 `finally` 里 `remove()`。

## 8. 场景六：CPU 飙高怎么排查

题目可能这样问：

- 服务 CPU 突然飙高，你怎么查？

回答主线：

1. 先确认是哪台机器、哪个进程 CPU 高
2. 再定位高 CPU 线程
3. 再看线程栈
4. 最后回到具体代码位置

常见原因：

- 死循环
- 高频自旋
- 热点计算逻辑
- 锁竞争导致线程频繁抢占

面试口径：

- CPU 飙高最终要落到具体线程和具体代码栈，不能只停留在“看监控”。

## 9. 场景七：大量线程都在等待，怎么区分原因

题目可能这样问：

- 你看到很多线程处于等待状态，怎么判断是锁问题还是下游问题？

回答主线：

- 如果栈里大量卡在锁、同步块、`park`，更像锁竞争或线程协作问题
- 如果大量卡在数据库、网络、RPC、IO 读写，更像下游依赖阻塞

面试口径：

- 我会结合线程状态和线程栈判断等待位置，不能只看状态名，还要看线程到底卡在哪个方法上。

## 10. 场景八：定时任务为什么不用 `Timer`

题目可能这样问：

- 定时任务你会用什么？
- 为什么不用 `Timer`？

回答主线：

- `Timer` 更接近单线程调度模型
- 一个任务执行久了，后续任务容易受影响
- 异常处理不好也可能影响调度线程
- `ScheduledThreadPool` 是线程池模型，更适合项目统一管理

面试口径：

- 简单任务 `Timer` 能做，但工程里更常用 `ScheduledThreadPool`，因为它支持线程池模型，多个任务之间影响更小，也更方便统一管理。

## 11. 场景九：生产者消费者怎么写

题目可能这样问：

- 一个线程生产数据，一个线程消费数据，怎么实现？

推荐方案：

- `BlockingQueue`

基础回答：

- 生产者 `put`
- 消费者 `take`
- 队列为空时消费者阻塞
- 队列满时生产者阻塞

也可以提：

- `wait/notify`
- `Condition`

但项目里一般更推荐先想到：

- `BlockingQueue`

## 12. 场景十：顺序打印 / 交替打印

题目可能这样问：

- 三个线程按顺序打印 A、B、C
- 两个线程交替打印奇偶数

核心考点：

- 线程协作
- 条件等待
- 唤醒时机

常见方案：

- `synchronized + wait/notifyAll`
- `ReentrantLock + Condition`
- `Semaphore`

面试口径：

- 这类题不是考打印本身，而是考“谁该执行、谁该等待、谁来唤醒下一个线程”。

## 13. 场景题最容易犯的错误

1. 一上来就报工具名，不分析问题性质
2. 线程池打满就说加线程
3. 异步任务不考虑超时和异常
4. `ThreadLocal` 不清理
5. 所有业务共用一个线程池
6. 只看线程状态，不看线程栈
7. 把“并发执行”和“顺序控制”混在一起

## 14. 当前阶段最值得背的 5 个口径

### 14.1 线程池打满

- 先看活跃线程数、队列长度、拒绝次数和任务耗时，再判断是线程少、任务慢还是下游卡住。

### 14.2 RPC 并行汇总

- 独立任务可以用 `CompletableFuture` 并行编排，但要配合自定义线程池、超时和异常兜底。

### 14.3 线程池隔离

- 不同业务耗时、优先级和失败代价不同，所以核心和非核心任务最好隔离线程池。

### 14.4 `ThreadLocal` 污染

- 线程池线程会复用，不清理 `ThreadLocal` 可能导致旧值残留和请求间数据污染。

### 14.5 CPU 飙高

- 最终要定位到高 CPU 线程和具体代码栈，不能只停在监控层。

## 15. 一句话总结

线程场景题的核心不是背 API，而是先判断问题性质，再选择合适的并发工具，并说明风险控制和排查思路。
