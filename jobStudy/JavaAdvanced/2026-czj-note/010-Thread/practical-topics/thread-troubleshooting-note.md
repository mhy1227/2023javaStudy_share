# 线程排障补充笔记

## 1. 为什么这一份要单独补

只要面试里开始带项目经验，很容易继续追：

- 线程出问题了你怎么查
- CPU 飙高怎么办
- 线程池打满了怎么看
- 死锁怎么定位

所以这一份不是再讲线程定义，而是补：

- 线程真的出问题时，你有没有排查思路

## 2. 先一句话理解线程排障在补什么

可以先记一个口径：

- 线程排障的核心，不是背工具名，而是先会判断“线程到底卡在哪一类问题上”

例如常见就包括：

1. 死锁
2. 线程池打满
3. 任务阻塞
4. CPU 飙高
5. 大量等待下游资源

## 3. 死锁怎么查

这个点前面的死锁文档已经讲过主线了，这里更偏排障口径。

可以先记：

- 出现疑似死锁时，先看线程 dump

高频会提到的工具是：

- `jstack`

它的核心价值是：

- 帮你看线程状态、锁等待关系、是否互相卡住

## 4. CPU 飙高时先想什么

如果面试官问：

- 某个服务 CPU 飙高，你怎么查？

你先不要一上来乱答命令。

先给主线：

1. 先确认是不是线程层问题
2. 再找占 CPU 高的线程
3. 再结合线程栈看它在跑什么逻辑

也就是说，重点不是只会说：

- 看 CPU

而是要知道：

- 最后要落到线程栈和代码位置

## 5. 线程池打满怎么理解

这是项目面试非常爱问的一类。

你至少要先能想到：

- 活跃线程数是不是已经打满
- 队列是不是已经堆积
- 任务是不是执行太慢
- 下游依赖是不是卡住了

这类题更想听的是：

- 你会不会先区分“线程池太小”和“任务本身太慢”

## 6. 为什么线程排障不能只盯线程池参数

因为很多问题表面上像：

- 线程池配置不合理

但本质可能是：

- SQL 太慢
- RPC 卡住
- 死循环
- 大量锁竞争

所以一个更成熟的回答应该包含：

- 参数只是表象，任务执行逻辑和下游依赖也要看

## 7. 线程 dump 为什么重要

因为很多线程问题最后都要回到：

- 线程当前到底在干什么

而线程 dump 正好能帮助你看：

- 线程状态
- 是否在等待锁
- 是否在 `WAITING` / `BLOCKED`
- 是否卡在某段业务代码

所以它是线程排障里特别高频的关键词。

## 8. 面试里怎么回答“线程池打满了怎么办”

可以直接背一版：

如果线程池打满，我不会只盯着把线程数调大，而是会先看活跃线程数、队列长度、任务耗时和拒绝次数这些指标，先判断是线程池太小、任务执行太慢，还是下游依赖阻塞了。然后再结合线程 dump 看线程卡在哪些代码上，必要时再调整线程池参数、拆分业务线程池、优化慢任务或者从业务侧限流。

## 9. 面试里怎么回答“CPU 飙高怎么查”

可以背一版短的：

如果 CPU 飙高，我会先确认是不是某些线程长期占用 CPU，再定位高 CPU 线程，结合线程栈看它在执行什么代码，比如死循环、热点计算、频繁重试等。线程问题排查不能只看系统指标，最后还是要落到线程栈和具体代码位置。

## 10. 当前阶段最值得先记的排障关键词

如果你现在不想学太散，可以先记住这些词：

1. `jstack`
2. 线程 dump
3. 线程状态
4. 活跃线程数
5. 队列堆积
6. 拒绝次数
7. 下游依赖阻塞

## 11. 当前阶段记忆重点

这一份最少要记住：

1. 线程排障先分问题类型
2. 死锁常看线程 dump / `jstack`
3. CPU 飙高最后还是要落到线程栈
4. 线程池打满要区分参数问题和任务慢问题
5. 很多线程问题本质不只是线程池配置，而是业务逻辑或下游依赖问题
