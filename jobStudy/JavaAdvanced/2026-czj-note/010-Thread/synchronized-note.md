# synchronized 说明

## 1. 为什么先讲 `synchronized`

在 Java 并发里，`synchronized` 基本可以看成：

- 最基础
- 最经典
- 面试最常问

的一种同步手段。

前面线程安全那份已经讲过：

- 多线程为什么会出问题
- 什么是原子性、可见性、有序性

这一份就顺着往下讲：

- Java 最常见的同步关键字 `synchronized` 到底是什么
- 它锁的是什么
- 它能解决哪些问题
- 面试里最容易怎么追问

## 2. 什么是 `synchronized`

可以先给一个容易口述的定义：

- `synchronized` 是 Java 提供的内置同步关键字，用来控制多个线程对共享资源的访问顺序

更直白一点说：

- 同一时刻，只让一个线程进入被它保护的那段代码

这样就能减少并发访问同一资源时的冲突。

## 3. 为什么需要 `synchronized`

因为多线程环境下，共享资源竞争会导致结果不正确。

最典型例子：

```java
class Counter {
    int count = 0;

    public void add() {
        count++;
    }
}
```

如果多个线程同时执行 `add()`，`count++` 不是原子操作，就可能出现丢失更新。

这时如果把关键代码保护起来：

```java
class Counter {
    int count = 0;

    public synchronized void add() {
        count++;
    }
}
```

同一时刻就只有一个线程能进入 `add()`，结果会更稳定。

## 4. `synchronized` 的核心作用

在当前阶段，可以先记住它的三个核心作用：

1. 保证临界区代码的互斥执行
2. 保证一定的可见性
3. 保证一定的有序性

但面试里最先强调的，通常还是：

- 互斥
- 原子性

## 5. 什么是临界区

临界区可以简单理解成：

- 那段访问共享资源、必须小心处理的代码

例如：

```java
count++;
```

如果多个线程同时执行，就可能出问题。

所以这段代码就属于典型临界区。

## 6. `synchronized` 有哪几种常见写法

最常见的有三种：

1. 同步实例方法
2. 同步静态方法
3. 同步代码块

## 7. 第一种：同步实例方法

### 7.1 写法

```java
public synchronized void add() {
    count++;
}
```

### 7.2 锁的是什么

锁的是：

- 当前对象实例，也就是 `this`

也就是说，多个线程如果访问的是同一个对象的这个同步方法，就会竞争同一把锁。

## 8. 第二种：同步静态方法

### 8.1 写法

```java
public static synchronized void add() {
    count++;
}
```

### 8.2 锁的是什么

锁的是：

- 当前类对应的 `Class` 对象

也就是说，它锁的不是某个具体实例，而是类级别的锁。

## 9. 第三种：同步代码块

### 9.1 写法

```java
synchronized (lock) {
    count++;
}
```

### 9.2 这一种的意义

同步代码块比同步方法更灵活，因为你可以自己指定锁对象，也可以缩小锁的范围。

例如：

```java
private final Object lock = new Object();

public void add() {
    synchronized (lock) {
        count++;
    }
}
```

## 10. `synchronized(this)`、实例同步方法、静态同步方法有什么区别

这是面试高频。

### 10.1 `synchronized(this)`

锁的是当前对象实例。

### 10.2 同步实例方法

本质上锁的也是当前对象实例，也就是 `this`。

### 10.3 同步静态方法

锁的是类对象，也就是 `当前类.class`。

所以最关键的区别在于：

- 实例锁和类锁不是同一把锁

## 11. 锁对象为什么很重要

因为 `synchronized` 能不能起到同步效果，关键不只是“有没有写这个关键字”，而是：

- 多个线程是不是在竞争同一把锁

例如下面这样就可能没有达到你以为的同步效果：

```java
public void add() {
    synchronized (new Object()) {
        count++;
    }
}
```

为什么？

因为每次进来都是一个新对象，相当于每次拿的都不是同一把锁。

## 12. `synchronized` 主要解决什么问题

### 12.1 原子性

这是它最直观的作用。

临界区代码被同一把锁保护后，同一时刻只能有一个线程执行，复合操作更容易保持完整。

### 12.2 可见性

线程释放锁前，对共享变量的修改，对后续获得同一把锁的线程通常是可见的。

### 12.3 有序性

`synchronized` 也能在一定程度上约束执行顺序。

当前阶段面试里你可以先稳妥地说：

- `synchronized` 不只是互斥，也能保证一定的可见性和有序性

## 13. 一个简单例子：卖票

```java
class TicketWindow {
    private int ticket = 100;

    public synchronized void sell() {
        if (ticket > 0) {
            System.out.println(Thread.currentThread().getName() + " sell " + ticket);
            ticket--;
        }
    }
}
```

这里如果不加 `synchronized`，多个线程可能同时读到相同票数，出现超卖。

加上后：

- 同一时刻只有一个线程能执行 `sell()`

## 14. `synchronized` 的优点

1. 语法简单
2. 是 Java 内置机制，使用直观
3. 对基础线程安全场景非常常用
4. 面试里容易展开讲

## 15. `synchronized` 的局限

也不要把它讲成“万能解法”。

它也有一些局限：

1. 不够灵活
2. 锁的获取和释放不如 `Lock` 明确
3. 不能像 `Lock` 一样方便做可中断、可超时、可公平等更细控制

所以后面才会继续补 `Lock` 和 `ReentrantLock`。

## 16. `synchronized` 和线程安全三要素的关系

结合前一份线程安全文档，可以这样记：

1. 它通过互斥进入来强化原子性
2. 它通过加锁和释放锁过程保证一定的可见性
3. 它也能在同步范围内保证一定的有序性

所以这也是为什么它能成为最基础的同步手段。

## 17. 面试里最容易漏的点

### 17.1 只会说“`synchronized` 能保证线程安全”

这还不够。

至少要补一句：

- 它本质上通过同一把锁控制临界区互斥访问

### 17.2 说不清锁的到底是谁

这是最常见的问题。

一定要分清：

- 实例方法锁 `this`
- 静态方法锁 `Class` 对象
- 代码块锁括号里的对象

### 17.3 以为写了 `synchronized` 就一定安全

如果不同线程根本没竞争同一把锁，那同步效果可能就达不到。

### 17.4 把 `sleep()` 和锁释放搞混

前面已经讲过：

- `sleep()` 不释放锁
- `wait()` 释放锁

这个点经常和 `synchronized` 一起被追问。

## 18. 面试里怎么回答“`synchronized` 锁的是什么”

可以这样答：

`synchronized` 锁的具体对象要看写法。如果是同步实例方法，锁的是当前对象实例 `this`；如果是同步静态方法，锁的是当前类对应的 `Class` 对象；如果是同步代码块，锁的是括号里指定的那个对象。能不能真正起到同步效果，关键在于多个线程是不是竞争同一把锁。

## 19. 面试里怎么回答“`synchronized` 能解决什么问题”

可以这样答：

`synchronized` 是 Java 的内置同步关键字，主要通过让同一时刻只有一个线程进入临界区代码，来解决共享资源并发访问时的线程安全问题。它最核心地保证了互斥执行，从而帮助解决原子性问题，同时也能保证一定的可见性和有序性。

## 20. 当前阶段最该记住的结论

1. `synchronized` 是最基础的同步手段
2. 它的核心是让同一时刻只有一个线程进入临界区
3. 同步实例方法锁的是 `this`
4. 同步静态方法锁的是 `Class` 对象
5. 同步代码块锁的是括号里的那个对象
6. 关键不只是“有没有锁”，而是“是不是同一把锁”

## 21. 一句话总结

`synchronized` 这一块最重要的不是死背语法，而是先讲清楚：为什么需要它、它锁的到底是谁、它通过什么方式控制临界区访问，以及它在原子性、可见性和有序性里分别起了什么作用。
