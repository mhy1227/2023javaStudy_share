# 线程面试题示例回复

## 1. 这一份怎么用

这一份不是标准唯一答案，而是偏“面试能说顺”的示例回复。

使用建议：

- 先自己答
- 再对照这一份查漏补缺
- 最后把表述改成你自己的话

## 2. 进程和线程有什么区别？

示例回复：

进程可以理解成一个正在运行中的程序实例，它更偏资源分配单位；线程是进程中的执行单元，更偏 CPU 调度单位。进程之间资源相对独立，而同一进程中的多个线程会共享大部分资源，所以线程切换成本通常比进程低，但也更容易出现线程安全问题。

## 3. 什么是多线程？为什么要有多线程？

示例回复：

多线程就是一个进程里同时存在多个线程，共同推进多个任务。之所以要有多线程，核心是为了提高程序响应能力、提高资源利用率，以及更自然地拆分任务。不过多线程带来效率的同时，也会带来共享资源竞争和线程安全问题。

## 4. 并发和并行有什么区别？

示例回复：

并发强调的是在同一段时间内多个任务都在推进，不一定同一时刻真的同时执行；并行强调的是同一时刻多个任务真正同时执行。面试里一般可以顺带补一句，单核 CPU 更偏并发，多核 CPU 更容易体现真正的并行。

## 5. 创建线程有哪几种方式？

示例回复：

Java 里常见的线程创建方式有：继承 `Thread`、实现 `Runnable`、实现 `Callable` 并配合 `FutureTask`，以及通过线程池提交任务。面试里如果问基础，我会先讲 `Thread` 和 `Runnable`；如果问实际开发，我会补充更常用的是线程池。

## 6. `run()` 和 `start()` 有什么区别？

示例回复：

`run()` 只是线程任务逻辑的方法体，直接调用它本质上还是普通方法调用；`start()` 才是真正启动线程的方法，调用后会创建新的执行路径，再由新线程去执行 `run()` 里的逻辑。

## 7. `Runnable` 和 `Callable` 有什么区别？

示例回复：

`Runnable` 没有返回值，也不能直接抛出受检异常；`Callable` 可以有返回值，也可以抛受检异常，通常会配合 `Future` 或 `FutureTask` 使用。如果任务需要拿结果，通常就会考虑 `Callable`。

## 8. 线程有哪些状态？

示例回复：

Java 线程通常有 6 种状态：`NEW`、`RUNNABLE`、`BLOCKED`、`WAITING`、`TIMED_WAITING` 和 `TERMINATED`。其中 `RUNNABLE` 表示具备运行条件，不一定就是正在跑；`BLOCKED` 常见于等锁；`WAITING` 和 `TIMED_WAITING` 的区别主要在于是否带超时。

## 9. 什么是线程安全？

示例回复：

线程安全指的是在多线程并发访问共享资源时，程序仍然能够得到正确、可预期的结果。线程安全问题通常出现在多个线程同时访问或修改同一份共享数据时，核心往往和原子性、可见性、有序性有关。

## 10. 为什么 `i++` 在线程下不安全？

示例回复：

因为 `i++` 不是原子操作，它通常包含读取、计算、写回三个步骤。多个线程同时执行时，这三步可能交错进行，就会产生丢失更新，所以 `i++` 在并发环境下通常是不安全的。

## 11. 什么是原子性、可见性、有序性？

示例回复：

- 原子性：一个操作要么全部完成，要么全部不做，中间不能被打断。
- 可见性：一个线程修改了共享变量，其他线程能够及时看到这个变化。
- 有序性：程序执行顺序在并发环境下是否仍然符合预期，是否会受到重排序影响。

## 12. `sleep()` 和 `wait()` 有什么区别？

示例回复：

`sleep()` 是 `Thread` 类的静态方法，用来让当前线程休眠一段时间，不会释放锁；`wait()` 是 `Object` 类的方法，通常必须在同步代码块中使用，它会让当前线程进入等待状态并释放当前对象锁，通常需要 `notify()` 或 `notifyAll()` 来唤醒。

## 13. `wait()` 为什么必须在同步代码块中使用？

示例回复：

因为 `wait()` 本质上是和对象监视器配套使用的，只有先持有这个对象的监视器锁，才有资格调用它的 `wait()`。如果不在同步代码块里直接调用，通常会抛出 `IllegalMonitorStateException`。

## 14. `notify()` 和 `notifyAll()` 有什么区别？

示例回复：

`notify()` 只会唤醒一个等待线程，`notifyAll()` 会唤醒所有等待线程。但被唤醒线程并不是立刻执行，而是要重新竞争锁，最终谁先拿到锁谁先继续执行。

## 15. `join()` 是做什么的？

示例回复：

`join()` 的作用是让当前线程等待目标线程执行结束。也就是说谁调用 `join()`，谁就进入等待，不是让被调用的那个线程去等别人。

## 16. 什么是上下文切换？

示例回复：

上下文切换可以理解成 CPU 从当前线程切走、保存它的执行现场，再切换到另一个线程继续执行。这个过程不是免费的，所以线程数量并不是越多越好，线程太多反而可能把时间浪费在切换上。

## 17. 什么是 `synchronized`？

示例回复：

`synchronized` 是 Java 提供的内置同步关键字，用来控制多个线程对共享资源的访问顺序。它的核心是让同一时刻只有一个线程进入被保护的临界区，从而帮助解决线程安全问题。

## 18. `synchronized` 锁的是什么？

示例回复：

要看具体写法。如果是同步实例方法，锁的是当前对象实例 `this`；如果是同步静态方法，锁的是当前类对应的 `Class` 对象；如果是同步代码块，锁的是括号里指定的那个对象。能不能真正起到同步效果，关键在于多个线程是不是在竞争同一把锁。

## 19. 什么是 `volatile`？

示例回复：

`volatile` 是 Java 提供的关键字，用来修饰共享变量，主要保证这个变量在多线程环境下的可见性，并在一定程度上保证有序性。最经典的使用场景就是停止标记或开关变量。

## 20. 为什么 `volatile` 不能保证 `i++` 线程安全？

示例回复：

因为 `volatile` 主要解决的是可见性问题，而 `i++` 的问题本质上是原子性。`i++` 包含读取、计算、写回三个步骤，`volatile` 不能把这三个步骤变成一个不可分割的原子动作，所以它不能单独保证 `i++` 的线程安全。

## 21. 什么是 `ReentrantLock`？

示例回复：

`ReentrantLock` 是 `Lock` 接口最常见的实现类之一，可以理解成更灵活的显式可重入锁。它和 `synchronized` 的目标类似，都是保护共享资源，但它提供了更多控制能力，比如 `tryLock()`、可中断获取锁、公平锁和非公平锁等。

## 22. 什么是可重入？

示例回复：

可重入指的是同一个线程在已经持有某把锁的前提下，还可以再次获得这把锁，而不会把自己阻塞住。`ReentrantLock` 是可重入锁，`synchronized` 也是可重入的。

## 23. 为什么 `unlock()` 一般要放 `finally`？

示例回复：

因为如果业务代码中间抛了异常，而你没有在 `finally` 中释放锁，就可能导致锁一直不被释放，其他线程一直拿不到锁。所以 `Lock` 通常要和 `try-finally` 配套使用，确保锁一定会被释放。

## 24. 为什么不推荐频繁 `new Thread()`？

示例回复：

因为线程的创建和销毁都有成本，线程过多会增加调度和上下文切换开销，也不方便统一管理线程数量和任务堆积情况。所以实际开发里更推荐用线程池来统一管理线程。

## 25. 什么是线程池？

示例回复：

线程池是一种对线程进行复用和统一管理的机制，它会预先或按需创建一批线程，用来执行提交进来的任务。核心价值就是复用线程、控制并发数量、统一管理任务执行过程。

## 26. `execute()` 和 `submit()` 有什么区别？

示例回复：

`execute()` 更偏向执行一个不需要返回值的任务；`submit()` 更灵活，它可以提交任务并返回 `Future`，便于后续获取执行结果或处理异常。

## 27. 线程池核心参数有哪些？

示例回复：

`ThreadPoolExecutor` 常见核心参数有：核心线程数 `corePoolSize`、最大线程数 `maximumPoolSize`、空闲线程存活时间 `keepAliveTime`、时间单位 `unit`、任务队列 `workQueue`、线程工厂 `threadFactory` 和拒绝策略 `handler`。面试里最好还能顺着讲任务流转过程。

## 28. 为什么不建议无脑用 `Executors` 默认工厂？

示例回复：

因为 `Executors` 的默认工厂屏蔽了很多关键参数，不同工厂可能带来队列过长或线程数失控等风险。生产环境里更常见的建议是直接显式使用 `ThreadPoolExecutor`，这样更方便控制核心线程数、最大线程数、队列和拒绝策略。

## 29. `ConcurrentHashMap` 为什么比 `Hashtable` 更常用？

示例回复：

它们都能用于线程安全的 `Map` 场景，但 `Hashtable` 的设计更老、控制更粗，而 `ConcurrentHashMap` 更适合现代并发场景，通常性能和伸缩性更好，所以实际开发和面试里更常提到 `ConcurrentHashMap`。

## 30. `CopyOnWriteArrayList` 适合什么场景？

示例回复：

`CopyOnWriteArrayList` 更适合读多写少的并发场景，因为它采用写时复制思路，读操作体验较好，但写操作成本更高，所以不适合频繁修改的数据结构。

## 31. `BlockingQueue` 有什么用？

示例回复：

`BlockingQueue` 是带阻塞能力的队列接口，常用于生产者消费者模型和任务缓冲。队列空时消费者可以等待，队列满时生产者也可以等待，所以它在并发任务协作里非常常见，线程池内部也会涉及类似的任务排队思路。

## 32. 一段比较稳的总回答

如果面试官让你整体讲线程这一章，可以这样说：

Java 线程这一块我一般先从基础概念去理解，比如进程、线程、多线程、并发和并行的区别。再往下是线程创建方式、线程状态，以及 `sleep`、`wait`、`notify`、`join` 这些常见方法。真正进入并发核心后，重点是线程安全，也就是原子性、可见性和有序性。对应的解决手段里，`synchronized` 更偏互斥和原子性，`volatile` 更偏可见性，`Lock` 则提供了更灵活的显式锁控制。再往实际开发走，就会讲到线程池和并发容器，比如 `ThreadPoolExecutor`、`ConcurrentHashMap`、`CopyOnWriteArrayList` 和 `BlockingQueue`。

## 33. 面试前最后背的结论

1. 线程这章的主线不是工具名，而是“问题和解决手段”
2. `run()` 和 `start()`、`sleep()` 和 `wait()`、`synchronized` 和 `volatile` 都是高频区别题
3. 线程安全一定要能讲到原子性、可见性、有序性
4. `volatile` 不能保证 `i++` 是非常高频的考点
5. 线程池要会讲参数和任务流转过程
6. 并发容器要会讲 `ConcurrentHashMap`、`CopyOnWriteArrayList`、`BlockingQueue`
