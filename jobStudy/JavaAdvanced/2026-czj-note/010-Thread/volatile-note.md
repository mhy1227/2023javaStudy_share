# volatile 说明

## 1. 为什么在 `synchronized` 后面讲 `volatile`

因为很多人学完 `synchronized` 后，很容易产生两个误区：

1. 以为线程安全只能靠加锁
2. 以为 `volatile` 也能像 `synchronized` 一样把所有线程安全问题都解决掉

这两个理解都不够准确。

所以这一份要解决的核心就是：

- `volatile` 到底解决什么问题
- 它不解决什么问题
- 它和 `synchronized` 到底有什么本质区别

## 2. 什么是 `volatile`

可以先给一个容易口述的定义：

- `volatile` 是 Java 提供的关键字，用来修饰共享变量，主要保证这个变量在多线程环境下的可见性，并在一定程度上保证有序性

更直白一点说：

- 一个线程改了这个变量，其他线程能尽快看到最新值

## 3. 为什么需要 `volatile`

前面线程安全那份已经讲过：

- 可见性问题是并发里的三大核心问题之一

也就是说：

- 线程 A 改了值
- 线程 B 不一定马上看得到

如果程序恰好依赖“另一个线程能及时看到这个变化”，那就可能出问题。

这时 `volatile` 就有用了。

## 4. `volatile` 最核心的作用：可见性

这是必须先记住的第一点。

如果一个共享变量被 `volatile` 修饰，那么：

- 一个线程对它的修改
- 对其他线程会更快可见

面试里可以先用这句稳妥表达：

- `volatile` 的核心作用是保证共享变量修改后的可见性

## 5. 一个最典型的例子：停止标记

例如有一个线程不停循环工作：

```java
class Worker implements Runnable {
    private volatile boolean running = true;

    @Override
    public void run() {
        while (running) {
            // do something
        }
    }

    public void stop() {
        running = false;
    }
}
```

这里的关键是：

- 子线程在循环里不断读 `running`
- 另一个线程可能会把它改成 `false`

如果没有 `volatile`，子线程可能一直看不到最新值；
加上 `volatile` 后，这个停止标记更容易及时生效。

## 6. `volatile` 的第二个作用：一定程度上的有序性

除了可见性，`volatile` 还和有序性有关。

当前阶段可以先这样理解：

- 它能够在一定程度上禁止相关指令重排序

也就是说，在某些场景下，它不仅让“值”更容易被看到，也让“顺序”更符合预期。

但如果你现在还在打基础，先牢牢记住：

- `volatile` 最重要的是可见性

## 7. `volatile` 不能保证什么

这是最关键的边界。

`volatile` 不能直接保证：

- 复合操作的原子性

也就是说，它不能单独解决：

- `i++`
- `count--`
- 先判断再执行

这种多步骤操作的线程安全问题。

## 8. 为什么 `volatile` 不能保证 `i++` 线程安全

这是高频必问题。

因为 `i++` 不是单一步骤，而通常包含：

1. 读取 `i`
2. `i + 1`
3. 写回结果

`volatile` 只能让线程更容易看到最新值，但它不能把这三步合并成一个不可分割的原子动作。

所以就算写成：

```java
volatile int count = 0;
```

再执行：

```java
count++;
```

在多线程下还是可能出错。

## 9. 一个简单例子：`volatile` 修饰计数器为什么还不安全

```java
class Counter {
    volatile int count = 0;

    public void add() {
        count++;
    }
}
```

这里虽然 `count` 是 `volatile` 的，但 `add()` 依然不是线程安全的。

原因不是“看不到最新值”，而是：

- 多个线程仍然可能交错执行读、改、写这三步

所以最终结果还是可能偏小。

## 10. `volatile` 适合什么场景

`volatile` 更适合下面这些场景：

1. 一个线程写，多个线程读
2. 状态标记类变量
3. 配置开关类变量
4. 不涉及复合操作的简单赋值场景

最常见的就是：

- 停止标记
- 开关状态
- 是否完成初始化的标志

## 11. `volatile` 不适合什么场景

下面这些场景通常不能只靠 `volatile`：

1. 计数器自增自减
2. 库存扣减
3. 余额修改
4. 多步骤状态更新
5. 需要整体互斥的临界区逻辑

这类场景更常要考虑：

- `synchronized`
- `Lock`
- 原子类

## 12. `volatile` 和 `synchronized` 的关系

当前阶段不要急着死背对比表，可以先记它们解决问题的重心不同。

### 12.1 `volatile`

更偏：

- 可见性
- 一定程度上的有序性

### 12.2 `synchronized`

更偏：

- 互斥
- 原子性
- 同时也能保证一定可见性和有序性

所以最关键的一句可以先记：

- `volatile` 不是轻量版万能锁

## 13. 为什么有人说 `volatile` 比加锁“轻”

因为它不做完整的互斥控制。

也就是说：

- 它不会像 `synchronized` 那样让多个线程排队进入临界区
- 它主要处理的是“值什么时候能被看见”这个问题

所以在某些简单状态标记场景下，`volatile` 会显得更直接、更轻量。

但轻，不等于它能力更全面。

## 14. 一个面试里常见的延伸点

很多面试官会追问：

- 既然 `volatile` 能保证可见性，为什么还需要 `synchronized`？

答案核心就是：

- 因为很多线程安全问题不只是“看不看得到值”，还涉及“多个步骤是否被打断”

而这个问题本质上是原子性问题，不是单靠 `volatile` 就能解决的。

## 15. 面试里最容易漏的点

### 15.1 只会说 `volatile` 能保证线程安全

这不准确。

更准确的说法应该是：

- `volatile` 只能解决一部分并发问题，主要是可见性

### 15.2 说不清为什么 `i++` 不安全

一定要补一句：

- 因为 `i++` 是复合操作，不具备原子性

### 15.3 只会说“可见性”，不会给出场景

至少要补一个：

- 停止标记

### 15.4 把 `volatile` 和 `synchronized` 混成一类

这也是高频错误。

必须分清：

- 一个偏“让别的线程看到”
- 一个偏“同一时刻只让一个线程进来”

## 16. 面试里怎么回答“`volatile` 能解决什么问题”

可以这样答：

`volatile` 主要解决的是共享变量在多线程环境下的可见性问题，也能在一定程度上保证有序性。它适合用在状态标记、开关变量这类场景，一个线程修改后，其他线程能更快看到最新值。但它不能直接保证复合操作的原子性，所以像 `i++` 这种场景不能只靠 `volatile`。

## 17. 面试里怎么回答“为什么 `volatile` 不能保证 `i++` 线程安全”

可以这样答：

因为 `i++` 不是一个原子操作，它通常包含读取、计算、写回三个步骤。`volatile` 只能保证线程读到的是较新的值，但不能保证这三个步骤在并发环境下不被其他线程打断，所以多个线程同时执行 `i++` 仍然会产生线程安全问题。

## 18. 当前阶段最该记住的结论

1. `volatile` 主要保证可见性
2. `volatile` 还能保证一定程度上的有序性
3. `volatile` 不能直接保证复合操作的原子性
4. 停止标记是最经典的使用场景
5. `volatile` 不能替代 `synchronized`

## 19. 一句话总结

`volatile` 这一块最重要的不是背定义，而是先讲清楚：它最擅长解决的是“一个线程改了值，别的线程能不能及时看到”的问题，但它解决不了像 `i++` 这种复合操作的原子性问题，所以它不是锁的替代品。
